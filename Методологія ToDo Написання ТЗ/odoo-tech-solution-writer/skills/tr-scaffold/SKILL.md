# tr-scaffold — каркас документа «Технічне рішення»

## Мета
Перетворити сирий вхід (бриф, транскрипт созвону, задачу з ERP) на готовий каркас ТР:
заповнити шапку, обрати формат і лишити чіткі плейсхолдери під секції, які наповнять інші скіли.
Єдиний каркас гарантує, що всі ТР виглядають однаково, а аналітик не збирає шаблон руками і не
забуває секції.

## Інструменти та середовище
**Вхід (будь-що з):** текстовий бриф у чаті/файлі; транскрипт Fathom (`list_meetings` /
`search_meetings` → `get_meeting_transcript`); посилання/ID задачі `erp.todo.ltd`.
**Збереження:** `ТР_[коротка-назва].md` у робочій папці → `Write` (пробіли в імені файлу заміняй на `_`).
**Після завершення:** `present_files` + підкажи наступний скіл.
**Глибокий довідник:** `docs/шаблон-ТР.md` (повний каркас і правила).

## Крок 0: Визнач вхідні дані
- Якщо дано **транскрипт/созвон** — витягни вимоги через Fathom; бери лише те, що реально сказано.
- Якщо дано **ID/посилання задачі** — це «Номер запиту». Немає доступу до ERP — попроси текст задачі.
- Якщо **вхідних даних замало** (незрозуміла суть доробки) — задай 1–2 точкові питання, перш ніж
  генерувати порожній каркас.
- Якщо на созвоні **кілька різних доопрацювань** — спитай: робимо один ТР з кількома «Задачами»
  чи окремі ТР. За замовчуванням: непов'язані фічі → окремі ТР; пов'язані дрібні → один ТР, Формат B.

## Крок 1: Обери формат
- **Формат A (по фічах/сутностях)** — велика розробка: новий модуль, сутності `td.*`, багато полів.
- **Формат B (по задачах)** — точкова доробка існуючого, 1–3 невеликі задачі.
- **Формат C (Правки/Доопрацювання)** — зміни до **вже описаного/розробленого** модуля: патч після
  зустрічі, набір доопрацювань по моделях, двошаровий (клієнт→BA) чи живий трекер. Це **не повний ТР** —
  легкий формат; будуй за `docs/правки-та-доопрацювання.md` (групування по об'єктах, маркери статусу
  `Готово/OLD/НЕ РОБИТИ`, обов'язкове посилання на вихідне ТЗ + ID задачі, дедлайни замість годин).
Евристика: новий функціонал/сутності → A; 1–3 правки нового опису → B; **зміни до вже існуючого
опису/розробки → C**. Сумніваєшся — запропонуй вибір.

**Великі фічі — дроби.** Якщо фіча завелика для одного ТР, розділи на окремі документи
«<Назва> FEATURE 1 / FEATURE 2 / …» або «… Блок 1 / Блок 2 / …». Практика з корпусу:
- зведений (батьківський) документ містить повний опис; дочірні — підмножини, **кожен зі своїм
  «Номером запиту»** (окрема задача в ERP);
- групуй фічі в блоки і за логікою, і **за ризиком** — неузгоджені/ризиковані фічі винось в окремий
  блок (напр. позначка «не підтверджено — потрібно погодити з замовником»), щоб не блокувати решту;
- ревізія після зустрічі з клієнтом — нова версія `V2` з ремаркою «Відбулись зміни після зустрічі
  з замовником» і явним переліком дельти відносно `V1`.

**Новий функціонал — у новому модулі** з префіксом проекту (`<проект>_<область>` чи `td_<область>`,
напр. `olimpius_delivery`, `td_olimpius_box_packing`); моделі лишаються з префіксом `td.`.
**Спеціальні типи** мають власну структуру: Звіт-Pivot і Обмін/інтеграція — `docs/звіти-та-обміни.md`;
повторювані рішення (wizard, ir.cron, версійність BoM, portal-форма, QWeb-друк, движок сценаріїв/стадій,
AI-поле…) — `docs/патерни-Odoo-рішень.md`.

**Мова підписів полів/UI — параметр проекту:** Elfa/Олімпіус — українська, Bixter (HR) — англійські
імена полів і UI з описами українською. Бери мову лейблів з наявних документів проекту; технічні
імена (`td_*`) завжди латиницею. На ранніх ТЗ допустимий чернетковий стиль із відкритими питаннями
в тексті (див. `tr-functional-requirements`).

## Крок 2: Створи файл з каркасом
**Спершу обери набір секцій** (за проектом / наявними документами проекту — див. `docs/шаблон-ТР.md`):
- *Скорочений (Elfa):* Загальна інформація → Обсяг робіт (5 етапів) → Технічне рішення → Архітектура → Інструкція.
- *Розширений (Олімпіус):* Загальна інформація → Обсяг робіт (7 етапів + «Виконавець») → **Функціональні
  вимоги** (окрема секція) → Технічне рішення → Архітектура → **Тестовий сценарій** → **Протокол
  тестування** → Інструкція.
Невідомо який — бери скорочений, але спитай/познач припущення.

Візьми каркас з `docs/шаблон-ТР.md` (українські заголовки не змінюй) і заповни:
- **Шапка:** контакти ToDo і `Від __.__.20__ року` лиши як у шаблоні (постав поточну дату).
- **Загальна інформація:** «Найменування модифікації», «Проект», «Система» — версія Odoo з картки
  проекту (напр. `Odoo 19 EE` для Elfa, `Odoo Enterprise Edition 18.0` для Олімпіус); **не хардкодь
  версію, бери з входу/проекту**. «Замовник/Ініціатор», «Номер запиту» (= ID задачі), «Прийнято»,
  «Виконавець» — з входу; чого немає, лишай порожнім (імен людей не вигадуй).
- **Обсяг робіт:** лиши таблицю порожньою (заповнить `tr-effort-instruction`).
- **АРХІТЕКТУРА** і **Інструкція:** лиши шаблонними.

## Крок 3: Розстав плейсхолдери під секції
Під кожну незаповнену секцію встав позначку, який скіл її закриває:
```
ТЕХНІЧНЕ РІШЕННЯ

1. Загальний опис процесу та функціональні вимоги.
   <!-- TODO: tr-functional-requirements -->

2. Опис технічної реалізації.
   <!-- TODO: tr-odoo-tech-design -->
   <!-- TODO: tr-usecases-acceptance — Use Cases + Acceptance Criteria -->
```
Для **розширеного набору** додай окремі секції з їхніми скілами:
```
Функціональні вимоги            <!-- TODO: tr-functional-requirements -->
ТЕХНІЧНЕ РІШЕННЯ                 <!-- TODO: tr-odoo-tech-design + tr-usecases-acceptance -->
Тестовий сценарій               <!-- TODO: tr-effort-instruction (кроки з AC) -->
Протокол тестування             <!-- лишити шаблоном: заповнює тестувальник -->
```
Для **Формату B** створи заготовки «Задача 1 / Задача 2 …» з порожніми «Проблема / Очікуваний
результат» і позначками TODO.

## Pre-save верифікація
- [ ] Ім'я файлу `ТР_[реальна назва].md` — не шаблонне;
- [ ] «Система» = версія Odoo з проекту (не хардкод 19); «Номер запиту» = ID задачі, якщо є;
- [ ] обраний формат (A чи B) відображений у структурі тіла;
- [ ] усі секції шаблону на місці, заголовки українською дослівно;
- [ ] плейсхолдери `<!-- TODO: ... -->` розставлені під кожну незаповнену секцію.

## Наступний крок
Покажи файл (`present_files`) і повідом: «Каркас ТР готовий (Формат [A/B]). Рекомендований порядок:
tr-functional-requirements → tr-usecases-acceptance → tr-odoo-tech-design → tr-effort-instruction → tr-review.
З якої секції почати?»

## Чого НЕ робити
- **НЕ ВИГАДУЙ нічого поза джерелами.** Працюй лише з даними входу (бриф/транскрипт/задача), узгодженими
  конвенціями/довідниками цього плагіна та офіційною документацією Odoo. Чого немає в джерелах і не
  описано в скілах — не видавай за факт; невідоме лишай порожнім або познач «потребує уточнення».
  Тут конкретно: не вигадуй найменування, проект, замовника, номер задачі, дати, суть доробки.
- Не вигадуй вимоги, яких немає у вході (бриф/транскрипт/задача).
- Не заповнюй секцію АРХІТЕКТУРА — це зона архітектора після узгодження.
- Не змінюй українські назви секцій і підписи полів.
- Не конвертуй одразу в .docx — спершу збирається і узгоджується .md.
