# tr-review — QA-перевірка консистентності ТР

## Мета
Перед узгодженням клієнтом перевірити, що ТР внутрішньо несуперечливий і повний: вимоги →
сценарії/AC → технічна реалізація → оцінка → інструкція утворюють зв'язний ланцюг без дірок.
Це фінальний фільтр, який ловить «вимога без AC», «AC без реалізації», «кнопка без пост-ефекту».

## Інструменти та середовище
**Вхід:** `ТР_[назва].md` → `Read`. Глибокі довідники для звірки: `docs/шаблон-ТР.md`,
`docs/odoo-конвенції.md`, `docs/патерни-UC-AC.md`.
**Збереження:** `ЗВІТ_QA_[назва].md` → `Write`; коротке резюме в чат.
**Після завершення:** `present_files`.

## Крок 0: Визнач вхідні дані та тип документа
Знайди `ТР_*.md` (кілька — спитай який перевіряємо). Немає — попроси шлях.
**Визнач тип:** повний ТР (Формат A/B) чи **документ змін (Формат C** — патч/доопрацювання). Від цього
залежить чек-лист: для Формату C застосовуй полегшений набір (нижче), а **не** повний — інакше валідний
патч помилково вважатиметься «неповним» (відсутність повних секцій / «Обсяг робіт» — не помилка).

## Крок 1: Прогони чек-лист консистентності
Перевір по пунктах і для кожного зафіксуй статус (✅ / ⚠️ / 🔴) і конкретику:

**Структура**
- [ ] Усі секції шаблону присутні, заголовки українською дослівно;
- [ ] Не лишилось `<!-- TODO: ... -->` чи незаповнених плейсхолдерів;
- [ ] «Загальна інформація» заповнена (мінімум: Найменування, Проект, Система, Номер запиту);
- [ ] **Назва документа/файлу збігається з «Найменування модифікації» в шапці** (у корпусі траплявся
      рассинхрон назви файлу і задачі — це плутає розробку);
- [ ] **«Тестовий сценарій»/«Протокол тестування» заповнені реальними кроками** (а не порожній шаблон) —
      часта прогалина; порожні тести = не готово до приймання.

**Зв'язність вимоги → AC → реалізація**
- [ ] Кожна функціональна вимога (секція 1) має сценарій та/або AC;
- [ ] Кожен AC має технічне втілення в «Опис технічної реалізації» (поле/модель/правило/кнопка);
- [ ] Кожна кнопка-дія з тех. реалізації має AC (коли видно, що робить, що після);
- [ ] Є AC на «нещасливі» шляхи (немає даних, помилка, права, межі/constrains).

**Технічна якість**
- [ ] Іменування за конвенціями (`td.`/`td_`, маска `<Проект>:`); біля людських назв є технічні в дужках;
- [ ] Кожне поле має тип, джерело, і де доречно — domain/required/read-only/store;
- [ ] Довідники мають значення і шлях у UI; права мають мінімум Користувач/Адміністратор.

**Відповідність Odoo 19 (обов'язково)**
- [ ] **Кожен пункт реалізації звірено з офіційною документацією Odoo 19** — вказано джерело
      (`довідник §N` або офіційний URL). Пункт без звірки/джерела — 🔴;
- [ ] синтаксис лише v19: немає `attrs`/`states`, `<tree>`, `t-raw`, `group_operator`, xmlrpc/jsonrpc —
      якщо є застаріле, це 🔴 з виправленням на v19-аналог (`invisible`/`<list>`/`t-out`/`aggregator`/`/json/2`);
- [ ] стандартні моделі/поля Odoo, у яких не було впевненості, або звірені з docs, або позначені як припущення.

**Оцінка та інструкція**
- [ ] «Обсяг робіт»: усі етапи оцінені, `Всього` = сума (перевір арифметику), дано діапазон/допущення;
- [ ] «Інструкція»: кроки відтворюють AC, однозначно відтворювані.

**Межі**
- [ ] Секція АРХІТЕКТУРА лишена шаблоном (не вигадана за архітектора) — це коректно, не помилка.

**Полегшений чек-лист для Формату C (Правки/Доопрацювання)** — застосовуй замість повного; повні
секції / «Обсяг робіт» / суцільне покриття AC тут НЕ вимагаються (їх відсутність — не помилка):
- [ ] зміни згруповані по об'єктах/моделях; кожна зрозуміла й однозначна;
- [ ] є **посилання на вихідне ТЗ + ID задачі** (часта прогалина — це 🔴);
- [ ] проставлені маркери статусу (Готово/Почато/OLD/НЕ РОБИТИ); замінена логіка перенесена в `OLD`, не видалена;
- [ ] для нетривіальних змін є критерії приймання / тест-кроки (інакше зміну не прийняти однозначно);
- [ ] поля подані таблицею з «Розташування»; умовна видимість і країнові умови зафіксовані явно;
- [ ] двошарові правки: збережено і сирі зауваження клієнта, і технічну інтерпретацію BA.

## Крок 2: Сформуй звіт `ЗВІТ_QA_[назва].md`
```
# QA-перевірка: ТР «[назва]»
Дата: [сьогодні]   Файл: ТР_[назва].md

## Підсумок
Готовність: [Готовий до узгодження / Потребує доопрацювання]. Знайдено: 🔴 [N] / ⚠️ [M].

## Проблеми за секціями
### 🔴 Критичні (блокують узгодження)
- [Секція] — [проблема] → [конкретне виправлення]
### ⚠️ Зауваження (бажано виправити)
- [Секція] — [проблема] → [рекомендація]

## Що добре
- [сильні місця, які лишити як є]
```
Кожна проблема — з посиланням на конкретне місце і **готовим виправленням**, а не лише констатацією.

## Pre-save верифікація
- [ ] Пройдено весь чек-лист (нічого не пропущено);
- [ ] Кожна проблема має секцію, критичність і конкретне виправлення;
- [ ] Підсумок чесний (не занижуй і не завищуй готовність).

## Наступний крок
Повідом: «QA готова: 🔴[N]/⚠️[M]. [Готовий / Потребує правок]. Виправити критичні зараз, чи
далі — tr-registry-update і експорт у .docx?»

## Чого НЕ робити
- **НЕ ВИГАДУЙ нічого поза джерелами.** Звіт QA будуй лише з реального вмісту ТР, узгоджених конвенцій/
  довідників плагіна та офіційної документації Odoo. Не вигадуй проблеми і не вигадуй «правильні»
  значення за автора — фіксуй лише реальні розбіжності з конкретним виправленням.
- Не «дописуй» сам відсутній контент мовчки — фіксуй проблему і запропонуй виправлення
  (виправлення наповнення — задача профільних скілів).
- Не позначай ТР готовим, якщо є 🔴; не вважай порожню АРХІТЕКТУРА помилкою.
