# tr-functional-requirements — «Загальний опис процесу та функціональні вимоги»

## Мета
Відповісти на «навіщо бізнесу» і «що система має робити» — без технічної реалізації (моделі,
поля — це `tr-odoo-tech-design`). Якість цієї секції визначає, чи зрозуміє клієнт цінність, а
розробка — межі задачі.

## Інструменти та середовище
**Вхід:** файл `ТР_[назва].md` (плейсхолдер `<!-- TODO: tr-functional-requirements -->`) → `Read`/`Edit`;
бриф / транскрипт Fathom / задача ERP.
**Збереження:** впиши контент у `ТР_[назва].md` на місце плейсхолдера → `Edit`.
**Глибокий довідник:** `docs/патерни-UC-AC.md` (розділи «Основна бізнес-ціль», «As-Is/To-Be»).

## Крок 0: Визнач вхідні дані
- Знайди файл `ТР_*.md`. Якщо кілька — спитай, для якої задачі/проекту працюємо.
- Якщо файлу немає — запропонуй спершу `tr-scaffold` або спитай шлях.
- Якщо вхід — транскрипт, бери вимоги лише з реального тексту (`get_meeting_transcript`), не додумуй.
- **Куди писати залежить від структури документа:** у *скороченій* (Elfa) — це пункт 1 «Загальний
  опис процесу та функціональні вимоги» всередині «Технічне рішення»; у *розширеній* (Олімпіус) —
  **окрема секція «Функціональні вимоги»** перед «Технічне рішення». Пиши в ту, що вже є в каркасі.

## Крок 1: Розділи шари
У голові розділи: «навіщо» (бізнес-ціль), «що зараз/що буде» (As-Is/To-Be), «що має вміти»
(функціональні вимоги). Не змішуй з «як» (технічна реалізація — інший скіл).

## Крок 2: Напиши секцію (вихід українською)

### Основна бізнес-ціль
1–3 абзаци: яку задачу закриваємо, хто користувач (ролі, кількість), який вимірюваний результат.
Без техніки. Є цифри у вході (кількість користувачів, обсяги, частота) — обов'язково використай.

### Процес As-Is / To-Be (якщо це доробка існуючого)
Покажи різницю «як зараз» → «як має бути». Для нового функціоналу As-Is = «функціонал відсутній».

### Функціональні вимоги
Перелік того, що система має вміти з точки зору користувача/процесу. Кожна вимога — окремий пункт,
перевірюваний і однозначний; групуй за підпроцесами. Формулюй «Система має …», «Користувач повинен
мати можливість …». Погано: «зробити зручний каталог». Добре: «дати можливість фільтрувати товари
за брендом і додавати відфільтровані позиції в документ лістингу однією дією».

## Крок 3: Познач прогалини та відкриті питання
Якщо даних бракувало — додай блок «Припущення» з відкритими питаннями (чесно, щоб аналітик уточнив
у клієнта), а не ховай прогалину вигадкою.
**Чернетковий стиль (реальна практика).** На ранніх ТЗ нормально лишати **відкриті питання прямо в
тексті** біля місця, якого вони стосуються (напр. жирним: «**Уточнити: статуси визначає HR?**»),
і маркери `СПЕЦИФІКУВАТИ` / «потребує обговорення» для нерозкритих місць. Не вигадуй відповіді за
клієнта — познач питання, щоб закрити його на зустрічі. Веди їх так, щоб потім легко зібрати
(окремий блок «Відкриті питання» або однакова позначка для пошуку).

## Pre-save верифікація
- [ ] Є «Основна бізнес-ціль» без технічних деталей;
- [ ] для доробки існуючого є As-Is/To-Be;
- [ ] функціональні вимоги — перевірювані пункти, без «зручно/швидко» без критерію;
- [ ] використані реальні цифри з входу; вигаданих метрик немає;
- [ ] невідоме винесено в «Припущення».

## Наступний крок
Повідом: «Секцію 1 заповнено. Далі — tr-usecases-acceptance (сценарії + критерії прийняття)?»

## Чого НЕ робити
- **НЕ ВИГАДУЙ нічого поза джерелами.** Працюй лише з даними входу (бриф/транскрипт/задача), узгодженими
  конвенціями/довідниками цього плагіна та офіційною документацією Odoo. Чого немає в джерелах і не
  описано в скілах — не видавай за факт. Тут конкретно: не вигадуй вимоги, бізнес-цілі, метрики, цифри,
  імена; невідоме виноси в блок «Припущення» / «Відкриті питання», а не вигадуй.
- Не лізь у технічну реалізацію (`td.*`, поля, права) — це `tr-odoo-tech-design`.
- Не вигадуй факти; недостатнє → «Припущення».
- Не змінюй українські заголовки секцій.
