# tr-effort-instruction — «Обсяг робіт» + «Інструкція»

## Мета
Два фінальні блоки ТР. «Обсяг робіт» дає клієнту і команді оцінку трудозатрат по етапах.
«Інструкція» — покроковий сценарій перевірки, що доробку зроблено (протокол успішних випробувань).

## Інструменти та середовище
**Вхід:** `ТР_[назва].md` (секції тех. реалізації + AC) → `Read`/`Edit`.
**Збереження:** заповни таблиці «Обсяг робіт» і «Інструкція» у `ТР_[назва].md` → `Edit`.
**Глибокі довідники:** `docs/калібрування-оцінки.md` (орієнтири годин за реальними ТР ToDo),
`docs/шаблон-ТР.md` (точні таблиці).

## Крок 0: Визнач вхідні дані
- Знайди `ТР_*.md` (кілька — спитай яку задачу).
- Для «Обсяг робіт» потрібна наповнена «Опис технічної реалізації»; для «Інструкція» — AC.
  Якщо AC немає — попередь, що інструкція буде поверхневою, і запропонуй `tr-usecases-acceptance`.

## Частина 1: Обсяг робіт (оцінка)
Набір етапів залежить від структури проекту (звір з тим, що збудував `tr-scaffold`):

**Варіант А — скорочений (Elfa):**
```
| Етап робіт       | План люд./год. | Коментар |
| Технічне рішення |                |          |
| Архітектура      |                |          |
| Розробка         |                |          |
| Тестування       |                |          |
| Інструкція       |                |          |
| **Всього**       |                |          |
```
**Варіант Б — розширений (Олімпіус, + колонка «Виконавець»):**
```
| Етап робіт          | План люд./год. | Виконавець |
| Функціональні вимоги |               |            |
| Аналіз вимог         |               |            |
| Архітектура          |               |            |
| Тестовий сценарій    |               |            |
| Розробка             |               |            |
| Протокол тестування  |               |            |
| Інструкція           |               |            |
| **Разом**            |               |            |
```
Заповнюй той варіант, що вже стоїть у документі (не міняй набір етапів довільно).
Метод (деталі — у `docs/калібрування-оцінки.md`):
1. Розклади на архетипи Розробки, просумуй → `Розробка`. Орієнтири: зміна відображення 1–3 год;
   поле related/computed 1–3; domain/простий фільтр 2–4; кнопка-дія з логікою 3–6; новий тип
   контакту + compute 7; великий інтерактив 15–25; нова сутність/модуль — десятки.
2. `Тестування` = 30–50% від Розробки (+ оверхед, якщо «потрібно розгортати локальне середовище»).
3. `Технічне рішення` + `Архітектура` = аналітичний оверхед 3–6 год.
4. `Всього` = сума; **перевір арифметику**.
Правила: години через кому (`2,75`); при кількох задачах розбий Розробку в «Коментар» по задачах;
у «Коментар» познач суттєве (локальне середовище, ризики, допущення). Дай **діапазон** (мін–макс).
Якщо просять «просто цифру в чат» — дай діапазон і коротке обґрунтування, без редагування файлу.
Оцінка — судження, не факт: явно перелічуй допущення, від яких вона залежить.

**Дедлайни замість годин (патчі/живі трекери).** Для документів змін (Формат C) замість «Обсяг робіт»
у годинах часто ведуть задачі за **дедлайнами** (`ДД-ММ-РРРР`) і списками Done / To fix по об'єктах.
Якщо це патч-документ — підтримай цей формат (дедлайни + статуси), а не нав'язуй 5/7-етапну таблицю.

## Частина 2: Тест-кейс і інструкція
**Куди писати — залежить від структури:**
- *Скорочена (Elfa):* секція «Інструкція» = тест-кейс (сценарій + протокол успішного виконання).
- *Розширена (Олімпіус):* тест-кроки йдуть у секцію **«Тестовий сценарій»**; секція **«Протокол
  тестування»** — це лог фактичного проходження (Pass/Виявлено), лишай її шаблоном для тестувальника;
  «Інструкція» — користувацький гайд «як користуватись фічею».

Заповни таблицю кроків (для тест-сценарію/інструкції):
```
| Крок | Опис кроків |
| 1.   |             |
| 2.   |             |
```
Правила: кроки відтворюють AC з `tr-usecases-acceptance` (кожен значущий AC → перевірюваний крок:
передумова → дія → очікуваний результат). Конкретика: з якими даними, під якою роллю
(Користувач/Адміністратор), що саме дивимось на екрані. Покрий і «нещасливі» шляхи. Текст українською.

## Pre-save верифікація
- [ ] Усі етапи обраного варіанту (5 чи 7) оцінені; `Всього`/`Разом` = сума (арифметика перевірена);
- [ ] для розширеного варіанту: «Протокол тестування» лишено шаблоном (його заповнює тестувальник);
- [ ] оцінка спирається на обсяг тех. реалізації і кількість AC; дано діапазон і допущення;
- [ ] кроки інструкції відтворюють AC, однозначно відтворювані іншою людиною;
- [ ] інструкція покриває і «нещасливі» шляхи; заголовки таблиць не змінені.

## Наступний крок
Повідом: «Обсяг робіт і Інструкцію заповнено (≈[N] год). Далі — tr-review (QA перед узгодженням)?»

## Чого НЕ робити
- **НЕ ВИГАДУЙ нічого поза джерелами.** Оцінку бери з калібрування + обсягу тех. реалізації; кроки
  інструкції — з AC. Чого немає в джерелах і не описано в скілах — не видавай за факт. Тут конкретно:
  не вигадуй години/дедлайни «зі стелі» і не пиши кроки тесту без опори на AC; брак даних — познач.
- Не давай одну точну цифру там, де доречний діапазон; не ховай допущення.
- Не пиши інструкцію, відірвану від AC.
- Не змінюй набір етапів і українські заголовки таблиць.
