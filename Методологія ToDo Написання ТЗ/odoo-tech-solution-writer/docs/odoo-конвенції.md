## 1. Іменування об'єктів і полів
- Кастомні моделі: префікс `td.` → `td.listing`, `td.listing.line`. Документ-«шапка» та його
  рядки: `td.<entity>` + `td.<entity>.line` (за аналогією `sale.order`/`sale.order.line`,
  `stock.picking`/`stock.move`).
- Кастомні поля: префікс `td_` → `td_network_type_id`, `td_tt_category_id`, `td_tt_status`,
  `td_sales_channel_id`, `td_national_chain_id`, `td_formatted_name`. Суфікс `_id` для many2one,
  `_ids` для many2many/one2many.
- Маска імен прав/правил/груп: `<Проект>: <назва правила>` (наприклад `Elfa: ...`) — щоб
  доопрацювання клієнта візуально відділялися від стандартних об'єктів Odoo.
- У тексті завжди вказуй технічне ім'я в дужках: «Прайс-лист фактичної ціни (actual_price_list)».

## 2. Типи полів Odoo
| Людське (у ТР) | Тип Odoo | Коли |
| :-- | :-- | :-- |
| check-box | `boolean` | так/ні («Є Дистриб'ютором») |
| selection | `selection` | фікс. набір у коді («Статус ТТ»: Активна/Неактивна) |
| many2one | `many2one` | посилання на 1 запис (контакт, товар, прайс-лист) |
| many2many | `many2many` | набір значень з довідника («Місце на полиці») |
| one2many | `one2many` | рядки документа (`td.listing.line` у `td.listing`) |
| monetary | `monetary` | ціна/сума (валюта, за замовч. UAH) |
| date / datetime | `date`/`datetime` | дата (відображення `ДД.ММ.РРРР`) |
| char / text | `char`/`text` | короткий / багаторядковий текст |
| integer / float | `integer`/`float` | числа |

Атрибути поля, які прийнято фіксувати: **related** (тягнеться зі зв'язаного запису — вказуй
джерело, напр. «related на parent_id»), **computed** (розрахункове — вказуй тригер і логіку),
**store=True** (зберігається в БД — для швидкого пошуку/сортування), **read-only**
(«змінюється в налаштуваннях»), **required** («обов'язкове до заповнення для зберігання»),
**domain** (фільтр вибору), **default** (значення за замовчуванням), **constrains** (обмеження,
напр. «не більше одного власника на ТТ»).

## 3. Стандартні моделі Odoo для посилань
`res.partner` (контакти/ТТ/компанії, ієрархія через `parent_id`, `child_ids`, `display_name`,
`type`), `res.users` (`user_id`, продавець), `res.company`, `product.product`/`product.template`
(`default_code` = Артикул), `product.pricelist`/`product.pricelist.item`, `sale.order`/
`sale.order.line` (`partner_invoice_id`), `stock.picking`/`stock.move` (зразок архітектури
документ+рядки), `ir.sequence` (нумерація), `account.move`. Патерн «реалізувати на прикладі
моделі X» — прямо посилайся на стандартну модель як зразок.

## 4. Таблиця специфікації полів (обов'язковий формат)
```
| Назва поля | Тип поля | Звідки тягнути значення | Додаткова примітка |
| :-- | :-- | :-- | :-- |
| Дистриб'ютор | many2one | Контакти (res.partner) | domain: «Є Дистриб'ютором» = True |
| Дата | date | — | формат 21.05.2026; обов'язкове до зберігання |
```
Розширені колонки за потреби: «Опис», «Колір на скрін шотах». Завжди відмічай: джерело,
domain, required, read-only, store, синхронізацію.

## 5. Довідники (selection / many2many)
Під кожне selection/many2many-поле — таблиця значень:
```
| Довідник / Вибір | Значення |
| :-- | :-- |
| Статус ТТ | Активна; Неактивна |
| Фейсінг | 1; 2; 3; 3+ |
```
Шлях у UI: `<Модуль> — Налаштування` (напр. «Лістинги — Налаштування»). Вказуй значення за замовчуванням.

## 6. Права доступу та групи
Мінімум дві групи:
```
| Права доступу | Що відображати / що дозволено |
| :-- | :-- |
| (немає прав) | Модуль не відображається |
| Користувач | Лише свої записи; довідники без додавання/видалення; після збереження не видаляє рядки |
| Адміністратор | Усі записи; повний доступ до довідників |
```
Вказуй read-only поля по групах, дозволені дії (створити/видалити/редагувати стадію), маску
імен груп `<Проект>: ...`, record rules і constrains.

## 7. Подання, smart-кнопки, стадії
- Типи подань: **Канбан** (стадії), **Список**, **Карта** (координати/адреса), **Зведена таблиця** (аналітика).
- **Стадії (Kanban)**: довідник стадій, редагується Адміністратором; за потреби правило
  «перехід на наступний етап — повернення назад заборонено».
- **Smart-кнопки**: перехід у зв'язану сутність з фільтром (смарт-батон «Лістинг» у контакті
  → `td.listing` з фільтром по ТТ).
- **Кнопки-дії** («Створити...», «Зберегти та відправити на узгодження», «Оновити ціни...»):
  для кожної опиши коли видно (права + умова), що робить, що після (зміна стадії, запис у чатер,
  зникнення кнопки, повідомлення).

## 8. Related / Computed / Stored — синхронізація
Часте джерело багів — фіксуй явно: напрямок синхронізації (напр. «тільки згори вниз»:
parent→child, назад не пишемо); які саме поля синхронізуються, а які ні; коли значення
фіксується при створенні і НЕ оновлюється авто (тоді потрібна кнопка ручного оновлення);
поведінку при зміні джерела після створення.

## 9. Послідовності та автоназви
Автонумерація через `ir.sequence` з префіксом (`L00123`, префікс `L`). Складені імена:
`[Назва картки] + [Дата створення]` → `L00065 01.03.2026`.

## 12. Модулі, версія системи, локалізація
- **Версія Odoo — параметр проекту, не константа.** Elfa = `Odoo 19 EE`, Олімпіус =
  `Odoo Enterprise Edition 18.0`. Бери з картки проекту/завдання, не хардкодь.
- **Новий функціонал — у новому модулі з префіксом проекту:** `<проект>_<область>`
  (напр. `olimpius_delivery`). Моделі при цьому лишаються з префіксом `td.` (`td.delivery.route`).
  За потреби познач «згенерувати іконку модуля».
- **Локалізація («переклад полів!»):** усі нові поля/меню мають мати українські підписи.
- **Маппінг станів Odoo → українські підписи** наводь явно, коли логіка залежить від статусів:
  напр. `waiting` — «Очікує іншої операції», `confirmed` — «Очікування», `assigned` — «Підготовлено»,
  `done` — «Виконано», `cancel` — «Скасовано».
- **Натуральні ключі (для обмінів і дедуплікації):** ЄДРПОУ (`ref`), ІПН (`vat` /

## 15. Складні форми, видимість полів, розташування (з HR-проекту)
- **Умовна видимість/обов'язковість полів:** поле показується/ховається або стає обов'язковим
  залежно від значення іншого поля. Фіксуй умову словами: «видиме, якщо `Education status = In
  progress`», «обов'язкове, якщо `Driver license = yes`». **Синтаксис Odoo 17+/19 — прямі атрибути з
  Python-виразом** (`invisible="state == 'done'"`, `required="..."`, `readonly="..."`,
  `column_invisible="..."` у `<list>`), а **НЕ `attrs`/`states`** (упразднені) — деталі в
  `odoo19-довідник.md` §5. Це пронизує складні картки — описуй кожну залежність явно.
- **Колонка «Розташування» в таблиці полів:** для форм зі складним layout додавай 4-ту колонку —
  де поле на формі: `| Назва поля | Тип поля | Нотатки | Розташування |` (напр. «вкладка CV, секція
  справа», «картка, секція Work info»). Так розробник збере форму без здогадок.
- **Багатокрокові форми (Step 1…N):** для великих анкет/портальних форм розбивай на кроки з умовною
  логікою переходів; познач, які поля на якому кроці й за якою умовою з'являються (див. архетип
  portal-форми у `патерни-Odoo-рішень.md`).
- **Нові стандартні моделі:** `mail.template` (типізовані шаблони — див. архетип №19),
  `survey.survey` (опитування/сертифікації за типом — архетип №16), `mail.activity`/`mail.activity.type`
  (дії-нагадування), `res.users`/portal-користувачі (зовнішні партнери). Механіка mail/activity/template,
  Automation Rules (`base.automation`), `res.config.settings`, контролери/portal — у `odoo19-довідник.md` §10-13.
- **«Реалізувати за аналогією з CRM»:** часта вказівка — бери готові механіки CRM як зразок:
  команди продажу → сценарії/стадії; `Mark lost` (візард причина+коментар, архівація); `days_to_rot`;
  smart-button «Meetings» (календар). Прямо посилайся на CRM-аналог.

## 17. Робочі документи змін, UI-стани, дедлайни
- **Маркери статусу в робочих доках:** у документах правок/доопрацювань прийнято позначати прогрес
  прямо в тексті — `Готово` / `Почато` / `Очікує затвердження` / `Виконано` / `НЕ РОБИТИ` / `OLD`.
  Блок `OLD` зберігає замінену логіку (не видаляй). Деталі жанру — `правки-та-доопрацювання.md`.
- **Ribbon (плашка):** для особливих станів запису використовуй кутову плашку (напр. жовту
  «Need transfer») — див. архетип №25.
- **Виключення зі стандартного списку за прихованим статусом:** записи з технічним статусом
  (напр. `Transfer status = Need transfer`) не показуй у дефолтному list view, лише через окремий фільтр.
- **Дедлайни замість/поряд з годинами:** живі трекери ведуть задачі за дедлайнами `ДД-ММ-РРРР`
  і списками Done/To-fix; це валідна альтернатива «Обсяг робіт» для патч-документів.
- **Країнова/програмна варіація** (вкладки/поля/стадії за країною+громадянством+програмою через
  умовну видимість `invisible`/`required`, v19 — не `attrs`) — частий патерн; будуй за архетипом №22.

- **Розбиття/розширення типів операцій:** нове Selection-поле на `stock.picking.type` + кілька пунктів

- **List view доопрацювання:** перейменування колонок, show/hide полів, формат дат `ДД.ММ.РРРР` зі
  збереженням кольору — частий тип легких доробок.

### Префікс кастомних полів: `td_` (а не `x_`)
У корпусі траплялась неузгодженість: одні поля `td_*`, інші `x_*` (Studio-стиль). **Конвенція: для полів
у коді — `td_*`**; `x_*` лиши тільки для полів, доданих вручну/через Studio. У ТР пиши `td_*` і познач,
якщо поле справді Studio-поле.

## 19. Фінанси / PLM / CRM-driven (з проекту RSE)
- **Аналітичний облік:** `account.analytic.plan` (кілька планів — Проект/CashFlow/Установка),
  `account.analytic.account`, `account.analytic.distribution` (на `account.move.line`/`purchase.order.line`).
  Авто-створення/масове проставлення — архетип №35.
- **Approvals (Затвердження):** `approval.request` + `approval.category`; база для документ-процесів
  (напр. заявка на оплату) — архетип №33; тип обирається в `res.config.settings`.
- **PLM:** `mrp.eco` (ECR/ECO, інженерні зміни) → формує/версіонує `mrp.bom`.
- **CRM:** `crm.lead`/`crm.stage`; прапор-чекбокс на стадії запускає створення downstream — архетип №32;
  правила автоматизації етапів можна вести таблицею відповідності `crm.stage → automation/нагадування`.
- **Опитування:** `survey.survey` (за типом), share-wizard з автопідстановкою email, smart-button результатів.
- **Графік оплат:** кастомна `td.payment.schedule`, прив'язана до `sale.order`/`purchase.order`; валідація
  суми при підтвердженні; `ir.cron` створює `mail.activity`-нагадування перед датою.
- **Серійники/виробництво:** правило «1 `mrp.production` = 1 `stock.lot`» (авто-спліт кількості X → X ЗНВ
  по 1; wizard створення `stock.lot` при підтвердженні ЗНВ); чекбокси на `mrp.bom` («Окремий ордер»,
  «Миттєве створення серійного номеру»); розділення `stock.picking` за адресами доставки з рядків КП;
  «Відповідальний за замовчуванням» на `stock.picking.type`; ізоляція ЗНКП «1 проект/серійник = 1 документ».
- **Друковані форми (QWeb)** мають окрему структуру опису — `друковані-форми.md`.
