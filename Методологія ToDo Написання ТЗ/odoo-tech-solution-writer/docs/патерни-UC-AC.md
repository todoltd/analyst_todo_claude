## 1. Основна бізнес-ціль
1–3 абзаци: яку бізнес-задачу закриваємо, хто користувач (ролі, кількість), який результат.
Без техніки. Якщо в брифі є цифри (кількість користувачів, обсяги) — обов'язково використовуй.
> Створення єдиної бази даних торгових точок по всій Україні для фіксації домовленостей щодо
> асортименту, контролю цін на полицях та аналізу покриття. Інструмент для 17–20 торгових
> представників та база для аналітики менеджменту.

## 2. As-Is / To-Be
Для доробки існуючої поведінки завжди показуй різницю:
```
As is: при зміні рекомендованих цін, ціна в лістингах не змінюється.
To be: адміністратор натискає «Оновити ціни в лістингах», і значення оновлюються в усіх документах.
```
Для нового функціоналу As-Is = «функціонал відсутній».

## 3. User Story
**Як** `<роль>`, **я хочу** `<можливість>`, **щоб** `<вигода>`.
```
Як менеджер цін, я хочу мати в картці прайс-листа кнопку «Оновити ціни в лістингах»,
щоб усі лістинги приводились до актуальних цін за рішенням користувача, а не автоматично.
```
Доповнюй блоками «Контекст / проблема» та «Передумови».

## 4. Use Cases (Сценарії використання)
Кожен сценарій — реалістична ситуація з конкретикою (числа, ролі, обсяги).
```
#### Сценарій N: <коротка назва>
- Дія: <що робить користувач: «активно понад 500 позицій»>
- Поведінка системи: <по кроках 1., 2., 3. що показує/робить система>
```
Покривай: нормальний хід, межові ситуації (порожньо / багато / немає даних), скасування дії,
взаємовплив елементів (фільтри між собою). 3 сценарії — добрий мінімум на задачу.

## 5. Acceptance Criteria — Given/When/Then (основний формат)
Нумеруй AC1, AC2…; ключові слова **Given / When / Then** (і **And**) — жирним; текст українською.
```
##### AC1 — <що перевіряємо однією фразою>
- Given користувач не має прав «Адміністратор» модуля «Лістинг»
- When він відкриває картку прайс-листа
- Then кнопка «Оновити ціни в лістингах» не відображається.
```
ОБОВ'ЯЗКОВО додавай AC на «нещасливі» шляхи:
- немає даних / товар без ціни → система не падає, рядок не оновлює;
- помилка виконання → повідомлення, налаштування не змінюються, кнопка лишається для повтору;
- права/видимість → кнопка чи поле приховані для не-адміна;
- межі/constrains → напр. «друга спроба призначити власника → User Error».

Один AC = одна перевірювана думка. Покрий кожну функціональну вимогу хоча б одним AC.
Допустима нумерація групами (AC 1.1, 1.2 / AC 2.1…) для великих ТР.

## 6. Acceptance Criteria — нумерований список (для UI/візуальних вимог)
```
1. Критерій візуальної відповідності: значення фільтрів з нульовою кількістю товарів повністю
   приховані з DOM (а не disabled).
2. Критерій не-впливу на власну групу: активація фільтра в групі не приховує інші варіанти цієї ж групи.
3. Критерій збереження вибраного стану: активний фільтр не може зникнути з екрану.
```

## 7. Варіанти рішення
Якщо реалізація неоднозначна — дай варіанти з трейд-офами для вибору клієнтом/архітектором:
```
#### Варіант 1: <мінімальний>
#### Варіант 2: Варіант 1 + <надбудова>
```

## 8. Коли що застосовувати
- Новий функціонал/сутність → User Story + розгорнуті Given/When/Then.
- Точкова доробка → Проблема → Очікуваний результат → Use Cases → Критерії прийняття.
- UI/візуальна поведінка → нумеровані критерії (п.6).
- Будь-яка неоднозначність → додай «Варіанти рішення».
- Зміна існуючої поведінки → обов'язково As-Is/To-Be.

## 9. Acceptance Criteria — український формат (УМОВА/КОЛИ/ТОДІ)
У компанії використовуються **два формати AC** — обирай за проектом/наявними документами проекту,
у межах одного документа формат єдиний:
- **EN:** Given / When / Then / And (проект Elfa).
- **UA:** УМОВА / КОЛИ / ТОДІ / ТА, оформлені як **Scenario N** (проект Олімпіус, новіші ТЗ).
```
Scenario 2 — Спроба додати вже прив'язаний picking:
УМОВА: stock.picking школи №12 вже входить до маршруту "Рейс 23.05"
КОЛИ: Логіст намагається додати той самий picking до іншого маршруту
ТОДІ: Система показує помилку: "Відвантаження Школа №12 вже включено до маршруту Рейс 23.05."
ТА: Додавання не відбувається
```
Зміст вимог однаковий незалежно від формату: межові випадки, помилки, права — обов'язкові.

## 10. Формальна структура фічі (новіший стандарт)
Для складних фіч новіші ТЗ структуруються як окремі іменовані секції (а не суцільний опис):
```
USER STORY        — Як <роль>, я хочу <…>, щоб <…>.
BUSINESS RULES    — нумерований перелік правил (1, 2, 3…) з технічною конкретикою:
                    моделі/поля, формули назв, дефолти, домени, дослівні тексти помилок,
                    статусний pipeline (Чернетка → Підтверджено → Завершено → Скасовано).
ACCEPTANCE CRITERIA — Scenario N (УМОВА/КОЛИ/ТОДІ) або AC# (Given/When/Then).
UI/UX             — окремий розділ меню/модуль, форма (шапка, вкладки, колонки таблиці),
                    кнопки панелі дій (коли активні), статусний badge, орієнтовний вигляд.
```
**BUSINESS RULES** — найцінніша частина: кожне правило однозначне, з технічними іменами і
дослівними повідомленнями. **UI/UX** виноситься окремо, а не ховається в тексті.
Простіші доробки можна лишати у форматі нумерованих секцій (як у старіших ТЗ) — це теж припустимо.

## 11. Межі scope ТЗ (дисципліна обсягу)
Явно фіксуй, що входить у ЦЕЙ ТЗ, а що винесено в наступний — це запобігає роздуванню обсягу.
Приклад формулювань з реальних ТЗ:
> «В рамках даного ТЗ реалізовуємо лише поле, розрахунок цього поля додамо в наступному ТЗ.»
> «В рамках цього ТЗ ми зупиняємось лише на візуалізації; логіка після "Підтвердити" — наступне ТЗ.»
Додавай такий абзац наприкінці опису, коли фіча природно ділиться на етапи.

## 12. «Тестовий сценарій» як формат приймання (третій варіант)
У розширеній структурі (Олімпіус) приймання часто оформлюють не як Given/When/Then, а як таблицю
секції **«Тестовий сценарій»**: `| № | Крок | Очікуваний результат |`. Це припустимо для простіших
фіч — нумеровані кроки і є критеріями приймання. **Протокол тестування** — окрема таблиця з фактом
проходження (Pass/Виявлено/примітка).
Спостереження з корпусу: ці секції часто лишають порожніми — це слабке місце. Тому навіть у цьому
форматі **заповнюй тестовий сценарій реальними кроками**, що покривають кожну вимогу і «нещасливі»
шляхи; не лишай порожній шаблон.

Отже, три валідні формати приймання: (1) EN Given/When/Then; (2) UA УМОВА/КОЛИ/ТОДІ (Scenario N);
(3) нумерований «Тестовий сценарій» (Крок/Очікуваний результат). Обирай за проектом, але завжди
заповнюй фактичним змістом.

## 13. Зведена таблиця правил (для складної умовної поведінки)
Коли поведінка залежить від комбінації статус × роль × дія — стисло подай її матрицею (з проекту ДЕМЗ):
```
| Сценарій | Статус | Користувач | Дія системи |
| PO прийнято | picking=done | Звичайний | Редагування заблоковано, банер |
| PO прийнято | picking=done | Менеджер | Редагування дозволено + запис в audit |
| Скасовано всі picking | done→cancel | Будь-хто | Авто-розблокування |
```
Така таблиця доповнює (не замінює) AC/тест-кроки і добре читається розробкою та клієнтом.
