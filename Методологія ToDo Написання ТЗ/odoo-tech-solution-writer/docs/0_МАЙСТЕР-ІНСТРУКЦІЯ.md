# НАПИСАННЯ ТР — Майстер-інструкція

**Для кого:** бізнес-аналітик, архітектор, менеджер проекту
**Коли:** від задачі на доопрацювання Odoo (бриф / транскрипт зустрічі / задача в ERP) до готового Технічного рішення
**Результат:** узгоджений документ `ТР_[назва].md` + запис у реєстрі «Перелік технічних рішень»

---

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|----------|-----------------|-------------|
| Є бриф / транскрипт / задача — почати ТР | "створи ТР по [задача/проект]" | каркас `ТР_[назва].md` |
| Описати «що і навіщо» | "напиши функціональні вимоги для [назва]" | заповнена секція 1 у `ТР_[назва].md` |
| Описати поведінку і приймання | "напиши use cases і критерії прийняття для [назва]" | Use Cases + Acceptance Criteria у `ТР_[назва].md` |
| Описати реалізацію в Odoo | "опиши технічну реалізацію для [назва]" | секція «Опис технічної реалізації» у `ТР_[назва].md` |
| Оцінити і написати інструкцію | "оціни в годинах і напиши інструкцію для [назва]" | «Обсяг робіт» + «Інструкція» у `ТР_[назва].md` |
| Перевірити готовий ТР | "перевір ТР [назва]" | звіт QA-перевірки консистентності |
| Внести в реєстр | "онови реєстр технічних рішень по [назва]" | рядок для «Перелік технічних рішень» |

## Повний pipeline (рекомендований маршрут)

```
Бриф / транскрипт Fathom / задача в erp.todo.ltd
    ↓
1. tr-scaffold                →  ТР_[назва].md  (каркас + Загальна інформація + вибір формату)
    ↓
2. tr-functional-requirements →  секція 1: Основна бізнес-ціль, As-Is/To-Be, функціональні вимоги
    ↓
3. tr-usecases-acceptance     →  Use Cases (Сценарії) + Acceptance Criteria (Given/When/Then)
    ↓
4. tr-odoo-tech-design        →  Опис технічної реалізації (моделі td.*, поля, довідники, права, UI)
    ↓
5. tr-effort-instruction      →  Обсяг робіт (оцінка) + Інструкція (тест-кейс)
    ↓
6. tr-review                  →  QA-перевірка консистентності перед узгодженням
    ↓
[ УЗГОДЖЕННЯ КЛІЄНТА ]   →   [ АРХІТЕКТУРА заповнює архітектор ]
    ↓
7. tr-registry-update         →  запис у «Перелік технічних рішень»
```

Кожен скіл можна запускати окремо на потрібну секцію — усі дописують в один файл `ТР_[назва].md`.

---

## Мінімальний маршрут (є дедлайн)

Найшвидший шлях до робочого ТР для невеликого доопрацювання:
```
1. tr-scaffold
3. tr-usecases-acceptance   (Проблема → Очікуваний результат → AC)
4. tr-odoo-tech-design
5. tr-effort-instruction
6. tr-review
```

---

## Типові сценарії

**Сценарій А: велике доопрацювання / новий функціонал.**
Запускай 1→2→3→4→5→6 повністю (Формат A — по фічах/сутностях).

**Сценарій Б: точкова доробка (1–3 задачі).**
1 (Формат B — по задачах) → 3 → 4 → 5 → 6. Секція 2 може бути коротшою.

**Сценарій В: вхід — транскрипт зустрічі.**
`tr-scaffold` сам підтягне транскрипт через Fathom (`list_meetings` / `get_meeting_transcript`).
Якщо на зустрічі кілька різних доопрацювань — розділи на окремі ТР або на «Задачі» в одному.

**Сценарій Г: ТР уже написаний, треба здати клієнту.**
6 (tr-review) → виправити зауваження → 7 (tr-registry-update) → експорт у .docx.

## Формати ТР

- **Формат A — по фічах/сутностях** (флагман): нова сутність `td.*`, багато полів. Тіло секції
  «ТЕХНІЧНЕ РІШЕННЯ» іде по об'єктах. Приклад: «1. Технічне рішення: Розробка функціоналу лістингу».
- **Формат B — по задачах** (доопрацювання): 1–3 невеликі правки. Тіло іде блоками
  «Задача N → Проблема → Очікуваний результат → Use Cases → Критерії прийняття → Варіанти рішення».
- **Формат C — Правки/Доопрацювання** (зміни до вже описаного модуля): легкий патч/трекер, не повний
  ТР — групування по об'єктах, маркери статусу, посилання на вихідне ТЗ. Див. `правки-та-доопрацювання.md`.

---

## Межі відповідальності

- Секцію **АРХІТЕКТУРА** (об'єкти на рівні БД, «Налаштування системи») скіли **не заповнюють** —
  її пише архітектор після узгодження ТР клієнтом. Скіли лишають каркас і, опційно, перелік
  нових об'єктів як гіпотезу.
- Фінальна конвертація узгодженого `.md` у брендовий `.docx` (помаранчевий `#FF742F`, лого,
  «Too easy do simple», Times New Roman) — окремий крок через скіл `docx`.

## Глибокі довідники (docs/)

- `шаблон-ТР.md` — точний каркас документа і правила оформлення.
- `odoo-конвенції.md` — іменування `td.*`/`td_*`, типи полів, моделі, довідники, права, UI, чек-лист.
- `патерни-UC-AC.md` — User Story / Use Cases / Acceptance Criteria (EN і UA формати) / As-Is–To-Be /
  формальна структура фічі / межі scope.
- `звіти-та-обміни.md` — окремі типи: звіт (Pivot) і обмін/інтеграція (мапінг полів, синхронізація).
- `патерни-Odoo-рішень.md` — 35 повторюваних архетипів реалізації (wizard, ir.cron, singleton-флаг,
  версійність BoM, портал-форма, QWeb-друк, движок сценаріїв/стадій, країнова варіація, Transfer/
  Extension, документ-чек-лист, контроль доступу за складом/локацією, блокування за станом, Barcode/ТЗД,
  прикріплення документа→Bill, CRM→downstream, Approvals-процес, round-robin, аналітичний облік тощо).
- `друковані-форми.md` — структура опису QWeb друкованих форм (поля-джерела, сталі блоки, мультимовність,
  експортні реквізити, 2 шаблони за умовою, підміна стандартної форми).
- `odoo19-довідник.md` — офіційний довідник Odoo 19 (15 розділів): типи полів, `<list>`,
  `invisible`/`required` замість `attrs`, computed/обмеження, record rules, дії/cron, QWeb+звіти, домени,
  mail/activity/chatter, Automation Rules, res.config.settings, web/portal/OWL, окруження, зовнішній API
  (v19 `/json/2`+Bearer). Джерело правди по синтаксису v19.
- `правки-та-доопрацювання.md` — жанр документів **змін** до вже описаного модуля (патч після
  зустрічі / доопрацювання / двошаровий клієнт→BA / живий трекер; маркери статусу, дедлайни).
- `приклади.md` — розпізнані патерни з реальних ТР (Elfa — роздріб; Олімпіус — виробництво/логістика/обміни).
- `калібрування-оцінки.md` — орієнтири годин за реальними ТР ToDo (для `tr-effort-instruction`).

> **Домен і версія — параметри проекту.** Набір моделей Odoo залежить від галузі (роздріб / MRP /
> логістика / проекти / бухгалтерія / склад+закупівлі / HR-рекрутинг), а версія — своя на проект
> (Elfa `Odoo 19 EE`, Олімпіус `Odoo 18 EE`, Bixter HR, ДЕМЗ `Odoo 19 EE`). Не зашивай у скіли
> роздріб і версію 19 — бери з картки проекту.

> **ОБОВ'ЯЗКОВА звірка з документацією Odoo.** При написанні технічної реалізації **кожен пункт, що
> дорабатывается, звіряється з офіційною документацією Odoo** (для v19 — спершу `odoo19-довідник.md`,
> зведений з офіційного сайту; за потреби — наживо `https://www.odoo.com/documentation/19.0/`). Біля
> пункту вказується джерело; синтаксис — лише актуальний (не `attrs`/`<tree>`/`t-raw`/xmlrpc). Це
> робить `tr-odoo-tech-design` (Крок 1), а `tr-review` перевіряє наявність звірки. Деталі — у цих скілах.

Джерело методології: папка «Методологія ToDo Написання ТЗ» на Google Drive (шаблон, реєстр
«Перелік технічних рішень», реальні ТР Elfa).
