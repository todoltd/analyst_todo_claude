## Що змінилось у версії 1.1

**Структура 7 → 5 сесій** (номери стабільні — об'єднані сесії позначені, не перенумеровані):
- Сесія 3 (бухоблік Гонконгу) **об'єднана з Сесією 2** → спільна «облікова сесія» з бухгалтером (агентська закупівля + AP + план рахунків + банк/каса). Підстава: агентська трактовка напряму кормить план рахунків, обидві теми податково чутливі й потребують бухгалтера.
- Сесія 6 (звітність + права) **об'єднана з Сесією 5** → AR і прибутковість тісно пов'язані зі звітністю по маржі; модель прав тривіальна.
- Сесія 7 (перехід) — лишається, але **коротка**: міграція лише залишків, інтеграції майже всі «ні», NFR скромні — переважно підтвердження.

**Ризики:**
- R-1 (агентська закупівля): 🔴 → 🟡 — рамку отримано на С1; лишилась одна облікова розвилка (до бухгалтера).
- R-2 (кастомні транспортні документи): 🟡 → 🟢 — на С1 прояснено: екранні інтерфейси, одна модель на AWB/коносамент/CMR, без друку; обсяг кастома менший.
- R-5 (демо Odoo): прив'язано до Сесії 4, фокус — товарні партії + landed cost.
- **+ R-6 (новий)** 🔴: авто-рознесення витрат постачальника на собівартість партії (landed cost) — головний кастом, осуществимість не підтверджена.

**Нові виключення зі scope (підтверджено на С1, журнал X-14…X-18):** payroll у системі (ведеться в Excel), облік курсових різниць, друк документів/бланків, окремі коригувальні документи, автоматизація зміни маршруту/перевізника.

**Нові вимоги з С1, розселені по сесіях:** товарна партія + landed cost → С4; довідник рахунків для оплати → С5; помісячний звіт по закритих відправках (вхід для розрахунку ЗП) → С5; обов'язкові поля картки контрагента → С2; банківсько-касовий контур (конвертація, перекази, каса, банк-комісії) → С2.

## Синтез: що знаємо

### Підтверджений scope (за результатами УЗ + Сесії 1)
- **Система — виключно облікова** (фінансовий + управлінський + бухгалтерський облік), не операційна ERP — підтверджено.
- **Кастомні форми транспортних документів** (авіанакладна, коносамент, CMR) — **екранні інтерфейси, що візуально повторюють отриманий електронний документ; без друку**. Три типи обліково ідентичні (одна модель, відмінність — нумерація/реквізити); податкові поля прибираються. *(Уточнено на С1 — обсяг кастома менший, ніж закладали.)*
- Довідники UN Locations, авіакомпаній, перевізників — завантажити з 1С + оновити з безкоштовних офіційних UN-джерел; оновлюються рідко.
- **Облік логістичних послуг:** чернетка транспортного документа → позначення «відправлено» → вхідні рахунки субпідрядників → собівартість → вихідний рахунок клієнту (з котирування) → оплата клієнта → оплата агентам. Прибутковість по відправці (виручка − витрати).
- **Платежі:** документи надходжень/оплат — окремі; зв'язок «багато-до-багатьох» з відправками; прив'язка вхідних рахунків за номером транспортного документа; часткові/повні оплати.
- **Торгівля / товарні партії:** ExWorks купівля-продаж; примітивний box-moving; ФІФО, без партій/серійників. **Нова вимога — landed cost: рознесення витрат субпідрядників на собівартість партії, бажано авто з рахунку постачальника.**
- **Інвойсинг і AR:** рахунок по відправці, стейтменти, аванси/відстрочки, перевиставлення наскрізних витрат із націнкою.
- **Бухоблік Гонконгу:** план рахунків Гонконгу, мультивалюта (база USD), консолідація для аудиторів. Діяльність проста (за словами власника).
- **Конвертація валют** — окремий документ; курсові різниці не обліковуються.
- **Ролі:** менеджери з продажів бачать лише «сплачено/не сплачено»; можуть заводити контрагентів; правки рахунків — через бухгалтерію (ACL).
- **Користувачі:** максимум 6. **Обсяг:** до 50 відправок/місяць.

### Відкриті питання, що впливають на план
- **[КД] Облікова трактовка агентської закупівлі** (виручка vs транзит) — до бухгалтера на Сесії 2.
- **[КД] Перелік документів для Odoo** — клієнт довантажує до Сесії 2 (структури документів і довідники на Google Drive).
- **[ВД] Осуществимість landed cost** в Odoo 19 (авто-рознесення з рахунку постачальника) — перевірка BA/архітектором + демо до/на Сесії 4.

### Ризики, враховані в плані (актуалізовано v1.1)
- 🟡 **R-1 «Агентська закупівля»** — рамку отримано на С1 (комісія/націнка, товар напряму клієнту, дві можливі моделі). Лишилась облікова розвилка → винесено в облікову Сесію 2 з бухгалтером.
- 🟢 **R-2 Кастомні транспортні документи** — прояснено на С1 (екранні інтерфейси, одна модель, без друку). Специфікацію уточнюємо за наданими клієнтом зразками; залишковий ризик низький.
- 🟡 **R-3 Бухоблік Гонконгу** — тепер у Сесії 2; проста діяльність, але мультивалюта + трактовка агентської закупівлі потребують бухгалтера.
- 🟡 **R-4 Бізнес у фазі розвороту** (логістика → торгівля) — не фіксувати scope як остаточний до Сесії 5 (AR/прибутковість).
- 🟢 **R-5 Демо Odoo** — кероване демо по товарних партіях/landed cost, орієнтовно на/перед Сесією 4.
- 🔴 **R-6 Landed cost (новий)** — авто-рознесення витрат на собівартість партії: головний кастом, осуществимість не підтверджена. Перевірити технічно (BA + архітектор) до проєктування С4; підстрахувати демонстрацією.

### Сесія 2: Облікова сесія — агентська закупівля + розрахунки з постачальниками (AP) + бухоблік Гонконгу + банк/каса
Блок: A
Тривалість: 2 год (об'єднана сесія — за переповнення банк/касу винести в хвіст С5)
Учасники від клієнта: Станіслав; **бухгалтер/аудитор — обов'язково**
Ведучий від нас: BA (+ архітектор бажано)

> Об'єднує колишні Сесії 2 і 3. Агентська закупівля — НЕ повторна розвідка процесу (рамку отримано на С1), а одна облікова розвилка + уточнення, плюс структура бухобліку.

**Ціль сесії:**
Визначити облікову трактовку агентської закупівлі (виручка vs транзит) і структуру бухобліку (план рахунків Гонконгу, мультивалюта, контури); формалізувати AP і реєстр платежів; зафіксувати банківсько-касовий контур.

**Питання для дослідження:**
→ Облікова розвилка агентської закупівлі (до бухгалтера): виручка по повній угоді (купівля-продаж із націнкою) чи лише комісія як виручка, а кошти клієнта — транзит/обов'язок? Це визначає оборот, проводки і податок.
→ Хронологія платежів: клієнт завжди платить вперед, чи буває, що авансуєте фабрику зі своїх? Як обліковувати розрив у часі?
→ Конвертація: на якому моменті фіксуєте курс і чий курс (узгоджено з D-12 — окремий документ).
→ AP: як зіставляєте рядок вхідного рахунку з послугою у відправці; дублі/переплати; часткові оплати агентам; витрати, рахунок по яких ще не прийшов (accrued).
→ Перевиставлення наскрізних витрат із націнкою — облікова механіка (окремий рядок / у сумі).
→ План рахунків Гонконгу: готова структура? хто веде облік? розмежування контурів (фінансовий/управлінський/бухгалтерський)? мультивалюта (база USD)? формат і періодичність консолідації для аудиторів?
→ Банк/каса: конвертація валют, перекази між рахунками, зняття/внесення готівки, банк-комісії; довідник рахунків для оплати (банк, валюта); обов'язкові поля картки контрагента.

**Очікуваний результат:**
Облікова модель агентської закупівлі (підтверджена бухгалтером); вимоги до AP + реєстру платежів; рекомендація з Accounting (план рахунків, валюти, контури, межа «Odoo консолідує / аудитор звітує»).

**Ризики сесії:**
Податково/юридично чутлива зона — без бухгалтера частина перенесеться. Сесія об'ємна; якщо банк/каса не вміщується — винести у хвіст С5. Не фіксувати остаточну трактовку, якщо власник вагається — лишити варіанти до підтвердження.

### Сесія 4: Торгівля і товарний рух + товарні партії (landed cost)
Блок: B
Тривалість: 1.5–2 год
Учасники від клієнта: Станіслав; особа, що працює зі складом/закупівлями
Ведучий від нас: BA + архітектор (демо)

**Ціль сесії:**
Вимоги до обліку товарних партій, закупівель і складського руху (примітивний box-moving) + **механіка рознесення витрат на собівартість партії (landed cost)**.

**Питання для дослідження:**
→ **Landed cost (ключове):** як має працювати рознесення витрат субпідрядників на собівартість партії (приклад «100 + доставка 5 = 105»); бажано авто з рахунку постачальника, у т.ч. коли витрати додаються поступово кількома рахунками.
→ Два сценарії: ExWorks купив/продав без фізичного руху і консолідаційний склад + box-moving — як обидва відображаються документами.
→ Повноцінні документи закупівлі: хто оформлює, які поля, як пов'язані з продажем.
→ Складський рух партій (об'єднання/розділення коробок) — як обліковувати (ФІФО, без серійників — D-9).
→ Як визначається собівартість товару в партії для торгової маржі.
→ Чи перетинається товарний рух з логістичними послугами в одному замовленні.

**Очікуваний результат:**
Вимоги до спрощених Purchase/Inventory; модель руху товарних партій; **підтверджена осуществимість landed cost (Q-13) + результат керованого демо Odoo (R-5)**.

**Ризики сесії:**
R-6 — landed cost може потребувати кастому; перевірити технічно ДО сесії, на демо показати наявний механізм. Не перевантажити складським функціоналом — клієнт прямо сказав «все дуже просто».

### Сесія 5: Розрахунки з клієнтами (AR), прибутковість + управлінська звітність і права доступу
Блок: B/C
Тривалість: 2 год (об'єднана сесія)
Учасники від клієнта: Станіслав; бажано бухгалтер
Ведучий від нас: BA + архітектор

> Об'єднує колишні Сесії 5 і 6. Звітність по маржі і права доступу — пряме продовження AR/прибутковості; модель прав тривіальна (менеджер бачить лише «сплачено/не сплачено»).

**Ціль сесії:**
Модель інвойсингу клієнтам, взаєморозрахунків і розрахунку прибутковості по відправці (головний business case); перелік звітів/дашбордів і матриця прав доступу.

**Питання для дослідження:**
→ Рахунки: по кожній відправці + об'єднання у стейтмент (тижневий/місячний) — як формується стейтмент.
→ Аванси і відстрочки: як обліковувати (штрафних санкцій немає — X-8).
→ Прибутковість = виручка − витрати; у який момент маржа стає «фактичною»; accrued-витрати (зв'язок із С2).
→ Перевиставлення наскрізних витрат із націнкою — підтвердити облікову механіку (узгодити з С2).
→ Довідник рахунків для оплати (банк, валюта) у вихідному рахунку — структура.
→ Помісячний звіт по закритих відправках (вхід для розрахунку ЗП — сам payroll поза системою, X-15): склад, формула.
→ Розрізи маржі (клієнти, менеджери, періоди; маршрути/перевізники — другий етап); фінансові звіти власнику.
→ Матриця прав: менеджери бачать лише «сплачено/не сплачено», без фінансових документів; хто редагує довідники; чи потрібен журнал аудиту змін.

**Очікуваний результат:**
Модель AR + розрахунку маржі по відправці; перелік аналітичних розрізів і звітів MVP; матриця ролей і прав. Після цієї сесії — фіксація меж MVP (Q-8).

**Ризики сесії:**
Вузький спред маржі — точність рознесення витрат критична (залежить від С2 і С4). Бажання «бачити все» по звітах — пріоритизувати під MVP.

### Сесія 7: Перехід і технічні питання (наскрізні лінії) — коротка
Блок: C
Тривалість: 1–1.5 год (переважно підтвердження)
Учасники від клієнта: Станіслав; бухгалтер (по залишках)
Ведучий від нас: BA + архітектор + PM

**Ціль сесії:**
Закрити наскрізні лінії: міграція, межі інтеграцій, NFR, go-live, навчання.

**Питання для дослідження:**
→ Міграція: переносимо тільки залишки (D-7), історію — ні. Які саме залишки (AR/AP, банк, склад) і на яку дату cutover (Q-9)?
→ Довідники (UN Locations, авіакомпанії, перевізники) — формат завантаження, обсяг (джерела підтверджені на С1).
→ Інтеграції — підтвердити межі: банки (ні, вручну — X-3), трекінг (ні — X-4), пошта/парсинг (ні — X-6), Bitrix (окремо — X-5). Чи є щось, що таки треба обмінювати?
→ NFR: 6 користувачів, мова інтерфейсу, обсяги (~50 відправок/міс — отримано), вимоги до доступності.
→ Перехід: паралельна робота чи різкий cutover? Хто і коли навчається (6 користувачів)?

**Очікуваний результат:**
План міграції залишків + довідників; підтверджені межі інтеграцій; NFR; план go-live і навчання.

**Ризики сесії:**
Cutover фінансових залишків — вивірити дату і коректність вхідних залишків.
