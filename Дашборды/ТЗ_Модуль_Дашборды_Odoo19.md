# 1. Введение

## 1.1. Цель

Разработать нативный модуль Odoo 19 «Дашборды» — полноценную BI-платформу внутри Odoo, функционально сопоставимую с Google Looker Studio и Microsoft Power BI. Модуль позволяет бизнес-пользователю без программирования: подключиться к **любой** модели Odoo, пройти по связям произвольной глубины до любого поля, построить датасет с фильтрами, агрегациями и вычисляемыми полями, объединить несколько моделей (аналог blended data), сконструировать интерактивный дашборд из каталога визуализаций и поделиться им — внутри компании, по публичной ссылке, через embed, экспорт и рассылку по расписанию.

Ключевое отличие от внешних BI-систем: модуль исполняется внутри Odoo и **наследует модель безопасности Odoo автоматически** — каждая выборка данных выполняется от имени текущего пользователя через ORM, поэтому ACL (`ir.model.access`), record rules (`ir.rule`), ограничения видимости полей (`groups=`) и мультикомпанийность применяются без дублирования настроек. Это устраняет главный риск внешних BI: расхождение прав в учётной системе и в отчётности.

## 1.2. Контекст и рамки

- Целевая платформа: Odoo 19.0 Community. Модуль не должен зависеть от Enterprise-модулей (`spreadsheet_edition`, `documents`, `web_grid`, `web_map`, `web_cohort`). При наличии Enterprise модуль работает без конфликтов.
- Модуль строится поверх штатной инфраструктуры Odoo 19: OWL 2, реестры компонентов, сервис `orm` (`formatted_read_group`, `formatted_read_grouping_sets`, `web_search_read`), Chart.js 4.4.5 (vendored в `addons/web`), классы `Domain` (Python и JS), `useSortable`/`makeDraggableHook`, паттерн публичных ссылок из `spreadsheet_dashboard`.
- Важное ограничение версии: публичный метод `read_group` в Odoo 19 **удалён** (deprecated с 18.2). Вся агрегация в модуле выполняется через `formatted_read_group` / `_read_group` / `formatted_read_grouping_sets`. Атрибут агрегатора поля называется `aggregator` (бывший `group_operator`).

## 1.3. Термины

| Термин | Определение |
|---|---|
| Датасет (`td.bi.dataset`) | Настроенный источник данных: корневая модель Odoo + выбранные поля (включая поля связанных моделей по цепочкам), фильтры, вычисляемые поля; либо объединение нескольких источников (blend), либо SQL-источник (для администраторов) |
| Путь поля (field path) | Точечная цепочка от корневой модели до поля связанной модели, например `order_id.partner_id.country_id.name` |
| Измерение (dimension) | Поле, по которому выполняется группировка (ось, легенда, строка pivot) |
| Мера (measure) | Числовой показатель: поле с агрегатной функцией либо формула над агрегатами |
| Вычисляемое поле | Поле, заданное формулой на строчном уровне (аналог calculated column Power BI / calculated field Looker Studio) |
| Виджет (`td.bi.widget`) | Единица визуализации на странице дашборда: тип диаграммы + конфигурация данных + стиль + позиция в сетке |
| Контрол (`td.bi.control`) | Управляющий элемент фильтрации для зрителя: период, выпадающий список, поиск, диапазон и т. п. (аналог slicer Power BI / control Looker Studio) |
| Cross-filtering | Клик по элементу одного виджета фильтрует остальные виджеты страницы |
| Drill-down | Спуск по уровням иерархии группировок внутри виджета |
| Drill-through | Переход из точки данных к детальным записям (стандартный list view Odoo) или на страницу детализации |
| Временной интеллект | Расчёты относительно календаря: YTD/QTD/MTD, сравнение с прошлым периодом/годом, скользящие окна, накопительные итоги |
| RLS | Row-level security — ограничение видимости строк данных; в модуле реализуется record rules Odoo + дополнительными доменами аудитории дашборда |
| Frozen-снимок | Сохранённый результат данных дашборда для публичного доступа без живых запросов к БД |

## 1.4. Что берём от аналогов

**От Power BI:** разделение «вычисляемый столбец vs мера»; меры с модификацией контекста фильтра (аналог `CALCULATE` — фильтрованные меры, «% от итога»); библиотека быстрых мер (quick measures); временной интеллект; три уровня фильтров (виджет/страница/дашборд); матрица взаимодействий виджетов (Edit interactions); drill-down/drill-through; закладки (bookmarks); JSON-темы; подписки и data alerts; режим «View as» для проверки RLS.

**От Looker Studio:** трёхслойная модель «коннектор → источник (датасет) → отчёт»; переиспользуемые датасеты с переименованием/скрытием полей и агрегацией по умолчанию; blended data (до 5 таблиц, 5 типов JOIN, ключи-равенства, предагрегация до соединения); два уровня вычисляемых полей (датасет/виджет); параметры с наследованием отчёт→страница→группа→виджет; контролы с областью действия через группировку; сравнение с прошлым периодом «из коробки» у виджета; draft/publish и история версий; расписания рассылки с фильтрами на расписание.

**Чего сознательно НЕ берём в первой версии** (обоснование — раздел 14): cross-highlighting (затемнение долей вместо фильтрации) — заменяется cross-filtering; Q&A на естественном языке (в Power BI визуал Q&A объявлен deprecated, замена — Copilot; у нас — вне объёма, решение ОВ-11); кастомные community-визуалы как магазин; режимы Import/DirectQuery — модуль всегда работает «live» по PostgreSQL, поэтому шлюзы и scheduled refresh источников не нужны (расписания нужны только материализованным предагрегатам).

## 1.5. Критерии успеха

- Пользователь-аналитик (без прав разработчика) строит датасет «продажи по строкам заказов с клиентом, страной, категорией товара и командой продаж» и дашборд из 8 виджетов за 30 минут без обращения к разработчику.
- Любое поле любой модели, доступной пользователю по правам, достижимо через дерево полей не более чем за 5 переходов по связям.
- Продавец и руководитель открывают один и тот же дашборд и видят разные данные строго по своим record rules — без какой-либо настройки в самом модуле.
- Дашборд из 10 виджетов на модели в 1 млн строк рендерится не дольше 5 секунд (p95) при тёплом кэше и не дольше 15 секунд при холодном.

# 2. Концепция и архитектура слоёв

## 2.1. Слои

Модуль построен как конвейер из шести слоёв; каждый слой имеет собственные модели хранения и API:

1. **Источники данных.** Любая нетехническая модель Odoo (через `ir.model`), включая кастомные и Studio-модели. Для администраторов — дополнительно SQL-источники (паттерн `_table_query`). Слой отвечает на вопрос «куда можно дотянуться»: каталог моделей, дерево полей с навигацией по связям, метаданные `fields_get()`.
2. **Датасеты / модель данных.** `td.bi.dataset`: корневая модель + поля по путям + домен-фильтр + вычисляемые поля + объединения (blend). Датасет — переиспользуемая сущность (как data source в Looker Studio): один датасет питает много виджетов и дашбордов.
3. **Вычисления.** Меры (`td.bi.measure`), вычисляемые поля, параметры, временной интеллект, библиотека быстрых мер. Формулы — собственный безопасный DSL с AST-валидацией (НЕ `safe_eval` для пользователей).
4. **Визуализации.** `td.bi.widget`: ~25 типов на базе Chart.js 4 + vendored-плагины (geo, treemap) + MIT-плагины (matrix/heatmap, sankey, funnel) + табличные виджеты на стандартных механизмах Odoo.
5. **Дашборды.** `td.bi.dashboard` / `td.bi.dashboard.page`: grid-layout с drag-and-drop и resize, контролы, матрица взаимодействий, темы, закладки, draft/publish, версии.
6. **Доступ и публикация.** Права на дашборды (viewer/editor/manager), дополнительное сужение данных по аудитории, публичные ссылки (frozen/live), embed, экспорт PDF/XLSX/PNG/CSV, подписки, алерты.

## 2.2. Принципы архитектуры

- **П-1. Безопасность по умолчанию.** Все чтения бизнес-данных — от имени текущего пользователя (`su=False`), через ORM (`formatted_read_group`, `web_search_read`) либо через SQL, в который домен record rules внедрён явно (раздел 8.4). `sudo()` разрешён только для чтения собственных конфигурационных моделей `bi.*` и метаданных, никогда — для бизнес-данных.
- **П-2. Агрегация в PostgreSQL.** Никакой агрегации «в Python по записям»: группировки и агрегаты считает СУБД (`GROUP BY`, `GROUPING SETS`, window-функции). Клиенту передаются только агрегированные ряды (сотни строк, не тысячи).
- **П-3. Конфигурация — данные, не код.** Конфиг виджета, состояние фильтров, темы, закладки — JSON по задокументированной схеме (раздел 11.7), хранимый в полях `Json`/`Text`. Домены хранятся строкой и парсятся `ast.literal_eval` (как `ir.filters`), формулы — текстом DSL и компилируются на сервере. Никогда не исполняется пользовательский текст через `eval`/`safe_eval`.
- **П-4. Переиспользование ядра.** Всё, что есть в Odoo 19, не переписывается: см. таблицу 2.3.
- **П-5. Один источник правды для прав.** Модуль не имеет собственной системы «ролей на данные». Видимость строк = record rules Odoo (+опциональное сужение доменом аудитории). Видимость полей = `groups=` на полях (отрабатывается автоматически: `fields_get()` от имени пользователя не возвращает запрещённые поля).
- **П-6. Деградация вместо отказа.** Виджет, упавший по таймауту/правам, показывает карточку ошибки с диагностикой и кнопкой повтора — остальные виджеты страницы продолжают работать (загрузка виджетов параллельна и независима).

## 2.4. Поток данных при отрисовке виджета

1. Клиент (OWL-компонент виджета) собирает **эффективный домен**: `домен датасета ∧ домен аудитории дашборда ∧ значения контролов (дашборд/страница/группа) ∧ cross-filter-домены ∧ собственный домен виджета` — через `Domain.and(...)` (`@web/core/domain`). Record rules в домен НЕ включаются — их применит сервер.
2. Клиент вызывает RPC `td.bi.dataset.run_query(dataset_id, query_spec)` с контекстом пользователя (`tz`, `lang`, `allowed_company_ids`).
3. Сервер валидирует `query_spec` по схеме (раздел 11.7), проверяет доступ к датасету и к корневой модели (`has_access('read')`), компилирует запрос:
   - датасет-модель → `formatted_read_group(domain, groupby, aggregates, having, order, limit, offset)` от имени пользователя;
   - несколько виджетов одного датасета на странице → одна команда `formatted_read_grouping_sets` (раздел 10.2);
   - blend/SQL-датасет → генератор SQL (раздел 3.7, 8.4).
4. Сервер выполняет пост-обработку: формулы над агрегатами, временной интеллект (второй ряд со сдвинутым доменом), «% от итога», ранги; кэширует результат (`td.bi.cache`, раздел 10.3).
5. Клиент строит конфиг Chart.js / таблицы и рендерит; `__extra_domain` из ответа сохраняется на точках данных для drill-down и cross-filtering.
# 3. Источники данных и доступ к любой информации (ключевой раздел)

Этот раздел описывает конструктор датасетов — механизм, благодаря которому модуль «дотягивается» до любого объекта, поля и связи Odoo. Все утверждения о поведении ORM сверены с документацией и исходным кодом Odoo 19.0.

## 3.1. Каталог моделей

**Что делает.** Первый шаг создания датасета — выбор корневой модели. Пользователь видит каталог всех доступных ему моделей с человекочитаемыми названиями, поиском и группировкой по приложениям.

**Как настраивается.** Поиск по названию/техническому имени; переключатель «показать технические модели» (только для группы `td_bi_dashboard.group_bi_admin`).

**Реализация в Odoo 19.**

- Источник списка: `env['ir.model'].search([('transient', '=', False), ('abstract', '=', False)])`. Поля `transient` и `abstract` модели `ir.model` отфильтровывают мастера и абстрактные модели.
- Фильтр прав: модель попадает в каталог, только если `env['ir.model.access'].check(model, 'read', raise_exception=False)` истинен для текущего пользователя — каталог у каждого пользователя свой.
- Чёрный список по умолчанию (настраиваемый параметр `ir.config_parameter` `td_bi_dashboard.model_blacklist`): технические префиксы `ir.`, `bus.`, `base_import.`, `mail.followers`, `mail.message` (гигантская таблица, собственная security-логика), `ir.attachment` (переопределённый `_search` с собственной проверкой прав), `ir.logging` и т. п. В ядре Odoo формального флага «техническая модель» нет — список является продуктовым решением и редактируется администратором.
- Группировка по приложениям: через `ir.model.data` → модуль, давший модель, → категория модуля.

## 3.2. Дерево полей и визуальный построитель JOIN-цепочек

**Что делает.** После выбора корневой модели пользователь видит дерево полей. Обычные поля — листья; relational-поля (`many2one`, `one2many`, `many2many`) — раскрываемые ветки: раскрытие показывает поля связанной модели, и так далее на произвольную глубину. Добавление поля из глубины дерева автоматически фиксирует всю цепочку связи — пользователь не настраивает JOIN руками, он просто «идёт по ссылкам». Это и есть визуальный построитель JOIN: цепочка `order_id.partner_id.country_id.name` на канве датасета отображается как путь Заказ → Клиент → Страна → Название.

**Как настраивается пользователем.**

- Поиск поля по всему дереву (включая нераскрытые ветки — ленивый обход в глубину до настроенного лимита).
- У каждого поля видны: подпись, техническое имя, тип, признаки «можно фильтровать / сортировать / группировать / агрегировать».
- Поле добавляется в датасет с ролью: «измерение», «мера» (для числовых) или «атрибут» (для табличных виджетов и tooltip).
- Псевдоним (alias) и описание поля — переименование на уровне датасета, как field editing в Looker Studio.
- Лимит глубины — 5 уровней (параметр); защита от рекурсии: путь, в котором пара (модель, поле) повторяется (например, `partner_id.parent_id.parent_id...`), помечается предупреждением и требует явного подтверждения.

**Реализация в Odoo 19.**

- Метаданные ветки: `env[model].fields_get(attributes=['string', 'type', 'relation', 'relation_field', 'store', 'searchable', 'sortable', 'groupable', 'aggregator', 'selection', 'currency_field', 'related'])`, вызываемый **от имени текущего пользователя** (`su=False`). Это принципиально: `fields_get()` автоматически вырезает поля, закрытые атрибутом `groups=`, и возвращает переведённые подписи. Атрибуты `searchable`, `sortable`, `groupable` вычисляются ORM реально (с учётом search-методов и related-цепочек) — UI просто отражает их, не угадывая.
- `aggregator` (актуальное имя атрибута в 19; прежний `group_operator` устарел) определяет агрегатную функцию по умолчанию: `sum, avg, min, max, count, count_distinct, bool_and, bool_or`.
- UI-компонент: переиспользуем `ModelFieldSelector` (`@web/core/model_field_selector`) — штатный компонент выбора поля с навигацией по relations (props: `resModel`, `path`, `followRelations`, `filter(fieldDef)`); поверх него строится панель дерева с мультивыбором. Внутри он использует `fieldService.loadFieldInfo/loadPath` — кэширование метаданных на клиенте получаем бесплатно.
- Кэш метаданных на сервере: граф `fields_get` кэшируется `ormcache` с ключом (модель, пользователь-группы, язык) и сбрасывается по `CLEAR-CACHES`.

**Семантика путей по типам связей** (важно — поведение различается):

| Тип звена в пути | Фильтрация (domain) | Группировка (groupby) | Агрегация (мера) |
|---|---|---|---|
| `many2one` | Да: dot-path домены `('partner_id.country_id.code', '=', 'UA')` поддерживаются ORM нативно (автоматический подзапрос с учётом record rules комодели) | Да: в Odoo 19 `_read_group`/`formatted_read_group` принимают dot-path по цепочке many2one: `groupby=['partner_id.country_id']`, в т. ч. с гранулярностью дат на конце `'order_id.date_order:month'` | Нет (агрегаты по dot-path ORM не поддерживает — `ValueError`); решение ниже |
| `one2many` / `many2many` | Да: операторы `any` / `not any` — `('order_ids', 'any', [...])`; классические dot-path тоже работают | Прямо — нет (o2m); m2m — да, но JOIN дублирует строки и искажает sum/avg (см. ограничение Л-3) | Нет |

**Решение для мер «с дальней модели»** (выбранный вариант + альтернативы):

- **Выбрано: правило «факт — самая глубокая модель».** Конструктор предлагает в качестве корневой модели датасета модель фактов (строки заказа `sale.order.line`, проводки `account.move.line`, движения `stock.move`), а измерения подтягиваются вверх по many2one-цепочкам — это покрывает 95% аналитических сценариев и ложится на нативные возможности `formatted_read_group` без единой строчки SQL. Мастер создания датасета подсказывает: если пользователь выбрал `sale.order` и пытается мерить `order_line.price_subtotal`, мастер предлагает «перевернуть» датасет на `sale.order.line`.
- Альтернатива А (этап 2): меры через o2m-связь исполняются вторым `formatted_read_group` по дочерней модели с доменом `('order_id', 'in', ids-группы)` и склейкой на сервере — дороже, но универсально.
- Альтернатива Б (отклонена): создание related-полей «на лету» через `ir.model.fields` (state='manual') — загрязняет реестр полей, требует прав администратора, мешает обновлениям.

## 3.3. Поддержка всех типов полей Odoo

Полный перечень типов Odoo 19 (`binary, boolean, char, date, datetime, float, html, integer, json, many2many, many2one, many2one_reference, monetary, one2many, properties, properties_definition, reference, selection, text`) и их обработка в модуле:

| Тип | Роль в BI | Особенности реализации |
|---|---|---|
| `integer`, `float` | Мера / измерение | `digits` учитывается в форматировании; агрегаты sum/avg/min/max/count |
| `monetary` | Мера | Обязательно тянется `currency_field` (из `fields_get`); три стратегии агрегации — раздел 3.6 |
| `boolean` | Измерение / мера (bool_and/bool_or, count) | при группировке NULL приводится ORM к FALSE |
| `char`, `text` | Измерение / атрибут | переводимые (`translate=True`) поля хранятся jsonb и группируются по значению языка пользователя с fallback en_US — предупреждение в UI (стабильная группировка — по id связанной записи, не по строке) |
| `html` | Атрибут (таблицы) | вывод как очищенный текст |
| `date`, `datetime` | Измерение времени | гранулярности `:day/week/month/quarter/year` (+`hour` для datetime) и числовые части (`month_number`, `iso_week_number`, `day_of_week` и т. д.); datetime хранится в UTC — контекст `tz` пользователя обязателен в каждом RPC, иначе границы периодов «поедут»; неделя стартует по `lang.week_start` |
| `selection` | Измерение | данные группируются по техническому value; подпись берётся из `fields_get()['поле']['selection']` (переводится автоматически) — маппинг на клиенте |
| `many2one` | Измерение / звено пути | группа возвращается как `(id, display_name)`; display_name читается ядром в sudo — имя видно даже без прав на комодель (поведение ядра) |
| `one2many` | Звено пути (через `any`) | не группируема напрямую |
| `many2many` | Измерение (с ограничением Л-3) | **Л-3:** группировка по m2m выполняется через JOIN relation-таблицы и дублирует строки, искажая sum/avg; модуль запрещает sum/avg-меры в одном запросе с m2m-измерением либо автоматически декомпозирует запрос через `formatted_read_grouping_sets` |
| `properties` | Измерение / фильтр | в Odoo 19 группировка и фильтрация по properties поддерживаются нативно: groupby-спека `'props_field.property_name[:granularity]'`, домены `('props_field.prop', '=', v)`; дерево полей раскрывает properties по `definition_record` |
| `json` | Не поддерживается | скрыт из дерева |
| `binary`, `image` | Не поддерживается | скрыты (хранятся в `ir.attachment`) |
| `reference`, `many2one_reference` | Атрибут | без JOIN (нет FK); отображение «модель, id» |
| computed **stored** | Как обычное поле | searchable/groupable/sortable = true автоматически |
| computed **non-stored** | Ограниченно | фильтрация — только при наличии `search=`-метода; группировка невозможна (нет SQL-выражения) — поле серое в UI с подсказкой; related non-stored — группируемы (ORM разворачивает цепочку в JOIN, начиная с 17.2) |

UI опирается не на тип, а на фактические атрибуты `searchable/sortable/groupable/aggregator` из `fields_get()` — это гарантирует, что пользователь не сможет сконфигурировать запрос, который ORM отвергнет.

## 3.4. Domain-builder (фильтры датасета и виджетов)

**Что делает.** Фильтры на трёх уровнях (датасет → контролы дашборда/страницы → виджет) задаются визуальным конструктором условий с группами И/ИЛИ/НЕ произвольной вложенности.

**Как настраивается.** Пользователь добавляет условия: поле (через то же дерево полей) → оператор → значение. Поддерживаются предпросмотр количества записей («Найдено: 12 483») и режим разработчика (редактирование домена текстом).

**Реализация в Odoo 19.**

- UI: переиспользуем штатные `DomainSelector` / `TreeEditor` (`@web/core/domain_selector`, `@web/core/tree_editor`) — те же компоненты, что в «Add Custom Filter» и Studio; поддерживают вложенные ветки, relational-поля и properties.
- Хранение: домен — текст в формате Odoo (польская нотация), парсинг строго `ast.literal_eval` (как `ir.filters._get_eval_domain()`); запрещён eval.
- Доступные операторы: `=, !=, >, >=, <, <=, in, not in, like, ilike, not ilike, child_of, parent_of, any, not any` (для relational-полей `any`-домены дают фильтры вида «есть хотя бы одна строка заказа с условием...»).
- Динамические значения дат Odoo 19: строковые выражения `'today'`, `'now'`, относительные `'-3d +1H'`, якоря `'=monday -1w'` — конструктор предоставляет их как пресеты («сегодня», «начало недели», «N дней назад»), что делает сохранённые фильтры «вечно актуальными» без пересохранения.
- Доменные части дат: `('date_field.month_number', '=', 2)` — фильтры «только февраль», «только понедельники» без вычисляемых полей.
- Предпросмотр: `search_count(domain, limit=10000)` (limit как верхняя граница — не сканирует всю таблицу).
- Склейка уровней: только через `Domain.AND([...])` (Python) / `Domain.and(...)` (JS). Прямая конкатенация списков запрещена (известная атака `['|', ('id', '>', 0)]`, нейтрализующая последующее условие).

## 3.5. Агрегации (GROUP BY / HAVING)

**Что делает.** Виджет задаёт измерения (по чему группируем) и меры (что считаем). Дополнительно: HAVING-фильтр по агрегатам («показать клиентов с суммой > 10 000»), сортировка по любому измерению/мере, Top-N с группой «Прочие», лимит/смещение.

**Реализация в Odoo 19** — прямое использование публичного RPC-API:

```python
model.formatted_read_group(
    domain=effective_domain,
    groupby=['partner_id.country_id', 'date_order:month'],
    aggregates=['amount_total:sum', '__count', 'margin:avg'],
    having=[('amount_total:sum', '>', 10000)],
    order='amount_total:sum desc',
    limit=50, offset=0,
)
```

- Агрегатные функции: `sum, avg, min, max, count, count_distinct, bool_and, bool_or, array_agg, array_agg_distinct` + спец-агрегат `sum_currency` для monetary + `__count` (COUNT(*)).
- `having` — домен над агрегат-спеками (операторы `=, !=, <, >, <=, >=, in, not in`) — транслируется в SQL HAVING.
- Ответ содержит на каждую группу `__extra_domain` — готовый домен принадлежности к группе (для дат — корректный диапазон в UTC с учётом tz). Модуль использует его для drill-down, cross-filtering и drill-through без самостоятельного «реверс-инжиниринга» границ групп.
- Top-N + «Прочие»: основной запрос с `limit=N, order=мера desc` + второй запрос без группировки (итог) — «Прочие» = итог минус сумма топа; оба запроса объединяются в `formatted_read_grouping_sets` (один SQL).
- Пустые периоды временных рядов: контекст `fill_temporal=True` — ORM сам достраивает нулевые месяцы/недели.

## 3.6. Мультивалютность и мультикомпания в датасете

**Проблема.** SUM по `monetary`-полю записей в разных валютах математически некорректен.

**Поведение модуля.** Если в выборку попадает monetary-мера и среди записей более одной валюты (`count_distinct` по `currency_field`), применяется одна из стратегий (настройка меры):

1. **Конвертация в валюту компании по текущему курсу** — агрегат `sum_currency` (нативный в `_read_group` Odoo 19: `SUM(amount/rate)` с JOIN последнего курса). Быстро (один SQL), но курс «на сегодня». Стратегия по умолчанию; ограничение явно показано в подсказке виджета.
2. **Историческая конвертация по курсу даты документа** — для датасетов на моделях, где есть поле курса/даты: группировка дополнительно по `currency_id`, конвертация групп на сервере через `res.currency._convert(amount, to_currency, company, date)`. Точнее, дороже; паттерн штатных SQL-отчётов (`sale.report` использует currency table `res.currency._get_simple_currency_table()`).
3. **Группировка по валюте** — без конвертации, валюта становится измерением (легенда/колонки).

**Мультикомпания.** Запросы исполняются в контексте `allowed_company_ids` пользователя — record rules компаний отрабатывают автоматически. На дашборде доступен штатный переключатель компаний (виджет читает `env.companies`); дашборд может быть ограничен `company_ids` (виден только в этих компаниях). Целевая валюта конвертации — валюта `env.company`.

## 3.7. Объединение нескольких моделей (blend, аналог blended data)

**Что делает.** Когда измерения и меры нельзя получить одной моделью даже через цепочки (например, «выручка из продаж + затраты из закупок по месяцам и товарам»), пользователь строит **бленд**: до 5 таблиц-источников, соединённых JOIN.

**Как настраивается** (UI повторяет блендер Looker Studio, лучшие практики которого подтверждены документацией Google):

- Каждая таблица бленда = существующий датасет (или модель) + выбранные измерения/меры + собственный локальный фильтр + (для временных таблиц) своё поле даты.
- Соединения настраиваются между соседними таблицами: тип — `left outer / inner` (первая поставка; `right outer / full outer / cross` — этап 3, решение ОВ-6); условия — пары полей «слева = справа» (только равенство, одна или несколько пар); имена полей могут различаться.
- Итоговый набор полей бленда: выбор включаемых полей, скрытие повторяющихся ключей.
- Семантика «предагрегации до соединения» (как в Looker Studio): каждая таблица сначала группируется по своим измерениям, затем соединяется. UI предупреждает: «добавьте уникальный ключ в измерения таблицы, иначе строки схлопнутся до соединения».

**Реализация в Odoo 19.** Генератор одного SQL-запроса по схеме CTE:

```sql
WITH t1 AS (SELECT dim_a, dim_b, SUM(m1) AS m1 FROM sale_order_line
            WHERE <домен t1 + домен ir.rule пользователя> GROUP BY dim_a, dim_b),
     t2 AS (SELECT dim_a, SUM(m2) AS m2 FROM purchase_order_line
            WHERE <домен t2 + домен ir.rule пользователя> GROUP BY dim_a)
SELECT ... FROM t1 LEFT JOIN t2 ON t1.dim_a = t2.dim_a
GROUP BY <измерения виджета>;
```

- Построение SQL — исключительно через `odoo.tools.SQL` / `SQL.identifier` (никакой конкатенации строк), исполнение `env.execute_query(SQL(...))` с предварительным `flush_model()`.
- **Права в SQL** — критический пункт: WHERE каждого CTE включает домен record rules пользователя для соответствующей модели. Способ получения — раздел 8.4 (выбранный вариант: компиляция домена `ir.rule` + пользовательского домена через `Model._search(...)` в объект Query без исполнения, со встраиванием как подзапроса; запасной вариант — `WHERE id IN (search(domain).ids)`).
- Меры таблиц внутри бленда становятся числовыми колонками, доступными повторной агрегации на уровне виджета (паттерн reaggregation Looker Studio: «средний дневной чек» = AVG по дневным SUM).
- Бленд из одной модели (self-blend) поддерживается — это легальный способ «фиксированных мер» (мера со своим фильтром, не реагирующая на контролы).
- Ограничения (как у аналога, зафиксировать в UI): максимум 5 таблиц; вычисляемые поля уровня бленда не поддерживаются — только вычисляемые поля уровня виджета поверх результата; условия JOIN — только равенство.
- Лимит на размер промежуточных CTE (по умолчанию 100 000 строк на таблицу, параметр) + `statement_timeout` на курсор (раздел 10.6).

## 3.8. SQL-датасеты (для администраторов)

Для случаев, не покрываемых конструктором (оконные функции, рекурсивные CTE, внешние FDW-таблицы), администратор (`td_bi_dashboard.group_bi_admin`) создаёт датасет с режимом `sql`: произвольный SELECT, объявленные поля результата (имя, тип, роль). Запрос исполняется только от имени технического контекста с явным предупреждением: **record rules на такой датасет не действуют**, ответственность на авторе; доступ к таким датасетам по умолчанию закрыт группами. Альтернатива для разработчиков — зарегистрировать в системе модель с `_auto=False` и `_table_query` (паттерн `sale.report`) и использовать её как обычную модель-источник (тогда права применяются штатно через ACL/правила этой модели) — рекомендуемый путь, упоминается в документации модуля.

## 3.9. Датасет как переиспользуемая сущность

- Датасеты бывают **личные** (видит автор), **командные** (по `group_ids`) и **глобальные**.
- На уровне датасета фиксируются: переименования и описания полей, скрытие полей, агрегатор по умолчанию, формат отображения (валюта/процент/единицы), вычисляемые поля, базовый домен, политика кэша (TTL, раздел 10.3), стратегия мультивалютности.
- Дашборд может использовать несколько датасетов; виджет ссылается ровно на один датасет (или бленд).
- Изменение датасета версионируется (chatter + `td.bi.dashboard.version` затрагиваемых дашбордов помечаются «датасет изменился»); удаление поля, используемого виджетами, блокируется проверкой ссылочной целостности с перечнем затронутых виджетов.
# 4. Движок вычислений

Движок вычислений воспроизводит практически значимое ядро DAX (Power BI) и формульного языка Looker Studio, не копируя их синтаксис буквально: формулы компилируются в SQL-выражения PostgreSQL или вычисляются на сервере над готовыми агрегатами. Пользовательский текст формул никогда не исполняется интерпретатором Python.

## 4.1. Классификация вычислений

| Вид | Аналог | Когда вычисляется | Где задаётся |
|---|---|---|---|
| Вычисляемое поле (строчное) | calculated column (PBI), calculated field (LS) | На уровне строки, до агрегации; компилируется в SQL-выражение | Датасет (доступно всем виджетам) или виджет (локально) |
| Простая мера | implicit measure | Агрегатная функция над полем | Виджет (поле + агрегатор) |
| Формульная мера | measure / DAX | Формула над агрегатами, после агрегации | Датасет (`td.bi.measure`) или виджет |
| Фильтрованная мера | CALCULATE(..., FILTER) | Агрегат с собственным дополнительным доменом | `td.bi.measure.extra_domain` |
| Временной интеллект | time intelligence | Доп. ряд со сдвинутым доменом дат + дельты | Настройка меры или виджета |
| Быстрая мера | quick measure | Параметризованный шаблон из библиотеки | Мастер «быстрая мера» |
| Вычисление уровня визуала | visual calculation (PBI) | Над итоговой таблицей результата (running total, % of total, rank) | Виджет, без формул |

## 4.2. Формульный язык (DSL)

**Синтаксис.** Инфиксные выражения с функциями; идентификаторы — псевдонимы полей датасета в квадратных скобках: `[выручка]`, `[qty]`. Строки в одинарных кавычках, даты — `DATE '2026-01-01'`. Регистронезависимые имена функций.

**Примеры.**

```text
([price_unit] * [qty]) * (1 - [discount] / 100)            -- строчное поле
CASE WHEN [state] = 'sale' THEN [amount] ELSE 0 END        -- строчное поле
SUM([margin]) / NULLIF(SUM([amount_total]), 0)             -- формульная мера (ratio)
COUNT_DISTINCT([partner_id])                                -- мера
DATETIME_DIFF([date_done], [date_order], 'day')            -- строчное поле
REGEXP_EXTRACT([default_code], '^[A-Z]+')                   -- строчное поле
```

**Каталог функций (белый список, этап 1 → этап 2).**

| Группа | Функции |
|---|---|
| Агрегатные | SUM, AVG, MIN, MAX, COUNT, COUNT_DISTINCT, MEDIAN (percentile_cont 0.5), PERCENTILE, STDDEV, VARIANCE |
| Арифметика | + - * /, ABS, ROUND, CEIL, FLOOR, POWER, SQRT, LOG, MOD, NARY_MAX, NARY_MIN |
| Логика | CASE WHEN simple/searched, IF, IFNULL, NULLIF, COALESCE, AND, OR, NOT, IN |
| Текст | CONCAT, LOWER, UPPER, TRIM, LENGTH, LEFT, RIGHT, SUBSTR, REPLACE, CONTAINS, STARTS_WITH, ENDS_WITH, REGEXP_MATCH, REGEXP_EXTRACT, REGEXP_REPLACE (синтаксис POSIX PostgreSQL) |
| Дата/время | TODAY, NOW, YEAR, QUARTER, MONTH, WEEK, DAY, HOUR, WEEKDAY, DATE_TRUNC, DATETIME_ADD, DATETIME_SUB, DATETIME_DIFF, EXTRACT, FORMAT_DATE, PARSE_DATE |
| Преобразование | CAST (number/text/date/datetime), HYPERLINK(url, label) и IMAGE(url) — только для табличных виджетов |
| Параметры | подстановка `@имя_параметра` (раздел 4.6) |

**Компиляция и безопасность** (выбранный вариант + обоснование + альтернатива):

- **Выбрано:** собственный парсер на `ast.parse(..., mode='eval')` Python-совместимой грамматики с жёстким whitelist узлов AST (`BinOp, UnaryOp, BoolOp, Compare, Call, Name/идентификатор поля, Constant, IfExp`) и таблицей трансляции функций в SQL (имена колонок — только через `SQL.identifier` по списку реальных полей датасета; значения — bind-параметры). Строчные формулы превращаются в SQL-выражения внутри агрегатов (`SUM(price_unit * qty * (1 - discount/100))`); формульные меры декомпозируются на компонентные агрегаты, исполняемые одним `formatted_read_group`/SQL, затем выражение вычисляется на сервере над числами (безопасный численный evaluator без доступа к окружению).
- Обоснование: `odoo.tools.safe_eval` официально «ломает барьер между кодом и данными» и допустим только для доверенных привилегированных пользователей (так его использует ядро в `ir.rule.domain_force`); для формул, доступных широкой группе BI-редакторов, он неприемлем (ресурсное исчерпание, исторические обходы). Прецедент собственного мини-интерпретатора в ядре — клиентский `py_js` (`evaluateExpr`).
- Альтернатива (отклонена): `safe_eval` с ограниченным контекстом — оставлен только для админ-уровня (SQL-датасеты, раздел 3.8) по аналогии с `ir.rule`.
- Валидации при сохранении формулы: существование полей; запрет смешения агрегированных и неагрегированных операндов в одном выражении (правило Looker Studio: `SUM([a])/[b]` — ошибка, `SUM([a])/SUM([b])` — корректно); запрет агрегатов внутри строчных полей; контроль типов (текстовая функция над числом — ошибка с подсказкой); лимит длины и глубины AST; циклические ссылки полей (поле A использует B, B использует A) — отказ. Вычисляемые поля датасета могут ссылаться на другие вычисляемые поля датасета (разворачиваются при компиляции); поля уровня виджета — только на поля датасета (правило Looker Studio).

## 4.3. Меры

`td.bi.measure` — именованная мера датасета, доступная всем виджетам:

- **Простая**: поле + агрегатор (`amount_total` + `sum`). Создаётся в один клик из дерева полей.
- **Формульная (ratio и др.)**: выражение над агрегатами (4.2). Исполнение: компоненты добавляются в `aggregates` запроса (один проход СУБД), выражение считается на сервере по каждой группе. Деление на ноль → NULL (NULLIF навязывается автоматически с предупреждением).
- **Фильтрованная** (аналог `CALCULATE(SUM(x), фильтр)`): мера с собственным `extra_domain`, который складывается с доменом виджета по И — «выручка только по подтверждённым», «новые клиенты». Реализация: отдельная агрегат-колонка через SQL `FILTER (WHERE ...)`: на уровне `formatted_read_group` это второй запрос в `formatted_read_grouping_sets` с доменом `domain ∧ extra_domain`; в blend/SQL-датасетах — нативный `agg(...) FILTER (WHERE ...)`.
- **«% от итога»** (аналог `ALL()`): настройка меры «показать как % от: итога / итога по измерению X». Реализация: `formatted_read_grouping_sets(domain, [[dim1, dim2], [dim1], []], aggregates)` — детальные группы, промежуточные и общий итог одним SQL (GROUPING SETS), деление на сервере.
- Форматирование меры: тип (число/валюта/процент/длительность), точность, компактная запись (12,4 тыс.), префикс/суффикс.

## 4.4. Временной интеллект

Настраивается на мере или виджете при наличии измерения-даты (поле даты виджета — явный выбор, как date range dimension в Looker Studio; решает проблему нескольких дат в модели — аналог role-playing dimensions Power BI без дублирования таблиц).

| Расчёт | Аналог | Реализация |
|---|---|---|
| YTD / QTD / MTD / WTD | TOTALYTD и пр. | домен `[начало периода, конец выбранного диапазона]`, генерируемый сервером из границы диапазона контрола |
| Прошлый период (PoP) | DATEADD/PARALLELPERIOD | второй ряд: тот же запрос с доменом дат, сдвинутым на длину периода; рендер «призрачной» серией + дельта |
| Тот же период прошлого года (YoY) | SAMEPERIODLASTYEAR | сдвиг домена на 1 год; дельта абсолютная и % |
| Скользящее окно N периодов | DATESINPERIOD / Rolling average | оконная функция `AVG(...) OVER (ORDER BY период ROWS N-1 PRECEDING)` над агрегированным рядом (пост-обработка на сервере — ряд уже мал) |
| Накопительный итог (running total) | Running total / cumulated | `SUM(...) OVER (ORDER BY период)` над агрегированным рядом; для graph-подобных виджетов — флаг (аналог `cumulated` штатного graph view 19) |
| Период-к-дате на прошлый год (YTD PY) | комбинация | композиция правил 1 и 3 |

Сравнительный рендер единообразен для всех виджетов: KPI-карточка — значение + дельта (стрелка/цвет, % и абсолют); временной ряд — вторая серия пунктиром; таблица/pivot — доп. колонки «пред. период», «Δ», «Δ%». Оба ряда (текущий и сдвинутый) исполняются одним `formatted_read_grouping_sets`, либо двумя параллельными запросами при разных доменах — решает оптимизатор модуля.

Календарные нюансы: неделя — по `lang.week_start`; «неполные периоды» — опция «сравнивать только завершённые дни» (усечение обоих доменов по min(длина периодов)); финансовый год — смещение начала года (настройка датасета, по умолчанию из `res.company.fiscalyear_last_month` при наличии бухгалтерии). Нестандартные календари (4-4-5, ISO-недельный год) — этап 3 (решение ОВ-10).

## 4.5. Библиотека быстрых мер (quick measures)

Мастер «Быстрая мера»: пользователь выбирает шаблон, поля-параметры — формула и настройки создаются автоматически (по образцу quick measures Power BI; сгенерированная конфигурация видна и редактируема — обучающий эффект):

1. Доля от общего итога (% of grand total).
2. Доля внутри группы (% of parent/измерения).
3. Год-к-году, % (YoY growth).
4. Период-к-периоду, % (PoP growth).
5. Накопительный итог (running total).
6. Скользящее среднее N периодов (rolling average).
7. Среднее на категорию (average per category).
8. Ранг по мере (rank по убыванию, плотный/неплотный).
9. Разница от выбранного значения (difference from filtered value) — с фильтрованной мерой.
10. Счётчик новых сущностей (new customers: count_distinct с фильтром «первая покупка в периоде» — двухшаговый шаблон, этап 2).

## 4.6. Параметры

`td.bi.parameter` — именованная переменная (аналог parameters Looker Studio / what-if Power BI):

- Типы: text, number (c min/max/step), boolean, selection (список «значение + подпись»), date.
- Объявляется на датасете (доступен формулам и доменам датасета) или на дашборде (доступен контролам и виджетам).
- Использование: в формулах (`[price] * @ставка_ндс`), в доменах (`('user_id', '=', @менеджер)`), в заголовках виджетов (шаблонизация `{{параметр}}`).
- Значение задаётся: значением по умолчанию → контролом-инпутом на дашборде → переопределением на странице/виджете (наследование report → page → group → widget, побеждает нижний уровень — модель Looker Studio) → параметром URL действия дашборда (query string `?bi_params=` с URL-encoded JSON; белый список разрешённых к изменению параметров — настройка дашборда).
- Подстановка в SQL — только bind-параметрами; в домены — значением после валидации типа.

## 4.7. Вычисления уровня визуала

Без формул, переключателями на мере виджета (аналог visual calculations Power BI): % от итога, накопительный итог, ранг, разница с предыдущей строкой/периодом, % разницы. Вычисляются сервером над готовым результатом (ряды малы) — не порождают повторных запросов к данным.

## 5.3. Конфигурация виджета (общая схема)

Каждый виджет настраивается в трёх вкладках инспектора (правая панель конструктора):

**Данные:** датасет; измерения (1–2, для pivot — до 2×2); разбивка серий (breakdown, аналог Legend/Breakdown dimension); меры (1–N, для combo — раздельно для столбцов/линий с привязкой к осям); поле даты виджета; собственный фильтр виджета (DomainSelector); сортировка (по измерению или мере, asc/desc, вторичная); лимит / Top-N + «Прочие»; HAVING; сравнение периодов (выкл / пред. период / прошлый год / произвольный сдвиг); вычисления уровня визуала (4.7).

**Стиль:** заголовок и подзаголовок (шаблонизация параметров); легенда (позиция/выкл); оси (min/max, логарифмическая шкала, формат подписей, заголовки осей); подписи данных (вкл/значение/%); палитра (из темы / ручная по сериям / по значениям измерения — единая карта цветов значений на дашборд, как dimension value colors Looker Studio); условное форматирование (5.4); реперные линии (константа / среднее / медиана / процентиль / цель из параметра); формат чисел (компактный, точность, валюта по `currency_field`); пустое состояние (текст при отсутствии данных).

**Взаимодействия:** участвует ли в cross-filtering (источник/приёмник, раздел 6.4); поведение клика (drill-down / drill-through к записям / открыть URL / ничего); иерархия drill-down (упорядоченный список измерений, например Год → Квартал → Месяц → Категория); видимость по условию (скрыть виджет, если параметр = значение — для «переключаемых видов»).

## 5.4. Условное форматирование

Декларативные правила в конфиге виджета (массив, порядок = приоритет):

- Типы правил: «значение в диапазоне → цвет» (rules), «градиент от min до max» (color scale, 2–3 опорных цвета), «иконка по порогу» (▲▼●), «data bars» в ячейках таблиц.
- Область применения: ячейка меры, вся строка (таблица), сегмент/столбец (графики), фон/шрифт KPI-карточки.
- Источник порога: константа, параметр, другая мера (например, «красным, если факт < плана»).

Реализация: для Chart.js — колбэки `backgroundColor`/scriptable options, генерируемые из правил; для таблиц/pivot — классы OWL-шаблона. Правила сериализуются в JSON конфига виджета (формат — раздел 11.7).

## 5.5. Табличные виджеты — детали

**Таблица.** Колонки: поля-атрибуты + меры (при группировке) либо «плоский» режим записей (web_search_read, постранично по 80, виртуальный скролл); итоговая строка (sum/avg по настройке); перенос текста, ширины колонок, закреплённый заголовок; сортировка кликом (включая мультиколоночную Shift+клик); ссылки: m2o-поля кликабельны → form view записи (право `formview_ref` уважается), HYPERLINK/IMAGE-поля рендерятся; экспорт XLSX/CSV из меню виджета (право `base.group_allow_export`).

**Pivot.** До 2 измерений в строках и 2 в колонках (включая даты с гранулярностью), N мер; раскрытие/схлопывание уровней (lazy-подгрузка ветки через `web_read_group` с `unfold_read_default_limit`); подытоги уровней и общий итог (GROUPING SETS — один SQL); транспонирование; «значения в строках»; conditional formatting per мера; экспорт XLSX с сохранением структуры (паттерн штатного pivot: контроллер собирает workbook xlsxwriter из JSON-описания).

## 5.6. Гео-визуализации

- Уровни: страна (ISO-код из `res.country.code`), регион/штат (`res.country.state`), произвольный TopoJSON (загрузка файла администратором — кастомные регионы, план склада).
- Привязка данных: измерение-путь к `res.country`/`res.country.state` ИЛИ текстовое поле с гео-семантикой (настройка поля датасета «гео-роль»), как data semantics Looker Studio.
- Мера → цвет (хороплет, шкала с легендой) или размер пузыря (bubble, по центроидам).
- Реализация: `chartjs-chart-geo` (vendored), TopoJSON-карты мира/стран поставляются с модулем; проекции настройки не требуют (выбор из 3 пресетов).

## 5.7. Поведение всех виджетов

- Заголовок-тулбар: меню (обновить, развернуть на весь экран, экспорт данных XLSX/CSV, экспорт PNG, «показать как таблицу», информация об источнике: датасет, домен, время выполнения, кэш-статус).
- Tooltip: значения точки + дополнительные «поля для tooltip» (атрибуты без отображения на графике).
- Лоадер-скелетон при загрузке; карточка ошибки при таймауте/отказе прав с кнопкой «повторить» (П-6).
- Тёмная тема: палитры виджетов реагируют на `color_scheme` (как штатный GraphRenderer).
- Доступность: aria-подписи, навигация по таблицам с клавиатуры; цветовые палитры по умолчанию color-blind-safe.
# 6. Интерактивность

## 6.1. Уровни фильтрации

Эффективный домен виджета — конъюнкция уровней (аналог visual/page/report-фильтров Power BI):

| Уровень | Кто задаёт | Хранение | Видимость для зрителя |
|---|---|---|---|
| Домен датасета | автор датасета | `td.bi.dataset.domain` | нет (всегда активен) |
| Домен аудитории | владелец дашборда (раздел 8.3) | `td.bi.dashboard.access.extra_domain` | нет |
| Контролы дашборда | автор; значения — зритель | `td.bi.control` (scope=dashboard) | да |
| Контролы страницы/группы | автор; значения — зритель | `td.bi.control` (scope=page/group) | да |
| Cross-filter | зритель кликом | состояние страницы (не персистится в БД) | да, в панели активных фильтров |
| Домен виджета | автор виджета | конфиг виджета | опционально (иконка фильтра на виджете) |
| Record rules | администратор Odoo | `ir.rule` | нет (применяет сервер) |

У каждого контрола флаги `is_locked` (зритель видит, но не меняет) и `is_hidden` (скрыт от зрителя) — аналог Lock/Hide панели фильтров Power BI. Состояние значений контролов персистится per-пользователь (последнее состояние восстанавливается при открытии; кнопка «Сбросить к умолчаниям» возвращает авторские значения) — хранение в `td.bi.dashboard.state` с флагом `is_autosave`.

## 6.3. Область действия контролов и сопоставление полей

- Scope контрола: весь дашборд / страница / группа виджетов (группа создаётся выделением виджетов — аналог группировки Looker Studio «контрол действует только на группу»).
- **Сопоставление полей (field mapping).** Контрол хранит для каждого датасета страницы путь поля, на который он действует: контрол «Клиент» → `partner_id` в датасете продаж, `partner_id.commercial_partner_id` в датасете обращений. Автосопоставление по совпадению технического имени и типа; ручное переопределение в настройках контрола; датасет можно исключить («не действует»). Паттерн повторяет fieldMatching глобальных фильтров spreadsheet-дашбордов Odoo и «field ID matching» Looker Studio.
- Каскадность: списки значений зависимых контролов пересчитываются с учётом уже применённых доменов прочих контролов (опция «каскадные значения», по умолчанию вкл).

## 6.4. Cross-filtering

- Клик по элементу виджета (столбец, сегмент, строка таблицы, ячейка pivot, страна на карте) публикует в шину страницы фильтр-фрагмент: `{dataset_field_path, domain}` — домен берётся из `__extra_domain` группы, возвращённого сервером (корректен для дат/m2o/selection без клиентских вычислений).
- Остальные виджеты страницы, у которых включён приём, перезапрашиваются с этим доменом (сопоставление путей — через mapping 6.3, кросс-датасетно).
- Повторный клик/Esc снимает фильтр; активные cross-фильтры показываются чипами в панели фильтров страницы с возможностью снять каждый; мультивыбор — Ctrl+клик (домены объединяются по ИЛИ внутри одного поля, по И между полями).
- Виджет-источник подсвечивает выбранный элемент (выделение цветом). Режим cross-highlighting Power BI (показ доли без отфильтровывания) сознательно не реализуется (двойные запросы всех виджетов; cross-filtering покрывает сценарий) — помечено как возможное расширение этапа 3.
- **Матрица взаимодействий** (аналог Edit interactions Power BI): в режиме конструктора при выделении виджета-источника на остальных виджетах появляются переключатели «фильтровать / игнорировать»; матрица `источник × приёмник` хранится в JSON дашборда. По умолчанию: все фильтруют всех в пределах страницы.
- Brush-выделение диапазона на временных рядах (drag по оси X → фильтр диапазона дат) — этап 2.

## 6.5. Drill-down и drill-through

**Drill-down (в виджете).**

- Автор задаёт иерархию измерений виджета (например: Страна → Город; Год → Квартал → Месяц; Категория → Подкатегория → Товар). Для дат иерархия создаётся автоматически из гранулярностей.
- Жесты: клик по элементу → спуск в элемент (домен элемента добавляется, измерение заменяется следующим); кнопки в тулбаре виджета: «вверх», «следующий уровень целиком», «развернуть всё» (двухуровневый показ — для pivot).
- Реализация: стек уровней в состоянии виджета: `[{groupby, domain_фрагмент}]`; спуск = push(`__extra_domain` выбранной группы) + смена `groupby`; подъём = pop. Хлебные крошки текущего пути отображаются в заголовке виджета.
- Drill-down может транслироваться в cross-filter (опция «drill фильтрует другие виджеты», аналог Cross-drill Power BI).

**Drill-through (к деталям).**

- «Показать записи»: контекстное меню точки данных → стандартный list view Odoo с доменом = эффективный домен виджета ∧ `__extra_domain` точки (паттерн штатного graph view: `action service` + `ir.actions.act_window`). Из list view доступны form view, экспорт и пр. — права Odoo действуют как обычно.
- «Страница детализации»: автор помечает страницу дашборда как drillthrough-страницу и указывает входные поля; контекстное меню точки → переход на страницу с применёнными фильтрами по значениям точки + опция «переносить все фильтры исходной страницы» (аналог Keep all filters Power BI); автоматическая кнопка «Назад».
- Транзит между дашбордами (cross-dashboard drill-through) — этап 3.

## 6.6. Закладки и персональные состояния

- `td.bi.dashboard.state` — снимок состояния: значения контролов, активные cross-фильтры, drill-стеки, текущая страница, скрытые виджеты, выбранная тема. Виды: **авторские закладки** (видны всем зрителям; используются пресет-кнопками и навигацией «слайд-шоу») и **личные закладки** зрителя (включая «состояние по умолчанию для меня»).
- Применение закладки настраивается: данные (фильтры) / отображение (видимость) / всё (как Data/Display bookmarks Power BI).
- Персонализация виджета зрителем (аналог Personalize visuals): зритель меняет тип графика/меру/измерение «для себя»; overrides хранятся в личной закладке, авторская конфигурация не изменяется; сброс одним действием. Этап 2.

## 6.7. Состояние страницы и URL

Текущее состояние (страница, контролы, drill) сериализуется в hash-часть URL действия дашборда — ссылка из браузера воспроизводит вид; это же используется механизмом подписок («отправить с фильтрами расписания») и embed-параметрами. Белый список полей состояния, допустимых из URL, контролируется сервером (защита от подмены доменов: из URL принимаются только значения контролов и параметров, но не произвольные домены).
# 7. Конструктор дашбордов

## 7.1. Сетка и layout

- **Grid-layout: 24 колонки**, высота ряда фиксированная (настройка темы, по умолчанию 40 px), виджет занимает прямоугольник `(x, y, w, h)` в единицах сетки. 24 колонки дают совместимость с раскладками «3 × 8», «4 × 6», «2 × 12».
- Drag-and-drop перемещение и resize за маркеры углов/граней; снэп к сетке; авто-вытеснение (push) перекрываемых виджетов вниз; направляющие выравнивания.
- Реализация: собственный движок на CSS Grid + хук `makeDraggableHook`/`useSortable` из `@web/core/utils` (готового grid-движка в ядре нет — подтверждено исследованием; board использует колонки, spreadsheet — канву ячеек). Выбор CSS Grid + собственный контроллер против подключения сторонней библиотеки (gridstack.js, MIT) — решение в пользу собственного: меньше зависимость, полная интеграция с OWL-реактивностью; gridstack.js — запасной вариант, если собственный движок не уложится в смету этапа 1 (интерфейсы слоя layout проектируются так, чтобы замена не затронула модели).
- Страницы (вкладки) дашборда: добавление/переименование/дублирование/скрытие/порядок; навигация — вкладки сверху или скрытая (переходы кнопками).
- Responsive: брейкпоинты xl/lg/md/sm; ниже md (планшет/телефон) сетка схлопывается в одну колонку в порядке (y, x); опциональная ручная мобильная раскладка (отдельный набор координат) — этап 3. Высота виджетов на мобильном — из аспект-гайда типа виджета.

## 7.2. Режимы и работа конструктора

- Режимы «Просмотр» / «Редактирование» (кнопка — только при праве editor, раздел 8.2). Редактирование блокирует дашборд мягкой блокировкой (предупреждение о параллельном редакторе; этап 2 — без real-time-coedit).
- Палитра виджетов: drag из палитры на канву → виджет с настройками по умолчанию и автодатасетом страницы; двойной клик — добавить в первое свободное место.
- Буфер: копировать/вырезать/вставить виджет (в т. ч. между страницами и дашбордами — с переносом ссылки на датасет), дублировать (Ctrl+D).
- Undo/redo (Ctrl+Z/Ctrl+Shift+Z): команда-стек конфигурационных операций на клиенте; сохранение на сервер — явное («Сохранить») + автосохранение черновика каждые 60 с.
- Сетка-помощник, линейки, режим предпросмотра «как зритель» (включая выбор пользователя для проверки прав — раздел 8.6).
- Шаблоны дашбордов: сохранить дашборд как шаблон (без данных, со схемой требуемых полей); создать из шаблона с маппингом датасетов; поставляемые примеры (Продажи, Финансы, Склад, HR) — этап 2.

## 7.3. Темы и брендинг

- `td.bi.theme`: JSON-конфигурация — палитра данных (8+ цветов), цвета фона страницы/карточек, семейство и размеры шрифтов (заголовок виджета, подписи, KPI), радиусы и тени карточек, высота ряда сетки, дефолты per-тип-виджета (например, у всех line-виджетов — сглаживание; аналог visualStyles JSON-тем Power BI).
- Темы: системные (светлая/тёмная), корпоративные (создаёт администратор); тема по умолчанию в поставке — корпоративная тема ToDo (палитра, шрифты, логотип по брендбуку заказчика — решение ОВ-8); на дашборд — выбор темы + точечные переопределения. Прямое форматирование виджета сильнее темы (как в Power BI).
- Логотип/шапка дашборда (для экспорта и публичного вида).
- «Тема из логотипа» (извлечение палитры из изображения, аналог Looker Studio) — этап 3, опционально.

# 8. Безопасность

## 8.1. Группы доступа модуля

| Группа | XML id | Возможности |
|---|---|---|
| Пользователь BI | `td_bi_dashboard.group_bi_user` | просмотр опубликованных дашбордов, доступных по правам; личные закладки; экспорт при наличии `base.group_allow_export` |
| Дизайнер BI | `td_bi_dashboard.group_bi_designer` (implied: user) | создание/редактирование своих датасетов и дашбордов, формулы DSL, бленды; шаринг своих объектов |
| Администратор BI | `td_bi_dashboard.group_bi_admin` (implied: designer) | все объекты модуля; SQL-датасеты; публичные ссылки и embed; темы; глобальные датасеты; настройки модуля; просмотр аудита |

Схема повторяет проверенный паттерн `spreadsheet_dashboard` (группа менеджера + правила по `group_ids`/`company_ids`).

## 8.2. Права на объекты модуля (ACL + record rules)

- `td.bi.dashboard`: владелец (`owner_id`) + строки доступа `td.bi.dashboard.access` (пользователь или группа; роль `viewer/editor/manager`). Record rules: видимы дашборды, где пользователь — владелец, есть строка доступа, или дашборд опубликован «для всех сотрудников» (`group_ids` пуст → `base.group_user`); мультикомпанийное правило по `company_ids`. Manager может менять доступы и публиковать; editor — менять содержимое; viewer — только смотреть.
- `td.bi.dataset`: аналогично (личные/командные/глобальные); использование датасета в виджете требует права чтения датасета; **данные** при этом всё равно фильтруются правами зрителя, а не автора (см. 8.3).
- Прочие `bi.*`-модели наследуют доступ от родительского дашборда/датасета (ir.rule по связи).
- Полная матрица ACL — раздел 11.6.

## 8.3. Доступ к данным: наследование ACL и record rules Odoo (RLS)

**Главный принцип (П-1, П-5).** Запрос данных виджета исполняется на сервере от имени **текущего пользователя**: `formatted_read_group` сам вызывает `check_access('read')` (ACL) и внедряет домены `ir.rule` в WHERE и в подзапросы по dot-path/`any` (включая правила комоделей). Поэтому:

- продавец на дашборде «Продажи» видит агрегаты только своих заказов — таково его record rule на `sale.order`;
- пользователь без права чтения `hr.employee` получает на HR-виджете карточку «нет доступа» (AccessError перехватывается per-виджет), остальные виджеты работают;
- поля, закрытые `groups=`, не видны в конструкторе и недоступны в запросах (валидатор `query_spec` сверяет каждый путь с `fields_get()` пользователя; запрос с запрещённым полем отклоняется сервером, а не интерфейсом).

**Дополнительный слой — домен аудитории** (аналог ролей RLS Power BI, но только в сторону сужения): строка `td.bi.dashboard.access` может содержать `extra_domain` per-датасет (JSON: dataset → домен). Пример: дашборд один, аудитория «Менеджеры филиалов» получает `('team_id', '=', user.sale_team_id.id)` — динамическое правило через переменные `user` в пресетах конструктора (компилируются в домен на сервере при выполнении, по образцу eval-контекста `ir.rule`: доступны только `user`, `company_id`, `company_ids`). Сужение не может расширить видимость сверх record rules — оно добавляется по И.

**Чего модуль не делает принципиально:** не выполняет бизнес-запросы под `sudo()`; не кэширует данные кросс-пользовательски без включения ключа прав в ключ кэша (раздел 10.3); не доверяет доменам, пришедшим с клиента, сверх валидированных значений контролов (6.7).

## 8.4. Права в прямом SQL (бленды, SQL-датасеты)

Для blend-исполнителя домен record rules должен попасть в каждый CTE. **Выбранный вариант:** получение скомпилированного запроса через `Model._search(domain)` (возвращает объект Query с уже внедрёнными правилами; используется как подзапрос `WHERE id IN (<query>)` через `odoo.tools.SQL`). Обоснование: `_search` — стабильная точка входа, ровно так права применяет сам ORM. **Риск:** формально приватный API; митигируется интеграционными тестами на каждом релизе Odoo и инкапсуляцией в один адаптер `td.bi.query.compiler`. **Альтернатива (запасная):** двухфазно — `ids = model.search(domain).ids` → `WHERE id IN %s` (публичный API, дороже на больших выборках; включается флагом совместимости). Вариант `_where_calc` отклонён как более глубоко приватный.

## 8.5. Безопасность формул, доменов, SQL

- Формулы — только собственный AST-whitelist компилятор (4.2); идентификаторы → `SQL.identifier` по списку полей датасета; литералы → bind-параметры. `safe_eval` — нигде в пользовательском контуре.
- Домены — хранение текстом, парсинг `ast.literal_eval`, склейка `Domain.AND`; динамические даты — белый список синтаксиса ORM.
- SQL-датасеты: только `td_bi_dashboard.group_bi_admin`; SELECT-only (валидация: один statement, запрещены `;`, DML/DDL-ключевые слова вне строковых литералов — парсером, не regex); исполнение с `statement_timeout`; журналирование текста запроса в аудит.
- Параметры из URL: только значения объявленных параметров/контролов, серверная валидация типов.

## 8.6. Режим «Посмотреть как» и аудит

- «Посмотреть как пользователь/группа» (аналог View as roles Power BI): предпросмотр дашборда с `with_user(выбранный)` — доступно владельцу дашборда при наличии группы `bi_admin` (смена пользователя — чувствительная операция); для дизайнера — упрощённый режим «как аудитория X» (применение `extra_domain` без смены пользователя).
- Аудит (`td.bi.audit.log`, этап 2): просмотр дашборда, экспорт, создание публичной ссылки, изменение доступа, исполнение SQL-датасета — кто/когда/что; срок хранения — настройка (по умолчанию 180 дней). Экспортные события дополнительно журналируются по образцу `/spreadsheet/log`.

## 8.7. Публичный контур

См. раздел 9.3: токены `access_token` (uuid4) со сравнением `consteq`, только frozen-снимки (live-режим вне объёма — решение ОВ-4), ограничение «ссылка не шире прав автора» (`with_user(create_uid).has_access('read')` — паттерн `spreadsheet_dashboard.share`), опциональный срок действия и пароль ссылки, отключение ссылок при смене прав автора.
# 9. Публикация и шаринг

## 9.1. Жизненный цикл дашборда (draft / publish, версии)

- Состояния: `draft → published → archived`. Зрители видят только опубликованную версию; редакторы работают в черновике и публикуют кнопкой (модель Looker Studio). Публикация создаёт снимок конфигурации в `td.bi.dashboard.version` (полный JSON: страницы, виджеты, контролы, тема); поле `published_version_id` указывает текущую публикацию.
- История версий: список с автором/датой/комментарием; просмотр (read-only рендер версии), откат в черновик, публикация старой версии. Изменения датасетов (вне снимка) применяются сразу — это зафиксировано в документации пользователя (как в Looker Studio: правки reusable data source пробивают публикацию).
- Избранное и каталог: домашняя страница модуля — карточки дашбордов (превью-миниатюра, владелец, теги, избранное `favorite_user_ids`), поиск, группировка по тегам/разделам (`td.bi.dashboard.group` с sequence — аналог групп `spreadsheet.dashboard.group`).

## 9.2. Внутренний шаринг

- Кнопка «Поделиться»: добавление пользователей/групп с ролью viewer/editor/manager (8.2); ссылка-приглашение (внутренний URL действия); упоминание дашборда в Discuss-сообщении (chatter на `td.bi.dashboard`).
- «Опубликовать для всех сотрудников» — доступ группе `base.group_user` в роли viewer одним переключателем.
- Дашборд во вкладке приложения: пункт меню Odoo c действием дашборда (автогенерация `ir.ui.menu`+`ir.actions.client` по флагу «показывать в меню», с указанием родительского меню и групп) — дашборды встраиваются в навигацию приложений (как штатные Dashboards).

## 9.3. Публичные ссылки и embed

- **Frozen-ссылка (по умолчанию).** Создание ссылки рендерит снимок: данные всех виджетов при текущих правах и фильтрах автора фиксируются в JSON и сохраняются как attachment; анонимный зритель получает интерактивный, но не «живой» дашборд (работают переключения страниц, tooltips; не работают контролы по живым данным). Роут `/bi/share/<int:share_id>/<token>` (`auth='public'`), данные `/bi/share/data/...` — паттерн `spreadsheet_dashboard.share` (uuid-токен, `consteq`, проверка `with_user(create_uid).has_access('read')`).
- **Live-ссылки — вне объёма (решение ОВ-4).** Публичная ссылка всегда отдаёт frozen-снимок; живые данные доступны только аутентифицированным пользователям Odoo. Контур спроектирован так, что добавление live-режима в будущем не потребует миграции (data-роут уже отделён от роута страницы).
- **Embed.** Тот же контур ссылок в iframe: `/bi/embed/<id>?token=...`. Ядро Odoo не ставит запрещающие заголовки на кастомные публичные роуты (X-Frame-Options: DENY стоит только на `/web`, `/odoo`); модуль сам выставляет `Content-Security-Policy: frame-ancestors` по белому списку доменов из настроек ссылки (по умолчанию — запрещено всё, кроме self).
- Отзыв ссылки — мгновенный (удаление/деактивация записи `td.bi.dashboard.share`); ссылки автоматически деактивируются при архивации дашборда и при потере автором права чтения.

## 9.5. Подписки и рассылки

- `td.bi.subscription`: дашборд/страница, получатели (пользователи/партнёры/email), расписание (ежедневно/еженедельно с днями/ежемесячно + время; cron-выражение для админа), формат (PDF/XLSX/ссылка), **фиксированные значения фильтров на расписание** (один дашборд — разные срезы разным получателям, как filters per schedule Looker Studio), условие отправки «только если есть данные» (антиспам), активность/пауза, «отправить сейчас».

# 10. Производительность

## 10.1. Бюджеты (целевые показатели приёмки)

| Метрика | Цель |
|---|---|
| TTI дашборда 10 виджетов, модель 5 млн строк (профиль ОВ-3: 1–10 млн), тёплый кэш | ≤ 5 с (p95) |
| То же, холодный кэш (без материализации) | ≤ 20 с (p95) |
| То же, по материализованному предагрегату (10.4) | ≤ 5 с (p95) |
| Ответ одного виджета (агрегация ≤ 50 групп) | ≤ 1.5 с (p95) |

| Память браузера на дашборд | ≤ 300 МБ |
| Параллельные зрители одного дашборда | 50 без деградации > 2× |

## 10.2. Стратегия запросов

- Вся агрегация — в PostgreSQL (`GROUP BY/HAVING/GROUPING SETS/window`): клиент получает только готовые ряды (П-2).
- **Батчинг виджетов:** виджеты одной страницы группируются по (датасет, эффективный домен); совпадающие комбинации исполняются одним `formatted_read_grouping_sets` (несколько группировок одним SQL) — типовая страница «KPI-ряд + 2 графика + таблица по одному датасету» сводится к 1–2 запросам вместо 6–8.
- Ленивая загрузка: виджеты вне вьюпорта не запрашиваются до прокрутки; смена контролов дебаунсится (400 мс) и отменяет незавершённые запросы (`AbortController` + отмена RPC).
- Лимиты по умолчанию (настраиваемые): 50 групп на категориальные оси (Top-N + «Прочие»), 366 точек на временные ряды (огрубление гранулярности с подсказкой), 10 000 точек scatter, 80 строк страницы таблицы (виртуальный скролл), 200 000 строк underlying-экспорта.

## 10.3. Серверный кэш результатов

- `td.bi.cache`: ключ = SHA-256 от (датасет id+версия конфига, нормализованный домен, groupby, aggregates, having, order, limit, **uid-маркер прав**, lang, tz, company_ids); полезная нагрузка — JSON результата; TTL — политика датасета (0 = выкл; по умолчанию 10 мин; до 12 ч для тяжёлых датасетов — шкала аналогична data freshness Looker Studio).
- **uid-маркер прав**: по умолчанию `user_id` (консервативно — кэш персонален). Оптимизация (этап 2): хэш фактических доменов `ir.rule._compute_domain(model,'read')` пользователя по затронутым моделям — пользователи с одинаковыми правилами разделяют кэш.
- Инвалидация: по TTL (cron-очистка просроченных), вручную («Обновить» на виджете/дашборде — bypass cache), при изменении конфигурации датасета (bump версии). Инвалидация по записям источника не выполняется (стоимость надзора за write-хуками всех моделей несоразмерна; свежесть гарантируется TTL — задокументированное поведение, бейдж «данные на ЧЧ:ММ» на виджете).
- Клиентский слой: `orm.cache` (штатный RPC-кэш Odoo 19) для метаданных; результаты данных на клиенте не кэшируются дольше сессии страницы.
- Метаданные (`fields_get`-граф, каталог моделей) — `ormcache` на сервере (10.4 риск утечки памяти при больших графах учтён: ключ включает только группы пользователя, не uid).

## 10.4. Материализованные предагрегаты (этап 2 — решение ОВ-3, профиль 1–10 млн строк)

Для датасетов, где даже SQL-агрегация по млн строк не укладывается в бюджет: `td.bi.materialization` — определение предагрегата (датасет + фиксированные измерения/меры + гранулярность даты) → физическая таблица/materialized view (паттерн `_auto=False` + `init()`/`REFRESH MATERIALIZED VIEW CONCURRENTLY`), обновление `ir.cron` по расписанию; роутер запросов автоматически использует предагрегат, если запрос виджета «накрывается» его измерениями (иначе — живой запрос). Права: предагрегат строится **без** пользовательских правил, поэтому доступен только датасетам, у которых RLS-вариативность отсутствует или компенсируется измерением-ключом правила (например, предагрегат включает `team_id`, домен правила применяется к предагрегату) — конструктор материализации проверяет это условие и отказывает в небезопасных конфигурациях.

## 10.5. Индексы и советник

- Анализатор датасета (этап 2): по полям фильтров/группировок проверяет наличие индексов (`ir.model.fields.index`, pg_indexes) и выдаёт администратору рекомендации («поле `date_order` фильтруется в 12 виджетах — рекомендуется btree-индекс»).
- Поставляемые индексы самого модуля: по FK всех `bi.*`-моделей, по `td.bi.cache.cache_key (unique)`, `td.bi.cache.expires_at`.

## 10.6. Защитные лимиты

- `statement_timeout` на каждый BI-запрос (по умолчанию 30 с, параметр) — устанавливается на курсор перед выполнением и сбрасывается после; таймаут → карточка ошибки виджета (П-6).
- Ограничение конкурентности: не более 6 параллельных data-RPC c одного клиента (очередь на клиенте); серверный rate-limit на публичные роуты.
- Размер ответа виджета ограничен (5 МБ JSON) — защита от запроса «группировка по id на млн строк»: сервер обрезает с предупреждением.
- Мониторинг: время выполнения каждого запроса пишется в ответ (видно в «информации о виджете»); медленные запросы (> порога) — в лог с полным `query_spec` для разбора.

## 11.7. Схема `query_spec` и конфига виджета (контракт клиент-сервер)

```json
{
  "dataset_id": 12,
  "domain_parts": {
    "controls": [["date_order", ">=", "2026-01-01"]],
    "cross_filter": [["partner_id.country_id", "in", [233]]],
    "widget": []
  },
  "groupby": [
    {"path": "partner_id.country_id"},
    {"path": "date_order", "granularity": "month"}
  ],
  "measures": [
    {"key": "revenue", "field": "amount_total", "agg": "sum"},
    {"key": "orders", "agg": "__count"},
    {"key": "margin_pct", "measure_id": 7}
  ],
  "having": [["revenue", ">", 10000]],
  "order": [{"key": "revenue", "dir": "desc"}],
  "limit": 50, "offset": 0, "top_n_others": true,
  "comparison": {"mode": "prev_year", "date_field": "date_order"},
  "options": {"fill_temporal": true}
}
```

Сервер: валидирует по JSON-схеме; каждый `path` сверяет с `fields_get()` пользователя; домены — c literal-парсером; компилирует и исполняет (2.4). Ответ: `{rows: [{keys, values, extra_domains}], comparison_rows, totals, meta: {duration_ms, cache: hit/miss, currency, generated_at}}`.

Конфиг виджета (`td.bi.widget.config`) хранит секции `data` (ссылки на поля/меры по ключам, sort, limit, comparison, drill_hierarchy), `style` (заголовки осей, легенда, палитра, подписи, conditional_rules[], reference_lines[]), `interactions` (cross_source/cross_target, click_action, drill_enabled) — полная JSON-схема прилагается к модулю как `static/schema/widget_config.schema.json` и валидируется при сохранении.

# 12. Use Cases и Acceptance Criteria

Формат: основной поток + Given/When/Then. Полный реестр сценариев ведётся в трекере; здесь — ключевые, обязательные к приёмке.

## UC-1. Создание датасета с многоуровневым JOIN

**Актор:** дизайнер BI. **Предусловие:** право чтения `sale.order.line` и связанных моделей.

**Поток:** (1) Создать датасет → каталог моделей → поиск «строка заказа» → выбор `sale.order.line`. (2) В дереве полей раскрыть `order_id` → `partner_id` → `country_id`, добавить `name` как измерение «Страна» (путь `order_id.partner_id.country_id.name`, авто-JOIN зафиксирован). (3) Добавить `product_id.categ_id` (измерение «Категория»), `order_id.user_id` («Менеджер»), `price_subtotal` (мера sum), `order_id.date_order` (поле даты). (4) Задать домен датасета: `order_id.state in ('sale','done')` через DomainSelector. (5) Сохранить; предпросмотр-таблица показывает данные.

**Acceptance Criteria:**

- Given пользователь без группы `bi_designer`, When он открывает меню BI, Then пункт «Создать датасет» недоступен.
- Given дерево полей `sale.order.line`, When пользователь раскрывает `order_id.partner_id`, Then показываются только поля, возвращённые `fields_get()` этого пользователя (поля с `groups=`, недоступные ему, отсутствуют).
- Given путь глубины 6, When пользователь пытается раскрыть следующий уровень, Then ветка не раскрывается и показана подсказка о лимите глубины (5).
- Given поле `order_id.partner_id.parent_id.parent_id`, When путь содержит повтор (модель, поле), Then конструктор требует явного подтверждения рекурсии.
- Given датасет сохранён, When дизайнер открывает предпросмотр, Then выполняется ровно один RPC `run_query` и таблица показывает ≤ 80 строк с пагинацией.
- Given computed-поле без `store` и без `search`-метода, When оно отображается в дереве, Then оно неактивно (серое) с подсказкой «нельзя фильтровать/группировать».

## UC-2. Построение мер: ratio и фильтрованная

**Актор:** дизайнер BI. **Предусловие:** датасет UC-1.

**Поток:** (1) В датасете создать меру «Средний чек» = `SUM([price_subtotal]) / NULLIF(COUNT_DISTINCT([order_id]), 0)`. (2) Создать фильтрованную меру «Выручка (доставлено)»: поле `price_subtotal`, agg sum, extra_domain `('order_id.state', '=', 'done')` — складывается с фильтрами виджета по И. (3) Использовать обе меры в KPI-виджете и в таблице по менеджерам.

**Acceptance Criteria:**

- Given формула `SUM([price_subtotal]) / [qty]`, When дизайнер сохраняет, Then сервер отклоняет с ошибкой «смешение агрегированного и неагрегированного операндов» и подсветкой позиции.
- Given формула с несуществующим полем `[pric]`, When валидация, Then ошибка «неизвестное поле» с подсказкой ближайшего имени.
- Given мера-ratio на группировке по 12 месяцам, When виджет запрашивает данные, Then выполняется один агрегирующий запрос (компоненты SUM и COUNT_DISTINCT в одном `formatted_read_group`), а деление выполнено на сервере; деление на ноль даёт NULL (пустую ячейку), не ошибку.
- Given фильтрованная мера и контрол периода за март, When зритель смотрит таблицу, Then «Выручка (доставлено)» учитывает И март, И state=done.
- Given формула, содержащая `__import__` или любые имена с двойным подчёркиванием, When валидация, Then формула отклонена (AST-whitelist), инцидент записан в лог.

## UC-3. Drill-down

**Актор:** зритель. **Предусловие:** виджет «Выручка по странам» с иерархией Страна → Менеджер → Категория и временной ряд по месяцам.

**Поток:** (1) Клик по столбцу «Германия» → виджет показывает менеджеров по Германии, хлебная крошка «Все › Германия». (2) Кнопка «вверх» возвращает уровень. (3) На временном ряду «развернуть уровень целиком» меняет гранулярность месяц → неделя.

**Acceptance Criteria:**

- Given drill в «Германию», When данные запрошены, Then домен спуска взят из `__extra_domain` группы (а не сконструирован клиентом) и добавлен к эффективному домену.
- Given включена опция «drill фильтрует другие виджеты», When выполнен спуск, Then остальные виджеты страницы получили cross-filter «Германия», что отражено чипом в панели фильтров.
- Given drill-стек глубины 2, When зритель сохраняет личную закладку и переоткрывает дашборд по ней, Then состояние drill восстановлено.
- Given группа по месяцу с datetime-полем и пользователь в TZ Europe/Kyiv, When спуск в «март», Then диапазон `__extra_domain` соответствует марту по киевскому времени (UTC-границы корректно сдвинуты сервером).

## UC-4. RLS: record rules и домен аудитории

**Актор:** администратор Odoo, два зрителя. **Предусловие:** на `sale.order` действует штатное правило «продавец видит свои заказы»; руководитель — в группе менеджеров продаж (видит команду).

**Поток:** (1) Оба открывают один дашборд «Продажи». (2) Владелец добавляет аудиторию «Филиал Запад» с extra_domain по `team_id`. (3) Владелец проверяет режимом «как аудитория».

**Acceptance Criteria:**

- Given продавец А и руководитель Б, When каждый открывает дашборд, Then суммы KPI у А равны сумме только его заказов, у Б — заказов команды; конфигурация дашборда при этом одна (никаких настроек «ролей» в BI).
- Given у пользователя нет ACL-read на `account.move.line`, When страница содержит финансовый виджет, Then этот виджет показывает «нет доступа», остальные работают.
- Given аудитория с extra_domain `team_id=Запад`, When входит её участник, Then его данные = record rules ∧ extra_domain (extra_domain не расширяет видимость).
- Given «Посмотреть как пользователь» выполняет bi_admin, When предпросмотр открыт, Then все data-RPC исполнены `with_user(выбранный)` и факт просмотра записан в аудит.
- Given кэш включён, When А и Б открывают один виджет, Then они не получают результат из одного кэш-ключа (ключ включает маркер прав).

## UC-5. Шаринг и публикация

**Актор:** дизайнер (владелец), зритель, аноним.

**Поток:** (1) Владелец публикует дашборд (draft → published, создана версия). (2) Делится с группой «Продажи» (viewer) и коллегой (editor). (3) Создаёт frozen-ссылку для партнёра. (4) Правит черновик — зрители продолжают видеть публикацию; публикует v2; откатывается на v1.

**Acceptance Criteria:**

- Given зритель-viewer, When открывает дашборд, Then видит published-версию без кнопки редактирования; editor видит черновик и кнопку «Опубликовать».
- Given frozen-ссылка, When аноним открывает URL с верным токеном, Then рендерится снимок без живых запросов к бизнес-моделям; неверный токен → 403 (сравнение `consteq`); истёкшая ссылка → страница «ссылка недействительна».
- Given автор ссылки потерял право чтения дашборда, When аноним открывает ссылку, Then доступ закрыт (проверка `with_user(create_uid).has_access('read')`).
- Given откат на v1, When зритель обновляет страницу, Then видит v1; история версий содержит v1, v2 с авторами и датами.
- Given диалог создания публичной ссылки, When пользователь создаёт ссылку, Then доступен только режим frozen-снимка — живых публичных запросов к данным не существует (решение ОВ-4).

## UC-6. Экспорт и подписка

**Актор:** зритель с правом `base.group_allow_export`; менеджер.

**Поток:** (1) Зритель экспортирует таблицу виджета в XLSX и дашборд в PDF (с текущими фильтрами). (2) Менеджер создаёт подписку: каждый понедельник 08:00, PDF, фильтр «team = Запад», получатели — 3 пользователя. (3) Настраивает алерт на KPI «Просрочка > 100 000».

**Acceptance Criteria:**

- Given пользователь без `base.group_allow_export`, When открывает меню виджета, Then пункты экспорта данных скрыты, а прямой вызов контроллера экспорта возвращает 403.
- Given экспорт PDF, When файл получен, Then состояние фильтров на момент экспорта отражено в колонтитуле, каждый график присутствует растром, постраничная разбивка не режет виджеты.
- Given подписка с фильтром, When наступает время, Then каждый внутренний получатель получает PDF, собранный от его имени (данные = его права ∧ фильтры подписки); при сбое рендера письмо не отправляется, ошибка в логе подписки, следующий запуск по расписанию.
- Given алерт с порогом и троттлингом «раз в день», When условие выполняется дважды за день, Then уведомление отправлено один раз, оба срабатывания в журнале алерта.
- Given XLSX pivot-виджета, When файл открыт, Then структура строк/колонок и подытоги соответствуют экрану, числа — числовые (не текст), валюта отформатирована.

# 13. Нефункциональные требования

- **Производительность:** бюджеты раздела 10.1 — критерии приёмки нагрузочного теста (профиль по решению ОВ-3: модель 5 млн строк, 10 виджетов, 50 одновременных зрителей).
- **Сопровождение:** автоматический регресс-прогон тестов модуля после каждого обновления Odoo 19.x включается в контракт сопровождения (решение ОВ-12); матрица совместимости ведётся в CI (связано с рисками Р-1, Р-2, Р-8).
- **Совместимость:** Odoo 19.0 Community и Enterprise; PostgreSQL ≥ 13; браузеры: последние 2 версии Chrome/Edge/Firefox/Safari; мобильный просмотр (read-only) от 360 px.
- **Локализация:** все строки UI переводимы (ru, uk, en в поставке); форматы чисел/дат/валют — по `res.lang` (babel); недели — по `week_start`; RTL не требуется (этап 3 — по запросу).
- **Безопасность:** разделы 8, 5.4; OWASP-чек публичного контура (IDOR на share-роутах, инъекции, XSS в заголовках/текстовых виджетах — санитайзинг), pen-test перед релизом публичных ссылок.
- **Надёжность:** деградация per-виджет (П-6); фоновые задания идемпотентны и батчируются `_commit_progress`; миграционные скрипты конфигов между версиями модуля (`migrations/`).
- **Качество кода:** соответствие Odoo guidelines; покрытие тестами: unit (компилятор DSL, генератор SQL, маппинг контролов) ≥ 85%, интеграционные (права/RLS-матрица, кэш-ключи, grouping sets), tour-тесты UI (создание датасета, дашборда, drill, share); CI на ветке Odoo 19.
- **Документация:** руководство пользователя (дизайнер/зритель), руководство администратора (права, лимиты, материализация), README разработчика (схемы JSON, точки расширения: регистрация кастомного типа виджета и кастомной функции DSL через registry).
- **Расширяемость:** новые типы виджетов и функции DSL добавляются дочерними модулями через реестры (клиентский registry `td_bi_widgets`, серверный — реестр функций компилятора) без патчей ядра модуля.

# 16. Протокол исполнения для Claude Code (агентный цикл разработки)

Раздел адресован Claude Code как исполнителю. Разделы 1–15 — ЧТО строить; раздел 16 — КАК строить: принудительный цикл «план → реализация субагентами → проверка соответствия ТЗ → правки», повторяемый до закрытия всех требований. Отступление от протокола — само по себе несоответствие ТЗ.

## 16.1. Главный цикл качества

```text
СТАРТ:
  1. Прочитать ТЗ ЦЕЛИКОМ (разделы 1–15). Вопросов заказчику не задавать:
     все решения зафиксированы (15.3); неоднозначности решать по приоритету 16.8.
  2. Построить TRACEABILITY.md — трассировочную матрицу требований (16.4).
  3. Поднять окружение (16.6). Убедиться: пустой каркас модуля устанавливается зелёно.
  4. Вести список задач (инструмент задач): WP = задача, статусы обновлять немедленно.

ДЛЯ каждого этапа (этап 1, затем этап 2; этап 3 — только по явной команде заказчика):
  ДЛЯ каждого WP по графу зависимостей (16.2); независимые WP — параллельно (16.3):
    ЦИКЛ WP (максимум 5 итераций):
      PLAN      → субагент-планировщик: список файлов/моделей/компонентов/тестов
                  + сверка каждого используемого API ядра с исходниками Odoo 19 (16.5, правило A)
      IMPLEMENT → субагент-исполнитель: код + тесты по плану
      GATE-1    → механические ворота (16.5): установка модуля, lint, unit,
                  integration, tour — ВСЁ зелёное; красное → FIX → снова GATE-1
      GATE-2    → субагент-верификатор (НЕ тот, кто писал код) сверяет результат
                  с пунктами ТЗ пакета → отчёт несоответствий (16.4)
      ЕСЛИ отчёт пуст → WP завершён: обновить TRACEABILITY.md, атомарный коммит [WP-NN]
      ИНАЧЕ        → FIX по отчёту → повторить с GATE-1
    ЕСЛИ после 5 итераций остались несоответствия → DEVIATIONS.md (16.8), продолжить дальше
  ЭТАПНАЯ ПРИЁМКА (16.9): все AC этапа из раздела 12 + перф-смоук (10.1) + полный регресс
ФИНАЛ: ОТЧЁТ_ПРИЁМКИ.md (16.9)
```

Жёсткие правила цикла:

- Верификатор никогда не правит код; исполнитель никогда не «принимает» собственный пакет. Это разные субагенты с разным контекстом.
- WP не помечается завершённым при красном GATE-1 или непустом GATE-2 — без исключений «доделаю потом».
- Любое требование, которое невозможно выполнить, фиксируется в DEVIATIONS.md с обоснованием и выбранной заменой — молча пропускать запрещено.
- После слияния параллельных WP — обязательный интеграционный прогон GATE-1 на объединённом коде.
- Коммиты атомарные, по одному на итерацию WP, сообщение: `[WP-NN] <суть>`; финальный коммит пакета: `[WP-NN] done: <закрытые R-ID>`.

## 16.3. Субагенты: роли и параллельность

| Роль | Когда | Вход | Выход (обязательный формат) |
|---|---|---|---|
| Планировщик (Plan) | начало WP | разделы ТЗ пакета + текущий код | `plans/WP-NN.md`: файлы к созданию/изменению, модели/поля, компоненты, список тестов, таблица сверки API (16.5-A) |
| Исполнитель (Implement) | после плана | план + ТЗ-разделы | код + тесты + краткий журнал решений в `plans/WP-NN.md` (секция «Реализация») |
| Верификатор (Verify) | после зелёного GATE-1 | R-ID пакета + полные тексты разделов ТЗ + diff пакета | `reports/WP-NN-verify-K.md`: таблица несоответствий (16.4); пустая таблица = принято |
| Тест-инженер | WP-09/WP-21 и по запросу | AC раздела 12 | tour/integration-тесты, генератор нагрузочных данных, перф-отчёт |
| Интегратор | после слияния параллельных WP | объединённый код | зелёный полный GATE-1 + отчёт о конфликтах |

Правила параллельности: одновременно не более 3 исполнителей; параллелить только WP без общих файлов и без зависимостей по графу 16.2 (примеры пар: WP-10‖WP-11, WP-14‖WP-16‖WP-18); каждый параллельный WP — в изолированном git worktree; слияние — по порядку номеров; после слияния — Интегратор. Верификатор всегда запускается с чистым контекстом (без диалога исполнителя) — он читает ТЗ и код, а не объяснения.

Шаблон задания верификатору (использовать дословно, подставив пакет):

```text
Ты — аудитор соответствия. НЕ исправляй код. Вход: (1) выдержки ТЗ: <разделы WP>,
(2) список требований: <R-ID из TRACEABILITY.md>, (3) diff/файлы пакета.
Для КАЖДОГО R-ID верни строку таблицы:
| R-ID | Статус ✅/❌/⚠️ | Файл:строка | Чего не хватает / что отличается от ТЗ | Минимальная правка |
⚠️ = реализовано частично или иначе, чем в ТЗ. Отдельно перечисли найденные
нарушения правил безопасности раздела 8 (sudo на бизнес-данных, eval, raw SQL без
SQL-обёртки, домены без Domain.AND) — они всегда ❌ независимо от R-ID.
```

## 16.4. Трассировочная матрица (TRACEABILITY.md)

- Строится Планировщиком на старте проекта до первого WP: каждый проверяемый пункт разделов 3–11 и каждый AC раздела 12 получает ID `R-<раздел>-<NN>` (пример: `R-3.5-02 having-домен транслируется в SQL HAVING`; AC используют родные ID `UC-N-ACk`). Гранулярность: одно требование = одно проверяемое утверждение; ожидаемый объём — 250–400 строк.
- Колонки: `R-ID | Раздел | Формулировка (сжатая) | WP | Статус (todo/in_progress/✅/deviation) | Доказательство (тест или файл:строка)`.
- Обновляется при закрытии каждого WP; «✅ без доказательства» запрещён: доказательство — имя автотеста (предпочтительно) или файл:строка реализации.
- Этапная приёмка не начинается, пока у этапа есть R-ID в статусе todo/in_progress.

## 16.5. Тестовые ворота (GATE-1) и правило сверки API

**Правило A — сверка API ядра (анти-галлюцинация).** Перед использованием любого API Odoo из ТЗ (минимум: `formatted_read_group`, `formatted_read_grouping_sets`, `web_read_group`, `web_search_read`, `fields_get` (атрибуты `searchable/sortable/groupable/aggregator`), `Domain` py/js, `_search`, `odoo.tools.SQL`/`execute_query`, `ormcache`, `ir.cron._commit_progress`, `useSortable`/`makeDraggableHook`, `loadBundle`, `ModelFieldSelector`, `DomainSelector`, `name_search`, паттерн `spreadsheet.dashboard.share`) Планировщик обязан найти фактическую сигнатуру grep-ом по локальным исходникам Odoo 19 (`$ODOO_SRC`) и занести в таблицу сверки плана: `API | файл ядра:строка | сигнатура | примечание`. Кодировать по памяти запрещено. Если фактическая сигнатура расходится с ТЗ — следовать исходникам, расхождение записать в DEVIATIONS.md (это покрывает риски Р-1, Р-2).

**Команды GATE-1** (порядок обязателен, останов на первом красном):

```bash
# 1. Линт
ruff check addons_custom/td_bi_dashboard && eslint addons_custom/td_bi_dashboard/static/src
# 2. Чистая установка + апгрейд
odoo-bin -c odoo.conf -d bi_ci --no-http -i td_bi_dashboard --stop-after-init
odoo-bin -c odoo.conf -d bi_ci --no-http -u td_bi_dashboard --stop-after-init
# 3. Серверные тесты (unit + integration)
odoo-bin -c odoo.conf -d bi_ci --no-http --test-enabled \
  --test-tags /td_bi_dashboard --stop-after-init
# 4. Tour-тесты (HttpCase, нужен headless chrome в контейнере)
odoo-bin -c odoo.conf -d bi_ci --test-enabled \
  --test-tags /td_bi_dashboard:tour --stop-after-init
```

Минимальный состав тестов по типам кода: query engine — unit на merge доменов, ключ кэша, Top-N, sum_currency, tz-границы периодов (фиксированные фикстуры в 2 таймзонах); безопасность — integration-матрица «2 пользователя × record rule × groups=-поле» (AC UC-4); DSL (WP-11) — unit на каждую функцию каталога 4.2 + фаззинг отклонения запрещённых конструкций; UI — tour на каждый Use Case раздела 12. Перф-смоук: генератор данных (скрипт в `tests/perf/`) + замер бюджетов 10.1; результаты в `reports/perf-этап-N.md`.

## 16.8. Приоритет источников и отклонения

При противоречии внутри ТЗ или между ТЗ и фактическим поведением Odoo 19 решение принимается без вопросов заказчику, по приоритету:

1. Решения заказчика — таблица 15.3 (ОВ-1…ОВ-12).
2. Фактические сигнатуры/поведение исходников Odoo 19 (правило 16.5-A).
3. Правила безопасности и данных — разделы 8, 3.4–3.7, П-1…П-6.
4. Acceptance Criteria раздела 12 (буква AC сильнее описательного текста).
5. Описательный текст разделов; при равных вариантах — минимальный по сложности.

Каждое применение приоритета 2–5 вместо буквы ТЗ — строка в `DEVIATIONS.md`: `номер | пункт ТЗ | что сделано вместо | почему | влияние | требуется ли решение заказчика`. DEVIATIONS.md включается в финальный отчёт.

## 16.9. Definition of Done проекта и финальный отчёт

Проект завершён, когда: (1) все WP этапов 1 и 2 закрыты; (2) TRACEABILITY.md не содержит статусов todo/in_progress для этапов 1–2; (3) все AC раздела 12 зелёные автотестами; (4) перф-бюджеты 10.1 выполнены на профиле 5 млн строк (ОВ-3); (5) полный GATE-1 зелёный на чистой БД; (6) переводы ru/uk/en присутствуют (`*.po` без fuzzy); (7) собран `ОТЧЁТ_ПРИЁМКИ.md`: сводка по WP, итоговая TRACEABILITY (счётчики ✅/deviation), результаты перф-прогона, DEVIATIONS.md целиком, известные ограничения, инструкция установки. Этап 3 в DoD не входит (раздел 14).
