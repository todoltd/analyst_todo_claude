# Kickoff Question Builder

## Мета

Дати BA готовий, структурований опитувальник для Установчої Зустрічі: які саме питання поставити
клієнту, в якому пріоритеті, і **навіщо** — до якого блоку майбутнього плану дискавері веде кожна
відповідь. Опитувальник перетворює порядок денний (`agenda`) із "тем зустрічі" на конкретні питання,
а потім гарантує, що зібраної інформації вистачить для побудови плану дискавері.

Головна цінність — **повнота**: скіл звіряє питання з усіма входами `discovery-plan-builder`
і явно показує, де даних не вистачить, ще ДО зустрічі.

> **Формат Установчої Зустрічі:** 60 хвилин фіксовано (15 хв організаційні + 45 хв змістовна частина).
> Мета зустрічі — узгодити **scope**. Питання опитувальника — рівня scope, а не технічної глибини.
> **Не виноси на цю зустріч:** деталі міграції даних, точну кількість майбутніх користувачів,
> матриці прав, інфраструктуру. Це теми окремих сесій дискавері. На Установчій Зустрічі ми лише
> з'ясовуємо, **які з цих ліній взагалі релевантні** — щоб закласти їх у план.

---

## Інструменти та середовище

**Читання вхідних файлів:**
- `agenda_[назва_компанії].md` → інструмент `Read` (основний вхід — теми і блоки зустрічі)
- `підготовка_УЗ_[назва_компанії].md` → `Read` (реєстр невідомого, ризики, профіль — джерело питань)
- `бриф_[назва_компанії].md` → `Read` якщо потрібні деталі про стейкхолдерів

**Збереження результату:**
- Збережи у папку `outputs` як `питання_УЗ_[назва_компанії].md` → інструмент `Write`

**Після завершення:**
- Викличи `present_files` щоб показати файл користувачу
- Коротко повідом: "Опитувальник збережено. [N] питань, з них [M] критичних. Карта покриття плану: [скільки ліній покрито / усього]."
- Запропонуй наступний крок: "Готово. Далі — `facilitation-guide-builder` для сценарію ведення зустрічі. Опитувальник можна тримати поруч і фіксувати відповіді прямо в ньому."

---

## Рекомендований порядок запуску скілів блоку

```
1. kickoff-agenda-finalizer   → agenda_[назва_компанії].md         (обов'язково першим)
2. kickoff-question-builder   → питання_УЗ_[назва_компанії].md     (після agenda — деталізує її блоки)
3. facilitation-guide-builder → facilitation_[назва_компанії].md   (після — сценарій спирається на питання)
   discovery-charter-writer   → charter_[назва_компанії].md        (паралельно — для клієнта)
```

`kickoff-question-builder` потребує готової `agenda` — він розгортає блоки порядку денного в конкретні питання.
Без `agenda` опитувальник буде надто загальним.

## Крок 0: Перевір вхідні дані

Прочитай доступні файли. Мінімум для роботи:
- `agenda_[назва_компанії].md` — структура зустрічі і її блоки

**Якщо в outputs є кілька `agenda_*.md` або кілька `підготовка_УЗ_*.md`:**
Запитай: "Знайшов файли для кількох компаній. Для якої готуємо опитувальник?"

**Якщо `agenda` не знайдена, але є `підготовка_УЗ`:**
Працюй від `підготовка_УЗ` (там уже є питання блоками) і познач у шапці:
`⚠️ Опитувальник побудовано без agenda — рекомендується спочатку згенерувати порядок денний.`

**Якщо немає ні `agenda`, ні `підготовка_УЗ`:**
Запитай: "Надішли `agenda_[назва_компанії].md` або `підготовка_УЗ_[назва_компанії].md`. Без них опитувальник буде шаблонним."

**Таблиця інвентаризації:**

| Файл | Статус | Що дає |
|------|--------|--------|
| `agenda_[назва_компанії].md` | є / немає | Блоки зустрічі, теми → основа для питань |
| `підготовка_УЗ_[назва_компанії].md` | є / немає | Реєстр невідомого [КД], ризики, профіль |
| `бриф_[назва_компанії].md` | є / немає | Стейкхолдери, поточна система |

## Крок 1: Визнач теми опитувальника

Візьми змістовні блоки з **внутрішньої версії** `agenda` (зазвичай: Розуміння бізнесу, Ключові процеси і scope).
Розгорни їх у тематичні блоки опитувальника нижче. Якщо `agenda` має додаткові/специфічні блоки
(наприклад "Узгодження очікувань", "Бачення клієнта") — додай під них окремий тематичний блок.

**Базовий набір тематичних блоків (адаптуй під клієнта, не вигадуй зайвого):**

1. **Бізнес-контекст і цілі проекту** — навіщо проект, критерії успіху, головний біль
2. **Ключові процеси і межі scope** — що в scope, що поза ним, що найкритичніше
3. **Поточні системи (високий рівень)** — чим користуються зараз, що замінюється
4. **Інтеграції (високий рівень)** — з якими зовнішніми системами треба обмін
5. **Користувачі і ролі (без підрахунку кількості)** — які відділи/ролі працюватимуть
6. **Обмеження: терміни, очікування, регуляторика** — дедлайни, очікування, вимоги
7. **Стейкхолдери і прийняття рішень** — хто спонсор, хто точка контакту, власники процесів
8. **Запуск і навчання (високий рівень)** — як уявляють go-live, попередній досвід

> Не обов'язково мати всі 8 блоків. Якщо для блоку немає підстав у `agenda`/`підготовка_УЗ` — пропусти його,
> але зазнач у "Карті покриття плану" (Крок 3), що відповідна лінія не покрита.

## Крок 2: Сформуй питання

Для **кожного** тематичного блоку згенеруй 3–7 питань. Кожне питання — за форматом:

```
**[Пріоритет] Питання?** `[тег-плану]`
> Навіщо: [одне речення — на яке рішення плану дискавері вплине відповідь]
> Відповідь: _________________________________________________
```

**Пріоритети (постав на кожне питання):**
- 🔴 **Критично** — без відповіді не можна планувати дискавері (scope, цілі, спонсор, обов'язкові інтеграції)
- 🟡 **Важливо** — суттєво уточнює план, але дискавері можна почати без цього
- ⚪ **Опційно** — добре мати, але не блокує

**Теги прив'язки до плану дискавері** (постав 1 тег на питання — це вхід `discovery-plan-builder`):
- `[scope]` — межі проекту: що в/поза scope
- `[процеси]` — бізнес-процеси → функціональні сесії дискавері
- `[дані]` — наскрізна лінія: дані та міграція
- `[інтеграції]` — наскрізна лінія: інтеграції
- `[ролі]` — наскрізна лінія: ролі, права доступу, безпека
- `[NFR]` — наскрізна лінія: обсяги, продуктивність, регуляторика
- `[go-live]` — наскрізна лінія: навчання і запуск/cutover
- `[стейкхолдери]` — учасники сесій, власники процесів
- `[рішення]` — ключові рішення і залежності, критерії успіху
- `[бюджет]` — очікування щодо бюджету/термінів (для reality-check SoW)

**Правила формулювання питань:**
- Конкретні, відкриті, без жаргону. Не "розкажіть про продажі", а "як у вас зараз формується ціна і хто її затверджує?".
- Рівень scope, не глибина дискавері. Для наскрізних ліній (`[дані]`, `[інтеграції]`, `[ролі]`, `[NFR]`, `[go-live]`)
  став питання-**фільтр**: "чи це взагалі релевантно і наскільки критично?" — деталі парку для дискавері (Крок 4).
- **Бюджет — без цифр.** Питання `[бюджет]` формулюй так, щоб з'ясувати наявність і власника бюджету,
  а не суму: "Чи є вже затверджений бюджет на проект і хто ним розпоряджається?" Не проси назвати цифру на зустрічі.
- **Кількість користувачів НЕ запитуємо** на Установчій Зустрічі. Тільки які ролі/відділи будуть користувачами.

### Приклади (не копіюй дослівно — адаптуй під клієнта)

**Блок: Бізнес-контекст і цілі проекту**
```
**🔴 Чому ви вирішили міняти/впроваджувати систему саме зараз?** `[scope]`
> Навіщо: визначає головну невизначеність і пріоритет блоку A плану.
> Відповідь: _________________________________________________

**🔴 Як ви зрозумієте через рік, що проект був успішним?** `[рішення]`
> Навіщо: критерії успіху → deliverables і фокус дискавері.
> Відповідь: _________________________________________________
```

**Блок: Ключові процеси і межі scope**
```
**🔴 Які процеси точно мають увійти в першу чергу, а що свідомо лишаємо на потім?** `[scope]`
> Навіщо: підтверджений scope → кількість і теми функціональних сесій.
> Відповідь: _________________________________________________
```

**Блок: Інтеграції (високий рівень)**
```
**🟡 З якими зовнішніми системами нова система обов'язково має обмінюватись (банк, маркетплейси, логістика, держсервіси)?** `[інтеграції]`
> Навіщо: визначає, чи потрібна окрема сесія дискавері по інтеграціях.
> Відповідь: _________________________________________________
```

> Згенеруй питання для КОЖНОГО релевантного блоку — не лише для наведених трьох.

## Крок 3: Карта покриття плану

Це **головна перевірка скілу**. Побудуй таблицю: для кожного входу плану дискавері познач,
чи є хоча б одне питання, яке його закриває. Так BA бачить прогалини ще до зустрічі.

| Вхід плану дискавері | Покрито питанням? | № / коротко |
|----------------------|-------------------|-------------|
| Цілі та критерії успіху `[рішення]` | ✅ / ❌ | [яким питанням] |
| Scope: межі проекту `[scope]` | ✅ / ❌ | |
| Бізнес-процеси `[процеси]` | ✅ / ❌ | |
| Дані та міграція `[дані]` | ✅ / ❌ | |
| Інтеграції `[інтеграції]` | ✅ / ❌ | |
| Ролі, права, безпека `[ролі]` | ✅ / ❌ | |
| Обсяги / NFR / регуляторика `[NFR]` | ✅ / ❌ | |
| Навчання і go-live `[go-live]` | ✅ / ❌ | |
| Стейкхолдери і власники процесів `[стейкхолдери]` | ✅ / ❌ | |
| Бюджет / очікування `[бюджет]` | ✅ / ❌ | |

**Для кожного ❌:** додай рядок у розділ "Прогалини" з поясненням — питання не вмістилось у 45 хв
чи лінія неактуальна для цього клієнта. Якщо неактуальна — так і напиши ("виробництва немає → лінія обсягів не критична").
Якщо просто не вмістилось — це сигнал винести в першу сесію дискавері.

## Крок 4: Парковка для дискавері

Питання, які важливі, але **занадто глибокі для 60-хвилинної Установчої Зустрічі**. Не задаємо їх на зустрічі —
фіксуємо, щоб закласти в план дискавері. Типово сюди потрапляють:
- Детальний маппінг і якість даних для міграції
- Точна кількість користувачів і ліцензій
- Матриці прав доступу по ролях
- Технічна інфраструктура, середовища, продуктивність
- Деталі кожної інтеграції (протоколи, формати, частота обміну)

Сформуй короткий список "→ [тема] — на сесію дискавері по [лінія]".

## Важливі правила

**Опитувальник — для команди, не для клієнта.** Поля "Навіщо" і теги-плану — внутрішні. Клієнту не надсилати.

**Питання служать плану.** Кожне питання має вести до конкретного входу `discovery-plan-builder`.
Якщо питання нікуди не веде (немає тега) — або додай тег, або прибери питання.
