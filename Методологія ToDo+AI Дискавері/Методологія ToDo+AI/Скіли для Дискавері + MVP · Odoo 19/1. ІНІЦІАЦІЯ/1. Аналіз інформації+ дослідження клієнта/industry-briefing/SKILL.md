# Industry Briefing

## Мета

Дати команді фундамент для впевненої розмови з клієнтом: що типово для його галузі,
які болі вже відомі без запитань, які процеси точно будуть у scope, які регуляторні
вимоги впливають на Odoo-конфігурацію. Це не дослідження конкретного клієнта —
це підготовка BA як фахівця в галузі.

---

## Інструменти та середовище

**Читання брифу (якщо є):**
- `бриф_[назва_компанії].md` на диску → використай `Read` перед початком

**Веб-дослідження (обов'язково для регуляторики):**
- Пошук актуальних даних по галузі, законодавству, Odoo-локалізаціях →
  `WebSearch` (завантажити через `ToolSearch` перед першим використанням)
- Відвідати офіційний сайт регулятора або документацію Odoo →
  `mcp__workspace__web_fetch`
- Якщо сторінка JS-рендерингу → Claude in Chrome:
  `mcp__Claude_in_Chrome__navigate` + `mcp__Claude_in_Chrome__get_page_text`

**Збереження результату:**
- Збережи у папку `outputs` як `галузь_[назва_галузі]_[назва_компанії].md` → інструмент `Write`

**Після завершення:**
- Викличи `present_files` щоб показати файл користувачу
- Коротко повідом: "Галузевий брифінг збережено. Ключовий момент для команди: [1 речення]"
- Запропонуй наступний крок: "Готово. Далі — `discovery-context-builder` для зведеного пакета підготовки до Установчої Зустрічі."

## Крок 0: Визнач вхідні дані

**Потрібно мінімум:**
- Галузь / ніша (наприклад: "оптова торгівля будматеріалами", "виробництво меблів на замовлення")
- Країна або регіон (впливає на регуляторику)

**Бажано також:**
- Розмір компанії (мікро / малий / середній / великий) — впливає на глибину процесів
- Модель бізнесу: B2B / B2C / B2B2C, виробництво / торгівля / послуги
  (впливає на набір модулів — деталі в кроці 2)
- Що вже відомо з брифу (`бриф_[назва_компанії].md`) — не повторювати те що вже є

**Якщо галузь не вказана:**
Запитай: "Яка галузь і країна клієнта? Якщо є бриф — надішли його."
Без галузі брифінг буде занадто загальним і некорисним.

**Якщо галузь дуже широка** (наприклад "торгівля" або "послуги"):
Уточни: "Яка саме торгівля — опт, роздріб, e-commerce? Які товари або послуги?"
Чим точніша галузь — тим кориснішим буде брифінг.

## Крок 1: Загальний огляд галузі

Використовуй знання про галузь + обов'язково виконай веб-пошук для актуальних даних.
Внутрішніх знань достатньо для загальних паттернів — але не для поточних регуляторних вимог,
свіжих змін у законодавстві і стану локалізації Odoo 19.

**Шаблони пошуку (адаптуй під галузь і країну):**
```
"[галузь]" "[країна]" автоматизація ERP 2024 OR 2025
"[галузь]" "[країна]" регуляторні вимоги законодавство
Odoo "[галузь]" modules best practices
"l10n_[код_країни]" Odoo 19 OR "Odoo 17" OR "Odoo 18"
"[галузь]" "[країна]" обов'язкова звітність OR "галузевий регулятор"
```

> ℹ️ Якщо пошук не дав актуальних результатів — зазнач це у розділі регуляторики
> і порекомендуй верифікувати вимоги у локального податкового консультанта або
> на офіційному сайті регулятора.

### 1.1 Типова бізнес-модель
- Як влаштований бізнес у цій галузі: хто виробляє, хто продає, хто обслуговує
- Ланцюжок постачань або надання послуги (схематично)
- Ключові гравці: постачальники — компанія — клієнти

### 1.2 Типові больові точки галузі
Конкретні проблеми, а не загальні слова.
- Добре: "Виробники меблів страждають від ручного розрахунку специфікацій і втрат матеріалу"
- Погано: "Проблеми з автоматизацією"

Якщо відомо що клієнт озвучував болі в брифі — зістав: типова проблема галузі чи специфіка клієнта?

### 1.3 Сезонність і ключові події
- Чи є пікові сезони, дедлайни, регуляторні звітні періоди?
- Що може впливати на терміни впровадження?

### 1.4 Рівень цифрової зрілості галузі
- Галузь переважно на 1C / SAP / Excel / власних розробках / без автоматизації?
- Чи є вже прийняті рішення для цієї галузі на ринку?

## Крок 2: Типові процеси та модулі Odoo

### 2.1 Вплив бізнес-моделі на набір модулів

Перш ніж перераховувати модулі — визнач базовий набір за моделлю:

| Модель | Обов'язкові модулі |
|--------|-------------------|
| B2B продажі | Sales, CRM, Invoicing, Inventory |
| B2C / роздріб | Point of Sale, eCommerce, Inventory |
| Виробництво | Manufacturing (MRP), Inventory, Purchase |
| Послуги / проекти | Project, Timesheets, Helpdesk, Invoicing |
| Змішана / B2B2C | комбінація вищенаведеного |

Це базовий шар. Галузева специфіка додається поверх нього в кроці 2.2.

> ⚠️ Таблиця — груба евристика, не догма. Реальний набір залежить від конкретного бізнесу:
> напр. дистриб'ютор без власного складу може не потребувати Inventory, але потребувати
> повноцінний Accounting (не лише Invoicing). Не подавай ці модулі як остаточний scope —
> це гіпотеза для перевірки на дискавері.

> **Наскрізні лінії scope (не забути в жодній галузі):** окрім функціональних модулів,
> у scope майже завжди входять: міграція даних, інтеграції, ролі/права доступу,
> нефункціональні вимоги (обсяги, продуктивність), навчання і go-live. Познач їх як
> теми, які треба окремо дослідити — вони часто найризикованіші.

### 2.2 Ключові бізнес-процеси галузі

Перелічи 5–10 типових процесів специфічних для цієї галузі.
> ⚠️ Не копіюй приклади — заповни таблицю реальними процесами галузі клієнта.

| Процес | Модуль Odoo | Пріоритет |
|--------|-------------|-----------|
| [Процес 1 для цієї галузі] | [Модуль] | [Обов'язково / Зазвичай / Опціонально] |
| [Процес 2] | [Модуль] | [...] |
| ... | ... | ... |

Пріоритети:
- **Обов'язково** — є у 90%+ компаній цієї галузі, без цього Odoo не замінить поточне рішення
- **Зазвичай потрібно** — є у більшості, але залежить від розміру і моделі
- **Опціонально** — нішеві потреби, уточнити на Установчій Зустрічі

### 2.3 Специфічні модулі для галузі
Що відрізняє цю галузь від типового впровадження:
- Нестандартні модулі (Manufacturing, MRP, PLM, Helpdesk, Events тощо)
- Кастомізації яких зазвичай потребує галузь
- Інтеграції типові для галузі (банки, маркетплейси, держпослуги, логістика)

### 2.4 Червоні прапорці scope
Що у цій галузі часто розширює scope несподівано:
- Наприклад: "У виробництві замовних виробів часто виявляється потреба в PLM або складній варіативності продуктів"
- Якщо нічого специфічного — напиши "Специфічних розширювачів scope не виявлено"

## Крок 3: Регуляторні та локальні вимоги

### 3.1 Локалізація Odoo
Залежно від країни клієнта:

**Україна:**
- ПДВ і податкова накладна + **реєстрація в ЄРПН** (Єдиний реєстр податкових накладних) у законодавчі строки — критично для обліку ПДВ; стандартний `l10n_ua` часто потребує доопрацювання або стороннього рішення
- ПРРО / РРО якщо є роздрібні продажі (для ТОВ, ПП, AT з готівковими розрахунками)
- Банківські інтеграції (Приватбанк, Monobank, ПУМБ тощо)

**Польща:**
- JPK (Jednolity Plik Kontrolny) — обов'язкова податкова звітність
- KSeF (Krajowy System e-Faktur) — національна система обовʼязкових структурованих е-фактур (формат FA(3), опубліковано 25.06.2025 на gov.pl). Обовʼязковість **поетапна**, офіційно анонсований графік МФ Польщі: **з 1 лютого 2026** — великі платники (обіг > 200 млн PLN за 2024), **з 1 квітня 2026** — решта платників ПДВ. `[звірити актуальний статус/дати на ksef.podatki.gov.pl — графік історично зсувався; для клієнтського документа дату підтвердити з джерела]`
- Faktura VAT — стандарти рахунків-фактур
- Модуль l10n_pl

**ЄС (загально):**
- GDPR — захист персональних даних
- e-Invoice вимоги (залежать від країни)

**Міжнародна компанія / кілька юросіб:**
- Multi-currency, multi-company
- Консолідована звітність

### 3.2 Галузева регуляторика
Специфічні вимоги до галузі:
- Сертифікація, ліцензування (наприклад: медицина, харчова промисловість, фармацевтика)
- Обов'язкова звітність галузевим регуляторам
- Маркування, відстеження партій (lot/serial tracking)

Якщо галузь без специфічної регуляторики — напиши "Специфічних галузевих вимог не виявлено".

> ⚠️ **Застереження щодо актуальності:** Регуляторні вимоги і стан локалізацій Odoo змінюються.
> Інформація у цьому брифінгу — відправна точка, не юридична консультація.
> Перед Установчою Зустріччю рекомендується верифікувати критичні вимоги:
> - Локалізація: офіційна документація Odoo або community-репозиторій `l10n_[країна]`
> - Регуляторика: офіційний сайт відповідного органу (ДПС, ZUS, податковий орган тощо)

## Крок 4: Питання для Установчої Зустрічі з галузевого контексту

**Спочатку перевір бриф** (`бриф_[назва_компанії].md` якщо є):
- Якщо питання вже має відповідь у брифі — не включай його, воно вже відоме команді
- Якщо відповідь є але неповна — постав питання з позначкою "уточнити"
- Питання = тільки те що ще невідомо або потребує підтвердження

> Це **кандидати**, а не остаточний перелік питань Установчої Зустрічі — фінальні
> пріоритизовані питання (макс. 15) зводить `discovery-context-builder`, щоб не дублювати.

Сформуй 5–8 кандидатів-питань з галузевого контексту:

Формат:
```
→ [Питання]
Навіщо: [чому це важливо для scope або конфігурації Odoo]
```

Приклади для виробника меблів:
```
→ "Ви виробляєте за стандартними специфікаціями чи під індивідуальне замовлення?"
Навіщо: якщо під замовлення — потрібен MRP з варіативністю або PLM, це суттєво впливає на scope.

→ "Як зараз розраховуєте собівартість виробу?"
Навіщо: якщо вручну або в Excel — Cost of Goods у Odoo потребує правильного налаштування BOM.
```

## Крок 6: Перевір брифінг перед збереженням

Перед збереженням перечитай весь документ і перевір:
- [ ] Назва файлу `галузь_[назва_галузі]_[реальна назва компанії].md` — не шаблонна
- [ ] Регуляторні вимоги та локалізація Odoo 19 мають позначку `[перевірити]` або посилання на офіційне джерело — не подані як остаточна юридична консультація
- [ ] Модулі Odoo мають пріоритети (Обов'язково / Зазвичай / Опціонально)
- [ ] Червоні прапорці scope виділені окремо
- [ ] Застереження "Чого НЕ чекати від цього брифінгу" присутнє
- [ ] "Головне для команди" заповнене реальними даними
- [ ] Немає рядків `[⚠️ ІНСТРУКЦІЯ:...]` — всі замінено або видалено

Якщо знайшов щось із переліку — виправ до збереження.

## Важливі правила

**Розрізняй знання та здогадки.** Загальновідомі галузеві факти — без позначки.
Висновки що потребують уточнення — `[уточнити на Установчій Зустрічі]`.

**Не підміняй дискавері.** Цей брифінг — підготовка, не замінник Установчої Зустрічі.
Не роби висновків про конкретного клієнта якщо вони не підкріплені брифом.

**Нішеві галузі.** Якщо галузь вузька і відомостей мало — напиши чесно:
> ℹ️ Галузь нішева, загальнодоступних даних обмаль. Брифінг базується переважно
> на загальних принципах. Рекомендується розширена Установча Зустріч з детальнішим
> з'ясуванням процесів.

**Глибина.** Ціль — дати команді достатньо для впевненої розмови, не академічний огляд.
Орієнтовний час підготовки: 20–30 хвилин.

**Зберігай брифінг у файл.** Збережи як `галузь_[назва_галузі]_[назва_компанії].md` — він буде
використаний в `discovery-context-builder` для фінальної підготовки до Установчої Зустрічі.
