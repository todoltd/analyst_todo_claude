# Зразки реальних фрагментів протоколів (за типами зустрічей)

Фрагменти з реальних протоколів (EDELWEIS SSERVICE — 41 протокол; SUPPLYCHAINS — польовий тест).
Це **зразки стилю і структури**, не тексти для копіювання: підставляй факти своєї зустрічі.

---

## 1. Установча — «Команда і ролі» ПЕРЕЛІКОМ по рядках (не прозою)

```
1. Команда проєкту та ролі учасників
З боку ToDo проєкт ведуть:
[Ім'я] — керівник проєкту і провідний аналітик, який проводитиме обстеження;
[Ім'я] — керівник відділу управління проєктами, наглядова роль;
[Ім'я] — менеджер з комерційних питань;
[Ім'я] — координація зустрічей і комунікації.
З боку замовника проєкт супроводжує власник — [Ім'я]: він ухвалює рішення і погоджує подальші кроки.
(Внутрішні практики виконавця — «топ-менеджмент особисто веде проєкти…» — у протокол НЕ вносити.)

2. Інструменти співпраці, комунікація та графік роботи
• Проєктна документація ведеться в Google-документах — ви матимете до неї постійний доступ.
• Оперативна комунікація — у спільній групі [месенджер] (створена одразу під час зустрічі).
• Графік роботи команди ToDo — будні, 9:00–18:00 за Києвом.
```

## 2. Збір вимог / статусна — формат «Процес: Опис / Кроки / Відповідальні / Вимоги»

```
Процес: Управління CRM та проєктами
• Опис: ведення клієнтської бази та проєктів за визначеними етапами.
• Кроки: створення картки проєкту -> проходження етапів (налаштованих відповідно до попередніх зустрічей).
• Відповідальні: менеджери з продажів/проєктів.
• Вимоги до автоматизації: реалізація додаткових полів у CRM, що відповідають специфіці бізнесу замовника.
```

## 3. Збір вимог — ключові вимоги списком (зрозуміло клієнту)

```
Ключові вимоги:
• Форми транспортних документів — авіаційні накладні та морські коносаменти — з потрібними довідниками
  (UN/LOCODE — міжнародний довідник локацій). Це найперший пріоритет.
• Рахунки по кожній відправці; зведені рахунки (стейтменти) за тиждень або місяць; аванси й відстрочки.
• Поза межами системи: митні операції, інтеграції з банками (виписки опрацьовуються вручну);
  наявна CRM залишається працювати окремо.
```

## 4. Інтеграції — формат «лейбл: значення»

```
Синхронізація номенклатури
• Майстер-система: Odoo (довідник товарів ведеться тут, WMS отримує оновлення).
• Ідентифікація: штрихкод — атрибут товару, не окрема сутність.
• Процес передачі: створення/зміна товару в Odoo -> автоматична передача у WMS через API.
• Обмеження: вагогабарити заповнюються на стороні WMS після первинного приймання.
```

## 5. Навчання — кроки через «->» + операційні деталі

```
Процес: створення підзадачі
• Кроки: відкрити задачу -> кнопка «Підзадача» -> заповнити назву і відповідального -> зберегти.
• Особливість: кожна підзадача може мати власні підзадачі (вкладеність без обмежень).
• Доступи: виконавець бачить лише свої задачі; керівник проєкту — всі задачі проєкту.
```

## 6. Демонстрація — «Етап: Відповідальний / Дії / Особливість»

```
Етап 1. Створення нагоди
• Відповідальний: менеджер з продажів.
• Дії: створює картку нагоди, заповнює клієнта і маршрут.
• Особливість: поля «Водій» і «Авто» необов'язкові — на момент надання ставки ці дані часто невідомі.
```

## 7. Розбір правок — «Суть проблеми → Рішення»

```
3. Логіка розрахунку ваги у КП
• Суть проблеми: рядки наповзали один на одного, через що текст розривався при переході на нову сторінку.
• Рішення та крок автоматизації: міжрядковий інтервал 1.15 і заборона розриву всередині рядка таблиці;
  правка у шаблоні друкованої форми.
```

## Humizer: пари «було → стало» (з реальної практики)

| Було (внутрішнє/жаргон) | Стало (для клієнта) |
|---|---|
| Зафіксовано scope-розрив по WMS-інтеграції | Узгодили, що саме передається між системами, а що лишається поза межами проєкту |
| Стейкхолдер підтвердив реквайрмент | [Ім'я, посада] підтвердив(ла) цю вимогу |
| Odoo — master data для номенклатури | Довідник товарів ведеться в Odoo; склад (WMS) отримує оновлення автоматично |
| Мігруємо історичні транзакції частково | З наявної системи переносяться лише поточні залишки, історія операцій не переноситься |
| NFR: concurrent users без деградації | Кілька людей можуть працювати в системі одночасно без обмежень |
| Імплементуємо FEFO | Першим відвантажується те, що швидше псується (принцип FEFO) |

**Нормативна лексика:** торговельний (не «торгівельний»), навчання (не «обучення»), наступний (не «слідуючий»).

**Прибрати повністю (правки методолога, польовий тест SUPPLYCHAINS):**
- метафори й «образно кажучи…» («система — бухгалтерська книга бізнесу, а не пульт керування» — видалено методологом);
- розмовні довіски: «так зручніше рахувати», «по суті», «ваш зворотний зв'язок допомагає нам покращуватися»;
- внутрішні практики виконавця («топ-менеджмент особисто веде проєкти…»);
- захисні умови виконавця («ToDo може призупинити проєкт при неоплаті…»);
- комерційні параметри (ставка/валюта) — навіть як відкрите питання;
- зайві часові прив'язки («з цього літа») і дубль дати наступної зустрічі в пунктах.
