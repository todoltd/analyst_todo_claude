# Інструкція: Аналіз Установчої Зустрічі + план дискавері (Claude)

**Для кого:** BA, менеджер проекту, будь-хто хто провів Установчу Зустріч  
**Коли використовувати:** після Установчої Зустрічі — є транскрипт, нотатки або запис  
**Результат:** структурований аналіз зустрічі, офіційний протокол для замовника і готовий план дискавері за 45–80 хвилин

---

## Місце в загальному pipeline

```
БЛОК 1: Аналіз інформації + дослідження замовника
        ↓ підготовка_УЗ_[назва_компанії].md
БЛОК 2: Підготовка до Установчої Зустрічі
        ↓ agenda, питання_УЗ, facilitation, charter
[ УСТАНОВЧА ЗУСТРІЧ ]
        ↓ транскрипт / нотатки + заповнений питання_УЗ
БЛОК 3: Аналіз Установчої Зустрічі + план дискавері  ← ти тут
  1. kickoff-transcript-analyzer  → аналіз_Установчої Зустрічі_[назва_компанії].md
  2. meeting-protocol-builder      → протокол_[тип]_[дата]_[компанія].md + .docx   (документ замовнику)
  3. discovery-plan-builder       → план_дискавері_[назва_компанії].md
        ↓
[ СТАРТ ДИСКАВЕРІ ]
```

---

## Що тут є

3 скіли які перетворюють матеріали зустрічі на документи й план дій:

```
транскрипт / нотатки Установчої Зустрічі
        ↓
1. kickoff-transcript-analyzer  →  аналіз_Установчої Зустрічі_[назва_компанії].md   (внутрішній аналіз)
        ↓
2. meeting-protocol-builder      →  протокол_[тип]_[дата]_[компанія].md + .docx  (офіційний документ замовнику)
        ↓
3. discovery-plan-builder       →  план_дискавері_[назва_компанії].md               (внутрішній план)
```

**Мінімальний маршрут:** кроки 1 + 3 (аналіз + план)  
**Повний маршрут:** усі 3 кроки (додатково — протокол для замовника на погодження)

> Кроки 2 і 3 обидва будуються з `аналіз_УЗ` (крок 1). Протокол — для замовника, план — для команди.

---

## Крок 1 — kickoff-transcript-analyzer

**Що робить:** аналізує транскрипт або нотатки Установчої Зустрічі → виявляє підтверджені вимоги,
нові відкриття, ризики, відкриті питання, зміни scope  
**Час:** 15–25 хвилин

### Як запустити

Напиши у Claude одне з:
- "проаналізуй Установчу Зустріч [назва компанії]"
- "розбери транскрипт Установчої Зустрічі"
- "що вийшло з Установчої Зустрічі [назва компанії]"

Потім надішли або прикріп транскрипт / нотатки.

### Що потрібно мати

Обов'язково:
- Транскрипт або нотатки Установчої Зустрічі (txt, md, або текст у повідомленні)

Бажано:
- `підготовка_УЗ_[назва_компанії].md` — для порівняння з очікуваннями
- `питання_УЗ_[назва_компанії].md` — заповнений на зустрічі опитувальник (структуровані відповіді)

### Що отримаєш

Файл `аналіз_Установчої Зустрічі_[назва_компанії].md` з:
- Профілі учасників і їхні позиції
- Що підтверджено vs що з'явилось вперше
- Нові ризики з пріоритетами 🔴🟡🟢
- Відкриті питання з пріоритетами [КД] / [ВД] / [ПП]
- Зміни scope (розширення / звуження / уточнення)
- Резюме для тих хто не був на зустрічі

### На що звернути увагу

- Це **внутрішній** документ команди (не для замовника — для замовника є протокол, крок 2)
- Якщо транскрипт неповний або тільки нотатки — скіл позначить обмеження

---

## Крок 2 — meeting-protocol-builder

**Що робить:** перетворює внутрішній аналіз на **офіційний документ для замовника** — Протокол обстеження
(проведення інтерв'ю): учасники, порядок денний, результати, прийняті рішення, погодження. Має вбудований
крок **Humizer** — олюднення формулювань, щоб усе було зрозуміло замовнику.  
**Універсальний:** працює не лише для Установчої — на будь-яку зустріч (збір вимог, статусна, інтеграції,
навчання, демо, фінанси), адаптуючи порядок денний і формат РЕЗУЛЬТАТІВ під тип. Калібровано на 41 реальному
протоколі EDELWEIS з виправленнями методолога (v0.7.x): РЕЗУЛЬТАТИ — за темами порядку денного, дзеркально 1:1;
«Обговорювали» проставляється («Так»/«Ні»); рішення — **рядками в таблиці «ВИРІШИЛИ»** (№/Рішення/Термін/Відповідальний);
у «Погодженні» вписується ПІБ погоджувача від замовника (відмітку ставить замовник); «Додаток 1» — порожній. Без метафор, розмовних вставок і чутливого (ставки, внутрішні умови виконавця).  
**Час:** 15–25 хвилин

### Як запустити

Напиши:
- "оформи протокол обстеження для [назва компанії]"
- "склади протокол інтерв'ю для замовника [назва компанії]"
- "протокол зустрічі для замовника"

### Що потрібно мати

Обов'язково:
- `аналіз_Установчої Зустрічі_[назва_компанії].md` (з кроку 1) — або нотатки зустрічі

Бажано:
- `підготовка_УЗ` / `agenda` / `питання_УЗ` — для мети, учасників і порядку денного
- `бриф_[назва_компанії].md` — назва замовника, проект, № договору, контакти

### Що отримаєш

Два файли:
- `протокол_[тип]_[дата]_[компанія].md` — робоча версія (і для команди)
- `протокол_[тип]_[дата]_[компанія].docx` — брендований документ ToDo (шапка, логотип, таблиці) для замовника

Розділи: Загальна інформація, Учасники, Порядок денний, Результати інтерв'ю, Вирішили, Погодження, Додаток 1 (опис бізнес-процесу).

### На що звернути увагу

- Це **документ для замовника** — жодних внутрішніх маркерів (🔴, [КД], «scope», оцінки людей). Це робить крок Humizer.
- Порожні поля (Дата, Місце, № договору) скіл лишає як `‹заповнити›` — додай вручну перед відправкою.
- Факти (дати, імена, домовленості) Humizer не змінює — лише спрощує мову.

---

## Крок 3 — discovery-plan-builder

**Що робить:** будує структурований план дискавері — скільки сесій, які теми,
в якому порядку, хто потрібен, що є deliverable  
**Час:** 20–30 хвилин

### Як запустити

Напиши:
- "побудуй план дискавері для [назва компанії]"
- "скільки сесій потрібно для дискавері [назва компанії]"
- "склади план дискавері"

### Що потрібно мати

Обов'язково:
- `аналіз_Установчої Зустрічі_[назва_компанії].md` (з кроку 1)

Бажано:
- `підготовка_УЗ_[назва_компанії].md` — для контексту scope
- `питання_УЗ_[назва_компанії].md` — заповнені відповіді за входами плану

### Що отримаєш

Файл `план_дискавері_[назва_компанії].md` з:
- Кількість і структура сесій (Блок A / B / C)
- Деталі кожної сесії: ціль, питання, очікуваний результат, учасники
- Ключові рішення і залежності
- Deliverables по завершенні дискавері

### На що звернути увагу

- План — гіпотеза. Він уточнюється після кожної сесії дискавері
- Якщо аналізу Установчої Зустрічі немає — план можна будувати на `підготовка_УЗ`, але він буде менш точним

## Типові сценарії

### Сценарій А: є транскрипт і повний пакет підготовки
→ Запускай усі 3 кроки по черзі. Займе 45–70 хвилин.

### Сценарій Б: замовник чекає офіційний протокол зустрічі
→ Кроки 1 + 2: аналіз → протокол обстеження (.docx). Надішли замовнику на погодження, план — пізніше.

### Сценарій В: є тільки нотатки (не повний транскрипт)
→ Крок 1 спрацює, але позначить обмеження. Кроки 2 і 3 будуються на тому що є.

### Сценарій Г: Установча Зустріч ще не відбулась або не буде
→ Пропусти крок 1, запускай крок 3 напряму на основі `підготовка_УЗ`. Скіл попередить про відсутність аналізу.

### Сценарій Д: замовник відмовився від повноцінного дискавері
→ Крок 2 (протокол — зафіксувати домовленості) + крок 3 з мінімальним контекстом (базовий план).

## Часті запитання

**Чим протокол обстеження відрізняється від аналізу Установчої Зустрічі?**  
`аналіз_УЗ` — внутрішній документ команди (з ризиками, оцінками, внутрішніми маркерами). Протокол обстеження —
офіційний документ для замовника на фірмовому бланку, простою зрозумілою мовою (крок Humizer), з розділом «Погодження».

**Що робить крок Humizer?**  
Переписує формулювання так, щоб їх зрозумів замовник без IT-бекграунду: прибирає жаргон і внутрішні маркери,
розкриває скорочення, пояснює терміни — але не змінює факти.

**Чи можна запустити план дискавері без аналізу Установчої Зустрічі?**  
Так. Але план буде менш точним — частину питань доведеться залишити відкритими.

**Чи план дискавері фіксований?**  
Ні. Це версія 1.0. Після кожної сесії дискавері план оновлюється (версії 1.1, 1.2...).

**Що після блоку 3?**  
Блок ІНІЦІАЦІЯ завершено. Наступний етап — цикл зустрічей дискавері (плагін `odoo19-discovery-sessions`): "підготуй питання до сесії 1" — скіл `session-question-builder` сам збере журнал циклу з плану, аналізу УЗ і протоколів.
