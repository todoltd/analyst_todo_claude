# MVP / БІЛД — Майстер-інструкція

**Для кого:** розробник Odoo, архітектор, BA (контроль відповідності ТР)
**Коли:** від затвердженого ТР (реєстр «Перелік технічних рішень» → «Погодження клієнта») до змердженого, протестованого `td`-модуля
**Результат:** робочий модуль Odoo 19, покритий тестами, що відповідає AC затвердженого ТР

---

## Модель (Варіант А)

ТР — єдине джерело істини. Пакет не переписує вимоги, а **проектує затверджений ТР**
у машинно-читний `SPEC.md` і веде його до коду. `approved`-гейт SDD = ваш статус
реєстру «Погодження клієнта».

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|---|---|---|
| Є затверджений ТР, готуємо SPEC | "збери SPEC з ТР [назва]" | `SPEC.md` (status approved, AC-01) |
| Модуль уже існує, нема SPEC | "зроби reverse-spec для модуля [шлях]" | `SPEC.md` з коду |
| SPEC готовий, потрібен дизайн | "згенеруй ARCHITECTURE для [модуль]" | `ARCHITECTURE.md` |
| Дизайн готовий, потрібен скелет | "згенеруй скелет модуля [модуль]" | файли модуля з `# TODO: AC-01` |
| Реалізувати критерій | "реалізуй AC-03 у [модуль]" | код одного AC + прибраний TODO |
| Потрібні тести | "напиши тести для [модуль]" | тести (1 на AC) + аудит покриття |
| Перевірити PR | "перевір PR проти SPEC" | вердикт APPROVE / REQUEST CHANGES / BLOCK |

## Ланцюжок і гейти

```
ТР «Погодження клієнта»
  → (reverse-spec, якщо td_-модуль існує)
  → spec-writer: ТР → SPEC.md           [гейт: ТР approved]
  → architecture: SPEC → ARCHITECTURE   [гейт: SPEC approved, 0 open questions]
  → scaffolding: скелет + # TODO: AC-01 [гейт: SPEC + ARCHITECTURE approved]
  → code-dev (по 1 AC) ⇄ testing (1 тест/AC)
  → pr-review → APPROVE → merge → реєстр «Виконано»
```

Гейти жорсткі: `architecture`/`scaffolding` не стартують без `approved` SPEC;
`pr-review` ставить **BLOCK** на будь-яке відхилення коду від ТР/SPEC. AI не
сам-approve — затвердження ТР робить клієнт.

## AC-01 — наскрізна трасування

`AC-01` веде від ТР до тесту: `# TODO: AC-01` → докстрінг коду → `test_ac01_…` →
чек pr-review. **Передумова:** `tr-usecases-acceptance` має давати AC у форматі
`AC-01` (див. `README_інтеграція.md`).

## Зв'язок із реєстром «Перелік технічних рішень»

- ТР «Погодження клієнта» = SPEC `approved` = старт білду.
- «Виконано» = після pr-review APPROVE і зелених тестів (`tr-registry-update`).

## Чого НЕ робити

- Не писати вимоги/AC на цьому етапі — це шар `tr-*` (аналіз/ТР).
- Не додавати поля/моделі поза ТР — спершу оновити ТР.
- Не перенумеровувати AC; не стартувати `architecture` без approved SPEC.
