# ДИСКАВЕРІ (цикл зустрічей) — Майстер-інструкція

**Для кого:** BA, менеджер проекту
**Коли:** від затвердженого плану дискавері до закриття всіх тем циклу
**Результат кожної ітерації:** опитувальник сесії + лист замовнику + актуальний журнал циклу

---

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|----------|-----------------|-------------|
| Готуєшся до сесії (першої чи будь-якої наступної) | "підготуй питання до сесії [N] для [Назва компанії]" | `питання_сесії-[N]_[компанія].md` + `журнал_дискавері_[компанія].md` |
| Треба надіслати замовнику підготовку | "лист замовнику до сесії [N] для [Назва компанії]" | робочі .docx для зустрічі (`питання_сесії-[N]_[компанія].docx` + `план_дискавері_[компанія].docx`, якщо оновився) + `підготовка_замовника_сесії-[N]_[компанія].md` + `.docx` |
| Зустріч позапланова | "підготуй питання до зустрічі про [тема]" | `питання_сесії-[дата]_[компанія].md` (тема — у журнал) |
| Після зустрічі — протокол замовнику | "оформи протокол обстеження для [Назва компанії]" | `протокол_[тип]_[дата]_[компанія].md` + `.docx` (скіл ІНІЦІАЦІЇ) |
| Після зустрічі — обіцяні замовнику шаблони | "підготуй обіцяні шаблони для [Назва компанії]" | список обіцяних артефактів (A-N) + згенеровані Excel-форми де можливо + запис у журнал (розділ 8) |
| Після зустрічі — переглянути план дискавері | "перевір, чи треба оновити план дискавері для [Назва компанії]" | перелік сигналів зі журналу + рекомендація: оновити план вручну / перезапустити `discovery-plan-builder` / лишити як є |

## Рекомендований цикл (кожна зустріч)

```
1. session-question-builder    → опитувальник + журнал (bootstrap або reconcile — автоматично)
2. BA правит опитувальник (прибрати/додати питання — за потреби)
3. session-client-prep-builder → робочі .docx на зустріч (опитувальник + план дискавері, якщо оновився) + лист замовнику (.md + .docx) → лист надіслати за 2–3 робочі дні до зустрічі
4. ЗУСТРІЧ — відповіді фіксуються прямо в питання_сесії-[N]
5. meeting-protocol-builder    → протокол замовнику на погодження
6. Обіцяні шаблони (опція BA) → "підготуй обіцяні шаблони для [компанія]": Claude витягує з протоколу
   обіцяні артефакти, генерує що може (Excel-форми) і фіксує в журналі (розділ 8, A-N)
7. Перегляд плану дискавері (опція BA) → "перевір, чи треба оновити план дискавері для [компанія]":
   Claude дає сигнали зі журналу, BA вирішує — оновлювати план чи лишити
8. Ще зустрічі? → наступна ітерація з кроку 1 (скіл сам внесе нові протоколи в журнал)
```

## Журнал дискавері — пам'ять циклу

`журнал_дискавері_[компанія].md` створюється автоматично на першій ітерації і оновлюється на кожній наступній:

- **розділ 1** — реєстр сесій: статуси (запланована/проведена/скасована), дати, протоколи;
  позапланові зустрічі — рядком «поза планом»
- **розділ 2** — статус тем плану (✅ закрито / 🔶 частково / ⬜ не починали)
- **розділ 3** — перенесені питання (C-N): підуть першими в наступний опитувальник
- **розділ 4** — виключено зі scope (X-N): чинні виключення скіли не питають; скасоване замовником
  виключення (рішення «все ж робимо») повертає тему в роботу
- **розділи 5–7** — рішення (D-N), нові теми (T-N), відкриті питання (Q-N)
- **розділ 8** — обіцяні артефакти (A-N): що МИ обіцяли надати замовнику (шаблони/форми/зразки), статус обіцяно→надіслано

Журнал можна правити руками — скіли це переживають. Головне: виключення (X-N) — лише з посиланням
на протокол; «Враховані джерела» в шапці не редагувати без потреби.

## Обіцяні шаблони/артефакти (опція після кожної зустрічі)

На зустрічі BA часто обіцяє замовнику надіслати робочі матеріали — шаблон для збору даних, форму, зразок
документа, мапу процесу. Цей крок збирає такі обіцянки в одному місці, готує що можна автоматично і
трекає їх до закриття. Опція, одразу після протоколу.

**Як запустити:** "підготуй обіцяні шаблони для [компанія]" (або "що ми обіцяли замовнику на сесії [N]").

**Що робить Claude:**

1. Витягує з протоколу зустрічі action items, де відповідальний — МИ (виконавець), а суть — надати замовнику
   шаблон/форму/зразок/мапу. BA доповнює перелік тим, що не потрапило в протокол.
2. Генерує те, що може автоматично — насамперед Excel-форми збору даних (номенклатура, контрагенти,
   залишки, користувачі): шапка колонок, приклад рядка, підказки заповнення. Складніші чи специфічні
   артефакти лишає в списку зі статусом «обіцяно» для ручної підготовки BA.
3. Фіксує кожен артефакт у журналі — розділ 8, A-N (обіцяно / згенеровано), з адресатом і терміном.
4. За потреби — чернетка супровідного листа замовнику з переліком вкладень.

**Після надсилання** BA позначає A-N статусом «надіслано (дата)» — вручну в журналі або це підхопить
наступний reconcile із протоколу. Незакриті A-N скіл показує в «Контексті циклу» наступної сесії як
борг виконавця перед замовником.

**Межа:** обіцянки ЗАМОВНИКА надати нам дані — це не A-N, а відкриті питання (C-N/Q-N «чекаємо…»).

## Перегляд плану дискавері (опція після кожної зустрічі)

Після кожної зустрічі збору вимог — перш ніж готувати наступну сесію — BA звіряє план дискавері з
тим, що накопичив журнал. Це **опція, не обов'язок**: план оновлюємо лише коли для цього є привід.
Журнал перераховується щоразу (reconcile), а план — стабільний кістяк, який міняється точково.

**Як запустити:** "перевір, чи треба оновити план дискавері для [компанія]". Claude читає журнал
(і свіжий протокол) і повертає перелік сигналів — рішення лишається за BA. Сам план Claude не редагує.

**Сигнали, що план варто переглянути:**

- **Нові теми поза планом** (журнал, розділ 6 / T-N) — кандидати в нову або окрему сесію.
- **Перенесені питання накопичуються** (розділ 3 / C-N) — тема не закривається за відведену сесію → можливо потрібна додаткова.
- **Зміни scope** (розділ 4): нове виключення X-N (тема зникає з плану) або скасоване виключення (тема повертається в роботу).
- **Статус тем зрушив** (розділ 2): тема закрилась швидше або виявилась ширшою → перерозподіл сесій.
- **Зсув дат** (розділ 1) — графік сесій розійшовся з планом.
- **Нові рішення** (розділ 5 / D-N), що змінюють напрям дискавері.

**Якщо вирішили оновлювати:**

- *Дрібні правки* (додати сесію, зрушити дату, уточнити фокус) — BA редагує `план_дискавері_[компанія].md`
  вручну і піднімає версію (напр. 1.0 → 1.1). Наступний `session-question-builder` побачить новішу версію
  і звірить журнал з планом сам.
- *Суттєве перепланування* — перезапустити `discovery-plan-builder` (скіл ІНІЦІАЦІЇ) з урахуванням журналу.

Номери вже проведених сесій не перенумеровуються — нові отримують наступний вільний № (контракт журналу,
правило 9). Перегляд виключень X-N — окрема розмова BA з замовником, не питання на сесії.

## Перша ітерація (одразу після ІНІЦІАЦІЇ)

Потрібні в outputs: `план_дискавері_[компанія].md` (наполегливо рекомендовано; без нього скіл працює
в деградованому режимі — без карти покриття, файл з датою замість N), `аналіз_Установчої Зустрічі_…`,
заповнений `питання_УЗ_…`, `протокол_установча_…`. Скіл збере з них журнал сам — окремих дій не треба.

## Чого НЕ робити

- Не надсилати замовнику `питання_сесії-[N]` — для замовника є лист від `session-client-prep-builder`
- Не питати на сесії про виключене зі scope (X-N) — перегляд виключень це окрема розмова BA
- Не вести два журнали на одну компанію — джерело істини одне
