# Рантайм-тест циклу: 3 ітерації (виконати після інсталяції плагіна)

Головний e2e-тест пакета `odoo19-discovery-sessions`. Перевіряє пам'ять циклу:
bootstrap → carry-over → anti-repeat → стійкість до ручних правок.
Фікстури — за мотивами реального кейса SUPPLYCHAINS (установча 03.06.2026).

## Підготовка: покласти в outputs фікстури

1. `план_дискавері_ТестЗамовник.md` — 3 сесії: 1) Продажі та агентська закупівля, 2) Логістика
   (маршрути, стейтменти), 3) Інтеграції (1С, митниця, CRM); у кожної — ціль і 5 питань-заготовок;
   розділ «Ключові рішення і залежності» — 1 рішення-блокер.
2. `аналіз_Установчої Зустрічі_ТестЗамовник.md` — 2 відкриті питання: [КД] «Як нараховується комісія
   агента при частковій доставці?» і [ВД] довільне; у «Зміни scope» — 1 звуження («бухоблік лишається
   у провайдера»).
3. `питання_УЗ_ТестЗамовник.md` (заповнений) — «Парковка для дискавері»: 2 теми з прив'язкою до сесій.
4. `протокол_установча_2026-06-03_ТестЗамовник.md` — таблиця ВИРІШИЛИ з 2 рішеннями.

## Ітерація 1 — bootstrap

**Запуск:** "підготуй питання до сесії 1 для ТестЗамовник"

**Очікувано:**
- [ ] Створено `журнал_дискавері_ТестЗамовник.md`: розділ 1 — 3 сесії (УЗ у розділі 1 НЕМАЄ — вона
      не зустріч циклу); розділ 2 — всі теми ⬜; розділ 4 — X-1 (бухоблік, статус «чинне», джерело —
      аналіз/протокол УЗ); розділ 5 — D-1, D-2; розділ 7 — питання з аналізу УЗ + 2 з парковки
      + 1 рішення-блокер як [КД], у кожного Q-N заповнена колонка «Сесія (план)»; [КД] з аналізу УЗ
      збережено як критичне до сесії 1
- [ ] `питання_сесії-1_ТестЗамовник.md`: контекст циклу = «перша сесія»; секція 0 = «Перенесених немає»;
      питання нумеровані наскрізно (1, 2, 3…); питання-заготовки сесії 1 розгорнуті;
      жодного питання про бухоблік (X-1 чинне)
- [ ] Кожне 🔴 питання має 2–3 навідні («Навідні:» під питанням), конкретні під ТестЗамовник
      (не абстрактні шаблони), прийоми чергуються
- [ ] Питання з парковки УЗ, прив'язані до сесії 1, — в опитувальнику

## Між ітераціями: фікстура результатів сесії 1

5. `протокол_збір-вимог_2026-06-09_ТестЗамовник.md` — ВИРІШИЛИ: 1) майстер-система — Odoo;
   2) «все ж переносимо бухоблік Гонконгу в Odoo» (скасування виключення X-1);
   текстом: «повернення агентів — окрема тема» (нова тема); «тарифікацію агентів відклали —
   чекаємо приклади рахунків» (перенесення); питання про комісію агента лишилось без відповіді
   (воно вже існує в журналі як Q-N з аналізу УЗ — перевірка дедуплікації);
   action item ВИКОНАВЦЯ: «надамо замовнику шаблон Excel для збору номенклатури до 13.06» (обіцяний артефакт).
6. `питання_сесії-1_ТестЗамовник.md` — частину полів «Відповідь:» заповнити, 2 питання лишити порожніми.

## Ітерація 2 — reconcile + carry-over

**Запуск:** "підготуй питання до сесії 2 для ТестЗамовник"

**Очікувано:**
- [ ] Журнал оновлено БЕЗ перезатирання: «Враховані джерела» + протокол 09.06
- [ ] **Розділ 1: сесія 1 → статус «проведена», дата 09.06.2026, файл протоколу в колонці «Протокол»**
- [ ] Розділ 4: X-1 → статус «скасовано (09.06.2026, протокол)», тема бухобліку повернута
      в розділ 2 (⬜/🔶) або розділ 7 — питання про бухоблік далі ДОЗВОЛЕНІ
- [ ] Розділ 3: C-1 (тарифікація — чекаємо рахунки); питання про комісію агента НЕ продубльовано
      як C-N — натомість Q-N доповнено приміткою «ставилось на сесії 1» (дедуплікація)
- [ ] Розділ 5 — D-3, D-4; розділ 6 — T-1 (повернення агентів); розділ 2 — тема 1 → 🔶 з приміткою
- [ ] **Розділ 8: A-1 (шаблон номенклатури) статус «обіцяно», адресат + термін 13.06**; обіцянка
      ЗАМОВНИКА (приклади рахунків) НЕ в розділі 8 — вона лишається C-1
- [ ] Контекст циклу опитувальника сесії 2 містить незакритий A-1 як борг виконавця
- [ ] Крок «обіцяні шаблони»: "підготуй обіцяні шаблони для ТестЗамовник" → A-1 у журналі + згенеровано
      `шаблон_номенклатура_ТестЗамовник.xlsx` (колонки SKU/од.виміру/штрихкод + приклад рядка); статус A-1 → «згенеровано»
- [ ] `питання_сесії-2_…`: секція 0 відкривається перенесеним C-1 і Q-N про комісію (з ID, один раз);
      anti-repeat: немає питань про вирішене (D-1…D-4); підсумок anti-repeat присутній
- [ ] Лист: "лист замовнику до сесії 2" → спершу створено `питання_сесії-2_ТестЗамовник.docx`
      (опитувальник 1:1, з тегами/ID/навідними — без Humizer; на робочому бланку ToDo:
      обкладинка «ОПИТУВАЛЬНИК СЕСІЇ 2» ціла, рендер без накладань) і `план_дискавері_ТестЗамовник.docx`
      (теж на робочому бланку, обкладинка «ПЛАН ДИСКАВЕРІ»)
      (першого запуску .docx плану ще немає → створюється), потім `підготовка_замовника_сесії-2_…` .md + .docx:
      приклади рахунків (C-1) — у «Що підготувати»; фіндиректор — у «Кого запросити» як критичний;
      «Мета зустрічі» однакова в .md і таблиці .docx; нуль внутрішніх маркерів/ID;
      нуль ‹плейсхолдерів› у .docx (окрім легітимного ‹уточнюється› в даті, якщо дата невідома); ≤ 1 МБ
- [ ] Рендер обкладинки листа коректний: заголовок «ПІДГОТОВКА ДО ЗУСТРІЧІ» + назва проєкту видно,
      текст не накладається на фото/смуги/логотип, контент починається зі сторінки 2

## Ітерація 3 — стійкість до ручних правок

7. Вручну в журналі: C-1 → статус «закрито (сесія 2)»; додати руками рядок X-2 (статус «чинне»);
   A-1 → статус «надіслано (12.06.2026)».

**Запуск:** "підготуй питання до сесії 3 для ТестЗамовник"

**Очікувано:**
- [ ] Скіл пережив ручні правки (читає за заголовками розділів, не за точним текстом)
- [ ] Закритий вручну carry-over НЕ з'являється в секції 0; питань по X-2 (чинне) немає
- [ ] A-1 зі статусом «надіслано (12.06.2026)» збережено — скіл не перезатирає ручну правку розділу 8
- [ ] Позапланова перевірка: "підготуй питання до зустрічі про повернення агентів" →
      файл з датою замість N; у розділі 1 журналу — рядок «поза планом» (тема, дата, «запланована»);
      розділ 6 НЕ дублюється (використано наявний T-1)
- [ ] Лист для позапланової: вихідні файли теж з датою замість N (дзеркально до опитувальника)

## Граничні випадки (окремо)

- [ ] Порожньо: видалити всі фікстури → "підготуй питання до сесії" → скіл НЕ вигадує,
      питає про план/тему (поведінка Кроку 0)
- [ ] Два замовники в outputs → скіл уточнює, для якого
- [ ] План v1.1 (вставити нову сесію МІЖ наявними) → reconcile додає її з наступним вільним №

**Критерій проходження:** усі чекбокси; знахідки → раунд правок → повторний прогін зміненого кроку.
