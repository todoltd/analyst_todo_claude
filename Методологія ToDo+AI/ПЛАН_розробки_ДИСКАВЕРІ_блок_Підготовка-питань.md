## 1. Місце блоку в процесі

Канонічна BPMN, цикл зустрічей дискавері:

```
БА Планування (план_дискавері готовий — фінал ІНІЦІАЦІЇ)
    ↓
┌─→ ПІДГОТОВКА ПИТАНЬ (Claude)  ← ЦЕЙ БЛОК
│       ↓ (2 файли: опитувальник БА + лист клієнту)
│   Порядок денний (БА — доповнює, фіналізує, розсилає)
│       ↓
│   ЗУСТРІЧ (запис + транскрипт)
│       ↓
│   Аналіз транскрипту (Claude)  ← наступний блок, ще не розроблений
│       ↓
│   Протокол (Claude — вже покрито `meeting-protocol-builder`)
│       ↓
│   Review (БА)
│       ↓
└─ Так ── Шлюз «Ще зустрічі?» ── Ні → Маппінг scope (вихід з циклу)
```

**Ключова особливість блоку — він циклічний.** Виконується перед КОЖНОЮ зустріччю:

- **Ітерація 1** (перша зустріч після УЗ): контексту циклу ще немає. Входи — `план_дискавері`, `аналіз_Установчої Зустрічі`, заповнений `питання_УЗ` (включно з секцією «Парковка для дискавері» — її питання зобов'язані потрапити у відповідні сесії).
- **Ітерація N**: додаються протоколи і результати попередніх зустрічей. Скіл зобов'язаний знати: що **вже обговорено** (не перепитувати), що **не встигли / перенесли** (поставити першим), що **виключили зі scope** (не питати взагалі).

Носій цього знання — журнал (розділ 2).

## 2. Артефакти і контракти

### 2.1 Карта файлів

| Файл | Пише | Читає | Призначення |
|---|---|---|---|
| `журнал_дискавері_[компанія].md` | скіл питань (bootstrap + reconcile); згодом — скіл аналізу транскрипту (основний writer) | обидва скіли блоку, майбутні скіли циклу | Стан циклу: статус тем, перенесення, виключення, рішення |
| `питання_сесії-[N]_[компанія].md` | скіл 1 | БА (на зустрічі), скіл 2, майбутній аналіз транскрипту | Повний внутрішній опитувальник сесії. Тільки команда |
| `підготовка_клієнта_сесії-[N]_[компанія].md` + `.docx` | скіл 2 | клієнт (до зустрічі) | Що підготувати, кого запросити. Надсилається заздалегідь |

Іменування `сесії-[N]` — номер сесії з плану дискавері; для позапланової зустрічі — дата замість N (за зразком `протокол_[тип]_[дата]_[компанія]`).

### 2.2 Контракт журналу (проєктується ПЕРШИМ — від нього залежать обидва скіли)

Розділи журналу:

1. **Шапка** — компанія, версія плану дискавері, дата останнього оновлення, список джерел (які протоколи/аналізи вже враховано). Дата + перелік джерел = механізм виявлення відставання.
2. **Реєстр сесій** — N | тема | дата | статус (запланована / проведена / скасована) | посилання на протокол.
3. **Статус тем плану** — кожна тема/наскрізна лінія з плану дискавері: ✅ закрита / 🔶 частково (що лишилось) / ⬜ не починали | у якій сесії обговорено.
4. **Перенесені питання (carry-over)** — питання | з якої сесії | чому не закрито | пріоритет. Кандидати №1 у наступний опитувальник.
5. **Виключено зі scope** — що | коли вирішили | джерело (протокол №). Анти-список: ці теми в питання не потрапляють.
6. **Реєстр рішень** — однорядкові рішення з посиланням на протокол (щоб не перепитувати вирішене).
7. **Нові теми поза планом** — що з'явилось у ході циклу; сигнал БА оновити план дискавері.
8. **Відкриті питання** — наскрізний реєстр із пріоритетами `[КД]/[ВД]/[ПП]` (успадковує глосарій ІНІЦІАЦІЇ; саме `[ВД]` = «важливо під час дискавері» — код етапу дискавері, тоді як `[ВК]` лишається кодом етапу зустрічі в скілах ІНІЦІАЦІЇ. При перенесенні питань із `підготовка_УЗ` код переоцінюється — правило вже є в `kickoff-transcript-analyzer`).

Правила: наскрізні ID (як формат ризиків `R-N` у ІНІЦІАЦІЇ); записи стискаються до рядка + посилання на протокол (журнал — індекс, не архів); правки руками БА дозволені — скіли мусять переживати ручне редагування.

**Життєвий цикл:**
- **Bootstrap** — журналу немає → скіл 1 створює його з `план_дискавері` + `аналіз_УЗ` + парковки `питання_УЗ`.
- **Reconcile** — журнал є, але в outputs з'явились протоколи, яких немає в його списку джерел → скіл 1 спочатку доносить їх у журнал, потім генерує питання.
- **Update** (майбутнє) — після розробки блоку «Аналіз транскрипту» основним writer'ом стає він; контракт журналу проєктуємо вже зараз із цим розрахунком — щоб наступний блок вписався без зміни формату.

## 3. Етапи розробки

### Етап 0 — каркас пакета (0.5 сесії)

- Правило інсталяції незмінне: `.plugin` доставляти через папку outputs сесії.

### Етап 1 — контракт журналу (1 сесія)

- Спроєктувати шаблон журналу (розділ 2.2) як файл-зразок із заповненим прикладом.
- Прописати bootstrap- і reconcile-сценарії як алгоритми (стануть Кроками скіла 1).
- Перевірка: вручну зібрати журнал на реальному кейсі (SUPPLYCHAINS: аналіз УЗ + протокол від 03.06.2026) — чи лягають реальні дані в структуру без натягування.

### Етап 2 — скіл 1: `session-question-builder` (1.5–2 сесії)

Внутрішній опитувальник сесії. Каркас кроків — за зразком `kickoff-question-builder`, з відмінностями:

- **Крок 0 (входи):** план дискавері (обов'язково), журнал (bootstrap/reconcile за відсутності/відставання), аналіз УЗ + питання_УЗ (перша ітерація), протоколи. Кілька компаній в outputs → запитати. Немає плану → працювати від теми, яку назвав БА, з попередженням.
- **Крок «Контекст циклу»:** зведення з журналу на початку файлу — що закрито, що перенесено сюди, що виключено. БА бачить стан циклу без читання журналу.
- **Крок «Питання»:** розгортання «Питань для дослідження» сесії з плану (5–10) у повний опитувальник (орієнтир 15–25 на 1–2 год сесії — відкалібрувати на тесті). Формат успадковується: пріоритети 🔴/🟡/⚪, поле «Відповідь:», теги (`[процеси]`, `[дані]`, `[інтеграції]`, `[ролі]`, `[NFR]`, `[go-live]`…), але прив'язка тегів — до deliverables дискавері (FRD, AS-IS карта, gap-list), а не до входів плану.
- **Перенесені питання — першою секцією** (борг попередньої зустрічі).
- **Anti-repeat-перевірка:** жодне питання не дублює закрите (розділ 3 журналу), вирішене (розділ 6) чи виключене (розділ 5). Дублі — або прибрати, або явно позначити «повторне підтвердження: [причина]».
- **Карта покриття цілі сесії:** замість «карти покриття плану» УЗ — чи закривають питання «Ціль сесії» та «Очікуваний результат» із плану дискавері.
- **Парковка:** питання не цієї сесії → у яку сесію за планом.
- **Важлива відмінність від УЗ:** обмеження «тільки scope, без глибини» НЕ діє — сесії дискавері і є місцем глибини (міграція, кількість користувачів, матриці прав — тепер можна). Лишається чинним фільтр комерційно чутливого для клієнтських матеріалів (правило v0.7.3).
- Pre-save верифікація (чекліст за зразком ІНІЦІАЦІЇ) + підказка наступного кроку (скіл 2).

### Етап 3 — скіл 2: `session-client-prep-builder` (1–1.5 сесії)

Лист клієнту для підготовки до зустрічі. Похідний від опитувальника сесії:

- **Вхід:** `питання_сесії-[N]` (після правок БА — тому окремий запуск, а не побічний продукт скіла 1).
- **Відбір:** тільки те, до чого клієнт реально може підготуватись — документи/вивантаження, приклади (рахунки, накладні, прайси), цифри, доступи, **хто потрібен на зустрічі** (ролі з плану сесії). Внутрішні питання («Навіщо», теги, пріоритети) не витікають.
- **Структура:** тема і мета зустрічі → що підготувати (список) → над чим подумати (3–7 простих питань) → кого запросити → дата/тривалість.
- **Humizer** — переюз каліброваних правил з `meeting-protocol-builder` (стриманий діловий тон, без метафор, пояснення термінів у дужках).
- **Фільтр чутливого (v0.7.3)** — ставки, внутрішні практики виконавця, захисні умови в клієнтський документ не потрапляють.
- **Вихід:** `.md` + брендований `.docx`. Задача всередині етапу: зробити з бланку протоколу (`assets/protocol_template.docx`, 750 КБ) полегшений шаблон листа (шапка/лого без протокольних таблиць), вшити у скіл як `assets/`.
- Обсяг — 1 сторінка: клієнт має підготуватись за 15–30 хв, це не анкета.

### Етап 4 — інтеграція з наявною методологією (0.5 сесії)

- `discovery-plan-builder` («Наступний крок») → згадати запуск циклу: «перед першою сесією — питання сесії-1».
- `meeting-protocol-builder` → у входи додати заповнений `питання_сесії-[N]` (аналогічно питання_УЗ для установчої). Це правка плагіна ІНІЦІАЦІЇ → підняти його patch-версію за стандартним циклом.
- Тригери скілів розвести з `kickoff-question-builder`: там — «Установча Зустріч», тут — «сесія», «зустріч дискавері», «наступна зустріч». У description обох — явне розмежування (відомий ризик конфлікту тригерів — прецедент із дублем `sales-handover-analyzer`).
- Майстер-ІНСТРУКЦІЯ.md етапу ДИСКАВЕРІ (стартова версія: цикл + 2 скіли), README, docs/ плагіна.

### Етап 5 — QA за прийнятим процесом (1.5–2 сесії)

1. **Валідатор:** адаптувати `валідатор_скілів.py` (Архів) під новий пакет — заборонені терміни, парність code-fence, «з'їдені імена», крос-посилання, збіг плагіна з джерелом. Редагування шаблонних блоків — тільки закавичені heredoc / Write (урок «з'їдених імен»).
2. **4 ітерації перевірки** (механіка → структура → доменна якість → рантайм) у стислому варіанті для 2 скілів.
3. **Скептик-рев'ю** (лінза senior BA / Odoo-впроваджувача).
4. **Головний e2e-тест — тест циклу, 3 ітерації поспіль:** сесія-1 (bootstrap) → фіктивний протокол → сесія-2 (reconcile: carry-over + виключення зі scope) → ручна правка журналу БА → сесія-3 (перевірка: нічого закритого/виключеного не перепитано, перенесене — першим). Фікстури — реальні протоколи EDELWEIS (41 шт.) і кейс SUPPLYCHAINS.
5. **Граничні випадки:** порожній вхід (не вигадувати — питати), позапланова зустріч, план оновився до v1.1 (звірка журналу з планом), два клієнти в outputs.

### Етап 6 — пакування і синхронізація (0.5 сесії)

За правилом 4 CLAUDE.md, у порядку: скіли → інструкції (docs/ + авторське дерево) → BPMN (канон: позначити вузли «Підготовка питань» статусом скілів + панель деталей із назвами скілів і файлів; синхронізувати копію) → майстер-інструкція ДИСКАВЕРІ → запис в ІСТОРІЮ. Збірка `.plugin` v0.1.0 → інсталяція через outputs → smoke-тест тригерів.

## 4. Ризики

| Ризик | Мітигація |
|---|---|
| Конфлікт тригерів із `kickoff-question-builder` («питання до зустрічі» — обидва) | Явне розмежування в description; рантайм-тест обох фраз на встановлених плагінах |
| Журнал розійшовся з реальністю (протокол не внесли) | Reconcile-крок у скілі 1 + список джерел у шапці журналу |
| Журнал розпухає з кожною зустріччю | Записи = рядок + посилання на протокол; деталі живуть у протоколах |
| Клієнтський лист зливає внутрішнє/чутливе | Окремий скіл (не побічний вихід), фільтр v0.7.3, Humizer, QA-пункт у pre-save |
| Наступний блок циклу (аналіз транскрипту) не впишеться в журнал | Контракт журналу проєктується на Етапі 1 одразу під двох writer'ів |
| `.docx`-бланк роздуває плагін | Прецедент вирішено у v0.5.1–v0.7.0: оптимізоване лого, шаблон ≤ 1 МБ |

## 6. Definition of Done

- [ ] Журнал: шаблон + bootstrap + reconcile, перевірений на реальному кейсі
- [ ] Скіл 1 генерує опитувальник для ітерації 1 і ітерації N (carry-over першим, anti-repeat працює)
- [ ] Скіл 2 дає .md + брендований .docx ≤ 1 МБ без чутливого
- [ ] Тест циклу на 3 ітерації пройдено; граничні випадки — за списком Етапу 5
- [ ] Валідатор зелений; плагін v0.1.0 встановлюється через outputs, тригери не конфліктують з ІНІЦІАЦІЄЮ
- [ ] BPMN (канон + копія), інструкції, README, ІСТОРІЯ — оновлені синхронно
- [ ] Польовий тест на найближчій реальній зустрічі (SUPPLYCHAINS) → раунд правок методолога

**Оцінка разом: ~6–8 робочих сесій.**
