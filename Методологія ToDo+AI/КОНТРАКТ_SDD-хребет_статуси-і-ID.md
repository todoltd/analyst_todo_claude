# Контракт SDD-хребта: статуси + ID-namespace

Нормативні правила, що зшивають етапи методології (ІНІЦІАЦІЯ → ДИСКАВЕРІ → ТР → MVP) в один
трасований конвеєр. Походить з `КАРТА_SDD-хребет_методології.md` (важіль P1). Це джерело істини
для майбутніх правок скілів. Застосовувати **прогресивно**: легко в дискавері, жорстко на білді.

## 1. Єдиний словник статусів

Канон (5 значень) — для будь-якого артефакта-результату (бриф, аналіз УЗ, план дискавері, ТР,
SPEC, ARCHITECTURE, модуль):

| Статус | Значення | Хто ставить |
|---|---|---|
| `чернетка` | у роботі, неповний | автор / Claude |
| `на розгляді` | готовий, чекає рев'ю / QA | автор |
| `погоджено` | затверджено; відкриває наступний етап | **лише людина / клієнт** |
| `виконано` | реалізовано й перевірено | людина |
| `застаріле` | замінено новішою версією / скасовано | людина |

Маппінг на наявні системи (нічого не ламаємо — лише зводимо до спільної мови):

| Канон | Реєстр «Перелік ТР» | SPEC.md | Плани / доки |
|---|---|---|---|
| чернетка / на розгляді | Аналіз та підготовка рішення | draft / review | чернетка |
| погоджено | Погодження клієнта | approved | погоджено |
| виконано | Виконано | модуль змерджено | — |
| застаріле | Скасовано | superseded | застаріле |

> Окрема вісь — **статус сесії** (запланована / проведена / скасована). Не плутати зі статусом артефакта.

## 2. Правило гейту (одне на всі межі)

- Наступний етап НЕ стартує, доки артефакт попереднього ≠ `погоджено`.
- `погоджено` ставить **лише людина / клієнт**. AI ніколи не self-approve.
- На межі AI лише **позначає drift** (розбіжність нижнього з верхнім) і блокує (як `pr-review` BLOCK), але не затверджує.

## 3. ID-namespace: AC як універсальна валюта вимоги

- Канонічний формат: `AC-01`, `AC-02`, … — **єдиний** (не `AC1` / `Scenario N` / `УМОВА-КОЛИ-ТОДІ`).
  Тіло AC може лишатися Given/When/Then або УМОВА/КОЛИ/ТОДІ — фіксований лише **ID = AC-NN**.
- Кожен AC несе backref **`origin`** на джерело вгору:
  `origin: <журнал D-N | сесія N | тема T-N | бриф | UAT>`.
- Наскрізний потік:
  `журнал (D-N / T-N)` → `ТР: AC-01 (+origin)` → `SPEC: AC-01` → `# TODO: AC-01` → `test_ac01_…` → `pr-review`.
- Правила: не перенумеровувати; один namespace на вимогу; downstream лише споживає, не вигадує.

## 4. Гейти й проекції на межах

| Межа | Вхід (`погоджено`) | Машинна проекція | Чим |
|---|---|---|---|
| УЗ → план | аналіз УЗ | план дискавері | kickoff-transcript-analyzer → discovery-plan-builder |
| цикл → ТР | журнал (теми / рішення) | ТР (US / AC з `origin`) | ✅ `journal-to-tr-builder` (closeout) — скаффолд `ТР_*.md` + секція 1 з `origin`; далі `tr-usecases-acceptance` |
| ТР → білд | ТР `погоджено` | SPEC.md (AC-01) | spec-writer |
| SPEC → код | SPEC `approved` | ARCHITECTURE → скелет → код | architecture / scaffolding / code-dev |
| код → merge | pr-review APPROVE + тести | — | pr-review → реєстр «виконано» |

## 5. Що вже діє / що додати

**Діє:** AC-01 наскрізь ТР→білд (`spec-writer`, `scaffolding`, `code-dev`, `testing`, `pr-review`).

**Додати (мінімально, за пріоритетом):**
- `origin` в `tr-usecases-acceptance` (upstream-правка — поза сесією) і збереження `origin` у `spec-writer`.
- посилання на цей словник статусів із реєстру / ТР / SPEC / планів.
- крос-консистентний валідатор: кожен AC має `origin`; кожне «виконано» має змерджений модуль; кожен AC у SPEC прийшов із ТР.

## 6. Прогресивний ригор

Легко в ІНІЦІАЦІЇ / ДИСКАВЕРІ (статуси й ID emergent, журнал редагований руками) →
жорстко на ТР / білді (AC-NN фіксовані, гейти, drift = BLOCK). Клієнтські документи —
людські й українські; хребет (статуси + ID) — службовий шар **під** ними, не замість.
