# План аналізу корпусу ТР (Етап 1)

Дата: 2026-06-12. Стратегія: **map-reduce** (168 ТР > 100) — стратифіковане глибоке читання
репрезентативної вибірки + семантичні зрізи по всій базі. Логи — append-only, пишуть субагенти
(контекст-сейф). Мета аналізу: довести ланцюг `tr-*` (R1–R5 + правила якості AC/Odoo) на реальних патернах.

| Вимір | Значення |

| Тип задачі | нова сутність/модуль · поле/доробка форми · звіт (Pivot) · обмін/інтеграція (API) · права/доступи · друкована форма (QWeb) · патч/доопрацювання (Формат C) |

## Вибірка глибокого читання (ціль ~80–100 ТР дослівно)

Принцип: **усі** ТР у малих різнорідних проєктах (максимум різноманіття патернів) + **семпл** з великих
однорідних серій (висока внутрішня надлишковість) + **усі** патчі (Формат C) + усі ідентифіковані скасовані.
ID субагент перерезолвить per-project через `find_documents` (частину ID втрачено при компакції реєстру).

|---|---|---|---|

Чесна цифра покриття фіксується у фіналі (саме прочитане, не заявлене).

## Семантичні зрізи (search_knowledge_base, по всій базі — окремий субагент → `_семантичні_зрізи.md`)

1. Формат AC: «критерії приймання Given When Then» / «УМОВА КОЛИ ТОДІ acceptance criteria» — який домінує.
2. UC: «сценарій використання поведінка системи user story».
3. Моделі: «кастомна модель td. поле many2one selection computed related».
4. Таблиця полів: «таблиця полів тип поля джерело значення domain required».
5. Права: «права доступу групи користувач адміністратор record rule constrains».
6. Подання/кнопки: «smart-кнопка подання канбан список кнопка-дія стадія».
7. Звіти/обміни: «Pivot зведена таблиця звіт обмін інтеграція API json».
8. Оцінка: «обсяг робіт оцінка людино-годин трудомісткість етапи».
9. Тест/інструкція: «тестовий сценарій протокол випробувань інструкція кроки».
10. Нещасливі шляхи: «немає даних помилка User Error права обмеження валідація».
11. Legacy-синтаксис (детектор): «attrs states tree t-raw xmlrpc group_operator».
12. Архітектура/БД: «архітектура об'єкти бази даних налаштування системи».
13. Формат C: «доопрацювання OLD НЕ РОБИТИ правки зауваження клієнта статус готово».
14. Варіанти/відкриті: «варіанти рішення трейд-оф відкриті питання».

## Шаблон видобутку (що субагент фіксує по КОЖНОМУ ТР у лог-файл батчу)

```
### [project] · «[filename]» · status:[з тіла/реєстру або н/д]
- Формат: A/B (повний) | C (патч). Секції присутні: 1вим / 2UC+AC / 3техреал / 4арх / 5обсяг / 6тест.
- Вимоги (с.1): є/нема; нумеровані (ВИМ-NN? інше?); backref на джерело; стиль.
- UC+AC (с.2): UC є? К-сть AC; ФОРМАТ {GWT-EN | УМОВА-UA | AC1 | AC 1.1 | Scenario N | нумер.список | НЕМАЄ}; happy+unhappy? кнопки покриті? 1 приклад AC дослівно.
- Техреал (с.3): моделі (td.*? інше?); таблиця полів є? типи; права/подання/smart-кнопки/related-computed? 1 приклад.
- Консистентність імен: поле під різними іменами між секціями? (так/ні + приклад)
- Odoo sanity: вигадані/сумнівні поля/механізми; LEGACY-синтаксис (attrs/states/<tree>/t-raw/xmlrpc)? версія згадана?
- Ручні правки BA / недопрацювання: порожні секції, шаблонні фрази, реалізація «на словах», розсинхрон назви файлу/задачі.
- Анти-патерн (якщо скасовано/слабко): чому.
```

Кожен субагент пише `_патерни_ТР_лог_[A..H].md`; наприкінці Етапу 2 зливаю у `_патерни_ТР_лог.md`.

## Чесні обмеження
- Статус ТР переважно невідомий (метадані порожні; реєстри лише в ельфа/демз/тов-2021). Скасовані поза цими реєстрами не ідентифікувати — беремо що є.
- Великі серії семплюються; якщо семпл покаже високу варіативність — добираємо ще.
- Бріолекс — SelfERP, не Odoo: джерело UC/AC-патернів, але не Odoo-точності.
