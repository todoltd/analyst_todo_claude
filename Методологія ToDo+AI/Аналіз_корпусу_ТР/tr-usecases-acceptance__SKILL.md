# tr-usecases-acceptance — Use Cases + Acceptance Criteria

## Мета
Use Cases описують поведінку в реальних ситуаціях, а Acceptance Criteria роблять вимоги
перевірюваними — це контракт між ToDo, клієнтом і розробкою. Без AC «готово» стає предметом
суперечки; з AC приймання об'єктивне.

## Інструменти та середовище
**Вхід:** `ТР_[назва].md` (секція 1 + плейсхолдер `<!-- TODO: tr-usecases-acceptance -->`) → `Read`/`Edit`;
за потреби бриф/транскрипт.
**Збереження:** впиши у `ТР_[назва].md` → `Edit`.
**Глибокий довідник:** `docs/патерни-UC-AC.md` (User Story, Use Cases, **обидва формати AC** —
EN Given/When/Then і UA УМОВА/КОЛИ/ТОДІ, формальна структура фічі, межі scope).

## Крок 0: Визнач вхідні дані
- Знайди `ТР_*.md` (кілька — спитай яку задачу). Немає — запропонуй `journal-to-tr-builder`
  (він скаффолдить ТР і пише секцію 1; окремих `tr-scaffold`/`tr-functional-requirements` у методології немає).
- Найкраще джерело — вже написана секція 1 (вимоги `ВИМ-NN`, автор — `journal-to-tr-builder`): кожна
  вимога звідти має отримати покриття сценарієм і AC тут. Якщо секції 1 ще немає — попередь, що покриття
  буде неповним, і запропонуй спершу `journal-to-tr-builder`.

## Крок 1: Use Cases (Сценарії використання)
Для кожної задачі/фічі — мінімум 3 сценарії з конкретикою (числа, ролі, обсяги):
```
#### Сценарій N: <коротка назва>
- Дія: <що робить користувач, з деталями: «активно понад 500 позицій»>
- Поведінка системи: <по кроках 1., 2., 3.>
```
Покривай: нормальний хід, межові випадки (порожньо/багато/немає даних), скасування дії,
взаємовплив елементів (фільтри між собою).

## Крок 2: User Story (для нового функціоналу/дії)
**Як** `<роль>`, **я хочу** `<можливість>`, **щоб** `<вигода>`. Додавай «Контекст/проблема» і «Передумови».

## Крок 3: Acceptance Criteria (канонічний ID + гнучке тіло)
> **Контекст корпусу (90 реальних ТР):** формальні AC є лише в ~3 з 90 — їх місце займають нумеровані
> вимоги й тест-кроки. Твоя задача тут — **ввести AC там, де їх немає**, а не лише переформатувати наявне.
**ID — завжди канонічний `AC-01`, `AC-02`…** — єдиний наскрізний namespace вимоги, що доживає
до SPEC → `# TODO: AC-01` → `test_ac01` → pr-review. **НЕ** `AC1` / `AC 1.1` / `Scenario N` —
інакше `spec-writer` на pre-flight зупиниться (контракт SDD §3, `КОНТРАКТ_SDD-хребет`).
Якщо ТР зібрано з дискавері — `AC-01` уже стоять із `journal-to-tr-builder`: не перенумеровуй.
**Тіло AC — за проектом** (у межах документа єдине):
- **EN:** Given / When / Then / And (як у проекті Elfa);
- **UA:** УМОВА / КОЛИ / ТОДІ / ТА (як у новіших ТЗ Олімпіус).
Ключові слова — жирним; текст українською.
```
##### AC-01 — <що перевіряємо однією фразою>
- Given <передумова>
- When <дія>
- Then <очікуваний результат>
```
Правила: один AC = одна думка; кожна вимога і кожна кнопка/дія покрита хоча б одним AC.
**Трасування вгору — `ВИМ-NN` + `origin`.** Кожен AC посилається на вимогу секції 1, яку покриває:
`AC-01 (… вимога: ВИМ-02)`. Якщо вимога несе backref `origin: D-N / T-N / сесія N` (ТР зібрано з
дискавері через `journal-to-tr-builder`) — перенеси і його: `AC-01 (origin: D-3 · вимога: ВИМ-02)`.
Так нитка «рішення дискавері → ВИМ → AC → код/конфіг» цілісна (контракт SDD: один namespace `AC-NN`,
`origin` донизу не вигадується). Якщо ТР не з дискавері — `origin` пропусти (опційне); backref на
`ВИМ-NN` лишається обов'язковим, коли секція 1 має ID вимог.
**ОБОВ'ЯЗКОВО для КОЖНОЇ нетривіальної дії/кнопки — щонайменше один AC на «нещасливі» шляхи** (корпус
їх системно пропускає — саме вони ловлять баги на прийманні):
- **немає даних / порожньо** — товар без ціни, порожня вибірка → система не падає, рядок не оновлюється;
- **помилка/валідація** → `UserError`/`AccessError`/`ValidationError` з **дослівним текстом** помилки;
  стан не змінюється (НЕ «403»/HTTP-код — це не механізм Odoo, див. `tr-odoo-tech-design`);
- **немає прав / видимість** — кнопка/поле приховані для не-адміна (`groups`/`column_invisible`);
- **межі/constrains** — дублікат, друга спроба, від'ємний залишок → блок із повідомленням;
- **повтор/конкурентність** — повторне натискання не дублює ефект; **скасування/відкат** дії.
Для UI/візуальних вимог допустимий короткий нумерований формат критеріїв (див. довідник).

## Крок 4: Варіанти рішення (якщо реалізація неоднозначна)
Запропонуй 2 варіанти з трейд-офами для вибору клієнтом/архітектором.

## Крок 5: Формальна структура та межі scope
Для складних фіч (новіший стандарт) оформлюй як іменовані секції:
`USER STORY → BUSINESS RULES (нумеровані правила) → ACCEPTANCE CRITERIA → UI/UX`. Технічну частину
BUSINESS RULES/UI деталізує `tr-odoo-tech-design`; тут — story, сценарії і AC.
**Межі scope:** якщо фіча природно ділиться на етапи — додай абзац, що входить у ЦЕЙ ТЗ, а що
винесено в наступний (напр. «розрахунок поля — наступне ТЗ»). Це стримує роздування обсягу.

## Pre-save верифікація
- [ ] Кожна функціональна вимога (`ВИМ-NN`) з секції 1 має сценарій і/або AC;
- [ ] кожен AC має backref `вимога: ВИМ-NN` (і `origin: D-N`, якщо вимога його несе);
- [ ] на кожну кнопку/дію є AC;
- [ ] **на кожну нетривіальну дію є AC на «нещасливі» шляхи** (немає даних; помилка з дослівним текстом; права; межі; повтор);
- [ ] AC перевірювані (немає «працює коректно»);
- [ ] **ID канонічні `AC-01`** (не `AC1`/`AC 1.1`/`Scenario N`) — інакше spec-writer зупиниться;
- [ ] Given/When/Then (або УМОВА/КОЛИ/ТОДІ) оформлені, нумерація наскрізна й послідовна.

## Наступний крок
Повідом: «Use Cases і AC готові. Далі — tr-odoo-tech-design (реалізація в Odoo)?»

## Чого НЕ робити
- Не описуй технічну реалізацію (це `tr-odoo-tech-design`) — AC про спостережувану поведінку.
- Не пиши розпливчасті AC; не пропускай гілки помилок і прав — саме вони ловлять баги на прийманні.
