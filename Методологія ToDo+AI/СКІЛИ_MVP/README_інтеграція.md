# СКІЛИ_MVP — джерело істини пакета MVP-білд

Похідне від стороннього SDD-пакета (7 скілів) для **етапу MVP / білд** методології
ToDo+AI. Повний план — `../ПЛАН_інтеграції_SDD-пакета_MVP.md`.

## Ухвалені рішення
- **Модель зведення:** Варіант А — ТР є єдиним джерелом істини; `spec-writer`
  перероблено на конвертер «ТР → SPEC.md»; `architecture` лишається як
  архітекторський шар (заповнює порожню секцію АРХІТЕКТУРА ТР).
- **AC-ID:** канонічний формат `AC-01`, `AC-02` … — наскрізний від ТР до тестів
  (`# TODO: AC-01` → докстрінг → `test_ac01_…` → pr-review).
- **Бренд:** ToDo / todo.ltd. Моделі `td.*`, модулі `<проект>_<область>`.
- **Мова:** інструкції скілів — англійською (як решта пакета); тексти артефактів,
  видимі клієнту/в коді (US, AC, помилки `_()`) — українською (мовою ТР).

## Зроблено
- **Етап 0:** ребренд Triodoo→ToDo; створено `spec-writer/references/odoo-groups.md`;
  прибрано хардкод `/mnt/user-data/uploads` у `reverse-spec`.
- **Етап 1:** `spec-writer/SKILL.md` перероблено на конвертер «ТР→SPEC.md»

- **Етап 2:** `architecture` — ToDo-конвенції (`td.*`, `<проект>_<область>`, маски) +
  обов'язкова звірка кожного пункту дизайну з офіційною докою Odoo 19 з джерелом;
  `scaffolding` — імена модуля `<проект>_<область>`, версія з картки проекту, маркери AC-01.
- **Етап 3:** `code-dev` — AC-01, гілка/ком з ID задачі ERP, маски; `testing` — прив'язка
  до «Тестового сценарію»/«Протоколу тестування» ТР; `pr-review` — еталон = затверджений ТР,
  розведення з `tr-review` (документ vs код).
- **Перенос і пакування:** оформлено пакет `СКІЛИ_MVP` (plugin.json `odoo19-mvp-build`,
  README, docs, `ЗІБРАТИ_plugin.sh`); додано стадію «3. MVP» (3 блоки) в авторське дерево;
  обидві копії синхронні. Версія пакета — `0.2.0`.

## ⚠️ Потрібна upstream-правка (поза цією сесією)
Конвертер `spec-writer` **вимагає AC у форматі `AC-01`**. Зараз
`tr-usecases-acceptance` дозволяє кілька форматів (`AC1`, `AC 1.1`, `Scenario N`,
`УМОВА/КОЛИ/ТОДІ`). Щоб трасування працювало, треба:

> У скілі **`tr-usecases-acceptance`** (Крок 3 «Acceptance Criteria»):
> зробити канонічним єдиний формат ідентифікатора — **`AC-01`, `AC-02`, …**
> (тіло AC лишається Given/When/Then або УМОВА/КОЛИ/ТОДІ — змінюється лише **ID**).
> Оновити Pre-save верифікацію: «нумерація у форматі AC-NN».

Цю зміну **не можна внести з поточної сесії** — встановлені скіли тут read-only.
Внести у вашому джерелі скілів `tr-*` (де воно лежить) або через
**Settings → Capabilities** / `skill-creator`. Якщо лишити старі формати —
тоді треба перемикатися на «автомаппінг» (інша модель, ми її відхилили).
