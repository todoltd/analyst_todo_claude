# odoo19-mvp-build

Методологія етапу **MVP / БІЛД** для проєктів впровадження Odoo 19 — SDD-ланцюжок
(Spec-Driven Development) від затвердженого ТР до перевіреного `td`-модуля.
Стартує там, де завершується аналіз: вхід — **затверджений ТР** (реєстр «Перелік
технічних рішень», статус «Погодження клієнта»).

Модель інтеграції — **Варіант А**: ТР лишається єдиним джерелом істини, а пакет
проектує його у `SPEC.md` і далі веде до коду й тестів. Деталі —
`../ПЛАН_інтеграції_SDD-пакета_MVP.md` і `README_інтеграція.md`.

## Скіли по блоках

### Блок 1 — Специфікація
1. `odoo-reverse-spec` — реверс існуючого `td_`-модуля у SPEC.md (для доробок наявних модулів).
2. `odoo-spec-writer` — конвертер «затверджений ТР → SPEC.md»; не пише вимоги з нуля, зберігає `AC-01`.

### Блок 2 — Архітектура і скелет
3. `odoo-architecture` — SPEC.md → ARCHITECTURE.md (файлове дерево, скелети класів, трасування AC, data flows, міграції, Odoo.sh). Заповнює «архітекторський» шар, який ТР лишає порожнім.
4. `odoo-scaffolding` — SPEC + ARCHITECTURE → повний скелет модуля з маркерами `# TODO: AC-01`.

### Блок 3 — Розробка, тести, рев'ю
5. `odoo-code-dev` — реалізація по одному AC за раз (spec-locked).
6. `odoo-testing` — юніт/інтеграційні тести Odoo, один тест на AC; реалізує «Тестовий сценарій» ТР у коді.
7. `odoo-pr-review` — рев'ю PR проти SPEC/ТР (вердикт APPROVE / REQUEST CHANGES / BLOCK).

## Ланцюжок

```
ТР затверджено (реєстр: «Погодження клієнта»)
    ↓
[існуючий td_-модуль?] → odoo-reverse-spec → SPEC.md
    ↓
odoo-spec-writer (ТР → SPEC.md, AC-01)
    ↓ approved
odoo-architecture → ARCHITECTURE.md
    ↓
odoo-scaffolding → скелет модуля (# TODO: AC-01)
    ↓
odoo-code-dev (по 1 AC) ⇄ odoo-testing (1 тест/AC)
    ↓
odoo-pr-review → APPROVE → merge → реєстр: «Виконано»
```

## AC-01 — наскрізна нитка
Ідентифікатор критерію `AC-01` веде від ТР до тестів: `# TODO: AC-01` (скелет) →
докстрінг (код) → `test_ac01_…` (тести) → чек pr-review. **Передумова:**
`tr-usecases-acceptance` має емітити AC у форматі `AC-01` (див. `README_інтеграція.md`).

## Збірка та інсталяція

```bash
cd "СКІЛИ_MVP" && bash ЗІБРАТИ_plugin.sh
```

Інсталяційна картка з кнопкою з'являється лише коли `.plugin` доставлено через папку **outputs** сесії Cowork.

## Статус

Етапи інтеграції 0–3 виконані: ребренд ToDo; фікс битого референсу й шляху;
`spec-writer` → конвертер «ТР→SPEC.md» (Варіант А, AC-01); у `architecture` —
обов'язкова звірка з Odoo 19 з джерелом; у `scaffolding`/`code-dev` — конвенції
імен (`<проект>_<область>`, `td.*`, маски, версія з картки); `testing` прив'язано
до «Тестового сценарію» ТР; `pr-review` розведено з `tr-review`.
Лишився **Етап 4**: синхронізація статусів із реєстром + оновлення
BPMN / майстер-інструкції / ІСТОРІЇ — див. `../ПЛАН_інтеграції_SDD-пакета_MVP.md`.
