# odoo19-mvp-build

Методологія етапу **MVP / БІЛД** для проєктів впровадження Odoo 19 — SDD-ланцюжок
(Spec-Driven Development) від затвердженого ТР до перевіреного `td`-модуля.
Стартує там, де завершується аналіз: вхід — **затверджений ТР** (реєстр «Перелік
технічних рішень», статус «Погодження клієнта»).

Модель інтеграції — **Варіант А**: ТР лишається єдиним джерелом істини, а пакет
проектує його у `SPEC.md`. Далі `SPEC.md` розгалужується на **два паралельні треки**
на спільному `AC-01`: **Трек 1 — код** (кастомні `td.*`-модулі) і **Трек 2 —
конфігурація** (БА налаштовує стандартний Odoo через UI). Деталі —
`../ПЛАН_інтеграції_SDD-пакета_MVP.md`, `../ПЛАН_розробки_MVP_блок_Конфігурація_Трек-2.md`,
`README_інтеграція.md`.

## Скіли по блоках

### Блок 1 — Специфікація
1. `reverse-spec` — реверс існуючого `td_`-модуля у SPEC.md (для доробок наявних модулів).
2. `spec-writer` — конвертер «затверджений ТР → SPEC.md»; не пише вимоги з нуля, зберігає `AC-01`.

### Блок 2 — Архітектура і скелет
3. `architecture` — SPEC.md → ARCHITECTURE.md (файлове дерево, скелети класів, трасування AC, data flows, міграції, Odoo.sh). Заповнює «архітекторський» шар, який ТР лишає порожнім.
4. `scaffolding` — SPEC + ARCHITECTURE → повний скелет модуля з маркерами `# TODO: AC-01`.

### Блок 3 — Розробка, тести, рев'ю (Трек 1)
5. `code-dev` — реалізація по одному AC за раз (spec-locked).
6. `testing` — юніт/інтеграційні тести Odoo, один тест на AC; реалізує «Тестовий сценарій» ТР у коді.
7. `pr-review` — рев'ю PR проти SPEC/ТР (вердикт APPROVE / REQUEST CHANGES / BLOCK).

### Форк — розподіл по треках
8. `build-allocator` — класифікує кожен `AC-01`: **config / code / hybrid** → `ALLOCATION.md`. Запускається одразу після `spec-writer`; живить обидва треки.

### Блок 4 — Конфігурація (Трек 2)
9. `config-spec` — config-AC → `CONFIG.md` (DOC-2): що налаштувати в UI, по областях, з `[AC-NN]` і джерелом Odoo 19. Аналог `spec-writer`+`architecture` для конфігурації.
10. `config-runbook` — `CONFIG.md` → впорядкований UI-сценарій для БА + `config-log` (доказ застосованого; замінює git diff). Аналог `scaffolding`+`code-dev`.
11. `config-review` — гейт перед UAT: вердикт APPROVE / REQUEST CHANGES / BLOCK проти `CONFIG.md`+`config-log`. Аналог `pr-review` для конфігурації.

## Ланцюжок

```
ТР затверджено (реєстр: «Погодження клієнта»)
    ↓
[існуючий td_-модуль?] → reverse-spec → SPEC.md
    ↓
spec-writer (ТР → SPEC.md, AC-01)   [гейт: ТР approved]
    ↓ approved
build-allocator → ALLOCATION.md (AC → config / code / hybrid)
    ├───────────────── Трек 1 · КОД ─────────────────┐
    │ architecture → ARCHITECTURE.md            │
    │ scaffolding → скелет (# TODO: AC-01)       │
    │ code-dev (1 AC) ⇄ testing (1 тест/AC) │
    │ pr-review → APPROVE → Деплой               │
    └───────────────── Трек 2 · КОНФІГУРАЦІЯ ─────────┤
      config-spec → CONFIG.md (DOC-2)            │
      config-runbook → runbook + config-log      │
      config-review → APPROVE                    │
                                                      ↓
                Майстер-дані + права (БА) → UAT → реєстр: «Виконано»
```

## AC-01 — наскрізна нитка (обидва треки)
Ідентифікатор критерію `AC-01` — спільна валюта обох треків.
**Трек 1 (код):** `# TODO: AC-01` (скелет) → докстрінг (код) → `test_ac01_…` (тести) → чек pr-review.
**Трек 2 (конфіг):** `[AC-01]` у `CONFIG.md` → крок runbook → запис у `config-log` → чек config-review.
Гібридний AC ділиться між треками, але `AC-NN` не дублюється (один namespace).
**Передумова:** `tr-usecases-acceptance` має емітити AC у форматі `AC-01` (див. `README_інтеграція.md`).

## Збірка та інсталяція

```bash
cd "СКІЛИ_MVP" && bash ЗІБРАТИ_plugin.sh
```

Інсталяційна картка з кнопкою з'являється лише коли `.plugin` доставлено через папку **outputs** сесії Cowork.

## Статус

**Трек 1 (код)** — етапи інтеграції 0–3 виконані: ребренд ToDo; фікс битого референсу
й шляху; `spec-writer` → конвертер «ТР→SPEC.md» (Варіант А, AC-01); у `architecture` —
обов'язкова звірка з Odoo 19 з джерелом; у `scaffolding`/`code-dev` — конвенції
імен (`<проект>_<область>`, `td.*`, маски, версія з картки); `testing` прив'язано
до «Тестового сценарію» ТР; `pr-review` розведено з `tr-review`.

**Трек 2 (конфігурація)** — v0.3.0: додано форк `build-allocator` і блок конфігурації
(`config-spec` → `config-runbook` → `config-review`) на спільному `AC-01`. План —
`../ПЛАН_розробки_MVP_блок_Конфігурація_Трек-2.md` (відкриті питання §6).
