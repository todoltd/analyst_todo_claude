# Блок 3 — Розробка, тести, рев'ю

Реалізує AC у коді, покриває тестами і контролює відповідність ТР перед мерджем.

## Скіли

### `code-dev`
По одному AC за раз; spec-locked (нічого поза scope).
Docstring із номером AC; прибирає `# TODO: AC-01` після реалізації.
Конвенції Odoo: `_()` для рядків, `invisible=`, `@api.model_create_multi`, `ir.sequence td.{module}.{model}`. Гілка — з ID задачі ERP.

### `testing`
Юніт/інтеграційні тести Odoo, **1 тест на AC**, теги `post_install`, `with_user` для перевірки прав.
Реалізує у коді «Тестовий сценарій»/«Протокол тестування» ТР. Таблиця «AC coverage audit» — точка приймання.

### `pr-review`
Рев'ю PR за 5 вимірами (spec-compliance, AC coverage, конвенції Odoo, тести, маніфест).
Вердикт **APPROVE / REQUEST CHANGES / BLOCK**; будь-яке відхилення від ТР/SPEC → BLOCK.
Перевірки v19 (`attrs`/`<tree>`/`t-raw`/`group_operator`/xmlrpc) збігаються з конвенціями ToDo.
**Розведення з `tr-review`:** `tr-review` — QA документа ТР до підпису клієнтом; `pr-review` — QA коду до мерджу.

## Гейт на вихід блоку
- `pr-review` = APPROVE, тести зелені → merge → реєстр «Виконано».

## Чого НЕ робити
- Не реалізовувати кілька AC в одному комі; не додавати «nice to have» поза AC; не мерджити з BLOCK.
