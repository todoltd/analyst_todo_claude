# MVP / БІЛД — Майстер-інструкція

**Для кого:** розробник Odoo, архітектор, BA (контроль відповідності ТР + конфігурація)
**Коли:** від затвердженого ТР (реєстр «Перелік технічних рішень» → «Погодження клієнта») до прийнятого MVP
**Результат:** робочий MVP Odoo 19 — кастомний код (покритий тестами) + конфігурація стандартного Odoo, разом відповідні AC затвердженого ТР
**Два треки:** після `SPEC.md` робота йде паралельно — **Трек 1 (код)** і **Трек 2 (конфігурація БА через UI)** — на спільному `AC-01`. Розводить їх форк `build-allocator`.

---

## Модель (Варіант А)

ТР — єдине джерело істини. Пакет не переписує вимоги, а **проектує затверджений ТР**
у машинно-читний `SPEC.md` і веде його до коду. `approved`-гейт SDD = ваш статус
реєстру «Погодження клієнта».

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|---|---|---|
| Є затверджений ТР, готуємо SPEC | "збери SPEC з ТР [назва]" | `SPEC.md` (status approved, AC-01) |
| Модуль уже існує, нема SPEC | "зроби reverse-spec для модуля [шлях]" | `SPEC.md` з коду |
| SPEC готовий, потрібен дизайн | "згенеруй ARCHITECTURE для [модуль]" | `ARCHITECTURE.md` |
| Дизайн готовий, потрібен скелет | "згенеруй скелет модуля [модуль]" | файли модуля з `# TODO: AC-01` |
| Реалізувати критерій | "реалізуй AC-03 у [модуль]" | код одного AC + прибраний TODO |
| Потрібні тести | "напиши тести для [модуль]" | тести (1 на AC) + аудит покриття |
| Перевірити PR | "перевір PR проти SPEC" | вердикт APPROVE / REQUEST CHANGES / BLOCK |
| SPEC готовий, ділимо роботу | "розподіли AC по треках" | `ALLOCATION.md` (config / code / hybrid) |
| Потрібна специфікація налаштувань | "збери CONFIG з SPEC" | `CONFIG.md` (DOC-2): що налаштувати в UI, по AC |
| Потрібен сценарій для БА | "зроби runbook для конфігурації" | `CONFIG_RUNBOOK.md` + `config-log` |
| Перевірити конфігурацію | "перевір конфігурацію проти CONFIG" | вердикт APPROVE / REQUEST CHANGES / BLOCK |

## Ланцюжок і гейти

```
ТР «Погодження клієнта»
  → (reverse-spec, якщо td_-модуль існує)
  → spec-writer: ТР → SPEC.md            [гейт: ТР approved]
  → build-allocator: SPEC → ALLOCATION   [розподіл AC: config / code / hybrid]
    ├── ТРЕК 1 · КОД
    │   → architecture: SPEC → ARCHITECTURE   [гейт: SPEC approved, 0 open questions]
    │   → scaffolding: скелет + # TODO: AC-01  [гейт: SPEC + ARCHITECTURE approved]
    │   → code-dev (по 1 AC) ⇄ testing (1 тест/AC)
    │   → pr-review → APPROVE → Деплой
    └── ТРЕК 2 · КОНФІГУРАЦІЯ
        → config-spec: SPEC + ALLOCATION → CONFIG.md (DOC-2)
        → config-runbook: CONFIG → runbook + config-log
        → config-review → APPROVE
  → Майстер-дані + права (БА) → UAT → реєстр «Виконано»
```

Гейти жорсткі (однакові на обох треках): downstream не стартує без `approved` SPEC/CONFIG;
`pr-review` ставить **BLOCK** на відхилення коду від ТР/SPEC, `config-review` — на
відхилення конфігурації від CONFIG.md або на вигадане налаштування Odoo 19. AI не
сам-approve — затвердження ТР робить клієнт.

## AC-01 — наскрізна трасування (обидва треки)

`AC-01` — спільна валюта вимоги. **Трек 1:** `# TODO: AC-01` → докстрінг коду →
`test_ac01_…` → чек pr-review. **Трек 2:** `[AC-01]` у CONFIG.md → крок runbook →
запис у config-log → чек config-review. Гібридний AC ділиться між треками без
дублювання id. **Передумова:** `tr-usecases-acceptance` має давати AC у форматі
`AC-01` (див. `README_інтеграція.md`).

## Зв'язок із реєстром «Перелік технічних рішень»

- ТР «Погодження клієнта» = SPEC `approved` = старт білду.
- «Виконано» = після pr-review APPROVE і зелених тестів (`tr-registry-update`).

## Чого НЕ робити

- Не писати вимоги/AC на цьому етапі — це шар `tr-*` (аналіз/ТР).
- Не додавати поля/моделі поза ТР — спершу оновити ТР.
- Не перенумеровувати AC; не стартувати `architecture`/`config-spec` без approved SPEC.
- Не вигадувати налаштувань Odoo 19 у `config-spec` — кожен пункт із джерелом; як не цитується, це код/hybrid.
- Не дублювати `AC-NN` між треками; гібрид ділиться на слайси, не на нові id.
- Не налаштовувати майстер-дані в Треку 2 — це спільний вузол `t-masterdata` після обох треків.
