# Трек 2 — Конфігурація (БА налаштовує Odoo через UI)

**Для кого:** BA (виконує налаштування), архітектор (розподіл config/code), розробник (hybrid-зчеплення)
**Коли:** паралельно з Треком 1, після `SPEC.md` `approved` і розподілу `ALLOCATION.md`
**Результат:** налаштований стандартний Odoo 19, що покриває config-AC, з доказом (`config-log`) і пройденим гейтом `config-review` — готовий до UAT

---

## Навіщо окремий трек

Не кожна вимога — це код. Більшість MVP на Odoo закривається **налаштуванням
стандарту**: активувати модулі, налаштувати права, форми/звіти через Studio,
автоматизації, email-шаблони, інтеграції. На BPMN це вузол `t-config` + артефакт
DOC-2 «Специфікація конфігурації». Раніше під ним не було скілів — Трек 2 закриває цей пробіл,
симетрично коду.

## Ланцюжок Треку 2

```
SPEC.md approved
  → build-allocator → ALLOCATION.md   (кожен AC: config / code / hybrid)
  → config-spec      → CONFIG.md       (DOC-2: що налаштувати, по областях, [AC-NN] + джерело)
  → config-runbook   → CONFIG_RUNBOOK.md + config-log   (впорядкований UI-сценарій + запис)
  → (БА налаштовує Odoo через UI, заповнює config-log)
  → config-review    → APPROVE / REQUEST CHANGES / BLOCK   (гейт перед UAT)
  → Майстер-дані + права (БА) → UAT
```

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|---|---|---|
| SPEC готовий, ділимо роботу | "розподіли AC по треках" | `ALLOCATION.md` (config / code / hybrid) |
| Потрібна специфікація налаштувань | "збери CONFIG з SPEC" | `CONFIG.md` (DOC-2) |
| Сценарій для БА | "зроби runbook для конфігурації" | `CONFIG_RUNBOOK.md` + `config-log` |
| Перевірити налаштоване | "перевір конфігурацію проти CONFIG" | вердикт APPROVE / REQUEST CHANGES / BLOCK |

## Ключові принципи

1. **Спільний AC-01.** Конфіг-пункт несе `[AC-NN]` — дзеркало `# TODO: AC-01` у коді.
   Один namespace; гібрид ділиться на config-слайс і code-слайс без нових id.
2. **Дисципліна Odoo 19 (Крок 1).** Кожен пункт `CONFIG.md` — з посиланням на офіційну
   доку Odoo 19. Вигадане налаштування = головний ризик; не цитується → це код/hybrid.
3. **Config-log замість git diff.** Ручні кліки не лишають сліду — доказом застосованого
   є `config-log` + експорт (`base.automation` / `mail.template` / server action / cron /
   `ir.config_parameter`). `config-review` перевіряє проти артефактів, не «на віру».
4. **Гейт перед UAT.** `config-review` ставить **BLOCK** на відхилення від `CONFIG.md`
   або на неіснуюче налаштування. APPROVE → внесок у «Виконано» (разом із `pr-review` Треку 1).
5. **Межі.** Майстер-дані (продукти/контрагенти/план рахунків) — спільний вузол
   `t-masterdata` після обох треків, не в C1–C3. Studio (no-code) = конфігурація;
   версіонований `td.*` = код.

## Гейти й статуси

- `CONFIG.md` успадковує `approved` від ТР/SPEC (AI не сам-approve).
- UAT-петля (`gw-reason`): помилки конфігурації → БА переналаштовує → повторний
  `config-review` до APPROVE. Баги коду — у Трек 1 (`pr-review`), не сюди.

## Артефакти

| Файл | Пише | Читає |
|---|---|---|
| `ALLOCATION.md` | `build-allocator` | `architecture` (Трек 1), `config-spec` (Трек 2) |
