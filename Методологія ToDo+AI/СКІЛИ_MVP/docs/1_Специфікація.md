# Блок 1 — Специфікація

Перетворює затверджений ТР (або існуючий код) на машинно-читний `SPEC.md` — вхід для блоку «Архітектура і скелет».

## Скіли

### `reverse-spec`
**Коли:** `td_`-модуль уже існує, але `SPEC.md` немає (доробка наявного, онбординг, аудит техборгу).
**Вхід:** шлях до модуля (`__manifest__.py`).
**Вихід:** `SPEC.md` з коду — моделі, поля, подання, права, бізнес-логіка; невизначене → «Open questions».
**Правила:** факти лише з коду, нічого не вигадувати; кожна модель/поле з колонкою `Since` (= версія маніфесту).

### `spec-writer` — конвертер ТР → SPEC.md (Варіант А)
**Коли:** є **затверджений** ТР, треба підготувати SPEC для білду.
**Вхід:** `ТР_[назва].md` (статус «Погодження клієнта»).
**Вихід:** `SPEC.md` (status approved, успадковано від ТР).
**Правила:** не пише вимоги з нуля; зберігає `AC-01` дослівно; проектує «Опис технічної реалізації» у блоки моделей SPEC; DB-архітектуру лишає для блоку 2; тексти US/AC — мовою ТР.

## Гейт на вихід блоку
- ТР approved; усі AC у форматі `AC-01`; `Open questions` порожні; `Out of scope` заповнено.

## Чого НЕ робити
- Не авторити нові вимоги/US/AC (це `tr-*`); не міняти ID критеріїв; не вигадувати поля/моделі поза ТР.
