# Жанр «Правки / Доопрацювання» (зміни до вже описаного модуля)

Окрім повних ТР, команда ToDo пише легкі документи **змін** до вже специфікованого/розробленого
модуля. Це не повне ТЗ — у них інша мета і структура. Виведено з корпусу проекту Bixter (HR).

## Коли це, а не повний ТР
- Доробка/зміна вже описаної фічі або модуля (а не новий функціонал з нуля).
- Збір правок після зустрічі з клієнтом.
- Додавання кількох полів/вкладок/умов до існуючих моделей.
Якщо це новий модуль/сутність — пиши повний ТР (`tr-scaffold` Формат A/B).

## Типи документів змін
1. **Патч після зустрічі** — кілька точкових правок; часто без шапки, дата лише в назві
   («Правки після зустрічі 09.03»). Тримай коротким: нумеровані пункти по об'єктах.
2. **Доопрацювання (додавання)** — нумеровані пункти по моделях: які поля/вкладки/умови додати
   («Додаткові доопрацювання полів HR Operational»).
3. **Двошаровий (клієнт → BA)** — спершу сирі зауваження клієнта (часто EN, скріншоти, терсно),
   потім технічна інтерпретація BA з деталізацією по полях. Зберігай обидва шари.
4. **Живий трекер** — Done / To fix по кожному об'єкту + «TO to do list» з **дедлайнами**
   (`ДД-ММ-РРРР`) замість годин; таблиці очікуваних колонок UI.

## Структура та правила
- Групуй зміни **по об'єктах/моделях** (RO, Candidate, Candidate Operational, Contacts…).
- Поля давай таблицею з колонкою **«Розташування»** (вкладка/секція/позиція) — як у повному ТР.
- Для зміни поведінки став **«було → стало»** (As-Is/To-Be в мініатюрі).
- **Обов'язково (рекомендація — у корпусі цього бракувало): посилання на вихідне ТЗ + ID задачі**,
  щоб зміну можна було простежити до першоджерела.

## Маркери статусу (внутрішня версійність)
Прямо в тексті біля пунктів/секцій став статус виконання:
`Готово` / `Почато` / `Очікує затвердження` / `Можна брати` / `Виконано` / `НЕ РОБИТИ` / `OLD`.
- `OLD` — лиши блок зі **заміненою** логікою (не видаляй — це історія рішення).
- `НЕ РОБИТИ` — явно познач відхилені пункти, щоб їх не реалізували.
Так документ слугує і журналом прогресу.

## Оцінка та приймання
- Часто без «Обсяг робіт»; живі трекери використовують **дедлайни** (`ДД-ММ-РРРР`) і списки
  Done/To-fix. Якщо потрібна оцінка — клич `tr-effort-instruction` (підтримує дедлайни).
- AC у патчах зазвичай немає, але для **нетривіальних** змін все одно додай критерії/тест-кроки
  (інакше зміну неможливо прийняти однозначно) — клич `tr-usecases-acceptance`.

## Мова
Підписи полів/UI — мовою проекту (Bixter — англійською; описи українською).

## Чого НЕ робити
- Не перетворюй патч на повний ТР, якщо клієнт чекає швидку правку — тримай формат легким.
- Не видаляй замінену логіку — переноси в `OLD`.
- Не лишай зміну без посилання на вихідне ТЗ/задачу.
