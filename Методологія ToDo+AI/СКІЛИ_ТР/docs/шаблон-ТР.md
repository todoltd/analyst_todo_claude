# Шаблон «Технічне рішення» (ТР) — канонічний каркас

Точна структура документа ТР компанії ToDo. Будь-який ТР збирається з цих секцій у цьому
порядку. Заголовки і підписи українською — не змінювати (документ іде клієнту і в розробку
у цьому вигляді).

Джерело: «Шаблон технічне рішення» (Google Drive).

---

## Хто яку секцію заповнює

| Секція | Хто | Скіл |
| :-- | :-- | :-- |
| Загальна інформація | Аналітик | `tr-scaffold` |
| Обсяг робіт | Лід / аналітик + архітектор | `tr-effort-instruction` |
| Загальний опис + Функціональні вимоги | Аналітик | `tr-functional-requirements` |
| Use Cases + Acceptance Criteria | Аналітик | `tr-usecases-acceptance` |
| Опис технічної реалізації | Аналітик/архітектор | `tr-odoo-tech-design` |
| АРХІТЕКТУРА (нові об'єкти, Налаштування системи) | Архітектор (після узгодження) | — лишається шаблоном |
| Інструкція | Аналітик | `tr-effort-instruction` |

Секція **АРХІТЕКТУРА** свідомо лишається порожнім шаблоном на етапі аналітики — її наповнює
архітектор після узгодження ТР клієнтом.

---

## Два формати тіла секції «ТЕХНІЧНЕ РІШЕННЯ»

### Формат A — по фічах/сутностях (флагман)
Велика розробка (новий модуль, сутності `td.*`). Структура «Опис технічної реалізації» іде
по об'єктах: бізнес-ціль → потреба в сутності → опис реалізації → таблиці полів → довідники →
права доступу → процеси (User Story + AC).

### Формат B — по задачах (доопрацювання)
Точкова доробка (1–3 задачі). Структура по задачах:
**Задача N → Проблема → Очікуваний результат (+ Приклад) → Use Cases → Критерії прийняття →
Варіанти рішення.**

Обидва формати використовують той самий каркас (шапка + Обсяг робіт + Архітектура + Інструкція).

## Правила оформлення
- Заголовки секцій і підписи полів — українською, дослівно як у шаблоні.
- Дати в тексті — формат `ДД.ММ.РРРР` (наприклад `21.05.2026`).
- Інкрементальні доповнення позначай датою в заголовку: «Процес створення документу (09.01.2026)»
  — документ живе і дописується по ходу проекту, історія не затирається.
- «Номер запиту» = ID задачі в `erp.todo.ltd`.
- «Система» зазвичай `Odoo 19 EE`.
- Біля людської назви поля завжди давай технічне ім'я в дужках: «Торгова точка (sale_point)».

## Варіант структури секцій (за проектом)
Набір секцій залежить від процесу проекту:

- **Скорочений (Elfa):** Загальна інформація → Обсяг робіт → Технічне рішення (Загальний опис +
  функціональні вимоги; Опис технічної реалізації) → Архітектура → Інструкція.
- **Розширений (Олімпіус):** Загальна інформація → Обсяг робіт → **Функціональні вимоги** (окрема
  секція) → Технічне рішення → Архітектура → **Тестовий сценарій** → **Протокол тестування** →
  Інструкція. Тут «Тестовий сценарій» і «Протокол тестування» — окремі секції, а «Обсяг робіт»
  має 7 етапів (Функціональні вимоги / Аналіз вимог / Архітектура / Тестовий сценарій / Розробка /
  Протокол тестування / Інструкція) + колонку «Виконавець».

Обирай набір секцій за проектом (за наявними документами проекту). У межах проекту структура єдина.
Блок-ТЗ (умбрела над фічами) має ще розширенішу шапку (Назва завдання, Замовник + Ініціатор, Дата
отримання, Дата реалізації, Погоджений дедлайн) і не містить «Технічне рішення»/тестів — лише перелік
напрямків.
