## Приклад Acceptance Criteria
Given/When/Then:
```
##### AC8 — Якщо є товари без ціни в прайс-листі
- Given у деяких рядках товар не має значення в обраному прайс-листі
- When виконується оновлення
- Then система не падає, такі рядки не оновлює (залишає попереднє значення)
```
Згруповані AC (з реального ТР «Власник ТТ»):
```
AC 1.3: Система дозволяє призначити лише одного «Власника» для ТТ. При спробі додати другого —
        User Error: «Торгова точка вже має призначеного власника».
AC 2.1: Якщо створено дочірній контакт з типом «Власник» (Геліантус ТОВ), display_name набуває
        вигляду «Геліантус ТОВ , Аптека №2».
AC 5.1: Поле td_formatted_name збережене в БД (store=True) — швидкий пошук у базах 10 000+ контактів.
```

## Реєстр «Перелік технічних рішень» (структура)
```
| Назва доопрацювання | ID задачі | Посилання на ТР або задачу | Документ | Дата старту | Статус | Дата закінчення |
```
Статуси життєвого циклу: `Аналіз та підготовка рішення` → `Погодження клієнта` → `Виконано` / `Скасовано`.
Посилання на задачу: `https://erp.todo.ltd/odoo/action-388/<...>/tasks/<ID>`.

## Приклади з проекту Олімпіус (інші домени — щоб не звужувати до роздробу)
Проект «Олімпіус Консалт», `Odoo Enterprise Edition 18.0` — кейтеринг + виробництво + логістика +
інтеграції. Показує ширший набір моделей і новіші патерни.

**Блочна організація + рівень ТЗ:** робота поділена на Блоки (Меню, Виробництво…); над фічами
стоїть «Загальне ТЗ по блоку» з розширеною шапкою (Назва завдання, Замовник + окремо Ініціатор,
Дата отримання, Дата реалізації, Погоджений дедлайн) і розширеним «Обсяг робіт» (7 етапів:
Функціональні вимоги / Аналіз вимог / Архітектура / Тестовий сценарій / Розробка / Протокол
тестування / Інструкція + колонка «Виконавець», підписи з датами).

**Формальна фіча (ТТН, FEATURE 1), новіший формат:**
```
USER STORY: Як логіст, я хочу створити маршрутний документ (рейс), щоб зв'язати водія, ТЗ і
            впорядкований список відвантажень та отримати пакет ТТН для друку.
BUSINESS RULES: 1. Маршрут — окрема модель td.delivery.route (fleet.vehicle + hr.employee).
                3. Назва = [Шаблон маршруту] + " / " + [Дата рейсу], редагована вручну.
                10. Додавати лише stock.picking у статусі Ready/Waiting, тип «Відвантаження».
                12. Pipeline: Чернетка → Підтверджено → Завершено → Скасовано.
ACCEPTANCE CRITERIA (UA): AC-02 — тіло УМОВА/КОЛИ/ТОДІ/ТА (приклад у `патерни-UC-AC.md`).
UI/UX: окремий модуль olimpius_delivery, меню «Маршрути доставки», форма (шапка+вкладка «Маршрут»
       з колонками), кнопки «Підтвердити/Завершити/Скасувати/Друкувати ТТН».
```

**Дисципліна scope (візард MO):** «В рамках цього ТЗ ми зупиняємось лише на візуалізації, підрахунку
полів та створенні кнопки. Логіка після "Підтвердити" — наступне ТЗ.»

**Звіт (Звіт Охват):** Pivot у `Меню → Звіти → Замовлення`; виміри (Проект, Клієнт, Група споживачів,
Дата, Товар, Район…), міри (Сума з SO, Кількість) з fallback-логікою (корінець→заявка;
доставлена→замовлена×ціна). Деталі формату — `звіти-та-обміни.md`.

**Обмін (Обміни Олімпіус):** Odoo ↔ 1С BAS, оформлено таблицею: матриця документів (Інвойс
`account.move` → «Документ реалізації»), мапінг полів (Договір → `td_agreement_id`, Відповідальний
→ `invoice_user_id` по ІПН `hr.employee.identification_id`), правила синхронізації (після `posted`;
правки в 1С не повертаємо), ключі ЄДРПОУ/ІПН. Деталі — `звіти-та-обміни.md`.
