# Патерни: Звіти (Pivot) та Обміни (інтеграції)

Два спеціальні типи доопрацювань, що мають власну структуру опису. Виведено з реальних ТЗ
проекту Олімпіус («ТЗ Звіт Охват», «Обміни Олімпіус»).

---

## 1. Звіт (Pivot / Список / Зведена таблиця)

Звіт описується не як набір полів, а як **виміри + міри + логіка розрахунку**. Структура опису:

1. **Допоміжні об'єкти** (якщо потрібні): нові довідники/поля, від яких залежить звіт
   (напр. довідник «Район» + `selection` у `res.partner` і `project.project`, зі значенням,
   що тягнеться з картки партнера, але редагується вручну).
2. **Розміщення в UI:** `<Модуль> → вкладка «Звіти» → вкладка «<Назва звіту>»`.
3. **Тип подання:** Pivot (Зведена таблиця) / Список — вкажи явно.
4. **Об'єкти (виміри / dimensions):** перелік полів, за якими групуємо/розрізаємо
   (Проект, Клієнт, Група споживачів, Дата, Товар, Мітки, Відповідальний, Район…).
5. **Міри (measures):** що рахуємо (Сума, Кількість) із **явним джерелом** кожної
   (Сума — з `sale.order`; Кількість — з корінця/заявки).
6. **Логіка розрахунку мір із fallback-правилами** — ядро звіту. Опиши покроково, звідки
   беремо значення і що робимо, коли даних немає. Приклад (реальний):
   - Кількість: беремо з «корінця»; якщо корінець = 0 → беремо із «заявки»; якщо в проекті
     «корінець = false» → завжди по заявці.
   - Сума: з SO, прив'язаного до задачі; якщо «доставлена кількість = 0» → «замовлена × ціна»
     (по кожній номенклатурі); якщо «доставлена > 0» → «доставлена × ціна одиниці».
   - Результат: звіт формується завжди, незалежно від наявності корінця.
7. **Acceptance Criteria** — на ключові правила розрахунку і fallback (їх легко перевірити).

Чек-лист звіту: [ ] допоміжні об'єкти; [ ] розміщення; [ ] тип; [ ] виміри; [ ] міри з джерелом;
[ ] логіка розрахунку з усіма fallback; [ ] AC на розрахунок.

## 2. Обмін / Інтеграція (напр. Odoo ↔ 1С BAS)

Інтеграція часто оформлюється як **таблиця (Google Sheet)**, а не текстовий ТР, бо ядро —
це помаплені поля. Структура опису:

### 2.1 Зведення по блоках (з оцінкою)
```
| Блок | Документ Odoo | Документ 1С | Уточнення вимог, год | Написання ТЗ, год | Розробка, год | Тестування, год | Примітки |
```
(Ядро інтеграції, Контрагенти, Номенклатура, Договори, Документи реалізації…)

### 2.2 Матриця обміну документів
```
| Напрям обміну | Умова для передачі | Документ Odoo | Документ 1С | Коментар |
| Odoo → 1C | після підтвердження (posted) | Інвойс (account.move) | Документ реалізації | ... |
```

### 2.3 Мапінг полів (по кожному документу/довіднику)
```
| Поле в 1С | Поле Odoo (технічне) | Примітки |
| Контрагент | Клієнт (partner_id) | |
| Договір | Договір (td_agreement_id) | |
| Відповідальний | Продавець (invoice_user_id) | зв'язок по ІПН: hr.employee.identification_id |
| Сума без ПДВ | — | розраховується в 1С (не переносимо) |
```
Завжди познач: технічне ім'я поля Odoo; що **не переносимо**; що **розраховується на іншій
стороні**; трансформації значень (напр. radiobutton `service` → «Послуга = true»).

### 2.4 Правила синхронізації (обов'язково явно)
- **Напрямок:** Odoo→1C, 1C→Odoo або двосторонній.
- **Тригер/умова:** після підтвердження (`posted`), по чек-боксу «доступний до обміну», по крону
  (напр. співробітники — 1×/місяць).
- **Конфлікти/зворотність:** коригування в Odoo → актуалізуємо в 1С; правки в 1С → в Odoo НЕ
  повертаємо (типове правило); або двосторонньо.
- **Натуральні ключі та унікальність:** Контрагент — ЄДРПОУ (`ref`), ІПН (`vat`); співробітник —
  ІПН (`identification_id`); перевірка унікальності (не може бути 2 контакти з одним кодом).
- **Що синхронізуємо:** напр. `product.template` синхронізується, `product.product` — ні.

### 2.5 Статус мапінгу
Для відстеження тестування — колонка статусу по кожному полю: `Pass` / `FAIL` / `-` (не
застосовно) + примітка з проблемою. Зручно для приймання інтеграції по полях.

### 2.6 Транспорт/API (Odoo 19)
Для нових інтеграцій у v19 використовуй **новий endpoint `/json/2/<model>/<method>`** з аутентифікацією
**API Key (Bearer)** (старі XML-RPC/JSON-RPC deprecated, видалення у v22). Деталі — `odoo19-довідник.md` §15.
Кастомні точки обміну на стороні Odoo — контролери `@http.route(type='jsonrpc'|'http')` (§13).
