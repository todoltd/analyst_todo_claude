# odoo19-tr-authoring

Методологія шару **ТР (Технічне рішення)** для проєктів Odoo 19 — міст між аналізом і білдом.
Перетворює вхід (бриф / транскрипт / задача в ERP **або** журнал дискавері через
`journal-to-tr-builder`) на узгоджений `ТР_[назва].md`, який далі стає входом у MVP/БІЛД
(`spec-writer: затверджений ТР → SPEC.md`).

Це продуктизація шару, що раніше жив як вільні `.claude/skills` («поза пакетами методології»).

## Скіли (5)

1. `tr-usecases-acceptance` — Use Cases (сценарії) + Acceptance Criteria (Given/When/Then або УМОВА/КОЛИ/ТОДІ); несе `origin` на AC.
2. `tr-odoo-tech-design` — «Опис технічної реалізації»: моделі `td.*`, поля, довідники, права, подання, кнопки; кожен пункт звірено з докою Odoo 19.
3. `tr-effort-instruction` — «Обсяг робіт» (оцінка людино-годин, діапазон) + «Інструкція» (тест-сценарій). Калібрування на реальних ТР.
4. `tr-review` — QA-перевірка консистентності ТР перед узгодженням клієнтом (вимоги→AC→реалізація→оцінка).
5. `tr-registry-update` — внесення/оновлення статусу в реєстрі «Перелік технічних рішень».

## Ланцюжок

```
вхід: журнал дискавері (journal-to-tr-builder) АБО бриф/транскрипт/задача
    ↓ (каркас ТР + секція 1 — journal-to-tr-builder з пакета closeout, коли з дискавері)
tr-usecases-acceptance → Use Cases + AC (+origin)
tr-odoo-tech-design    → Опис технічної реалізації (td.*)
tr-effort-instruction  → Обсяг робіт + Інструкція
tr-review              → QA
    ↓ [ УЗГОДЖЕННЯ КЛІЄНТА · реєстр «Погодження клієнта» ]
tr-registry-update     → реєстр; АРХІТЕКТУРА — заповнює архітектор
    ↓ approved
→ spec-writer (ТР → SPEC.md) → MVP / БІЛД
```

Каркас ТР (`tr-scaffold`) і секцію 1 (`tr-functional-requirements`) у цьому циклі **виконує
`journal-to-tr-builder`** (пакет `odoo19-discovery-closeout`) — окремих скілів під них немає
(один namespace, контракт SDD).

## AC-01 — наскрізна нитка
`AC-01` народжується в `tr-usecases-acceptance` і йде наскрізь: `ТР → SPEC → # TODO: AC-01 →
test_ac01 → pr-review` (Трек 1) і `[AC-01] у CONFIG.md` (Трек 2). `origin: D-N/T-N/сесія N` —
backref угору на рішення дискавері.

## Довідники (docs/)
Кожен скіл несе у своєму `docs/` потрібні йому довідники (`шаблон-ТР`, `odoo-конвенції`,
`odoo19-довідник`, `патерни-UC-AC`, `звіти-та-обміни`, `патерни-Odoo-рішень`, `друковані-форми`,
`калібрування-оцінки`, `правки-та-доопрацювання`, `приклади`). Майстер-набір і
`0_МАЙСТЕР-ІНСТРУКЦІЯ.md` — у `docs/` пакета (джерело істини).

## Збірка та інсталяція

```bash
cd "СКІЛИ_ТР" && bash ЗІБРАТИ_plugin.sh
```

Інсталяційна картка з кнопкою з'являється лише коли `.plugin` доставлено через папку **outputs**
сесії Cowork. (Опис `plugin.json` — короткий: довгий валить картку.)
