## Універсальний цикл перевірки (4 кроки)

Кожна ітерація — і поскілова, і інтеграційна — проходить **один і той самий цикл**:

1. **Аналіз ризиків і помилок — дуже ретельно.** Зафіксувати всі дефекти/ризики у фокусі ітерації, з рівнем 🔴/🟡/🟢. Дивитися на **зміст**, а не лише на наявність рядків (урок «зʼїдених імен»).
2. **Пропозиція з оптимізації.** На кожен дефект — конкретна пропозиція правки (що, де, як). Узгодити спірне з власником методології.
3. **Внесення правок.** Застосувати погоджені зміни в SKILL.md (і, де треба, в інструкціях).
4. **Перевірка, що все внесено.** Прогнати `валідатор_скілів.py` + переглянути **фактичні діфи**; підтвердити, що кожна правка з кроку 2 реально на місці й нічого не зіпсовано. Ітерація закрита тільки коли крок 4 чистий.

**Інструменти кроку 4:** `валідатор_скілів.py` (інваріанти вмісту), `git diff` / порівняння з бекапом, повторний прогін тригерів.

## Частина A — по 5 ітерацій на КОЖЕН скіл

Кожен із 7 скілів проходить ту саму **драбину з 5 ітерацій** (фокус поглиблюється), і на кожній — цикл з 4 кроків вище.

| Ітерація | Фокус | Що перевіряємо |
|----------|-------|----------------|
| **И1 — Механіка/консистентність** | форма | frontmatter, тригери (без латиниці поза дозволеними іменами), плейсхолдери `[назва_компанії]`, імена артефактів, крос-посилання на інші скіли, парність code-fence, «зʼїдені імена», українізація |
| **И2 — Структура/повнота** | повнота | наявність усіх секцій (Крок 0, «що робить», «як запустити», вхід, шаблон-результат, «на що звернути увагу», верифікація, підказка «наступний крок»); повнота шаблону виводу; граничні гілки (немає матеріалів / частковий вхід); нумерація без дублів |
| **И3 — Логіка контенту + методологія** | сенс | коректність інструкцій; гейти (немає джерела → лише питання); гіпотеза vs факт; статус обіцянок; єдиний глосарій приоритетів і формат ризиків; відповідність політиці ToDo (Установча Зустріч / 60 хв / тільки scope / бюджет без цифр / без списку юзерів); відсутність домислів |
| **И4 — Доменна коректність** | домен | модулі Odoo 19 (імена/наявність), локалізації `l10n_ua`/`l10n_pl`, регуляторика (KSeF дата/етапність, JPK, ПДВ), якість галузевих питань/сценаріїв — **з веб-перевіркою + експертом** |
| **И5 — Рантайм + адверсаріал + регресія** | поведінка | реальні прогони через тригери на нормальному / порожньому / сміттєвому / суперечливому / надлишковому вході; гранична логіка (Крок 0); **регресія всіх правок И1–И4**; чистова верифікація валідатором |

**Критерій «скіл готовий»:** пройдені всі 5 ітерацій, 0 🔴, валідатор зелений, регресія чиста.

### Риск-хотспоти по скілах (вхід для И1 і И3–И4)

| Скіл | Ключові ризики під пильну увагу |
|------|--------------------------------|
| `kickoff-agenda-finalizer` | структура 60/45 хв; зовнішня vs внутрішня версія; відсутність тех-блоку; узгодженість з `discovery-context-builder`; блок «узгодження очікувань» при бюджет-ризику |
| `kickoff-transcript-analyzer` | коди `[КД]/[ВД]` і глосарій; повнота вилучення (вимоги/ризики/scope-зміни); поведінка на неповному транскрипті; формат ризику зі сквозним ID |
| `discovery-charter-writer` | чи не повернулись числа/обіцянки строків; тон під клієнта без досвіду; «що отримає клієнт / що потрібно від вас»; лише для клієнта; узгодженість із зовнішньою agenda |
| `industry-briefing` | таблиця модулів Odoo 19 з приоритетами; KSeF (дата/етапність), JPK, `l10n_*`; галузеві «червоні прапорці scope»; застарілі/вигадані дані |
| `client-web-researcher` | gate при недоступному джерелі; «1 вакансія = `[припущення, 1 джерело]`»; модулі лише при ≥2 сигналах; заблокований LinkedIn/JS-сайти; дедуплікація з competitor-mapper |
| `competitor-mapper` | «конкурент на Odoo = гіпотеза, перевірити»; IT-стек зі слабких сигналів (gate); дедуплікація з client-web; сигнали scope ✅/💡/⚠️ |
| `discovery-plan-builder` | кількість сесій від інвентаря процесів (не від ярлика); обовʼязкові лінії (дані/міграція, інтеграції, доступи, NFR, навчання+go-live) + pre-save чек; план = гіпотеза v1.0; робота без `аналіз_УЗ` |

## Частина B — 5 інтеграційних ітерацій (усі разом, «на звʼязці»)

Перевіряємо 7 скілів **у звʼязці** (і їхні стики з рештою 3). Кожна — той самий 4-кроковий цикл.

| Ітерація | Фокус | Що перевіряємо |
|----------|-------|----------------|
| **Інт-1 — Термінологія і сцепка артефактів** | словник | єдині терміни на весь етап; імена входів/виходів збігаються (вихід скіла N = вхід N+1); єдиний глосарій кодів приоритетів |
| **Інт-2 — Наскрізний handoff** | ланцюг | кожен скіл коректно приймає вихід попереднього; немає розривів у блок 1→2→3; спільні дані (профіль, ризики, scope) не губляться |
| **Інт-3 — Несуперечливість і єдине джерело правди** | конфлікти | фінальні питання генерує лише `discovery-context-builder` (решта — сигнали); ризики в єдиному форматі зі сквозним ID; немає суперечливих рекомендацій між скілами |
| **Інт-4 — End-to-end на 2–3 реальних кейсах** | поле | повний шлях від матеріалів менеджера до плану дискавері (Supplychains + 1–2 нових); звірка з еталоном/очікуваннями BA; дефекти, що зринають лише end-to-end |
| **Інт-5 — Регресія + go/no-go + пересборка** | реліз | повна регресія всіх 40 циклів; інтеграційний прогін валідатора; пересборка плагіна (нова версія) + dogfood; інтеграційний go/no-go і реєстр залишкових ризиків |

**Критерій «інтеграція готова»:** Інт-1…Інт-5 пройдені, end-to-end на реальних кейсах без 🔴, валідатор зелений, плагін пересібрано і провалідовано.
