# Базлайн ланцюга скілів `tr-*` (станом до правок)

Дата: 2026-06-12 · Етап 0 (інвентаризація). Джерело істини скілів: `СКІЛИ_ТР/skills/`,
`СКІЛИ_ЗАВЕРШЕННЯ/skills/journal-to-tr-builder`, `СКІЛИ_MVP/skills/spec-writer`.

## Реальний ланцюг (як стикуються скіли)

```
journal-to-tr-builder (closeout)         → секція 1 «Функціональні вимоги»: ВИМ-NN + origin (D-N/T-N/сесія)
  ↓                                         + скаффолд ТР (заголовки 1–6 укр. дослівно, плейсхолдери)
tr-usecases-acceptance (ТР)              → секція 2 «Сценарії використання та критерії приймання»: UC + AC
  ↓
tr-odoo-tech-design (ТР)                 → секція 3 «Опис технічної реалізації»: td.* моделі, поля, права, подання + звірка з Odoo 19
  ↓
tr-effort-instruction (ТР)               → секція 5 «Обсяг робіт» + секція 6 «Інструкція/Протокол тестування»
  ↓
tr-review (ТР)                           → QA-гейт консистентності (ЗВІТ_QA_*.md), Формат A/B і C
  ↓
tr-registry-update (ТР)                  → рядок у «Перелік технічних рішень», статус
  ↓ (статус «Погодження клієнта»)
spec-writer (MVP)                        → SPEC.md (проекція ТР, AC-01 verbatim) → architecture → … → pr-review
```

«4 цільові скіли» з промту = link1 (`journal-to-tr-builder`, бо окремого `tr-functional-requirements` НЕМАЄ),
`tr-usecases-acceptance`, `tr-odoo-tech-design`, `tr-review`. Дотичні: `tr-effort-instruction`,
`tr-registry-update`, `spec-writer`.

## Базлайн по кожному скілу

### journal-to-tr-builder — секція 1 (вхід ланцюга)
- **Вхід:** `журнал_дискавері_*.md` + `gap-аналіз_*.md` (+scope, +план). **Вихід:** `ТР_[назва].md` (шапка + секція 1 + плейсхолдери 2–6).
- **ID:** вимоги `ВИМ-NN`, кожна з `origin:` (D-N/T-N/сесія N) + `gap: GAP-N`. Канонічний формат AC униз — **`AC-NN`** (один namespace), origin має пропагуватись на AC.
- **Контракти:** заголовки секцій укр. дослівно; плейсхолдер входу UC/AC рівно `<!-- TODO: tr-usecases-acceptance -->`; не перенумеровувати; не вигадувати за наступні скіли.
- **Стале:** у скаффолді коментар «скіл tr-effort — у розробці», хоча `tr-effort-instruction` уже існує.

### tr-usecases-acceptance — секція 2
- **Вхід:** секція 1 + плейсхолдер. **Вихід:** UC (≥3 сценарії), за потреби User Story, AC.
- **Формати AC:** EN Given/When/Then **або** UA УМОВА/КОЛИ/ТОДІ; «Нумеруй **AC1, AC2… або групами AC 1.1, 1.2 чи Scenario 1, 2…**».
- origin переноситься на AC, якщо вимога його несе. Обов'язкові «нещасливі» шляхи (немає даних/помилка/права/межі). Pre-save: кожна вимога→AC, кожна кнопка→AC.
- **Стале/розрив:** Крок 0 посилається на неіснуючі `tr-scaffold` і `tr-functional-requirements`.

### tr-odoo-tech-design — секція 3
- **Вхід:** секція 1 + AC. **Вихід:** опис сутностей: призначення+`td.<entity>`, таблиця полів `| Назва | Тип | Звідки | Примітка |`, довідники, права (Користувач/Адмін), подання, smart-кнопки, кнопки-дії, related/computed, `ir.sequence`, UI/UX окремо, модуль/версія.
- **Крок 1 ОБОВ'ЯЗКОВА звірка з Odoo 19** по кожному пункту: спершу `docs/odoo19-довідник.md`, далі офіційна дока/`.rst`; вказати джерело; синтаксис лише v19 (`<list>`/`invisible`/`t-out`/`aggregator`/`/json/2`).
- Формат для складних фіч: `BUSINESS RULES` + `UI/UX`. Звіти/обміни — за `docs/звіти-та-обміни.md`.
- **Стале/розрив:** Крок 0 теж посилається на `tr-scaffold`.

### tr-review — QA-гейт
- **Вхід:** `ТР_*.md`. **Вихід:** `ЗВІТ_QA_*.md` (🔴/⚠️/✅), Формат A/B (повний) і C (патч, полегшений чек-лист).
- Перевіряє: структуру/плейсхолдери; назва файлу = «Найменування модифікації»; тестовий сценарій заповнений; вимога→AC→реалізація; кнопка→AC; нещасливі шляхи; конвенції `td.*`; **звірку з Odoo 19 + джерело** (пункт без джерела = 🔴); застарілий синтаксис = 🔴; оцінка (арифметика) + інструкція.
- **Не ловить (попередньо):** контроль **формату/namespace AC** (AC-NN vs AC1/Scenario) — а саме він рве стиковку зі spec-writer; ID-трасування ВИМ-NN→AC→реалізація (перевіряє «кожна вимога має AC», але не за id); пропаганду origin на AC.

### tr-effort-instruction / tr-registry-update / spec-writer (дотичні)
- effort: секції 5/6, оцінка діапазоном (калібрування `docs/калібрування-оцінки.md`). registry: рядок реєстру + статус, дедуп за ID задачі. spec-writer: проекція ТР→SPEC, **hard-stop** якщо AC не у форматі `AC-01`.

## Карта розривів стиковки (попередня — до корпусної валідації)

| # | Розрив | Деривація | Дія (Етап 4) |
|---|--------|-----------|--------------|
| R1 | **Колізія формату AC.** `tr-usecases-acceptance` друкує `AC1/AC 1.1/Scenario N`; `journal-to-tr-builder` вимагає єдиний `AC-NN`; `spec-writer` hard-stop на legacy. Середній скіл поза контрактом з обома сусідами; рветься хребет `AC-NN → test_acNN → pr-review`. | Контракт скілів (точно) + корпус (який формат реально вживають) | Уніфікувати на `AC-NN` у `tr-usecases-acceptance`; додати перевірку формату/namespace у `tr-review` |
| R2 | **Мертві посилання** на `tr-scaffold` / `tr-functional-requirements` у Крок 0 `tr-usecases-acceptance` і `tr-odoo-tech-design`. | Структура пакета (точно) | Замінити на `journal-to-tr-builder` |
| R3 | **ID-трасування ВИМ→AC→реалізація** не за id. journal друкує `ВИМ-NN`, але AC не зобов'язані backref-ити `ВИМ-NN`; review перевіряє «вимога має AC», не за id. | Контракт + корпус | Додати backref `ВИМ-NN` на AC; review ловить «ВИМ-NN без AC» |
| R4 | **Пропаганда origin на AC** прописана, але не входить у чек-лист `tr-review`. | Контракт | review-чек: вимога з origin → її AC мають origin |
| R5 | **Стале «tr-effort — у розробці»** у скаффолді journal-to-tr-builder. | Структура (точно) | Прибрати/оновити коментар |

R1, R2, R5 — деривуються з самих контрактів (можна правити без корпусу). R3, R4 і **усі правила якості AC/техдизайну/Odoo-точності** — потребують корпусної валідації (Етапи 2–3) + звірки з Odoo 19, перш ніж ставати правилами скіла.

## Інвентаризація корпусу (стисло)
- ~168 унікальних реальних ТР у 30/68 проєктів. Топ: esper-bionics ~35, universalna ~16, 2-архів ~16, демз 12, тов-стар-світ-юа-логістика 11.
- Метадані `find_documents` НЕ містять статусу/дати/розміру (`author`/`modified_at` = null). Статус — лише з 3 реєстрових аркушів «Перелік технічних рішень» (ельфа-лабораторія 18, демз ~36, тов-2021-солюшинс 1) або з тіла ТР.
- Методологічних `журнал_дискавері_*` у базі **немає** (0). SDD `SPEC.md` немає (матчі «Spec» = продуктові PDF RSE). Отже корпус = реальні ТР, написані вручну — це калібрувальна база (аналог 41 протоколу EDELWEIS), а не вихід методології.
- Стратегія: **map-reduce** (>100) — стратифікована вибірка глибокого читання + семантичні зрізи по всій базі.
