# tr-odoo-tech-design — «Опис технічної реалізації»

## Мета
Перекласти «що потрібно» в «як це влаштовано в Odoo»: моделі, поля, джерела значень, довідники,
права, подання, кнопки. Достатньо конкретно, щоб архітектор і розробник не здогадувались, і у
стилі ToDo (моделі `td.*`, поля `td_*`, маски імен `<Проект>: ...`).

## Інструменти та середовище
**Вхід:** `ТР_[назва].md` (секція 1 + AC + плейсхолдер `<!-- TODO: tr-odoo-tech-design -->`) → `Read`/`Edit`.
**Збереження:** впиши у `ТР_[назва].md` → `Edit`.
**Глибокі довідники (обов'язково прочитай):** `docs/odoo-конвенції.md` — іменування, типи полів,
стандартні моделі (за доменами: продажі/MRP/логістика/проекти/бухгалтерія), формат таблиці полів,
довідники, права, подання, синхронізація, версія/модулі/локалізація, маппінг станів, чек-лист.
Для **звітів (Pivot)** і **обмінів/інтеграцій** — `docs/звіти-та-обміни.md` (інша структура опису).
**Перед описом реалізації звіряйся з `docs/патерни-Odoo-рішень.md`** — якщо задача підпадає під
архетип (wizard, ir.cron, singleton-флаг «актуальна», версійність BoM за датами, авто-номер партії,
пакетне створення документів, portal-форма, QWeb-друк тощо), бери готову канву звідти.
**Для точних механізмів/синтаксису Odoo 19** (типи полів, `<list>`, `invisible`/`required` замість
`attrs`, record rules, дії, QWeb `t-out`, домени) — `docs/odoo19-довідник.md`.

## Крок 0: Визнач вхідні дані
- Знайди `ТР_*.md` (кілька — спитай яку задачу). Немає — запропонуй `journal-to-tr-builder`
  (він скаффолдить ТР і пише секцію 1; окремого `tr-scaffold` у методології немає).
- Спирайся на секцію 1 і AC: кожна вимога/критерій має технічне втілення (поле/модель/правило).
  Якщо їх ще немає — можна працювати з брифу, але познач, що покриття орієнтовне.

## Крок 1: ОБОВ'ЯЗКОВА звірка з офіційною документацією Odoo 19 (по кожному пункту)
**Для КОЖНОГО пункту, що дорабатывается, ПЕРЕД описом реалізації звір механізм з офіційною
документацією Odoo 19. Без цієї звірки реалізацію не пиши.** Навіщо: уникнути неіснуючих полів/методів,
застарілого синтаксису і рішень «по пам'яті» — реалізація має спиратися на актуальний v19, а не на старі
звички з корпусу.

Порядок на кожен пункт:
1. **Спершу `docs/odoo19-довідник.md`** — він зведений з офіційної документації Odoo 19.0 і покриває
   поля/представлення/безпеку/дії/cron/QWeb/mail/automation/settings/web/API/домени. Знайди розділ по темі пункту.
2. **Якщо пункт стосується конкретного стандартного модуля/поля/методу або деталі, якої немає в довіднику —
   відкрий офіційну документацію наживо і звір:**
   - `https://www.odoo.com/documentation/19.0/` (developer → reference);
   - якщо сторінка порожня/рендериться JS — першоджерело `.rst`:
     `https://raw.githubusercontent.com/odoo/documentation/19.0/content/developer/reference/<тема>.rst`.
3. **Вкажи джерело біля пункту** реалізації: `(Odoo 19: довідник §N)` або `(Odoo 19 docs: <URL>)`.
4. **Синтаксис — лише v19:** `<list>` (не `<tree>`); `invisible/required/readonly` (не `attrs`/`states`);
   `t-out` (не `t-raw`); `aggregator` (не `group_operator`); зовнішній API — `/json/2`+Bearer.
5. **Звіряй не лише механізм, а й ТОЧНЕ написання кожного імені стандартної моделі/поля** — у корпусі
   одруки тиражуються між ТР (`parther_id`→`partner_id`, `signateru_date`→`signature_date`,
   `tock.quant`→`stock.quant`, `srm.lead`→`crm.lead`, `read.only`→`readonly`). Наявність/залишки —
   модель `stock.quant` (а `product.product.qty_available` — обчислюване з неї), НЕ `product.product` як таблиця залишків.
Немає доступу до джерел → познач пункт «потребує звірки з Odoo 19», не видавай непідтверджене за остаточне.

## Крок 2: Опиши кожну сутність (вихід українською)
Для кожної нової/змінюваної сутності по порядку:
1. **Призначення** і технічне ім'я (`td.<entity>`, за потреби `td.<entity>.line`), зв'язок зі
   стандартними моделями; за доречності «реалізувати на прикладі моделі X» (напр. `stock.picking`).
   **ОБОВ'ЯЗКОВО:** кожна модель/поле згадується з технічним іменем — людська назва без `td_*`/`td.*`
   у дужках неприпустима (відсутні техімена — розрив консистентності №1 корпусу, що змушує архітектора домислювати).
2. **Таблиця полів:** `| Назва поля | Тип поля | Звідки тягнути значення | Додаткова примітка |`
   — тип Odoo (many2one, selection, monetary, boolean, related, computed, AI-поле…), джерело, domain,
   required / read-only / store, валюта для monetary (UAH). Для складних форм додай колонку
   **«Розташування»** (вкладка/секція/позиція) і фіксуй **умовну видимість/обов'язковість**:
   «видиме/обов'язкове, якщо <інше поле> = <значення>». Синтаксис Odoo 19 — прямі `invisible`/`required`/
   `readonly` з Python-виразом (НЕ `attrs`/`states`); табличне подання — `<list>` (не `<tree>`). Точні
   механізми/назви v19 — `docs/odoo19-довідник.md`. **Єдиний словник імен:** одне поле — одне технічне
   ім'я через УСІ секції ТР (у корпусі те саме поле зустрічається під 3–4 іменами — це ламає здачу в код);
   введи ім'я раз і вживай дослівно скрізь.
3. **Довідники** під selection/many2many: `| Довідник | Значення |` + шлях у UI (`<Модуль> — Налаштування`).
4. **Права доступу:** матриця мінімум Користувач / Адміністратор; read-only по групах; маска
   `<Проект>: ...`; record rules і constrains.
5. **Подання та UI:** Канбан / Список / Карта / Зведена таблиця; стадії (Канбан); smart-кнопки
   (перехід + фільтр); кнопки-дії (коли видно, що робить, що після: стадія, чатер, зникнення, повідомлення).
6. **Синхронізація related/computed:** напрямок (напр. parent→child без зворотного), перелік
   полів, що синхронізуються, поведінка при зміні джерела, випадки ручного оновлення.
7. **Автонумерація/автоназва** через `ir.sequence` з префіксом, якщо застосовно.
8. **UI/UX окремою секцією** (для фіч з інтерфейсом): розділ меню/новий модуль, форма (шапка,
   вкладки, колонки таблиці), кнопки панелі дій (коли активні/недоступні), статусний badge/pipeline.
   Не ховай UI в тексті — виноси окремо.
9. **Модуль і версія:** новий функціонал — у новому модулі з префіксом проекту
   (`<проект>_<область>`, напр. `olimpius_delivery`, + іконка); моделі лишаються `td.*`. Версію Odoo
   бери з картки проекту (18/19), не хардкодь.

**Формат для складних фіч (новіший стандарт).** Структуруй як `BUSINESS RULES` (нумерований перелік
правил з технічними іменами, формулами, дефолтами, доменами і **дослівними текстами помилок** у
лапках) + окремий `UI/UX`. Поведінкову частину (USER STORY, сценарії, AC) тримай у
`tr-usecases-acceptance`. Маппінг станів Odoo → українські підписи наводь явно (див. конвенції §12).

**Спеціальні типи.** Якщо це **звіт** (Pivot) або **обмін/інтеграція** — будуй опис за
`docs/звіти-та-обміни.md` (виміри/міри/логіка розрахунку з fallback; мапінг полів з технічними
іменами + правила синхронізації + натуральні ключі ЄДРПОУ/ІПН), а не за загальним форматом сутності.

Наприкінці пройди **чек-лист повноти** з `docs/odoo-конвенції.md`.

## Крок 3: Не змішуй шари
Тут — будова (поля/моделі/права), а спостережувана поведінка і перевірки — в Use Cases/AC.
**Реалізація = конкретика, не проза:** кожен AC/вимога лягає в поле/модель/метод/правило. Заголовок
секції без полів і моделей («реалізувати передачу примітки») — це НЕ реалізація (часта прогалина корпусу).
Якщо випливає нова поведінка — додай User Story + AC у `tr-usecases-acceptance`, а не «воно само працює».
Де реалізація неоднозначна — дай «Варіанти рішення», не обирай мовчки за архітектора.

## Pre-save верифікація (за чек-листом конвенцій)
- [ ] **кожне поле/модель має технічне ім'я** (`td.*`/`td_*`) — немає назв «лише UA-міткою»;
- [ ] **одне поле — одне технічне ім'я через усі секції** (єдиний словник, без синонімів/одруків);
- [ ] таблиці полів з типами/джерелами/domain/required/read-only/store;
- [ ] довідники зі значеннями і шляхом у UI;
- [ ] матриця прав + record rules/constrains;
- [ ] подання, smart-кнопки, кнопки-дії з пост-ефектами;
- [ ] правила синхронізації related/computed;
- [ ] іменування за конвенціями (`td.`/`td_`, маска `<Проект>:`); технічні імена в дужках біля людських;
- [ ] **кожен AC/вимога має втілення** (поле/модель/метод/правило) — немає реалізації «на словах»;
- [ ] **КОЖЕН пункт і кожне ім'я стандартної моделі/поля звірено з Odoo 19** — вказано джерело; без одруків стандартних імен (`partner_id`, `stock.quant`, `crm.lead`…);
- [ ] синтаксис лише v19 (немає `attrs`/`states`/`<tree>`/`t-raw`/`group_operator`/xmlrpc).

## Наступний крок
Повідом: «Технічну реалізацію описано. Далі — tr-effort-instruction (оцінка + інструкція)?»

## Чого НЕ робити
- Не заповнюй секцію **АРХІТЕКТУРА** (об'єкти на рівні БД, «Налаштування системи») — зона
  архітектора після узгодження. Перелік нових об'єктів можна лишити як гіпотезу, не як фінал.
- Не вигадуй і не одруковуй стандартні моделі/поля Odoo (`product.product` як таблиця залишків замість
  `stock.quant`; HTTP-`403` як механізм блокування замість `AccessError`/`UserError`) — звір або познач як припущення.
- **Не пиши реалізацію пункту без звірки з офіційною документацією Odoo 19** (Крок 1); не видавай
  застарілий синтаксис (`attrs`/`<tree>`/`t-raw`/xmlrpc) за актуальний.
- Не порушуй конвенції іменування і не змінюй українські заголовки.
