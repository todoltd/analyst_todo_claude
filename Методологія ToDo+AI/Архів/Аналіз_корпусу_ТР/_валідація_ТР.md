## 3. Покращення за 4 критеріями

### Критерій 1: Консистентність (чи новий гейт ловить розриви, що раніше проходили)

**Так, підтверджено в 6 з 7 змістовних ТР.** Ключові нові спрацювання:

| Розрив | Де спрацював | Новий vs старий |
|---|---|---|
| `res.sity` (одрук стандартної моделі) | H4 | **Тільки новий** — правило sanity-флагу підозрілих ідентифікаторів |
| Не-`AC-NN` namespace | H1, H3, H4, H5, H6, H7, H12 | **Тільки новий** — явне 🔴 за відсутності `AC-NN` |
| C2: одне поле під двома іменами | H12 (`fiscal_company_id`/`td_fiscal_company_id`) | **Тільки новий** — правило крос-секційної звірки |
| C1: нові моделі без техімені | H3 (Бренди, Знижки) | **Тільки новий** — розширена область: не лише поля, а й моделі |
| Арифметика «Всього» | H7 (27 год. не порахували) | **Тільки новий** — перевірка суми |

### Критерій 2: Odoo-точність (вигадані/одрукані поля)

- **H4 `res.sity`**: одрук `res.city` → спіймано новим гейтом (C3-правило).
- **H1**: відсутній маппінг до моделі (не вказано `crm.lead`/`res.partner`) — нова вимога техімені з моделлю.
- **H12**: `fiscal_company_id` без `td_` — невідповідність конвенції — новий гейт ловить.
- **Вигаданих/неіснуючих стандартних моделей** типу `tock.quant`, `srm.lead` у цьому held-out зрізі **не знайдено** — але правило sanity-flag у гейті є і H4 підтвердив реальність такого класу помилок.

**Підсумок**: у 3 з 7 змістовних ТР виявлено Odoo-іменні проблеми, всі 3 ловляться новим гейтом. Старий — жодну.

### Критерій 3: Менше ручних правок BA

**Так**, для 5 з 7 змістовних ТР нові правила дають BA **конкретний список** замість «щось не так»:
- H4: BA тепер знає що `res.sity` → `res.city` (конкретний виправ).
- H12: BA знає що `fiscal_company_id` → `td_fiscal_company_id` скрізь (конкретний виправ).
- H3: BA знає що для Бренди потрібно `td.brand`/`td_brand` — конкретне завдання для BA/архітектора.

**Обмеження**: для повністю порожніх ТР (H2, H8–H11) — BA отримає правильний висновок «документ не заповнений», але гейт не може «допомогти наповнити» — це задача профільних скілів.

### Критерій 4: Якість AC (unhappy paths)

**Покриття unhappy paths у held-out корпусі: 1 з 7 змістовних ТР** (H4 — частково).

- H1, H3, H5, H6, H7 — **нуль** unhappy AC.
- H12 — є сценарії повернення (частковий unhappy).
- H4 — є тест-крок «відкликання прав» (unhappy для права).

Новий гейт правильно позначає відсутність unhappy paths як 🔴 для кожної нетривіальної дії. Це підтверджує системну прогалину корпусу: **unhappy paths майже не пишуть**.

## 4. Залишкові розриви + правила, що НЕ спрацювали → беклог

### 4.1. Правила, що спрацювали некоректно або неповністю

**Б1 — Гейт не розрізняє «ТР у розробці» та «ТР готовий до узгодження»**
H1, H5, H6, H7 мають ознаки незавершеного документа (порожня «Інструкція», незаповнений «Обсяг»), але немає способу встановити, чи це «чернетка», чи «submitted for review». Гейт виставляє 🔴 однаково. Потрібна перевірка дати/статусу або явний disclaimer. Беклог: додати до Кроку 0 перевірку «це чернетка чи фінал?»

**Б2 — Правило C4 (copy-paste) не спрацювало**
H2, H8–H11 — 5 ТР є ідентичними копіями порожнього шаблону з різною назвою проєкту в шапці. Гейт виставляє «плейсхолдери = 🔴», але не виявляє крос-документну ідентичність (у цьому контексті це менш критично, але при автоматизованій перевірці — сліпа зона).

**Б3 — «Посилання на зовнішній файл» як реалізація не помічено явно**
H5 (звіт по xlsx-посиланню), H6 (посилання на «існуюче доопрацювання»), H7 (вебхук URL в тілі) — гейт ловить C4 («посилань „як у проєкті X" без опису = 🔴»), але правило сформульоване для крос-ТР посилань, а не для зовнішніх URL. Потрібна окрема перевірка: «посилання на зовнішній Google Doc як замінник технічного опису = 🔴».

**Б4 — Відсутність `AC-NN` у ТР з «Критерії успішного прийняття» (H4)**
H4 має секцію «Критерії успішного прийняття» з bullet-списком — це де-факто AC, але не `AC-NN`. Гейт виставляє 🔴, але виправлення тривіальне (пронумерувати як `AC-01`…). Варто додати до гейту підказку: «якщо є блок „Критерії прийняття / Acceptance Criteria" — пронумерувати як `AC-NN`, а не відкидати як відсутні AC».

**Б5 — Odoo-версія в тексті vs шапці**
H12 — у шапці порожня дата і версія, але в тексті вживається «Odoo» без версії. H4 — у шапці `Odoo 19CE` ✅. Гейт перевіряє «версія у шапці», але не перевіряє **суперечність між шапкою та текстом**. Якщо в шапці «Odoo 17», а в тексті описується синтаксис v19 — гейт пропустить. Беклог: додати правило «версія в тексті ≈ версії в шапці».

**Б6 — Відсутність довідника §N або URL для Odoo-звірки**
Гейт вимагає: «кожен пункт реалізації звірено — вказано джерело (довідник §N або офіційний URL)». У жодному з 7 змістовних held-out ТР такого джерела немає. Правило технічно правильне, але **ніколи не виконується** на практиці — або правило потрібно послабити до «⚠️ за відсутності джерела» замість 🔴, або очікувати, що воно буде масовим 🔴 по всьому корпусу.

## 6. Висновок

Оновлений гейт `tr-review` **суттєво сильніший** за попередню версію: він ловить клас помилок (не-`AC-NN`, C2 крос-секційна невідповідність, C3 одруки стандартних імен, C1 нові моделі без техімені), які корпус системно несе, а старий гейт пропускав.

Ключові беклог-пункти для наступної ітерації:
1. Послабити «відсутність джерела Odoo §N» до ⚠️ (зараз дає масовий нереальний 🔴).
2. Додати підказку для «Критерії прийняття → пронумерувати як AC-NN» замість «AC відсутні».
3. Додати перевірку: «зовнішній Google Doc / URL як замінник технічного опису = 🔴».
4. Крок 0 — питання «чернетка чи фінал» для пом'якшення помилок незавершених документів.
5. Правило суперечності версії в шапці та тексті.
