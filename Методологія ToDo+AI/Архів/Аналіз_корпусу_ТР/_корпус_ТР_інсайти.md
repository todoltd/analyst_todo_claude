# Інсайти корпусу ТР (Етап 3 — REDUCE + звірка з Odoo 19)

Дата: 2026-06-12. База: **90 ТР прочитано дослівно** (батчі A–H) з 168 у корпусі (30 проєктів) +
семантичні зрізи по всій базі + звірка механізмів з офіційною докою Odoo 19. Логи: `_патерни_ТР_лог.md`,
`_семантичні_зрізи.md`. Кожне твердження — з частотою (скільки ТР підтверджують).

Розподіл прочитаного: A демз 10 · B тов-стар 11 · C тов-2021+ельфа 11 · D бріолекс+ярослав 10 ·
E агро+одиночки 9 · F esper(семпл) 8 · G universalna+2-архів(семпл) 11 · H revolution+одиночки 20.

## 0. Головний висновок (перевертає рамку R1)

У реальному корпусі **формальних Acceptance Criteria як класу майже немає**: окрема секція UC+AC з
AC-ідентифікаторами трапилась у **~3 з 90** ТР (ельфа «Лістинг» GWT-EN ~26 AC; revolution «Кастомні
права» 11 AC; edelweis «Примітки» v2.0 прото-BDD). Домінанта — **нумерований список вимог (1.X / 2.1)
з inline-умовами «якщо X → Y»**; критерії приймання, де є, «живуть» у таблиці тест-кроків або
«Умова→Поведінка». Given/When/Then і `УМОВА/КОЛИ/ТОДІ` системно відсутні; `Scenario N` — лише в секції
тестування ~2 ТР.

Наслідок для ланцюга: проблема не лише «не той формат нумерації AC» (R1), а **AC треба ВВОДИТИ** там,
де їх немає. При цьому `journal-to-tr-builder` вимагає єдиний namespace `AC-NN`, а `spec-writer` робить
hard-stop на не-`AC-NN` → за нинішнього `tr-usecases-acceptance` (друкує `AC1/AC 1.1/Scenario N`) майже
весь реальний стиль корпусу заблокувався б на вході в білд. Це найсильніший, корпусно-підтверджений
аргумент за R1-фікс + посилення `tr-review`.

## 1. Канонічна структура ТР (частоти по 90 ТР)

| Секція | Присутня / заповнена | Якість |
|---|---|---|
| 1. Функціональні вимоги | ~100% присутня | Нумерований список, часто без ID `ВИМ-NN`; backref на джерело — майже ніколи |
| 2. UC + AC | окрема секція ~3% | Переважно немає; замінники — нумеровані вимоги, «Умова→Поведінка», тест-кроки |
| 3. Опис тех. реалізації | присутня ~70%, заповнена нерівномірно | Часто без технічних імен; «на словах» без полів/моделей у ~частині |
| 4. Архітектура | порожня ~85–90% | **Коректно** (заповнює архітектор) — НЕ дефект |
| 5. Обсяг робіт (години) | у шаблоні є, заповнена частково/рідко | Системний дефіцит → зона `tr-effort-instruction` |
| 6. Тест/Інструкція | у шаблоні є, заповнена ~30–40% | Решта — плейсхолдер; часто єдине місце, де є де-факто критерії |

Стабільні артефакти-перешкоди: задвоєння документів (експорт Google Docs), назва файлу ≠ «Найменування
модифікації» у шапці, версія Odoo відсутня/вгадана.

## 5. Топ причин ручних правок BA (ранжовано — наслідок розривів вище)

1. **Відсутні технічні імена** (C1) → архітектор/розробник реконструює вручну — найчастіший корінь доробки.
2. **Немає AC / приймання сховане в тестах** (п.0, C5) → «готово» неоднозначне → доробка на прийманні.
3. **Дрейф/одруки імен** (C2, C3) → конфлікти в коді при здачі кількох ТР разом.
4. **Порожня оцінка** (с.5) → години дописують щоразу руками (дефіцит `tr-effort-instruction`).
5. **Незакриті inline-нотатки / відкритий scope** (C7) → BA вичищає перед клієнтом.
6. **Copy-paste контамінація** (C4) → ручна корекція моделі/полів.
7. **Версія не вказана/вгадана** (C8) → уточнюють руками.

## 6. Чеклист «нещасливих» шляхів для `tr-usecases-acceptance` (корпус їх системно пропускає)

Для КОЖНОЇ нетривіальної дії/кнопки — мінімум один AC на:

- **Немає даних / порожньо** (товар без ціни, порожня вибірка → система не падає, рядок не оновлюється).
- **Помилка/валідація** → `AccessError`/`UserError`/`ValidationError` з **дослівним текстом** помилки; стан не змінюється (НЕ «403»).
- **Немає прав / видимість** (кнопка/поле приховані для не-адміна; `groups`/`column_invisible`).
- **Межі/constrains** (дублікат, друга спроба, від'ємний залишок → блок з повідомленням).
- **Повтор/конкурентність** (повторне натискання кнопки не дублює ефект).
- **Скасування/відкат дії**.

Корпус: структурованих error-flow немає; обробку помилок переважно «віддають розробнику» без специфікації
— саме це треба зробити обов'язковим у скілі.

## 7. Резюме: що міняємо в ланцюгу (вхід у Етап 4)

- **R1** (формат AC): `tr-usecases-acceptance` → єдиний namespace `AC-NN` (прибрати `AC1/AC 1.1/Scenario N` як паралельні схеми); `tr-review` → ловити «AC немає / не `AC-NN`». Підстава: п.0 (формальних AC ~3%), контракт `journal-to-tr-builder`+`spec-writer`.
- **R2** (мертві посилання): `tr-usecases-acceptance`/`tr-odoo-tech-design` Крок 0 → `journal-to-tr-builder` замість `tr-scaffold`/`tr-functional-requirements`. Підстава: структура пакета.
- **R3/R4** (трасування): backref `ВИМ-NN`/`origin` на AC; `tr-review` ловить «ВИМ без AC» і «AC без origin». Підстава: C5 + контракт.
- **C1/C2/C3** (імена): `tr-odoo-tech-design` — обов'язкові техімена + єдиний словник + звірка кожного імені; `tr-review` — sanity-флаг імен і крос-секційна звірка. Підстава: найчастіші розриви корпусу.
- **C6** (на словах): `tr-odoo-tech-design`/`tr-review` — реалізація = поля/моделі/методи, не проза.
- **C4/C7/C8/C9** (контамінація, нотатки, версія, назва): додати у чек-лист `tr-review`.
- **AC unhappy paths** (п.6): зробити обов'язковими в `tr-usecases-acceptance`.
- Odoo-точність: посилити звірку **імен** (не лише синтаксису) у `tr-odoo-tech-design` Крок 1; `tr-review` — флаг вигаданих/одруканих стандартних моделей (D1–D2 як приклади); 403→exceptions = 🔴.
