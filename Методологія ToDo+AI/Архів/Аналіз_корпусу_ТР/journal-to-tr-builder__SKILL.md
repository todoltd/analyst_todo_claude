# Journal to TR Builder

## Мета

Перетворити рішення й вимоги, накопичені в журналі дискавері, на **скаффолд Технічного Рішення (ТР)**
із заповненою **секцією 1 «Функціональні вимоги»** — точкою входу в ланцюжок `tr-*` (use cases / AC →
технічна реалізація → рев'ю) і далі в білд.

Це **внутрішній документ** (BA / розробка), не для замовника. Він закриває реальний розрив методології:
окремого `tr-functional-requirements` / `tr-scaffold` немає, тож цей скіл і створює кістяк ТР, і пише
секцію 1. Кожна вимога несе `origin:` — наскрізна трасованість «рішення дискавері → вимога → AC →
код → тест» (контракт SDD-хребта, `КОНТРАКТ_SDD-хребет_статуси-і-ID.md`, §3).

---

## Інструменти та середовище

**Читання вхідних файлів:**
- `журнал_дискавері_[назва_компанії].md` → `Read` (обов'язково — рішення D-N, теми T-N, джерело `origin`)
- `gap-аналіз_[назва_компанії].md` → `Read` (обов'язково — customization-list: що саме потребує ТР)
- `scope_[назва_компанії].md` → `Read` (підтвердження in-scope)
- `план_дискавері_[назва_компанії].md` → `Read` (контекст, назви напрямів)

**Збереження результату:**
- `ТР_[коротка-назва-доробки].md` → `Write` (один або кілька — див. Крок 1)

**Після завершення:**
- Викличи `present_files` для кожного створеного ТР
- Повідом: "Скаффолд ТР готовий ([K] шт.). Секція 1 заповнена: [N] вимог, кожна з origin. Далі —
  `tr-usecases-acceptance` (сценарії + AC) і `tr-registry-update` (внести в «Перелік ТР»)."

---

## Місце в процесі

```
discovery-gap-analyzer → gap-аналіз (customization-list)
    ↓
journal-to-tr-builder → ТР_[назва].md (шапка + секція 1 з origin + плейсхолдери ланцюжка)   ← ЦЕЙ СКІЛ
    ↓
tr-usecases-acceptance → секція 2 (Use Cases + AC) → tr-odoo-tech-design → tr-review
    ↓ (статус «Погодження клієнта»)
spec-writer → architecture → … → pr-review  (MVP / білд)
```

---

## Крок 0: Визнач вхідні дані

**Мінімум:** `журнал_дискавері` + `gap-аналіз`.
- Немає gap-аналізу → "Спершу `discovery-gap-analyzer` — ТР будується з customization-list."
- Немає журналу → "Журнал — джерело вимог і origin. Спершу проведіть цикл / `discovery-scope-mapper`."
- **Кілька компаній** → уточни, для якої.

---

## Крок 1: Визнач розбивку на ТР

З customization-list gap-аналізу визнач, **скільки ТР** створювати. ТР зазвичай відповідає одній
доробці / одній задачі ERP (під окремий `ID задачі`):

- **За замовчуванням** — один ТР на велику доробку (або кластер пов'язаних GAP-N однієї області).
- Дрібні споріднені доробки однієї області можна об'єднати в один ТР.
- Якщо розбивка неоднозначна — **запитай**: "Customization-list має [N] доробок: [перелік]. Зробити
  окремий ТР на кожну, чи згрупувати (як саме)?"

Для кожного ТР визнач коротку назву → ім'я файлу `ТР_[коротка-назва].md`.

---

## Крок 2: Заповни секцію 1 «Функціональні вимоги» (з origin)

Для кожного ТР збери функціональні вимоги з журналу і gap-аналізу. Вимога описує, **що система має
робити** (функціонально), — не як це реалізовано (це секція 3, `tr-odoo-tech-design`).

Формат (групуй за областю/доробкою):

```
### [Область / доробка]
**ВИМ-01.** [Функціональна вимога — що система має робити, конкретно]
  origin: D-3 / сесія 2 · gap: GAP-5
**ВИМ-02.** [...]
  origin: T-1 / сесія 3
```

Правила:
- **Кожна вимога — з `origin`** на джерело в журналі: рішення `D-N`, нова тема `T-N` або сесія `N`.
  Без origin вимога не пишеться (обрив трасування вгору) — якщо джерела немає, це сигнал, що вимога
  не з дискавері: познач `origin: ‹уточнити›` і додай у «Відкриті питання ТР».
- Прив'яжи `gap: GAP-N` там, де вимога походить від customization-list — зв'язок із fit-gap.
- Вимоги — функціональні й перевірювані (під них далі пишуться AC). Не технічний дизайн, не оцінка.
- Не вигадуй вимог поза журналом/gap. Прогалини познач явно.

---

## Крок 3: Скаффолд решти ТР (плейсхолдери ланцюжка)

Додай кістяк ТР із заголовками **українською дослівно** (їх читає `tr-review`) і плейсхолдерами для
наступних скілів. **Нічого за наступні скіли не вигадуй** — лише маркери:

- `## 2. Сценарії використання та критерії приймання` → `<!-- TODO: tr-usecases-acceptance -->`
- `## 3. Опис технічної реалізації` → `<!-- TODO: tr-odoo-tech-design -->`
- `## 4. Архітектура` → коментар «заповнює архітектор після узгодження» (не вигадувати — `tr-review`
  вважає порожню АРХІТЕКТУРУ коректною)
- `## 5. Обсяг робіт (оцінка)` → `<!-- Обсяг робіт: заповнює tr-effort-instruction (оцінка діапазоном) / discovery-sow-builder -->`
- `## 6. Тестовий сценарій / Протокол тестування` → коментар «заповнюється після узгодження реалізації»

> **Звірка зі стандартом ТР.** Якщо в проєкті є канонічний шаблон ТР або приклад `ТР_*.md` — вирівняй
> назви секцій під нього (заголовки мають збігатися дослівно, бо downstream-скіли шукають саме їх).
> Плейсхолдер входу для сценаріїв/AC — рівно `<!-- TODO: tr-usecases-acceptance -->`.

---

## Крок 4: Дисципліна ID та origin

- **`origin` обов'язковий** на кожній вимозі секції 1 — це валюта трасування дискавері→білд.
- **AC-NN — канонічний формат** для критеріїв приймання (секція 2), які напише `tr-usecases-acceptance`:
  один namespace `AC-01`, `AC-02` (не `AC1` / `Scenario N` паралельно) — щоб ID дожив до
  `# TODO: AC-NN → test_acNN → pr-review`. Origin має **пропагуватись** на AC (вимога §5 контракту
  SDD — upstream-правка `tr-usecases-acceptance`; зазнач це в шапці ТР як нагадування).
- **Не перенумеровуй** наявні ID; downstream лише споживає.

## Крок 5: Pre-save верифікація

- [ ] Назва файлу кожного ТР: `ТР_[коротка-назва].md` (не шаблонна)
- [ ] Шапка: назва, `ID задачі (ERP): ‹заповнити›`, дата, статус «Аналіз та підготовка рішення», автор, походження (журнал+дата)
- [ ] Секція 1 заповнена; **кожна вимога має `origin`** (D-N / T-N / сесія N) або явне `origin: ‹уточнити›` + запис у відкриті питання
- [ ] Вимоги функціональні, **атомарні й перевірювані** (одна вимога — одна думка, щоб під неї став ≥1 AC); технічного дизайну/оцінки в секції 1 немає
- [ ] Customization-list gap-аналізу покрита вимогами (нічого не загублено); зв'язок `gap: GAP-N` де доречно
- [ ] Заголовки секцій 2–6 — українською дослівно; плейсхолдер `<!-- TODO: tr-usecases-acceptance -->` присутній
- [ ] Секцію 2/3/5 не вигадував; АРХІТЕКТУРА (4) лишена шаблоном
- [ ] Усі `[⚠️ ІНСТРУКЦІЯ: ...]` замінені реальним контентом

## Наступний крок

> "Скаффолд ТР готовий ([K] шт.), секція 1 з origin заповнена. Далі:
> • **'напиши use cases і AC для ТР [назва]'** (`tr-usecases-acceptance`) — секція 2;
> • **'внеси ТР [назва] в реєстр'** (`tr-registry-update`) — «Перелік технічних рішень», статус «Аналіз та підготовка рішення».
> Після проходження ланцюжка `tr-*` і статусу «Погодження клієнта» вмикається MVP-білд (`spec-writer`)."

---

## Важливі правила

**ТР — внутрішній документ.** Це не Discovery-звіт замовнику. Тут технічна мова, ID, origin.

**Кожна вимога — з origin.** Без backref на журнал (D-N / T-N / сесія N) трасування рветься. Немає
джерела — `origin: ‹уточнити›` + у відкриті питання, не мовчазний пропуск.

**Секція 1 — вимоги, не реалізація.** Що система робить (функціонально). Поля, моделі td.*, права —
це секція 3 (`tr-odoo-tech-design`), не тут.

**Не вигадуй за наступні скіли.** Секції 2/3/5 — плейсхолдери; АРХІТЕКТУРА — шаблон. Заповнюють їх
`tr-*` і архітектор.

**AC-NN — єдиний формат критеріїв.** Готуй ґрунт для одного namespace AC-01 з origin наскрізь —
не плоди паралельних схем нумерації.

**Не перенумеровуй.** ID журналу й вимог стабільні; downstream лише споживає.
