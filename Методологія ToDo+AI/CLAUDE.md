# Контекст папки: Методологія ToDo+AI (ІНІЦІАЦІЯ → ДИСКАВЕРІ → ТР → MVP)

Робоча папка методології впровадження Odoo 19 — **5 плагінів** наскрізного процесу:
`odoo19-discovery-initiation` (ІНІЦІАЦІЯ, 12 скілів), `odoo19-discovery-sessions` (ДИСКАВЕРІ — цикл
зустрічей, 3 скіли), `odoo19-discovery-closeout` (ЗАВЕРШЕННЯ ДИСКАВЕРІ — вихід із циклу в ТР, 5 скілів),
`odoo19-tr-authoring` (шар **ТР** — авторинг Технічного рішення, 5 скілів) і `odoo19-mvp-build`
(**MVP / БІЛД** — два треки на AC-01: код + конфігурація, 11 скілів).

## Канонічні артефакти

- `СКІЛИ_виправлені/` — **джерело істини пакета ІНІЦІАЦІЯ**: 12 скілів + `.claude-plugin/plugin.json` (версія тут) + `docs/` (інструкції) + `README.md` + `ЗІБРАТИ_plugin.sh` (авто-версійна збірка).
- `СКІЛИ_ДИСКАВЕРІ/` — **джерело істини пакета ДИСКАВЕРІ** (цикл, 3 скіли): `session-question-builder` (опитувальник сесії + журнал циклу, **канон контракту журналу** в `references/journal_template.md`); `session-client-prep-builder` (лист замовнику, бланк у `assets/letter_template.docx`); `session-transcript-analyzer` (**головний writer журналу** після зустрічі; синхронна копія контракту в своєму `references/journal_template.md`) + `docs/` + `валідатор_скілів_дискавері.py` (перевіряє всі пакети методології) + власний `ЗІБРАТИ_plugin.sh`. Плани: `ПЛАН_розробки_ДИСКАВЕРІ_блок_Підготовка-питань.md`, `ПЛАН_розробки_ДИСКАВЕРІ_блок_Завершення.md`.
- `СКІЛИ_ЗАВЕРШЕННЯ/` — **джерело істини пакета ЗАВЕРШЕННЯ** (`odoo19-discovery-closeout`, 5 скілів): `discovery-scope-mapper` (гейт готовності + scope in/out), `discovery-gap-analyzer` (AS-IS→TO-BE Odoo 19→gap/customization), `discovery-report-builder` (звіт замовнику + брендований `.docx`, бланк `assets/report_template.docx`), `journal-to-tr-builder` (**міст**: скаффолд `ТР_*.md` + секція 1 з `origin`, плейсхолдер під `tr-usecases-acceptance`), `discovery-sow-builder` (КП/SoW + `.docx`, бланк `assets/sow_template.docx`) + `README.md` + `ЗІБРАТИ_plugin.sh`.
- `СКІЛИ_ТР/` — **джерело істини пакета ТР** (`odoo19-tr-authoring`, 5 скілів): `tr-usecases-acceptance` (Use Cases+AC, несе `origin`), `tr-odoo-tech-design` (опис реалізації td.*, звірка з Odoo 19), `tr-effort-instruction` (Обсяг робіт+Інструкція, калібрування), `tr-review` (QA ТР), `tr-registry-update` (реєстр «Перелік ТР»). У кожного скіла свій `docs/` з довідниками; майстер-набір 11 довідників + `0_МАЙСТЕР-ІНСТРУКЦІЯ.md` — у `docs/` пакета. `tr-scaffold`/`tr-functional-requirements` свідомо відсутні — їх виконує `journal-to-tr-builder` (closeout). + `README.md` + `ЗІБРАТИ_plugin.sh`.
- `СКІЛИ_MVP/` — **джерело істини пакета MVP/БІЛД** (`odoo19-mvp-build`, 11 скілів, 2 треки на AC-01): Трек 1 код (`reverse-spec`, `spec-writer`, `architecture`, `scaffolding`, `code-dev`, `testing`, `pr-review`), форк `build-allocator`, Трек 2 конфігурація (`config-spec`, `config-runbook`, `config-review`) + `docs/` + `README.md` + `ЗІБРАТИ_plugin.sh`. Плани: `ПЛАН_інтеграції_SDD-пакета_MVP.md`, `ПЛАН_розробки_MVP_блок_Конфігурація_Трек-2.md`.
- `Шаблони_ToDo/` — офіційні бланки ToDo (джерела): `Шаблон_Технічне_рішення.docx` (еталон структури ТР), `Шаблон_Звіт_про_обстеження.docx`, `Шаблон_Протокол_обстеження_інтервʼю.docx`, `ToDo_LTD_профіль_компанії.md`.
- `АУДИТ_пробілів_до_MVP.md` — аудит наскрізного процесу до MVP (П-1…П-6; П-1…П-4 закрито пакетом ТР + валідатором, P-5 — генератором; лишилось P-6 і рантайм-частина).
- `генератор_проекцій.py` (P4/P-5) — проекції з джерела істини: `status` (версії `plugin.json` → якорі `<!--P4:VER:*-->` BPMN + дашборд `СТАТУС_методології.html`) і `rtm <project_dir>` (матриця трасування AC: ТР→SPEC→код→тест→реєстр). Джерело статусів — `СТАТУС_методології.json`. Запуск при зміні версій пакета. Спека — `СПЕЦ_P4_генератор-проекцій.md`.
- `odoo19-discovery-initiation-X.Y.Z.plugin`, `odoo19-discovery-sessions-X.Y.Z.plugin`, `odoo19-discovery-closeout-X.Y.Z.plugin`, `odoo19-tr-authoring-X.Y.Z.plugin`, `odoo19-mvp-build-X.Y.Z.plugin` — зібрані інсталяційні файли (тримати лише актуальні).
- `bpmn_odoo19_discovery_СТАТУС-скілів.html` — **канонічна BPMN-діаграма** (єдина; паралельну копію в окремій папці прибрано 12.06.2026).
- `ІНІЦІАЦІЯ_Майстер-інструкція.docx` — друкована майстер-інструкція (12 скілів, діаграма вшита).
- `Шаблон_Протокол_обстеження.docx` — джерело фірмового бланку (оптимізована копія вшита в скіл `meeting-protocol-builder/assets/`; бланк листа підготовки похідний від нього — в `session-client-prep-builder/assets/`; цей файл — резерв).
- `діаграма_процесу_ІНІЦІАЦІЯ.png` — рендер канонічної діаграми (⚠️ станом на 04.06.2026 відстає від HTML — перегенерувати при нагоді).

## Правила роботи

1. Скіли редагувати в джерелі відповідного пакета: `СКІЛИ_виправлені/skills/` (ІНІЦІАЦІЯ), `СКІЛИ_ДИСКАВЕРІ/skills/` (ДИСКАВЕРІ), `СКІЛИ_ЗАВЕРШЕННЯ/skills/` (ЗАВЕРШЕННЯ), `СКІЛИ_ТР/skills/` (ТР) або `СКІЛИ_MVP/skills/` (MVP), потім підняти версію у відповідному `plugin.json` → `bash ЗІБРАТИ_plugin.sh` того пакета. (На mount сесії Cowork збірка падає на atomic-rename — збирати через `/tmp` → `cp` у папку/outputs.) **`plugin.json.description` тримати коротким (~≤400 симв): довгий валить інсталяційну картку Cowork (CLI цього не ловить). `name:` кожного скіла = імені його теки.**
2. **Інсталяція:** картка з кнопкою встановлення з'являється лише коли `.plugin` доставлено через папку **outputs** сесії Cowork (не з цієї папки напряму).
3. Паралельну авторську копію «Методологія ToDo+AI Дискавері» прибрано (12.06.2026) — **ця папка єдине джерело істини**, синхронізувати нікуди не треба. Унікальні історичні матеріали з неї збережено в `Архів/з_папки_Дискавері/`.
4. При зміні процесу оновлювати разом: скіли → інструкції (docs/) → BPMN (канон) → майстер-інструкцію .docx → ІСТОРІЮ.
5. Контракт журналу циклу (`журнал_дискавері_…`): головний writer — `session-transcript-analyzer`; `session-question-builder` робить bootstrap/reconcile як підстраховку. **Канон** контракту — `session-question-builder/references/journal_template.md`; у `session-transcript-analyzer/references/journal_template.md` — **синхронна копія**. Зміни контракту — лише правлячи канон + копію + ІСТОРІЮ (тримати ідентичними).
