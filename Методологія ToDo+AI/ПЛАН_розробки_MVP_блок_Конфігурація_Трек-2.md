## 1. Місце блоку в процесі

Зараз Трек 2 на BPMN — це **названі вузли без методології під ними**: є `t-config` («БА налаштовує
Odoo через UI») і артефакт DOC-2 «Специфікація конфігурації», але **жоден скіл їх не виробляє і не
перевіряє**. Трек 1 (код) має повний ланцюжок `spec-writer → … → pr-review`; Трек 2 — порожній,
симетричний пробіл. Цей блок його закриває.

Канонічна BPMN, фаза MVP/БІЛД (форк після SPEC, сходження на майстер-даних і UAT):

```
        ТР «Погодження клієнта» → spec-writer → SPEC.md (approved, AC-01)
                                      ↓
                       РОЗПОДІЛ AC: конфігурація / код / гібрид        ← НОВЕ (форк, C0)
                          ┌───────────────────┴───────────────────┐
              ТРЕК 1 · КОД (готово)                    ТРЕК 2 · КОНФІГУРАЦІЯ      ← НОВЕ
        architecture → scaffolding →            C1 config-spec  (DOC-2: CONFIG.md)
        code-dev → testing →                          ↓
        pr-review (гейт) → Деплой (БА)          C2 config-runbook (runbook + config-log)
                          │                            ↓  (БА налаштовує Odoo через UI)
                          │                     C3 config-review (гейт перед UAT)
                          └───────────────────┬───────────────────┘
                                      ↓
                       МАЙСТЕР-ДАНІ + права доступу (БА, `t-masterdata`) — спільний вузол
                                      ↓
                          Тренінг → UAT (Замовник, `t-uat`)
                                      ↓
                       Шлюз «Причина відмови?» (`gw-reason`)
                          ├─ Баги коду → Трек 1 (Claude → Code Review)
                          └─ Помилки конфігурації → Трек 2 (БА → C3 повторно)
```

Природа блоку:

- **Трек 2 — паралельний Треку 1**, стартує з того самого `SPEC.md` (проєкції затвердженого ТР).
  Різниця носія: Трек 1 реалізує `AC-01` **кодом** (`td.*`-модулі), Трек 2 — **стандартним Odoo
  через UI** (модулі, workflow, форми, звіти, email-шаблони, інтеграції, бізнес-правила).
- **Спільна валюта — `AC-01`.** Кожен конфіг-пункт несе `origin: AC-NN`, як код несе `# TODO: AC-NN`.
  Один namespace на вимогу (контракт SDD §3) — трасування не двоїться.
- **Майстер-дані (`t-masterdata`) і UAT (`t-uat`) — спільні** вузли сходження обох треків; цей блок
  їх НЕ дублює (межа в §3.2).

## 2. Інвентар скілів (вузька нарізка)

Усі скіли — у `odoo19-mvp-build` (поряд із Треком 1), імена в конвенції `odoo-*`, як решта пакета.

| # | Скіл | Дія | Вхід | Вихід | Аналог у Треку 1 |
|---|------|-----|------|-------|------------------|
| C0 | `build-allocator` *(форк; можливо — Крок 0 у C1 або колонка в `architecture`)* | Класифікувати кожен `AC-01`: реалізується **конфігурацією / кодом / гібридом** | `SPEC.md` (approved) + «Опис технічної реалізації» ТР | таблиця розподілу `AC-NN → {config\|code\|hybrid}` (+ обґрунтування); живить обидва треки | — (точка форку після `spec-writer`) |
| C1 | `config-spec` | Спроєктувати конфіг-частину `AC-01` у машинну специфікацію конфігурації (**DOC-2**) | `SPEC.md` + таблиця розподілу (C0) | `CONFIG.md`: по пунктах — модулі, налаштування, workflow, форми, звіти, email-шаблони, інтеграції, бізнес-правила; кожен пункт `origin: AC-NN` + посилання на доку Odoo 19 | `spec-writer` + `architecture` |
| C2 | `config-runbook` | Розгорнути `CONFIG.md` у впорядкований UI-runbook для БА + журнал змін конфігурації | `CONFIG.md` | `CONFIG_RUNBOOK.md`: кроки «клік-за-кліком» по областях, кожен — з перевіркою результату та `AC-NN`; шаблон `config-log` для фіксації застосованого | `scaffolding` + `code-dev` (тут виконавець — БА, скіл дає runbook + лог) |
| C3 | `config-review` **(гейт)** | Перевірити налаштований Odoo проти `CONFIG.md` і `AC-01`; вердикт | `CONFIG.md` + заповнений `config-log` / докази БА (експорт налаштувань, скріни) | вердикт **APPROVE / REQUEST CHANGES / BLOCK** по кожному `AC-NN`; список config-drift; готовність до UAT → реєстр «виконано» | `pr-review` (гейт перед мерджем/«виконано») |

**Чому C1 — ключовий.** DOC-2 «Специфікація конфігурації» на BPMN існує як артефакт, але **producer'а
немає** — Claude мав би її писати «з повітря». `config-spec` закриває цей стик: детермінована
проєкція тих самих `AC-01` у перевірюваний конфіг-план. Без неї Трек 2 не трасується і не приймається
по AC (на відміну від коду).

**Чому потрібен гейт C3.** У коду дрейф ловить `git diff` + `pr-review`. У конфігурації **немає diff'а** —
ручні кліки в UI не залишають сліду. `config-review` + `config-log` дають Треку 2 той самий рівень
доказовості приймання, що `pr-review` дає коду; без нього «налаштовано» лишається «на віру».

## 3. Артефакти і контракти

### 3.1 Карта файлів

| Файл | Пише | Читає | Призначення |
|------|------|-------|-------------|
| `SPEC.md` (approved, AC-01) | `spec-writer` (Трек 1, є) | C0, C1 | Спільне джерело вимог для обох треків |
| таблиця розподілу `AC → config/code/hybrid` | C0 | Трек 1 (`architecture`), C1 | Хто який AC реалізує — щоб треки не дублювали й не лишали діри |
| `CONFIG.md` (DOC-2) | C1 | C2, C3, замовник (як додаток до приймання) | Машинна специфікація конфігурації, трасована по `AC-NN` |
| `CONFIG_RUNBOOK.md` + `config-log` | C2 (runbook), БА (лог — руками під час налаштування) | C3, БА | Впорядкований сценарій UI-налаштування + запис застосованого |
| вердикт `config-review` | C3 | БА, реєстр «Перелік ТР» | Гейт перед UAT; підстава для статусу «виконано» |

### 3.2 Контракти і наскрізні правила (переюз існуючого)

1. **Спільний форк, спільна валюта (контракт SDD §3).** Обидва треки стартують з `SPEC.md` і ведуть
   `AC-01` наскрізь. Конфіг-пункт = `origin: AC-NN` (дзеркало `# TODO: AC-NN` у коді). Один namespace,
   не перенумеровувати. Гібридні AC (частина кодом, частина конфігом) явно позначені в C0 — обидва
   треки бачать свою частку.
2. **Гейт і статуси (контракт SDD §1–2).** `CONFIG.md` **успадковує `approved`** від ТР, не переоформлює
   (як `SPEC.md`, §6 `spec-writer`). Налаштування → `config-review` APPROVE → реєстр «виконано».
   Правило гейту єдине: `погоджено`/APPROVE ставить **людина**; AI лише позначає drift і BLOCK'ає
   (як `pr-review`), сам себе не затверджує.
3. **Дисципліна Odoo 19 (Крок 1, за зразком `tr-odoo-tech-design` / `architecture`).** Кожен пункт
   `CONFIG.md` звіряти з **офіційною докою Odoo 19** і вказувати джерело (`довідник §N` / URL). Не
   вигадувати меню/налаштувань, яких у v19 немає — це головний ризик якості конфігурації.
4. **Прогресивний ригор.** На білді — **жорстко**: `AC-NN` фіксовані, гейт C3, config-drift = BLOCK
   (симетрично коду). Жодного «легкого» режиму, як у дискавері.
5. **Межа з суміжними вузлами.** `t-masterdata` (продукти, контрагенти, план рахунків, ролі/права) і
   `t-uat` — **спільні** вузли сходження; Трек 2 їх НЕ дублює. Імпорт майстер-даних — окреме питання
   (§6, можливий майбутній скіл), не частина C1–C3.
6. **Межа config vs code (Studio).** No-code кастомізації (Odoo Studio, server actions через UI,
   automated rules) → Трек 2; Python/XML-модулі `td.*` → Трек 1. Розвилку фіксує C0 (відкрите питання §6).
7. **RTM (важіль P4).** Матриця трасування розширюється на конфігурацію: `AC → CONFIG-пункт → config-log
   → config-review → реєстр`. У RTM тепер два типи рядків реалізації — `code` і `config` — спільний доказ
   покриття приймання й аудиту.

## 5. Ризики

| Ризик | Мітигація |
|-------|-----------|
| **Немає diff'а конфігурації** — ручні кліки в UI не лишають сліду, дрейф невидимий | `config-log` (C2) як обов'язковий запис застосованого + `config-review` (C3) проти нього; за можливості — експорт налаштувань (`ir.config_parameter`, data export) як доказ |
| **Невідтворюваність** dev→prod (ручна конфігурація не переноситься автоматично) | `CONFIG_RUNBOOK.md` = відтворюваний сценарій; пізніше (P6 «реверс») — захоплення конфігурації в data-модуль |
| **Вигадана конфігурація Odoo 19** (меню/налаштування, яких немає у v19) | Дисципліна Крок 1: кожен пункт `CONFIG.md` — з посиланням на офіційну доку v19 (контракт §3.3) |
| **Розмита межа config vs code** (Studio, server actions через UI) | C0 фіксує розподіл за правилом §3.6; гібридні AC явно позначені; відкрите питання §6 про межу Studio |
| **Діра/дубль на форку** — AC не потрапив у жоден трек або потрапив у обидва | C0 — обов'язковий гейт перед стартом обох треків; кожен `AC-01` має рівно один запис розподілу (config/code/hybrid); валідатор перевіряє покриття |
| **Конфіг-only проєкт** — немає кастомного коду, Трек 1 (`architecture`) недоречний | Підтвердити, що ланцюжок працює config-only: `spec-writer → C0 → C1–C3` без коду (відкрите питання §6) |
| **Дубль майстер-даних** Трек 2 ↔ `t-masterdata` | Чітка межа §3.5: майстер-дані — спільний вузол сходження, не входять у C1–C3 |
| **Конфлікт тригерів** C1–C3 ↔ скіли Треку 1 (`spec`, `review`) | Явне розмежування в description («конфігурація / налаштування Odoo через UI» vs «код / SPEC / PR»); рантайм-тест фраз на Етапі 6 |
| **UAT-петля** (`gw-reason`: помилки конфігурації) не має власника | Повернення замикається на C3: після переналаштування БА — повторний `config-review` до APPROVE |

## 6. Відкриті питання (підтвердити до/під час розробки)

1. **C0 розподіл AC:** окремий тонкий скіл `build-allocator` на форку (рекомендую — працює і для
   config-only) чи Крок 0 у `config-spec` / колонка в `architecture` AC-trace?
2. **Config-only проєкти:** підтвердити, що фаза працює без Треку 1 (`architecture` пропускається,
   форк веде лише в Трек 2).
3. **Межа Studio:** no-code Odoo Studio — це Трек 2 (конфігурація) чи Трек 1 (експортується в модуль)?
   Зафіксувати в правилі розподілу C0.
4. **Докази для C3:** що є acceptance-доказом конфігурації — `config-log` .md, експорт налаштувань,
   скріни UI, чи комбінація? Визначає вхідний контракт `config-review`.
5. **Майстер-дані:** окремий майбутній скіл (шаблони імпорту + валідація) чи лишаємо ручним вузлом БА
   (`t-masterdata`)?
6. **Імена скілів:** `config-spec` / `config-runbook` / `config-review` (+ `build-allocator`) — ок?
7. **Артефакти:** машинна назва `CONFIG.md` (паралель до `SPEC.md`/`ARCHITECTURE.md`) — ок? Вміст —
   українською (БА-facing), як решта клієнто-/виконавчо-видимого.
