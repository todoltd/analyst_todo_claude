# Discovery SoW Builder

## Мета

Зібрати **комерційну пропозицію / SoW** на впровадження за результатами дискавері: що робимо
(обсяг робіт із gap-аналізу), орієнтовна оцінка, межі (що не входить), припущення, етапи й терміни.
Документ для замовника — підстава для рішення про старт впровадження.

Це **єдиний документ блоку завершення, де доречні оцінка й комерційні параметри** — на відміну від
Discovery-звіту (де їх немає). Точна per-AC оцінка з'являється пізніше, у ТР («Обсяг робіт»); тут —
орієнтовна, із застереженням.

---

## Інструменти та середовище

**Читання вхідних файлів:**
- `scope_[назва_компанії].md` → `Read` (обов'язково — обсяг in/out)
- `gap-аналіз_[назва_компанії].md` → `Read` (обов'язково — customization-list → позиції робіт)
- `discovery-звіт_[назва_компанії].md` → `Read` (контекст, формулювання — якщо є)
- `tr-effort`-вихід / оцінки BA → якщо є (точніша оцінка); інакше — орієнтовні діапазони (Крок 2)
- `бриф_*`, `протокол_*` → `Read` (назва замовника, контактна особа, мова)

**Шаблон для .docx (вшито):** `assets/sow_template.docx` — брендований бланк ToDo (обкладинка +
лінійний зміст після розриву). Бандл недоступний → віддай якісний `.md`, повідом про .docx пізніше.

**Збереження результату:**
- `discovery-КП_[назва_компанії].md` → `Write`
- `discovery-КП_[назва_компанії].docx` → заповнений шаблон (Крок 4)

**Після завершення:**
- Викличи `present_files` для обох файлів
- Повідом: "Комерційна пропозиція готова (.md + .docx). Позицій робіт: [N], орієнтовна оцінка: [діапазон].
  Увага: оцінка попередня — уточнюється в ТР. Перевір комерційні умови перед надсиланням."

---

## Місце в процесі

```
discovery-scope-mapper → scope · discovery-gap-analyzer → gap-аналіз
    ↓
discovery-sow-builder → discovery-КП_[компанія].md + .docx (замовнику)   ← ЦЕЙ СКІЛ
    ↓
(паралельно: journal-to-tr-builder → ТР; точна оцінка — в «Обсяг робіт» ТР)
```

---

## Крок 0: Визнач вхідні дані

**Мінімум:** `scope_[компанія].md` + `gap-аналіз_[компанія].md`.
- Немає scope/gap → "Спершу `discovery-scope-mapper` і `discovery-gap-analyzer` — КП будується з них."
- **Кілька компаній** → уточни, для якої.
- **Чернеткові входи** → у шапці `⚠️ Попередня пропозиція — дискавері ще триває`.

**Джерело оцінки** (за пріоритетом):
1. Вихід `tr-effort` / готові оцінки BA, якщо є → бери їх.
2. Якщо немає — **орієнтовні діапазони** за розміром доробки (S/M/L) із Кроку 2, з явним застереженням.
3. **Ніколи не вигадуй точних годин як остаточних.** Завжди діапазон + припущення + «уточнюється в ТР».

**Мова, контактна особа, назва проєкту:** як у Discovery-звіті (мова замовника; контакти з брифу/протоколів, не вигадувати).

---

## Крок 1: Обсяг робіт (work breakdown)

Зі scope (in-scope) + gap customization-list склади перелік позицій робіт за категоріями:

| Категорія | Позиція | Джерело |
|-----------|---------|---------|
| Налаштування (конфіг) | [що налаштовуємо] | scope / gap (тип «конфіг») |
| Доробки (розробка) | [кастомізація] | gap customization-list (GAP-N) |
| Інтеграції | [система] | gap (тип «інтеграція») |
| Міграція даних | [джерело → Odoo] | scope (дані) |
| Навчання і запуск | [тренінги, go-live] | план (go-live) |
| Підтримка (опц.) | [супровід після запуску] | — |

Кожна позиція доробки трасується до `GAP-N` — спадкоємність із fit-gap.

---

## Крок 2: Орієнтовна оцінка

Для кожної позиції — оцінка. Якщо немає точних даних (tr-effort/BA), став **розмір** і діапазон:

| Розмір | Орієнтир | Коли |
|--------|----------|------|
| S | [діапазон годин/днів] | проста конфігурація, типова доробка |
| M | [діапазон] | доробка із логікою, проста інтеграція |
| L | [діапазон] | складна доробка/інтеграція, міграція великого обсягу |

- Дай **діапазон** (від–до), не одне число; додай **припущення** під оцінкою.
- Підсумок — діапазон по проєкту (сума від / сума до). Перевір арифметику.
- **Застереження обов'язкове:** «Оцінка попередня, для планування; точна per-вимога оцінка — у ТР
  (розділ «Обсяг робіт») після узгодження технічної реалізації.»

---

## Крок 3: Humizer + межі + чутливе

**Humizer:** стриманий діловий тон, звертання на «ви», терміни Odoo пояснені, без внутрішніх ID/тегів.
«gap/доробка» — нейтрально; «AS-IS/TO-BE» не вживати.

**Що тут доречно (на відміну від звіту):** оцінка трудомісткості, ціни, комерційні умови — це і є
предмет КП.

**Що НЕ потрапляє:** внутрішня структура собівартості й маржі виконавця (тільки підсумкові цифри для
замовника); захисні договірні формулювання — у договір, не в КП; оцінки людей замовника. Технічні
деталі td.* — у ТР, не тут.

**Межі (що не входить)** — зі scope «поза scope»: явний список виключень. Це ключове для КП —
захищає від спорів про обсяг.

---

## Крок 4: .docx за бандл-шаблоном

1. Скопіюй `assets/sow_template.docx` як `discovery-КП_[назва_компанії].docx`.
2. Через навички скілу **docx** (`unpack.py` → правки `word/document.xml` → `pack.py`):
   - **Обкладинка — точкова заміна в наявних ранах:** ‹Назва документа› → «КОМЕРЦІЙНА ПРОПОЗИЦІЯ»
     (або «STATEMENT OF WORK»), ‹Найменування проєкту›, ‹Назва компанії замовника›. Якірні зображення,
     абзаци, розриви — НЕ чіпати (верстка обкладинки на точному потоці абзаців).
   - **Лінійний зміст** — ЗАМІСТЬ маркера ‹зміст документа› (після розриву сторінки), маркер прибери.
     Заголовки розділів ~13 pt жирні; таблиці (обсяг робіт, оцінка) — Table Grid ~9.5 pt; підсумок оцінки — жирним.
   - Мова ≠ українська → переклади заголовок, назви розділів, шапки таблиць.
   - **Не чіпай** шапку, логотип, контакти, стилі бланку.
3. **Інваріант:** якірних зображень = шаблонній; нових якорів у змісті нуль; розрив сторінки на місці;
   плейсхолдерів ‹…› нуль; ≤ 1 МБ. Порушено — перероби.
4. **Рендер-перевірка:** PDF (`soffice --headless --convert-to pdf`) + зображення стор. 1–2 (`pdftoppm -png`),
   ПОДИВИСЬ: обкладинка ціла, заголовок і назва проєкту видно, зміст зі стор. 2. Накладання — НЕ віддавай.

## Крок 5: Pre-save верифікація

- [ ] Назви файлів: `discovery-КП_[реальна компанія].md` і `.docx`
- [ ] Обсяг робіт покриває scope + customization-list (нічого не загублено); доробки трасуються до GAP-N
- [ ] Кожна оцінка — діапазон + припущення; підсумок = сума діапазонів (арифметика перевірена)
- [ ] **Застереження про попередність оцінки присутнє**; точна оцінка віднесена до ТР
- [ ] «Що не входить» зі scope «поза scope» — явний список
- [ ] Humizer пройдено; внутрішня собівартість/маржа не показані; технічних деталей td.* немає
- [ ] .docx: обкладинка ціла, плейсхолдерів ‹…› нуль (програмно + рендером), ≤ 1 МБ
- [ ] Мова = мова замовника; якщо не українська — заголовки .docx перекладені
- [ ] Усі `[⚠️ ІНСТРУКЦІЯ: ...]` замінені; створено й показано обидва файли

> "Комерційна пропозиція готова (.md + .docx). Переглянь оцінку й комерційні умови, надішли замовнику.
> Паралельно — `journal-to-tr-builder` (ТР на доробки): точна per-вимога оцінка з'явиться в розділі
> «Обсяг робіт» ТР і має замінити орієнтовну."

## Важливі правила

**Оцінка — попередня.** Точна per-вимога оцінка — у ТР. Завжди діапазон + припущення + застереження;
ніколи не видавай орієнтир за остаточну ціну.

**КП — місце для оцінки й цін.** На відміну від Discovery-звіту, тут оцінка доречна. Але внутрішня
собівартість і маржа — ні: лише підсумкові цифри для замовника.

**Межі — явно.** «Що не входить» захищає обидві сторони; бери зі scope «поза scope».

**Трасування.** Позиції доробок → GAP-N; обсяг → scope. КП — проекція готових документів, не нове джерело.

**Технічне — у ТР.** Моделі, поля, реалізація — не в КП.
