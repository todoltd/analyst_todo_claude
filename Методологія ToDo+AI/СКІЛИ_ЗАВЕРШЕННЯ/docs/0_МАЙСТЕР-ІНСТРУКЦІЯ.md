# ЗАВЕРШЕННЯ ДИСКАВЕРІ — Майстер-інструкція

**Для кого:** BA, менеджер проекту
**Коли:** коли всі теми плану в `журнал_дискавері` закриті (✅) — вихід із циклу зустрічей у фазу ТР/білд
**Результат:** scope-маппінг, gap-аналіз, Discovery-звіт замовнику, скаффолд ТР, комерційна пропозиція

---

## Одна дія → один результат

| Ситуація | Що пишеш Claude | Що отримуєш |
|----------|-----------------|-------------|
| Цикл закрито, зводимо обсяг | "збери scope-маппінг для [Назва компанії]" | `scope_[компанія].md` (in/out + кандидати на кастомізацію) + вердикт готовності циклу |
| Потрібен fit-gap | "збери gap-аналіз для [Назва компанії]" | `gap-аналіз_[компанія].md` (AS-IS → TO-BE Odoo 19 → gap-list → customization-list) |
| Підсумок замовнику | "збери Discovery-звіт для [Назва компанії]" | `discovery-звіт_[компанія].md` + брендований `.docx` |
| Міст у ТР | "збери ТР з журналу для [Назва компанії]" | один/кілька `ТР_[назва].md` (скаффолд + секція 1 з origin) |
| Комерційна пропозиція | "збери комерційну пропозицію для [Назва компанії]" | `discovery-КП_[компанія].md` + брендований `.docx` |

## Рекомендований порядок (вихід із циклу)

```
УСІ ТЕМИ ЗАКРИТІ (журнал ✅)  ← сигнал від session-transcript-analyzer
    ↓
1. discovery-scope-mapper    → scope_[компанія].md            (гейт готовності — Крок 0)
2. discovery-gap-analyzer    → gap-аналіз_[компанія].md       (AS-IS → TO-BE → gap → customization)
3. discovery-report-builder  → discovery-звіт_[компанія].md + .docx  → замовнику на погодження
4. journal-to-tr-builder     → ТР_[назва].md (секція 1 + origin)     → ланцюжок tr-* → MVP/білд
5. discovery-sow-builder     → discovery-КП_[компанія].md + .docx    (паралельно з ТР)
```

scope живить усе наступне; gap живить звіт, ТР і КП. Звіт і КП — замовнику; ТР — внутрішній.

## Гейт завершення

Блок стартує лише коли цикл готовий: усі теми плану ✅, немає відкритих `[КД]`, перенесення (C-N)
закриті/свідомо відкладені, обіцяні артефакти (A-N) трекаються, виключення (X-N) — з джерелом.
`discovery-scope-mapper` перевіряє це першим кроком; якщо цикл не закритий — збирає **чернетку** з
видимими блокерами (прогресивний ригор).

## Стик із ТР і білдом

`journal-to-tr-builder` закриває розрив «цикл → ТР»: скаффолдить `ТР_[назва].md` і пише секцію 1
«Функціональні вимоги» з `origin` (D-N/T-N/сесія N), лишаючи плейсхолдер `<!-- TODO: tr-usecases-acceptance -->`.
Далі — ланцюжок `tr-*` (`tr-usecases-acceptance` → `tr-odoo-tech-design` → `tr-review`) і реєстр
(`tr-registry-update`). На статусі «Погодження клієнта» вмикається MVP-білд (`spec-writer` → …).
Наскрізна валюта вимоги — `AC-NN` з `origin` (контракт `КОНТРАКТ_SDD-хребет_статуси-і-ID.md`).

## Чого НЕ робити

- Не запускати завершення на незакритому циклі без позначки чернетки (гейт у scope-mapper).
- Не вигадувати scope/gap поза журналом — усі документи блоку є **проекціями журналу**.
- Не змішувати документи: звіт і КП — замовнику (Humizer + фільтр чутливого); ТР — внутрішній (origin, ID).
- Не проєктувати моделі/поля td.* у gap чи звіті — це секція 3 ТР (`tr-odoo-tech-design`).
- Оцінку трудомісткості не давати у звіті — лише в КП (орієнтовно) і в ТР (точно).
