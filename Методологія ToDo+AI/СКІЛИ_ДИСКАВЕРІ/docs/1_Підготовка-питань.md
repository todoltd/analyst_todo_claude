# Блок «Підготовка питань» — інструкція

**Місце в процесі:** перший крок кожної ітерації циклу зустрічей дискавері
(вузли BPMN: «Підготовка питань (Claude)» → «Порядок денний (БА)»).

## Скіли блоку

### 1. session-question-builder — опитувальник сесії (тільки команда)

**Коли:** перед кожною зустріччю циклу — першою після Установчої Зустрічі і кожною наступною.
**Тригер:** "підготуй питання до сесії [N] для [компанія]".
**Вхід:** план дискавері (рекомендовано; без нього — деградований режим: файл з датою замість N,
без карти покриття); журнал, протоколи, аналіз УЗ, заповнений питання_УЗ — підхоплює сам.
**Вихід:** `питання_сесії-[N]_[компанія].md` + створений/оновлений `журнал_дискавері_[компанія].md`.

Що робить за тебе:
- тримає контекст циклу: закрите не перепитує, виключене (X-N) не питає, перенесене (C-N) ставить першим;
- розгортає питання-заготовки сесії з плану в повний опитувальник (15–25 питань, пріоритети 🔴/🟡/⚪,
  теги deliverables, поля «Відповідь:»; кожне 🔴 — з 2–3 навідними підказками на випадок неговіркого
  замовника: останній випадок, альтернативи, квантифікація, сценарій, артефакт, нещасливий шлях);
- будує карту покриття цілі сесії і парковку для інших сесій;
- веде журнал: bootstrap на першій ітерації, reconcile з нових протоколів на наступних (зокрема розділ 8 —
  обіцяні замовнику артефакти A-N: витягує з протоколів і показує незакриті як борг виконавця в контексті циклу).

Що лишається за BA: переглянути питання, прибрати зайве, додати специфіку — ПЕРЕД генерацією листа замовнику.

### 2. session-client-prep-builder — лист замовнику (.md + .docx)

**Коли:** після правок BA в опитувальнику; надіслати за 2–3 робочі дні до зустрічі.
**Тригер:** "лист замовнику до сесії [N] для [компанія]".
**Вхід:** `питання_сесії-[N]` (основний), журнал (виключення), план (учасники).
**Вихід:** робочі .docx для зустрічі — `питання_сесії-[N]_[компанія].docx` (конвертація 1:1, без Humizer) і `план_дискавері_[компанія].docx` (якщо .docx відсутній або план оновився) + `підготовка_замовника_сесії-[N]_[компанія].md` + брендований `.docx` (бланк ToDo, ≤ 1 МБ).

Структура листа: тема й мета → короткий порядок денний (3–5 пунктів) → що підготувати (≤ 7 конкретних
артефактів) → над чим подумати (≤ 7 простих питань) → кого запросити (ролі). Обсяг — 1 сторінка.
Вбудовано Humizer (стриманий діловий тон, без внутрішніх маркерів) і фільтр чутливого (без ставок,
цін, внутрішніх умов; виключені зі scope теми не згадуються).

## Порядок у блоці

```
session-question-builder → правки BA → session-client-prep-builder (опитувальник + план → .docx, лист) → лист надіслати замовнику
```

## Типові помилки

- Генерувати лист ДО правок BA в опитувальнику → лист доведеться перегенерувати
- Питати в опитувальнику про X-N «бо замовник може передумати» → сигнал для BA, не питання замовнику
- Лист на 2+ сторінки → замовник не підготується; краще 5 точних пунктів
- Забутий протокол поза журналом → скіл 1 сам зробить reconcile, але протоколи мають лежати в outputs
