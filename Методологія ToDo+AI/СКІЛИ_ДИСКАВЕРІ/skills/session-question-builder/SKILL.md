# Session Question Builder

## Мета

Дати BA повний опитувальник для конкретної сесії дискавері. На відміну від опитувальника Установчої
Зустрічі (рівень scope), тут глибина дозволена і потрібна: міграція даних, кількість користувачів,
матриці прав, деталі інтеграцій — це і є предмет сесій.

Головна цінність — **пам'ять циклу**: скіл знає з журналу, що вже закрито (не перепитує), що перенесли
з минулої зустрічі (ставить першим), що виключили зі scope (не питає взагалі).

---

## Інструменти та середовище

**Читання вхідних файлів:**
- `план_дискавері_[назва_компанії].md` → `Read` (обов'язково — сесії, цілі, питання-заготовки)
- `журнал_дискавері_[назва_компанії].md` → `Read` якщо є (стан циклу)
- `протокол_*_[назва_компанії].md`, `аналіз_*_[назва_компанії].md` → `Read` (для bootstrap/reconcile журналу)
- `питання_УЗ_[назва_компанії].md` (заповнений), `аналіз_Установчої Зустрічі_[назва_компанії].md` → `Read` (перша ітерація)
- `питання_сесії-*_[назва_компанії].md` попередніх зустрічей (з номером або датою) → `Read` якщо є (незакриті питання)

**Довідник:** `references/journal_template.md` — контракт журналу, шаблон, приклад, алгоритми
bootstrap/reconcile. **Прочитай перед роботою з журналом.**

**Збереження результату:**
- `питання_сесії-[N]_[назва_компанії].md` → `Write`
- `журнал_дискавері_[назва_компанії].md` → `Write` (створений або актуалізований)

**Після завершення:**
- Викличи `present_files` для обох файлів
- Повідом: "Опитувальник сесії [N] ([тема]) збережено: [K] питань, з них [M] перенесених з попередніх зустрічей. Журнал актуальний станом на [дата]."
- Запропонуй наступний крок: "Далі — `session-client-prep-builder`: лист клієнту для підготовки до цієї сесії (після твоїх правок в опитувальнику)."

## Місце в циклі зустрічей

```
план_дискавері (фінал ІНІЦІАЦІЇ)
    ↓
┌─→ 1. session-question-builder   → питання_сесії-[N] + журнал   ← ЦЕЙ СКІЛ
│   2. session-client-prep-builder → підготовка_клієнта_сесії-[N] (.md + .docx) — надіслати клієнту
│   3. БА фіналізує порядок денний → ЗУСТРІЧ (запис + транскрипт)
│   4. Аналіз зустрічі (вручну / майбутній скіл) → meeting-protocol-builder → протокол клієнту на погодження
└── 5. Ще зустрічі? Так → наступна ітерація (скіл побачить нові протоколи і оновить журнал)
```

## Крок 0: Визнач вхідні дані

**Мінімум для роботи:** `план_дискавері_[назва_компанії].md`.

**Якщо в outputs файли кількох компаній:** запитай: "Знайшов матеріали для кількох компаній: [список]. Для якої готуємо сесію?"

**Якщо плану дискавері немає:** запитай: "Не знайшов `план_дискавері_[назва_компанії].md`. Назви тему і мету
зустрічі — підготую опитувальник без плану, але без карти покриття цілі сесії." Якщо користувач погоджується —
познач у шапці: `⚠️ Опитувальник побудовано без плану дискавері — прив'язка до сесій плану відсутня.`
Журнал у цьому режимі все одно створюється/ведеться (розділи 1–2 порожні, позначка в шапці — див. довідник);
ім'я файлу — з датою, як для позапланової зустрічі.

**Визнач номер сесії:**
- Користувач назвав номер або тему → знайди відповідну сесію в плані.
- Не назвав → візьми з журналу першу сесію зі статусом «запланована»; якщо журналу немає — сесію 1 плану.
  Підтверди: "Готую питання до сесії [N]: [тема] ([дата якщо є]). Вірно?"
- Зустріч позапланова (теми немає в плані) → працюй від теми користувача, у файлі замість номера — дата
  (`питання_сесії-[РРРР-ММ-ДД]_…`). На Кроці 1 зафіксуєш тему в журналі: рядок «поза планом» у розділ 1
  + T-N у розділ 6.

**Таблиця інвентаризації (встав у файл):**

| Файл | Статус | Що дає |
|------|--------|--------|
| `план_дискавері_…` | є / немає | ціль сесії, питання-заготовки, учасники |
| `журнал_дискавері_…` | є / створено зараз / оновлено | стан циклу |
| протоколи попередніх зустрічей | [N] шт. | рішення, виключення, перенесення |
| `питання_УЗ_…` (заповнений) | є / немає | парковка для дискавері (перша ітерація) |

## Крок 1: Журнал — bootstrap або reconcile

Працюй за `references/journal_template.md`:

**Журналу немає** (типово — перша сесія після УЗ) → **bootstrap**: створи `журнал_дискавері_[назва_компанії].md`
з плану дискавері + аналізу УЗ + заповненого `питання_УЗ` (особливо секція «Парковка для дискавері» —
її питання внеси в розділ 7: якщо парковка вже містить прив'язки до сесій — бери їх, інакше розподіли сам;
пріоритет, якщо джерело не дає, — [ВД], відповідальний — «—») + протоколу УЗ (рішення → розділ 5,
явні звуження scope → розділ 4; дату першої сесії з «Наступна зустріч» / рішень — простав у розділ 1).

**Журнал є** → **перевір свіжість**: чи всі `протокол_*` / `аналіз_*` з outputs перелічені в «Враховані
джерела». Якщо ні → **reconcile** за алгоритмом довідника (рішення → 5; «не робимо» → 4; відкриті питання → 7;
незакриті питання минулої сесії → 3 як C-N; статуси тем → 2; нові теми → 6). Онови дату і джерела в шапці.

**Журнал свіжий** → нічого не змінюй, лише прочитай.

**Зустріч позапланова** → незалежно від свіжості журналу: додай рядок «поза планом» (тема, дата,
статус «запланована») у розділ 1 і тему в розділ 6 (T-N — якщо такої теми там ще немає), онови дату в шапці.

> Якщо при reconcile бачиш, що версія плану в шапці журналу старіша за актуальний план — звір розділи 1–2
> журналу з планом (нові/прибрані/перейменовані сесії; номери НЕ перенумеровуються) і онови версію.

**Звірка теми сесії (після актуалізації журналу):** якщо тема обраної сесії в розділі 2 вже ✅ закрита
(встигли раніше / закрили позаплановою зустріччю) — попередь користувача і запропонуй: скасувати сесію
(статус у розділі 1) або змістити фокус сесії на відкриті C-N/Q-N і нові теми T-N. Не генеруй мовчки
опитувальник, який anti-repeat потім спустошить.

## Крок 2: Контекст циклу (зведення для BA)

З актуального журналу сформуй короткий блок у початок опитувальника:

- **Закрито раніше:** теми зі статусом ✅, а 🔶 — з приміткою «що лишилось» (одним рядком кожна, з номером сесії)
- **Перенесено в цю сесію:** усі відкриті C-N, що стосуються теми сесії
- **Виключено зі scope:** чинні X-N — статус «чинне» у розділі 4 (нагадування: цього НЕ питати;
  скасовані виключення не згадуй)
- **Ключові рішення по темі:** D-N, що стосуються теми сесії (щоб не перепитати вирішене)

Перша ітерація (bootstrap, попередніх сесій немає): додатково до пунктів блоку (виключення і рішення
показуй завжди — вони критичні для anti-repeat) напиши: "Перша сесія циклу — попередніх зустрічей немає.
З Установчої Зустрічі успадковано [N] відкритих питань (з них [M] — з парковки УЗ)."
Таблицю інвентаризації з Кроку 0 встав сюди ж, підрозділом блоку.

## Крок 3: Питання сесії

### 3.1 Джерела питань (у порядку пріоритету)

1. **Перенесені (carry-over)** — відкриті C-N по темі сесії **або призначені протоколом на цю/наступну
   зустріч** (навіть якщо тема інша — борг не губиться), а також Q-N з приміткою «ставилось на сесії …».
   Ідуть **першою секцією** опитувальника: це борг перед клієнтом з минулої зустрічі.
2. **Відкриті питання журналу** — Q-N з розділу 7, що стосуються теми сесії (включно з успадкованими
   з парковки УЗ). Критичні [КД] з них теж можна підняти в секцію 0 — з ID і поясненням.
3. **Питання-заготовки плану** — «Питання для дослідження» цієї сесії з плану дискавері (5–10) —
   розгорни кожне в конкретні питання.
4. **Нові питання** — доповни до повного покриття цілі сесії (Крок 5).

### 3.2 Формат питання

```
**[№]. [Пріоритет] Питання?** `[тег]` (C-N / Q-N якщо з журналу)
> Навіщо: [на який deliverable дискавері працює відповідь]
> Навідні (якщо відповідь коротка чи мовчання): → [навідне 1] → [навідне 2] → [навідне 3]
> Відповідь: _________________________________________________
```

[№] — наскрізний номер питання через весь опитувальник (1, 2, 3…, починаючи з секції «0. Перенесені») —
на нього посилається карта покриття (Крок 5). [Пріоритет] — лише емодзі 🔴/🟡/⚪, без слів.

### 3.3 Навідні питання — детализація для неговірких клієнтів

Навідні — заготовлені підказки фасилітатору: чим розкрити питання, якщо клієнт відповів одним
реченням або мовчить. Правила:

- **🔴 — обов'язково 2–3 навідні; 🟡 — 1–2 за потреби; ⚪ — можна без них.**
- Навідні НЕ рахуються в ліміт 15–25 — це не окремі питання, а сходинки до того самого.
- Формулюй конкретно під клієнта і тему (з плану/брифу/протоколів), не залишай абстрактних заготовок.
- Чергуй прийоми — не став три однотипні підряд:

| Прийом | Приклад формулювання |
|--------|----------------------|
| **Останній випадок** | «Розкажіть про останню таку угоду — що відбувалося крок за кроком?» |
| **Альтернативи** | «Це робиться в Excel, у системі чи на папері?» (обрати легше, ніж сформулювати) |
| **Квантифікація** | «Скільки разів на тиждень? Скільки людей задіяно? Скільки часу займає?» |
| **Гіпотетичний сценарій** | «Уявімо, прийшла заявка з нестандартною умовою — що далі?» |
| **Артефакт** | «Покажіть, будь ласка, приклад документа чи екрана, де це видно» |
| **Нещасливий шлях** | «А коли щось пішло не так — помилка, повернення — як було востаннє?» |

**Пріоритети:** 🔴 без відповіді ціль сесії не досягнута · 🟡 суттєво уточнює · ⚪ якщо лишиться час.

**Теги — прив'язка до deliverables дискавері** (1 тег на питання):
- `[процеси]` → карта бізнес-процесів AS-IS / FRD
- `[вимоги]` → список вимог (FRD)
- `[gap]` → gaps і customization list (де стандартного Odoo не вистачить)
- `[дані]` → план міграції даних
- `[інтеграції]` → план інтеграцій
- `[ролі]` → ролі, права доступу, безпека
- `[NFR]` → обсяги, продуктивність, регуляторика
- `[go-live]` → навчання і план запуску
- `[рішення]` → реєстр рішень (що клієнт має вирішити)

**Правила формулювання:**
- Конкретні, відкриті, по одній речі за раз. Не "розкажіть про склад", а "хто і коли фіксує фактичні
  залишки, і в якій системі?"
- **Глибина дозволена** — це сесія дискавері, не Установча Зустріч: кількість користувачів, матриці прав,
  формати обміну, обсяги даних — питати можна і треба, якщо це тема сесії.
- На «нещасливі» шляхи теж: що буває при помилці, поверненні, частковій оплаті, відсутності даних.
- Орієнтир обсягу: **15–25 питань** на сесію 1–2 год (🔴 ~третина). Більше — познач, що 🟡/⚪ — буфер.

## Крок 4: Anti-repeat перевірка

Пройди КОЖНЕ питання проти журналу:

| Питання перетинається з… | Дія |
|--------------------------|-----|
| ✅ закритою темою (розділ 2) | прибери, або переформулюй на уточнення нового аспекту |
| Рішенням D-N (розділ 5) | прибери; якщо треба перепідтвердити — залиш з позначкою `(повторне підтвердження: [причина], D-N)` |
| **Чинним** виключенням X-N (розділ 4) | **прибери безумовно**; якщо вважаєш, що виключення варто переглянути — не питай клієнта, а додай рядок у "Сигнали для BA". Скасовані X-N не блокують |
| Відкритим Q-N / C-N (розділи 7, 3) | залиш і постав ID — це легітимний повтор. Якщо те саме питання є і як C-N, і як Q-N — встав ОДИН раз з обома ID |

У файл додай підсумок: "Anti-repeat: знято [N] питань (закриті/вирішені/виключені), збережено [M] легітимних повторів з ID."

## Крок 5: Карта покриття цілі сесії

Звір опитувальник з планом дискавері:

| Елемент плану | Покрито питаннями? | Чим |
|---------------|--------------------|-----|
| Ціль сесії: "[цитата з плану]" | ✅ / ❌ | [№№ питань] |
| Очікуваний результат: "[цитата]" | ✅ / ❌ | [№№] |
| Кожне питання-заготовка плану (5–10) | ✅ розгорнуто / знято / ❌ | [№№ або D-N/X-N] |
| Перенесені C-N по темі | ✅ всі включені / ❌ | [ID] |

Заготовку плану, зняту anti-repeat (вже вирішена D-N або виключена X-N), познач «знято (D-N/X-N)» —
це не прогалина. Кожен ❌ — або додай питання, або явно поясни в "Прогалинах" чому (наприклад:
"потребує демонстрації системи, не питання інтерв'ю").

---

## Крок 6: Парковка (не на цю сесію)

Питання, що виникли, але належать іншим сесіям плану → список "→ [питання] — сесія [M] ([тема])".
Додай їх у розділ 7 журналу (Q-N, з прив'язкою до сесії), щоб не загубились. Якщо питання ВЖЕ є
в журналі — посилайся на наявний ID, нового не створюй. Тема, що не належить жодній сесії плану
(наприклад, повернута зі скасованого виключення) → у парковці познач «кандидат: нова сесія / оновлення
плану» і додай рядок у «Сигнали для BA».

## Крок 7: Pre-save верифікація

- [ ] Назва файлу: `питання_сесії-[N]_[реальна компанія].md` (позапланова — дата замість N)
- [ ] Дата підготовки і дата/тема сесії заповнені
- [ ] Блок «Контекст циклу» заповнений з журналу (не шаблонний)
- [ ] Перенесені питання (C-N) — першою секцією, кожне з ID
- [ ] Кожне питання: пріоритет + тег + «Навіщо» + поле «Відповідь:»
- [ ] Кожне 🔴 питання має 2–3 навідні; навідні конкретні під клієнта (не абстрактні шаблони), прийоми чергуються
- [ ] Anti-repeat виконано, підсумок у файлі; жодне питання не дублює чинні X-N чи рішення D-N

## Важливі правила

**Журнал — перш за все.** Застарілий журнал = питання повз контекст. Bootstrap/reconcile — обов'язковий Крок 1.

**Перенесене — першим.** Незакриті питання минулої зустрічі — борг; вони відкривають опитувальник.

**Не питай закрите, вирішене, виключене.** Anti-repeat (Крок 4) — головна перевірка скілу. Повторне
підтвердження — тільки явно позначене з причиною.

**Питання служать deliverables.** Кожне питання має тег. Питання без тега — або додай тег, або прибери.

**Опитувальник — для команди.** Клієнту — лист від `session-client-prep-builder`.

**Зберігай обидва файли:** опитувальник + журнал.
