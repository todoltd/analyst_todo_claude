## 1. Місце блоку в процесі

Канонічна BPMN, кінець циклу зустрічей і вихід у білд:

```
   ┌─→ ПІДГОТОВКА ПИТАНЬ (готово: session-question-builder, session-client-prep-builder)
   │       ↓ робочі .docx + лист → Порядок денний (BA) → ЗУСТРІЧ (запис + транскрипт)
   │       ↓
   │   АНАЛІЗ ТРАНСКРИПТУ СЕСІЇ (Claude)   ← НОВЕ · головний writer журналу
   │       ↓ (журнал оновлено: D-N / X-N / T-N / C-N / Q-N / A-N; теми ✅/🔶)
   │   Протокол обстеження (готово: meeting-protocol-builder) → Review (BA)
   │       ↓
   └── Так ─ Шлюз «Ще зустрічі?» ─ Ні (ВСІ ТЕМИ ЗАКРИТІ)
                                      ↓
                          ЗАВЕРШЕННЯ ДИСКАВЕРІ   ← НОВЕ · вихід із циклу
                                      ↓
              scope-маппінг · Gap-аналіз · Discovery-звіт · (КП/SoW)
                                      ↓
                       МІСТ: журнал → ТР_[назва].md (секція 1 + origin)
                                      ↓
        tr-usecases-acceptance → tr-odoo-tech-design → tr-review → (Погодження клієнта)
                                      ↓
                   MVP / БІЛД: spec-writer → architecture → … → pr-review
```

Дві частини блоку з різною природою:

- **Аналіз транскрипту сесії** — *внутрішньоциклічний*, повторюється на КОЖНУ зустріч. Закриває пробіл «головний writer журналу», який контракт журналу вже передбачив («у майбутньому основним writer'ом стане скіл аналізу транскриптів сесій»). Сьогодні журнал наповнює лише `session-question-builder` (bootstrap + reconcile) + ручні правки BA.
- **Завершення** — *одноразовий* вихід із циклу, коли всі теми плану закриті. Консолідує журнал + протоколи у фінальні deliverables і будує міст у ТР.

## 2. Інвентар скілів (вузька нарізка)

### 2.A Внутрішньоциклічний (пакет циклу)

| # | Скіл | Дія | Вхід | Вихід |
|---|------|-----|------|-------|
| A1 | `session-transcript-analyzer` | Розібрати транскрипт/нотатки сесії → оновити журнал | транскрипт/нотатки сесії, заповнений `питання_сесії-[N]`, `журнал_дискавері`, `план_дискавері` | оновлений `журнал_дискавері` (теми ✅/🔶; D-N/X-N/T-N/C-N/Q-N/A-N); короткий `аналіз_сесії-[N].md` як вхід у протокол |

Аналог `kickoff-transcript-analyzer` (для УЗ), але: (1) **пише журнал** як основний writer; (2) працює в контексті циклу (що вже закрито/виключено — не дублювати). Вихід далі живить `meeting-protocol-builder` (протокол замовнику) — розвести: аналіз = внутрішній стан, протокол = документ замовнику.

### 2.B Завершення (новий плагін `odoo19-discovery-closeout`)

| # | Скіл | Дія | Вихід |
|---|------|-----|-------|
| B0 | `discovery-readiness-check` *(гейт; можливо — Крок 0 у B1)* | Перевірити, що цикл можна закривати | звіт готовності: усі теми ✅, немає відкритих `[КД]`, `C-N` закриті/свідомо відкладені, `A-N` закриті/трекаються, виключення з джерелом |
| B1 | `discovery-scope-mapper` | Консолідувати scope із журналу | `scope_[компанія].md`: in-scope (модулі/процеси), out-of-scope (`X-N`), кастомізації vs стандарт |
| B2 | `discovery-gap-analyzer` | AS-IS → TO-BE (Odoo 19) → gap | `gap-аналіз_[компанія].md`: карта AS-IS, рекомендована конфігурація TO-BE, gap-list + customization-list |
| B3 | `discovery-report-builder` | Клієнтський підсумок дискавері | `discovery-звіт_[компанія].md` + брендований `.docx` (Humizer + фільтр чутливого) |
| B4 | `journal-to-tr-builder` **(міст, ядро SDD-P3)** | Журнал → скаффолд ТР + секція 1 «Функціональні вимоги» з `origin` | `ТР_[назва].md` (секція 1 + плейсхолдери для `tr-*`) |
| B5 | `discovery-sow-builder` | Комерційна пропозиція / SoW | `discovery-КП_[компанія].md` + брендований `.docx` |

**Чому B4 — ключовий.** На диску немає `tr-functional-requirements`, хоча `tr-usecases-acceptance` (Крок 0) читає саме «секцію 1» ТР і за її відсутності пропонує `tr-scaffold`. Тобто **producer'а секції 1 у методології зараз немає** — ланцюжок ТР починається «з повітря». `journal-to-tr-builder` закриває цей стик: скаффолдить `ТР_[назва].md` і авторить секцію 1 з вимогами, кожна з backref `origin: D-N / T-N / сесія N`. Далі `tr-usecases-acceptance` підхоплює без змін свого контракту.

## 3. Артефакти і контракти

### 3.1 Карта файлів

| Файл | Пише | Читає | Призначення |
|------|------|-------|-------------|
| `журнал_дискавері_[компанія].md` | **A1 (основний)**, `session-question-builder` (bootstrap/reconcile), BA (руками) | усі скіли блоку | Єдине джерело стану циклу |
| `scope_[компанія].md` | B1 | B2, B3, B4, B5 | Консолідований scope (in/out/custom) |
| `gap-аналіз_[компанія].md` | B2 | B3, B4, B5 | AS-IS / TO-BE / gap-list |
| `discovery-звіт_[компанія].md` + `.docx` | B3 | замовник | Клієнтський підсумок |
| `ТР_[назва].md` (секція 1) | B4 | `tr-usecases-acceptance` → `tr-odoo-tech-design` → `tr-review` → `spec-writer` | Міст у білд |
| `discovery-КП_[компанія].md` + `.docx` | B5 | замовник | Комерційна пропозиція |

### 3.2 Контракти і наскрізні правила (переюз існуючого)

1. **Гейт виходу (правило гейту SDD).** Завершення НЕ стартує, доки журнал не «закриваний»: усі теми плану ✅, немає відкритих `[КД]`, `C-N` закриті або свідомо відкладені, `A-N` закриті/трекаються, кожне `X-N` має джерело. AI лише позначає drift — рішення закрити цикл ставить BA (`погоджено` — людина).
2. **Журнал — джерело істини; усі виходи — проекції з нього.** Парсити структурні поля (заголовки розділів, колонки таблиць), переживати ручні правки BA (правило журналу №6).
3. **AC з `origin` (контракт SDD §3).** Вимоги в секції 1 ТР несуть backref `origin: D-N / T-N / сесія N`. Один namespace `AC-NN`, не перенумеровувати — щоб трасування дожило до `# TODO: AC-NN → test_acNN → pr-review`.
4. **Прогресивний ригор.** scope/gap/звіт/вимоги — рівня **DRAFT** (редаговані); канонічні `AC-NN` фіналізує ланцюжок `tr-*`. B4 емітить чернетки вимог, не фінальні AC.
5. **Humizer + фільтр чутливого** для документів замовнику (B3, B5): стриманий діловий тон, пояснення термінів, без ставок/внутрішніх практик виконавця. Переюз правил `meeting-protocol-builder` / `session-client-prep-builder` (v0.7.3).
6. **Дисципліна Odoo 19** для TO-BE (B2): кожен пункт конфігурації звіряти з офіційною докою Odoo 19 і вказувати джерело — як `tr-odoo-tech-design` Крок 1.
7. **Брендований `.docx` ≤ 1 МБ** (B3, B5): переюз полегшеного бланку з `session-client-prep-builder/assets/`.

## 5. Ризики

| Ризик | Мітигація |
|-------|-----------|
| **Пакувальне тертя A1**: аналіз транскрипту — блок циклу, а не завершення | A1 → `odoo19-discovery-sessions` (блок 2 циклу); завершення (B1–B5) → новий плагін. Зафіксувати у відкритому питанні |
| **Тонкий журнал** → тонкі виходи завершення | A1 розробляється першим (Етап 1); до його впровадження — fallback на протоколи + reconcile |
| **Drift контракту B4 ↔ `tr-usecases-acceptance`** | Секцію 1 і плейсхолдери писати точно під вхід `tr-usecases-acceptance` (Крок 0); e2e-тест підхоплення на Етапі 7 |
| **Дубль ролі `tr-functional-requirements`** | Свідомо: B4 виконує роль producer'а секції 1; окремий `tr-functional-requirements` не плодити (один namespace, контракт SDD §3) |
| **B5 без скіла оцінки** | `tr-effort` ще не зібраний; або винести КП/SoW у фазу 2, або зробити поле оцінки ручним входом BA |
| **Конфлікт тригерів** A1 ↔ `kickoff-transcript-analyzer`; B3 ↔ `meeting-protocol-builder` | Явне розмежування в description («сесія дискавері» vs «Установча»; «звіт дискавері» vs «протокол зустрічі»); рантайм-тест фраз |
| **Витік чутливого** у звіт замовнику / КП | Окремі скіли (не побічний вихід), фільтр v0.7.3, Humizer, QA-пункт у pre-save |
| **Неточний TO-BE** (вигадана конфігурація Odoo) | Дисципліна звірки з докою Odoo 19 + вказання джерела (Крок 1 за зразком `tr-odoo-tech-design`) |
