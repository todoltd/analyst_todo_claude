# Карта: SDD-хребет через усю методологію ToDo+AI

Схема рішення (не правки скілів): як підняти принципи SDD з етапу MVP на рівень усієї
методології. Дата: 11.06.2026. Контекст — `ПЛАН_інтеграції_SDD-пакета_MVP.md`.

## Ідея за один абзац

Інгредієнти SDD у методології вже є, але розкидані й стадія-локальні: **ТР = специфікація**,
**реєстр «Перелік ТР» = життєвий цикл статусів**, **журнал_дискавері = реєстр пам'яті**,
**AC = критерії**, **валідатор = контроль**. Модернізація = зшити їх одним хребтом за трьома
принципами SDD: (1) наскрізний ID, (2) єдиний гейтований статус, (3) «людський документ + машинна
проекція» на кожній межі. Ригор нарощувати поступово — легко в дискавері, жорстко на білді.

## Карта етапів

| Етап | Людський артефакт | Машинна проекція | ID-трасування | Статус / гейт |
|---|---|---|---|---|
| ІНІЦІАЦІЯ | бриф, аналіз УЗ, план дискавері | план (структурований) | теми плану, ризики | чернетка → погоджено |
| ДИСКАВЕРІ (цикл) | протоколи, журнал | `журнал_дискавері` | C-N, D-N, X-N, T-N, Q-N, A-N | сесія: запланована/проведена · теми: ✅/🔶/⬜ |
| ТР | `ТР_[назва].md` (клієнту) | — *(пробіл)* | US-01, **AC-01**, ID задачі ERP | реєстр: Аналіз → Погодження → Виконано |
| MVP / БІЛД | — *(внутрішнє)* | `SPEC.md`, `ARCHITECTURE.md` | **AC-01** наскрізь: `#TODO→test_ac01→pr-review` | SPEC draft → approved · жорсткі гейти |

Видно: **AC-01** уже може бути спільною валютою вимоги. Бракує двох зшивок — машинної проекції на межі
ТР і єдиного словника статусів поверх усіх стовпців «Статус».

## Три зшивки (важелі)

1. **Наскрізний ID — AC як універсальна валюта вимоги.**
   AC-01 уже йде ТР→код→тест. Додати йому `origin:` — backref угору на джерело
   (журнал D-N / сесія N / тема T-N). Тоді кожен рядок коду трасується до конкретного
   рішення дискавері. Один namespace, не кілька конкурентних.

2. **Єдиний словник статусів + одне правило гейту.**
   Звести `draft→review→approved→superseded` (SPEC) і `Аналіз→Погодження→Виконано→Скасовано`
   (реєстр) до одного канону, що діє для брифа, плану, ТР, SPEC, модуля. Правило:
   «нічого нижче не стартує без `погоджено` вище; approve — лише людина/клієнт; AI лише
   позначає drift». (Статус *сесії* — окрема вісь, не чіпаємо.)

3. **Пара «людський документ + машинна проекція» на кожній межі.**
   Патерн `ТР (людям) → SPEC.md (машині)` вже є — розширити на інші межі
   (УЗ-аналіз→план уже частково; **головний пробіл — журнал→ТР**). Скрізь: читабельний
   документ для людей + структурована проекція, з якої Claude працює детерміновано.

## Що побудувати (пріоритети, поки без переписування скілів)

- **P1 · Спільний словник статусів** — один контракт `.md`, на який посилаються реєстр, ТР, SPEC, плани. Дешево, вмикає гейти всюди.
- **P1 · ID-namespace контракт** — AC-01 як єдиний ID вимоги + поле `origin:` (журнал/сесія). Зачіпає `tr-usecases-acceptance` (емітити origin) і `spec-writer` (вже зберігає AC-01).
- **P2 · Крос-консистентний валідатор** — розширити ідею `валідатор_скілів.py` на артефакти: кожен AC у ТР має origin; кожне «Виконано» в реєстрі має змерджений модуль; кожен AC у SPEC прийшов із ТР.
- **P3 · Конвертер «журнал → ТР»** (новий скіл, пізніше) — сідить функціональні вимоги + чернетки AC із тем/рішень журналу. Закриває останній ручний розрив дискавері→ТР.
- **P4 · Генератор проекцій (статус-дашборд + RTM)** — *середні зусилля.* Рендерити з джерела істини, а не вручну: (а) статуси стадій/скілів і версії в BPMN/дашборді — з реєстру + `plugin.json` (знімає повторювану ручну синхронізацію BPMN); (б) жива **матриця трасування (RTM)**: рішення журналу → AC ТР → SPEC → файл/тест → статус реєстру — авто-доказ покриття для приймання й аудиту.
- **P5 · AC як контракт: зміни + приймання + оцінка** — *низькі–середні.* (а) запит на зміну = ТР Формату C зі своїми AC і `origin: CR-N`, тим самим білд-ланцюжком (scope-creep видимий і білабельний); (б) авто-«звіт приймання за AC» на підпис замовнику (проходять конкретні AC, не «на віру»); (в) оцінка per-AC у ТР + факт per-AC з білду → калібрований прайсинг у реєстрі.
- **P6 · «Реверс» скрізь + пост-проєктна калібрація** — *вищі (нові скіли).* Скіл «as-is мапа» з процесу/експорту замовника в дискавері (front-load структури в розвідку); пост-проєктний reverse-check «код vs ТР» — дані для калібрування методології (як 41 протокол EDELWEIS калібрував скіл протоколу).

## Прогресивний ригор

Дискавері — нечітке за природою; жорсткі ID/гейти на кікофі вб'ють гнучкість (журнал навмисно
«можна правити руками» — це фіча). Тому: **легко** в ІНІЦІАЦІЇ/ДИСКАВЕРІ (emergent, редаговане) →
**жорстко** на ТР/білді (фіксовані AC, гейти, drift = BLOCK). Клієнтські документи лишаються
людськими й українськими; машинний хребет — під ними, не замість.

## Ризики

- Над-формалізація ранніх етапів (втрата гнучкості розвідки).
- Подвійне введення, якщо конвертери неповні — звідси P3.
- Спокуса завести другий ID поряд з AC-01 — не робити; один namespace.

## Перший крок (рекомендація)

Почати з **P1**: словник статусів + поле `origin` для AC. Дешево, нічого не ламає, і одразу вмикає
наскрізну трасованість «рішення дискавері → рядок коду» — це найвідчутніший виграш для scope-суперечок,
аудиту й біллінгу.

Далі за важелем — **P4** (генератор статусів + RTM): знімає ручну синхронізацію BPMN (повторювана
біль із ІСТОРІЇ) і дає авто-доказ покриття для приймання. P5/P6 — коли підуть реальні проєкти на новому циклі.
