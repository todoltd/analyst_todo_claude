# Специфікація P4 — Генератор проекцій (статус-дашборд + RTM)

Деталізація пункту **P4** з `КАРТА_SDD-хребет_методології.md`. Спирається на
`КОНТРАКТ_SDD-хребет_статуси-і-ID.md` (статуси + AC-01/`origin`). Це **специфікація**, не реалізація.

**Принцип:** усе, що зараз правиться руками (версії й «ГОТОВО» в BPMN) або взагалі не ведеться
(трасування покриття), — **проекція з джерела істини**. Джерело — артефакти й `plugin.json`;
проекція — згенерована й одноразова. Генератор нічого не авторить, лише рендерить.

Реалізація — Python-скрипт `генератор_проекцій.py` (як `валідатор_скілів.py`), дві підкоманди.

---

## P4a — Проекція статусів методології

### Вхід (джерело істини)
- `СТАТУС_методології.json` — **новий** декларативний реєстр: стадії → блоки → скіли, кожен зі
  `status` (канон з контракту: `чернетка|на розгляді|погоджено|виконано|застаріле`) і посиланням на пакет.
- 3× `.claude-plugin/plugin.json` (ІНІЦІАЦІЯ / ДИСКАВЕРІ / MVP) — версії беруться **автоматично**, не дублюються в JSON.

Приклад `СТАТУС_методології.json`:
```json
{
  "stages": [
    { "id": "initiation", "title": "ІНІЦІАЦІЯ", "package": "odoo19-discovery-initiation",
      "status": "виконано", "skills": 12 },
    { "id": "discovery", "title": "ДИСКАВЕРІ", "package": "odoo19-discovery-sessions",
      "status": "виконано", "skills": 2 },
    { "id": "mvp", "title": "MVP / БІЛД", "package": "odoo19-mvp-build",
      "status": "виконано", "skills": 7 }
  ]
}
```

### Вихід
1. **BPMN (канон + копія)** — точкова заміна версій і бейджів/легенди за **стабільними маркерами**
   (не регекс по всьому файлу). Разова підготовка: завести в BPMN явні якорі, напр.
   `<!--P4:VER:mvp-->0.2.0<!--/P4-->` і `<!--P4:STATUS:mvp-->ГОТОВО<!--/P4-->`. Генератор заміняє лише вміст між якорями.
2. **`СТАТУС_методології.html`** — простий дашборд: стадії × блоки × скіли × статус × версія (кольори статусів з контракту).

### Тригер
`python генератор_проекцій.py status` — запускати при кожному релізі / зміні версії пакета.

### Інваріанти (фейл = стоп)
- Версія кожного бейджа BPMN == версія відповідного `plugin.json`.
- Кожен скіл у джерелі має `status` з канону контракту.
- Канон BPMN == копія (побайтно) після генерації.
- Жоден якір `P4:` не втрачено й не задубльовано.

## P4b — RTM (матриця трасування вимог), per-project

### Вхід (артефакти проекту)
| Джерело | Що парситься |
|---|---|
| `ТР_[назва].md` | `AC-NN` + `origin:` (журнал D-N / сесія N / CR-N) |
| `SPEC.md` | `AC-NN` (звірка, що прийшов із ТР) |
| модуль `td_*` / `<проект>_<область>` | маркери `# TODO: AC-NN` і докстрінги з `AC-NN` (файл:метод) |
| `tests/` | методи `test_acNN_*` |
| експорт реєстру «Перелік ТР» | статус ТР (`Погодження клієнта` / `Виконано` …) |

### Вихід — `RTM_[проект].md` (і/або `.html`)
Рядок = один AC; колонки:

| AC | origin (джерело) | у ТР | у SPEC | реалізація (файл:метод) | тест(и) | статус реєстру | покриття |
|----|------------------|------|--------|-------------------------|---------|----------------|----------|
| AC-01 | D-3 / сесія 2 | ✓ | ✓ | `models/td_x.py:action_confirm` | `test_ac01_*` | Виконано | ✓ |
| AC-07 | CR-2 | ✓ | ✓ | — | — | Погодження | ⚠️ нема реалізації/тесту |

### Тригер
`python генератор_проекцій.py rtm <project_dir>`

### Перевірки (перетин з P2-валідатором)
- Кожен `AC` має `origin` (інакше ⚠️ — обрив трасування вгору).
- Кожен `AC` має ≥1 тест (інакше ⚠️).
- Кожен ТР зі статусом `Виконано` має змерджений модуль і зелені тести.
- AC у коді/тесті, якого немає в ТР (інакше 🔴 — drift, як `pr-review` BLOCK).

### Залежність
RTM **повноцінна лише після P1** (формат `AC-01` + поле `origin`). Без них — часткова (по тому, що вдалося розпарсити).

## Технічні рамки
- **Мова:** Python stdlib (json, re, pathlib); для HTML — простий рядковий шаблон. Без зовнішніх залежностей (як валідатор).
- **BPMN:** заміна лише між якорями `P4:` — детерміновано; **не** регекс по всьому SVG.
- **Запис у змонтовану папку:** через `/tmp` → `cp` (atomic-rename на mount блокується — відомо з практики сесій).
- **Ідемпотентність:** повторний запуск без змін джерела не міняє байтів виходу.

## Тригери як скіл (опційно, пізніше)
Якщо обгорнути в скіл: фрази «онови статуси діаграми», «згенеруй RTM для [проект]». Поки достатньо скрипта.

## Етапність впровадження P4
1. **Крок 1 (найбільший виграш):** `СТАТУС_методології.json` + якорі `P4:` у BPMN + `status`-підкоманда. Закриває ручну синхронізацію версій/бейджів.
2. **Крок 2:** дашборд `СТАТУС_методології.html`.
3. **Крок 3:** `rtm`-підкоманда (після P1 — формат AC-01 + origin).

## Ризики / межі
- Якість RTM = якість конвенцій (AC-01, `origin`, `test_acNN`, `# TODO: AC-NN`). Парсити **структуровані поля**, не вільний текст ТР.
- Генератор лише **проектує**; джерело правди — артефакти й `plugin.json`, а не дашборд (його не редагувати руками).
- Якорі `P4:` у BPMN — разова інвазія в розмітку; узгодити, щоб не ламали рендер (коментарі HTML безпечні).
