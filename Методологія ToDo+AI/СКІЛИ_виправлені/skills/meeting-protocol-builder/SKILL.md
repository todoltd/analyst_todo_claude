# Meeting Protocol Builder (Протокол обстеження) — універсальний

## Мета

Створити **офіційний документ для клієнта** — Протокол обстеження (проведення інтерв'ю) — на **будь-яку**
проєктну зустріч, не лише Установчу. Скіл адаптується під тип зустрічі, бере факти з аналізу/нотаток і
оформлює їх простою мовою на фірмовому бланку ToDo.

> Калібровано на 41 реальному протоколі EDELWEIS SSERVICE **з виправленнями методолога**: у старих протоколах
> «Обговорювали» і таблицю «ВИРІШИЛИ» лишали порожніми — це визнано недоліком, а не нормою. Скіл їх **заповнює**.

---

## Залізні правила реальної практики (з аналізу 41 протоколу)

1. **РЕЗУЛЬТАТИ — за темами порядку денного.** Кожен заголовок РЕЗУЛЬТАТІВ = пункт порядку денного, з підблоками.
2. **«Обговорювали» — ПРОСТАВЛЯТИ**: «Так» для розглянутих тем; «Ні» — лише якщо тему перенесли.
   Шаблонне «Так/Ні» в готовому протоколі лишати не можна.
3. **Таблицю «ВИРІШИЛИ» — ЗАПОВНЮВАТИ**: кожне рішення окремим рядком — № / Рішення / Термін / Відповідальний.
   Рішення НЕ дублювати текстовим списком у РЕЗУЛЬТАТАХ. У **«ПОГОДЖЕННЯ»** впиши ПІБ погоджувача
   від клієнта (ЛПР); колонку «Погоджено/Зауваження» лиши порожньою — її заповнює клієнт.
4. **«ДОДАТОК 1» — лишати порожнім** (опис процесів іде в тілі РЕЗУЛЬТАТІВ, не в додатку).
5. **Тема** — generic за типом («Статусна зустріч»); **Мета — конкретна** і змістовна (НЕ дублювати Тему).
6. **«Наступна зустріч»** — якщо домовились, писати дату/час прямо в кінці РЕЗУЛЬТАТІВ.
7. **Мова — проста, ділова, стримана.** Пояснення термінів у дужках — так; метафори, образність
   і розмовні вставки — НІ. Це крок Humizer.
8. **Фільтр чутливого.** У клієнтський протокол НЕ вносити: внутрішні практики виконавця,
   комерційні параметри (ставки, валюта, ціни) — навіть як відкриті питання.
   Місце цього — договір або окрема розмова, не протокол.
   **Виняток для установчої зустрічі:** блок «Причини зупинки проєкту з боку Замовника / Виконавця» —
   стандартний елемент установчого протоколу (є у 73% установчих у реальній практиці), вноси без обмежень.
   Чутливе — це ставки, ціни, внутрішні норми часу; умови організаційної зупинки проєкту — ні.
9. **«Замовник» і «Виконавець» — завжди з великої літери** як позначення сторін договору у тексті
   РЕЗУЛЬТАТІВ і ВИРІШИЛИ: «Замовник погоджується…», «Виконавець надає…». Не «клієнт», не «компанія»,
   не «ви». Після Кроку 3 — перевір пошуком слів «клієнт» і «компанія» в тілі → замінити.
10. **Мова — українська; канонічний правопис:** «проєкт» (з єром — сучасна норма); «Odoo» — перша
    велика, решта малі (не ODOO, не odoo у тексті); «бізнес-процес» через дефіс; «Discovery» — латиницею.
11. **Терміни у ВИРІШИЛИ — конкретна дата** (ДД.ММ.РРРР або РРРР-ММ-ДД), не «найближчим часом»,
    «в процесі», «після налаштування». Якщо дата невідома — «до наступної зустрічі (ДД.ММ.РРРР)».
    Псевдостроки — перший маркер слабкого протоколу.

## Інструменти та середовище

**Читання вхідних файлів:**
- `аналіз_…_[компанія].md` / `аналіз_Установчої Зустрічі_…` → `Read` (основний вхід — факти зустрічі)
- транскрипт / нотатки зустрічі (будь-який формат) → `Read`
- `підготовка_УЗ` / `agenda` / `питання_УЗ` → `Read` якщо є (мета, учасники, порядок денний)
- `питання_сесії-[N]_[компанія].md` (заповнений на сесії дискавері) → `Read` якщо є — структуровані відповіді за темами сесії
- `бриф_[компанія].md` → `Read` (назва клієнта, № договору, контакти)

**Шаблон для .docx:** бандл-шаблон скілу `assets/protocol_template.docx` (фірмовий бланк ToDo, вже оптимізований ≤1 МБ).
Резерв, якщо бандл недоступний: `Шаблон_Протокол_обстеження.docx` у робочій папці або наданий користувачем.

**Зразки формулювань:** `references/examples.md` — реальні фрагменти протоколів за типами зустрічей і пари
«було→стало» для Humizer. **Прочитай перед написанням РЕЗУЛЬТАТІВ.**

**Збереження:**
- `протокол_[тип]_[дата]_[компанія].md` → `Write` (напр. `протокол_статусна_2026-05-15_EDELWEIS.md`)
- `протокол_[тип]_[дата]_[компанія].docx` → заповнений шаблон (Крок 6)

**Після завершення:** `present_files` для обох; повідом «Протокол ([тип]) готовий (.md + .docx). Відкритих питань: [N]. Рішень: [M].»

## Крок 0: Визнач тип зустрічі і вхідні дані

Спершу визнач **тип зустрічі** (з назви/мети/контексту) — він задає порядок денний і фокус РЕЗУЛЬТАТІВ:

| Тип | Фокус | Типові пункти порядку денного |
|-----|-------|-------------------------------|
| **Установча** | старт, домовленості | команда і ролі; канали комунікації (оперативний — Telegram/чат, формальний — e-mail); графік статусних зустрічей; модель T&M (кратно 15 хв + гарантійний термін); причини зупинки проєкту з боку Замовника і Виконавця (симетрично); документація проєкту (перелік файлів Google Drive з призначенням); наступна зустріч |
| **Збір вимог** | бізнес-процеси | по процесах (CRM, фінмодель, склад, BB-карта…), вимоги, відкриті питання |
| **Статусна** | прогрес + уточнення | підсумки, бізнес-процеси, вимоги (функц./нефункц.), пропозиції, відкриті питання |
| **Інтеграції** | технічна стиковка | **напрямок потоку даних обов'язково**: «A → B» / «A ⇔ B» / «A → B → A»; теми 1:1 (номенклатура, партії, API, зони, тарифікація); ВИСНОВОК ДИСКУСІЇ якщо обирали між варіантами |
| **Навчання** | навчання користувачів | модулі/функції покроково, доступи, сповіщення |
| **Демонстрація** | показ розробок | (1) вступний абзац «що показувалось» — обов'язково першим; (2) по блоках/процесах: Відповідальний / Дії / Особливість; (3) Відкриті питання — окремим підрозділом |
| **Фінанси** | облік, P&L | рахунки, інвойси, звітність, взаєморозрахунки |
| **План / договори / ТЗ** | план і scope | план, зміни scope, модуль договорів, пункти ТЗ |

**Потрібно мінімум:** аналіз зустрічі АБО нотатки/транскрипт. Якщо нема — попроси надіслати. Якщо кілька компаній — уточни, для якої.

## Крок 1: Збери метадані (шапка)

- **Дата** — реальна дата зустрічі у форматі ДД.ММ.РРРР (звір рік! у реалі траплялись одруки «20256»). Якщо нема — `‹заповнити: дата›`.
- **Місце** — Google Meet / офіс / ‹заповнити›.
- **Тема** — generic за типом (напр. «Статусна зустріч», «Збір вимог»).
- **Мета** — **конкретна** (напр. «Узгодження логіки калькуляторів для розрахунку логістичних послуг»). НЕ дублюй Тему.
- **Документ створив** — просто ПІБ автора, **без посади в дужках** (так у всіх реальних протоколах).
- **№ договору** — взяти з брифу/договору. Якщо невідомий — **спитай користувача**; якщо договору ще немає
  (типово для установчої) — постав `‹заповнити: № договору›` або запропонуй прибрати рядок.
  **НЕ лишай шаблонне «№ __________ від __.__.20__р.»** — це перше, що бачить клієнт.
- **Назва клієнта** — один раз із брифу/договору, стало по всьому документу.
- **Учасники** — ПІБ + організація/роль (команда ToDo + клієнт).

## Крок 2: Порядок денний

Сформуй нумерований перелік тем за типом зустрічі (Крок 0) + з agenda/питання_УЗ, якщо є.
У колонці «Обговорювали» **простав фактичний стан**: «Так» — тема розглянута; «Ні» — тему перенесли.
Шаблонне «Так/Ні» в готовому документі — недопрацювання.

**Дзеркальність 1:1.** Фінальні службові блоки, які будуть у РЕЗУЛЬТАТАХ («Перелік файлів та документів»,
«Відкриті питання»), **включай у порядок денний останніми пунктами** — тоді кожен пункт повестки має свій
нумерований розділ у РЕЗУЛЬТАТАХ, і немає розділів поза повесткою. Прийняті рішення в повестку не виносяться —
вони йдуть у таблицю «ВИРІШИЛИ».

## Крок 3: РЕЗУЛЬТАТИ — за темами порядку денного

Для **кожного** пункту порядку денного зроби заголовок і розкрий його. Обери формат підблоку під тип теми
(усі формати взяті з реальних протоколів — змішувати можна):

> 📎 Перед написанням відкрий `references/examples.md` — там реальні зразки кожного формату за типами зустрічей.

- **Описовий** — «Що обговорювали / Рішення» (найпоширеніший; для оглядових тем).
- **Процес** — «Опис · Кроки (1→2→3 або через `->`) · Відповідальні · Особливості/Умови» (для бізнес-процесів, навчання, демо).
- **Вимоги** — окремо «Функціональні вимоги» і «Нефункціональні вимоги» списками.
- **Лейбл: значення** — атрибути сутності (для інтеграцій: Майстер-система, Ідентифікація, Процес передачі…).
- **Суть → Рішення** — «Суть проблеми» + «Рішення та крок автоматизації» (для розбору правок/багів).
- **ТЗ на доробку** — нумеровані технічні вимоги (коли узгоджують доробку).
- **AS-IS → TO-BE** — обов'язковий для типів «Збір вимог» і «Фінанси»: спочатку `**Поточний стан (AS-IS):**`
  з описом систем, процесу, проблеми — потім `**Вимоги до Odoo (TO-BE):**` зі списком вимог.
  Змішування без заголовків-роздільників — дефект: читач не розуміє, що є зараз, а що має бути.

> **Підказка для збору вимог:** якщо Замовник називає числа (кількість SKU, записів, відсотки операцій, часові нормативи) — фіксуй їх: вони роблять вимогу конкретною і вимірюваною.

**Специфічні блоки для установчої зустрічі** (обов'язкові при типі «Установча»):
- **Канали комунікації** — три елементи: де оперативно, де формально, коли статусні:
  `Оперативна комунікація: Telegram-група «[Назва]» / Формальна: e-mail / Статусні зустрічі: [день], [час].`
- **Модель співпраці** (якщо обговорювали):
  `Модель: Time & Material. Облік часу: кратно 15 хвилинам. Гарантійний термін: [X] місяці.`
- **Причини зупинки проєкту** — симетрично з обох сторін:
  `З боку Замовника: [умови] / З боку Виконавця: [умова 1]; [умова 2].`
- **Документація проєкту** — нумерований перелік файлів/папок Google Drive (мінімум 7 пунктів)
  з коротким поясненням призначення кожного файлу. Формат:
  `1. Протокол обстеження — фіксується після кожної зустрічі, підтверджується Замовником.`
  `2. Бізнес-процес (BPMN) — погоджений процес впровадження.` тощо.

**Фінальні розділи РЕЗУЛЬТАТІВ** (ті, що додаєш, мають бути й пунктами порядку денного — Крок 2; нумерація наскрізна):
- **Перелік файлів/документів** — що обговорили / отримали від замовника.
- **Пропозиції та коментарі учасників** — за потреби з іменами або знеособлено.
- **Ключові ризики** *(опціонально; рекомендовано для статусних і технічних зустрічей)*
  Якщо на зустрічі явно обговорювали ризики — виноси окремим блоком з рівнем:
  `– [Назва ризику] — ВИСОКИЙ / СЕРЕДНІЙ / НИЗЬКИЙ`
  ВИСОКИЙ = може заблокувати проєкт або фазу. СЕРЕДНІЙ = вимагає уваги. НИЗЬКИЙ = на контролі.
  Без рівня — не ризик, а «спостереження»; залишай у тематичному блоці.
- **Відкриті питання** — нумерований перелік; **кожне питання — з відповідальним і строком** («хто / до коли»)
  або явним «розглянемо на наступній зустрічі». Питання без власника — недопрацьоване.
  Формат кожного рядка: `[N]. [Питання] — Відповідальний: [ПІБ] — До: [дата або «наступна зустріч»]`
  **Нумеровані пункти відкритих питань лишаються ТІЛЬКИ всередині цього розділу.**
  НЕ виносити їх окремими рядками перед іншими розділами результатів — це типова помилка генерації.
- **Прийняті рішення** — НЕ окремим текстовим блоком у РЕЗУЛЬТАТАХ: кожне рішення йде **рядком у таблицю
  «ВИРІШИЛИ»** (№ / Рішення / Термін / Відповідальний) — див. Кроки 5–6.
- **Наступна зустріч** — день тижня + дата + час. День тижня звіряй **програмно**, не з пам'яті:
  `date -d "РРРР-ММ-ДД" +%A` у bash (так само перевір дати термінів у ВИРІШИЛИ).
  **Розміщення: ЗАВЖДИ останній рядок секції РЕЗУЛЬТАТІВ, безпосередньо перед «## Вирішили».
  НЕ на початку результатів, НЕ після заголовка секції — тільки в самому кінці.**

> **Фільтр чутливого (важливо):** комерційні параметри (ставка, валюта, ціни), внутрішні практики
> виконавця — у протокол НЕ вносити, навіть як відкриті питання. Місце цього — договір або окрема розмова.
> **Виняток:** блок «Причини зупинки проєкту» для установчої зустрічі — організаційна домовленість,
> вноситься без обмежень (див. правило 8 і «Специфічні блоки для установчої зустрічі» вище).

> **«Команда і ролі»:** перелік по рядках — кожна роль з нового рядка (не суцільною прозою); ролі за
> фактичною функцією (сейлз — «Sale», не «менеджер проєкту»). Дату наступної зустрічі в пунктах
> не дублюй — вона вказується один раз, наприкінці РЕЗУЛЬТАТІВ.

> Нумеруй наскрізно й чисто: пункт повестки ↔ розділ РЕЗУЛЬТАТІВ строго 1:1 (у реалі траплялись збої «111.»,
> подвоєні набори — не повторюй цього).
> **Порядок розділів РЕЗУЛЬТАТІВ = порядок порядку денного: 1 першим, N останнім. Ніколи у зворотному порядку.**
> Якщо генеруєш з кінця — перестав розділи перед здачею. Зворотний порядок (7→6→5…1) — критична помилка структури.

## Крок 4: Humizer — олюднення формулювань для клієнта

Пройди весь текст і зроби його зрозумілим замовнику (як у реальних протоколах EDELWEIS):

**Прибери/переклади:** внутрішні маркери (🔴, [КД], «scope», «стейкхолдер», «deliverable»), оцінки людей;
розкрий скорочення; поясни терміни в дужках при першій згадці.
**Прибери також (правки методолога з польових тестів):**
- метафори та образні порівняння, фрази «образно кажучи…» — повністю;
- пряму мову та цитування учасників — повністю; лише непряма або безособова форма («Замовник підтвердив…», «Було зазначено…»);
- розмовні вставки-коментарі та «продаючі» приписки: «так зручніше рахувати», «по суті»,
  «ваш зворотний зв'язок допомагає нам покращуватися» тощо;
- внутрішні практики виконавця й захисні умови (див. «Фільтр чутливого», правило 8);
- зайві часові прив'язки («з цього літа»), дублікати дати наступної зустрічі в пунктах.
**Зроби:** короткі речення; **стриманий діловий тон**, «ви»; пояснення «що це і навіщо» — прямо, без образності
(коротке функціональне визначення на кшталт «план проєкту — маршрутна карта впровадження» допустиме);
нормативна лексика (напр. «торговельний», а не розмовне «торгівельний»).
**Часова форма в РЕЗУЛЬТАТАХ** — лише минулий час активного стану або безособово: «Домовились», «Погодили»,
«Визначено», «Презентовано». Не «Планується», «Буде зроблено», «Обговорять» — вони перетворюють протокол
на плани, а не факти. Виняток: ВИРІШИЛИ — інфінітив («Надати», «Підготувати», «Провести»).
**Збережи точно:** дати, імена, домовленості, суми, рішення — без змін.

**Перевірка Humizer:** зрозуміє людина без IT-бекграунду? немає жаргону/внутрішніх маркерів? немає метафор
і розмовних вставок? немає чутливого (ставки, внутрішні умови)? тон стриманий? факти збережені?

## Крок 6: Сформуй брендований .docx за шаблоном

1. Візьми **бандл-шаблон скілу** `assets/protocol_template.docx` і скопіюй як `протокол_[тип]_[дата]_[компанія].docx`.
   Якщо бандл недоступний — резерв: `Шаблон_Протокол_обстеження.docx` з робочої папки, або попроси користувача надіслати.
2. Через навички скілу **docx** (`unpack.py` → правки `word/document.xml` → `pack.py`):
   - Заповни «Загальну інформацію» (Дата/Місце/Тема/Мета/Документ створив) **і рядок «до Договору № …»** (Крок 1 — не лишай шаблонних підкреслень).
   - Додай рядки в «Учасники»; шапку таблиці заповни в обох комірках («Учасники | Учасники») — як в оригіналі бланку.
   - Заповни «ПОРЯДОК ДЕННИЙ»; у колонці «Обговорювали» простав **«Так»** (або «Ні», якщо тему перенесли) —
     шаблонне «Так/Ні» прибери з кожного рядка.
   - У «РЕЗУЛЬТАТИ ЗУСТРІЧІ» або «РЕЗУЛЬТАТИ ІНТЕРВ'Ю» (залежно від типу — Крок 5) встав текст за темами
     (Крок 3, після Humizer) — нумеровані підзаголовки строго 1:1 з повесткою.
   - Таблицю «ВИРІШИЛИ» **заповни рядками рішень**: № / Рішення / Термін / Відповідальний (кожне рішення — окремий рядок).
   - У «ПОГОДЖЕННЯ» **впиши ПІБ погоджувача від клієнта** (ЛПР); колонку «Погоджено/Зауваження» лиши порожньою.
   - «ДОДАТОК 1» — **лиши порожнім**.
   - **Не чіпай** шапку, логотип, контакти, стилі.
3. **Вага файлу:** перевір зображення шаблону (`word/media/`); якщо є файли >1 МБ (типово — великий логотип) —
   стисни (resize до ~1000 px по ширині + PNG optimize, зберігаючи пропорції та прозорість). Мета — готовий **.docx ≤ 1 МБ**.
4. *(Опція, лише за явним запитом користувача)* Ущільни «хвіст»: прибери зайві розриви сторінок, щоб порожні
   ВИРІШИЛИ / ПОГОДЖЕННЯ / ДОДАТОК 1 не займали кожен окрему сторінку.
5. Текст у .docx = текст .md після Humizer (1:1).
6. `pack.py` → перевір валідацію.

> Якщо шаблон недоступний — віддай якісний `.md`, а .docx сформуєш коли буде шаблон.

## Важливі правила

**Це документ для клієнта.** Усе зрозуміле й коректне; жодних внутрішніх оцінок.
**Точність важливіша за гладкість.** Humizer спрощує мову, але не змінює факти.
**Службові поля заповнюються:** «Обговорювали» — проставлено по факту; «ВИРІШИЛИ» — рішення рядками з термінами
і відповідальними; «ПОГОДЖЕННЯ» — вписано ПІБ погоджувача від клієнта (відмітку ставить клієнт); «Додаток 1» — порожній.
**Протокол — не договір.** Комерційні параметри (ставки, ціни, валюта) і внутрішні практики виконавця в нього не потрапляють. Виняток: умови зупинки проєкту — стандартний елемент установчого протоколу, вносяться без обмежень.
**Адаптуйся під тип зустрічі** (Крок 0) — порядок денний і формат РЕЗУЛЬТАТІВ різні для установчої, збору вимог, статусної, інтеграцій, навчання, демо.
**Зберігай у файли** `.md` + `.docx`.
